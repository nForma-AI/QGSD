# Phase v0.20-01: Schema Enrichment - Research

**Researched:** 2026-02-28
**Domain:** JSON Schema evolution, NDJSON writer upgrade, caller site propagation
**Confidence:** HIGH

## Summary

Phase v0.20-01 is a foundational enrichment of the formal verification check-result schema. The current schema (`formal/check-result.schema.json`) defines only 5 fields — `tool`, `formalism`, `result`, `timestamp`, `metadata` — with `additionalProperties: false`. The v2.1 spec requires 7 additional top-level fields: `check_id`, `surface`, `property`, `runtime_ms`, `summary`, `triage_tags`, and `observation_window`. All downstream consumers (triage bundles, planning gate, dashboards) depend on these fields being present and typed on every NDJSON line.

There are exactly 25 STEPS entries in `run-formal-verify.cjs` (the header comment says 23 but two CI steps were added in v0.19-11-01, making the actual count 25). Every STEPS entry is a "caller" that must emit v2.1 fields. The callers dispatch through individual runner scripts (`run-tlc.cjs`, `run-alloy.cjs`, `run-prism.cjs`, etc.) that each call `writeCheckResult()` directly. The change must be made in the shared writer first, then propagated to all callers.

The schema change must be additive-only: existing 5-field NDJSON lines from prior runs must remain valid. This means all new v2.1 fields in the schema must be optional (not in `required`) OR the schema must use a discriminated union approach (v1 record | v2.1 record). Given the requirement that existing lines "remain valid," the simplest and most correct approach is to leave new fields as optional in the schema (no new entries in `required`) while making `writeCheckResult()` enforce them in code for all new writes.

**Primary recommendation:** Update `check-result.schema.json` to add v2.1 fields as optional properties, update `writeCheckResult()` to accept and write all 7 new fields (validating that callers pass the required ones: `check_id`, `surface`, `property`, `runtime_ms`, `summary`), then update all 25 STEPS callers.

## Architecture Patterns

### Current Schema Structure (v1)

```
formal/check-result.schema.json
  required: ["tool", "formalism", "result", "timestamp"]
  properties: tool, formalism, result, timestamp, metadata
  additionalProperties: false  ← BLOCKS new fields until schema is updated
```

**Critical:** `additionalProperties: false` means ANY new field added to a record will fail JSON Schema validation against the current schema. This must be addressed in Task 1 before any caller changes.

### Target Schema Structure (v2.1)

```
formal/check-result.schema.json
  required: ["tool", "formalism", "result", "timestamp"]   ← NO new fields added to required
  properties: tool, formalism, result, timestamp, metadata,
              check_id, surface, property, runtime_ms,
              summary, triage_tags, observation_window
  additionalProperties: false
```

New fields remain OPTIONAL in schema (backwards compatibility) but are ENFORCED by `writeCheckResult()` in code — callers must pass them or the writer throws.

### v2.1 Field Semantics

The following field semantics are derived from the STEPS registry and SURFACE_MAP:

| Field | Type | Source | Example Value |
|-------|------|--------|---------------|
| `check_id` | `string` | STEPS `id` field | `"tla:quorum-safety"` |
| `surface` | `string` | STEPS `tool` field | `"tla"`, `"alloy"`, `"prism"`, `"ci"`, `"petri"`, `"generate"` |
| `property` | `string` | Human-readable check description | `"MCsafety — no deadlock, no invariant violation"` |
| `runtime_ms` | `integer` | Milliseconds elapsed (Date.now() delta) | `1823` |
| `summary` | `string` | One-line outcome description | `"pass: MCsafety verified in 1823ms"` |
| `triage_tags` | `array of string` | Optional — set when result is `fail` or `warn` | `["needs-fairness", "liveness-suspect"]` |
| `observation_window` | `object` | Optional — already used by PRISM callers in metadata | `{ window_start, window_end, n_traces, n_events, window_days }` |

**Note on `surface` vs `formalism`:** `surface` duplicates the STEPS `tool` value (e.g., `"tla"`, `"alloy"`) while `formalism` is the existing field (same domain values). The planner should treat `surface` as the STEPS tool group name matching the STEPS `tool` field. The existing `formalism` field values are `['tla', 'alloy', 'prism', 'trace', 'redaction']`; for `surface`, the valid values should include all STEPS tool groups: `generate`, `petri`, `tla`, `alloy`, `prism`, `ci`.

### `writeCheckResult()` Upgrade Pattern

Current signature (implicit):
```javascript
// Source: /Users/jonathanborduas/code/QGSD/bin/write-check-result.cjs
writeCheckResult({ tool, formalism, result, metadata })
```

New signature (v2.1):
```javascript
writeCheckResult({
  tool,        // string, required (existing)
  formalism,   // string, required (existing)
  result,      // string, required (existing)
  check_id,    // string, required NEW — throws if missing
  surface,     // string, required NEW — throws if missing
  property,    // string, required NEW — throws if missing
  runtime_ms,  // number, required NEW — throws if missing or not a number
  summary,     // string, required NEW — throws if missing
  triage_tags, // string[], optional — defaults to []
  observation_window, // object, optional — defaults to undefined
  metadata,    // object, optional (existing)
})
```

The record written to NDJSON must include all provided fields at the top level (not nested in `metadata`). This changes the output format from the current flat 5-field record to a flat 10-12 field record.

### Backwards Compatibility Pattern

The schema's `required` array stays as `["tool", "formalism", "result", "timestamp"]`. Old 5-field NDJSON lines will still validate against the schema because the new fields are optional. New writes from `writeCheckResult()` will include all v2.1 fields.

Consumers reading `check-results.ndjson` must handle both 5-field and 10-field lines via null-coalescing: `record.check_id || record.tool`.

### Anti-Patterns to Avoid

- **Nesting new fields in `metadata`:** The REQUIREMENTS explicitly call out `check_id`, `surface`, etc. as top-level fields. Do NOT put them in `metadata: { check_id: ... }`.
- **Adding new fields to `required` in schema:** This would invalidate existing NDJSON lines. All new fields must be optional in the schema.
- **Updating run-formal-verify.cjs STEPS to pass new fields:** The STEPS entries do NOT call `writeCheckResult()` directly. They invoke runner scripts via `spawnSync`. The callers are the individual runner scripts, not the STEPS registry. STEPS' `id` field is the source of truth for `check_id` values — but the actual write happens inside each runner script.
- **Changing the NDJSON append behavior:** `writeCheckResult()` uses `fs.appendFileSync()` which is correct. Do not change to streaming or batch writes.

## Common Pitfalls

### Pitfall 1: `additionalProperties: false` Blocking New Fields
**What goes wrong:** The current schema has `additionalProperties: false`. Adding new fields to NDJSON records without updating the schema first will cause schema validation failures.
**Why it happens:** The schema was written conservatively to reject unexpected keys.
**How to avoid:** Task 1 MUST update the schema before Task 2 updates the writer. The schema update should add all 7 new properties to the `properties` object but NOT to `required`.
**Warning signs:** Schema validator rejects newly-written NDJSON lines.

### Pitfall 2: Conflating `surface` With `formalism`
**What goes wrong:** `surface` and `formalism` have overlapping domains (both use values like `"tla"`, `"alloy"`, `"prism"`). A caller might set `surface = entry.formalism`.
**Why it happens:** The field names are close in meaning but not identical.
**How to avoid:** `surface` maps to the STEPS `tool` field (group name); `formalism` is the existing field for the verification technique. For most callers they will have the same string value, but for `generate` and `petri` steps, `surface` will be `"generate"` or `"petri"` while `formalism` may need a different value or the schema may need `generate`/`petri` added to the `formalism` enum.
**Warning signs:** `generate:tla-from-xstate` step uses `surface: "generate"` but what `formalism` should it use? Clarify this in the plan.

**Note:** The current `formalism` enum is `["tla", "alloy", "prism", "trace", "redaction"]`. The `generate` and `petri` STEPS do not currently call `writeCheckResult()` — only runner scripts that produce formal artifacts call it. Check whether generate/petri steps need to be included in the "23 callers" count.

### Pitfall 3: Caller Count Discrepancy
**What goes wrong:** REQUIREMENTS.md says "23 callers" but the actual STEPS count is 25.
**Why it happens:** STEPS grew from 21 to 23 in v0.19-11-01 (added `ci:trace-redaction` and `ci:trace-schema-drift`); a `tla:mcp-environment` step was also added at some point. The header comment in `run-formal-verify.cjs` says "23 steps" but actual count is 25.
**How to avoid:** The planner should target all STEPS entries that dispatch through scripts calling `writeCheckResult()`. The `wasm-dot` type steps (e.g., `petri:account-manager`) do NOT call runner scripts that invoke `writeCheckResult()`. The `generate:*` steps call `xstate-to-tla.cjs` and `generate-formal-specs.cjs` which may not call `writeCheckResult()` at all.
**Warning signs:** Passing "23" as a hard count and missing 2+ steps.

**Caller inventory (from grep analysis):**
Scripts that call `writeCheckResult()`:
1. `run-tlc.cjs` — covers `tla:quorum-safety`, `tla:quorum-liveness`, `tla:mcp-environment`
2. `run-oscillation-tlc.cjs` — covers `tla:oscillation`, `tla:convergence`
3. `run-protocol-tlc.cjs` — covers `tla:deliberation`, `tla:prefilter`
4. `run-breaker-tlc.cjs` — covers `tla:breaker`
5. `run-account-manager-tlc.cjs` — covers `tla:account-manager`
6. `run-alloy.cjs` — covers `alloy:quorum-votes`
7. `run-audit-alloy.cjs` — covers `alloy:scoreboard`, `alloy:availability`
8. `run-transcript-alloy.cjs` — covers `alloy:transcript`
9. `run-installer-alloy.cjs` — covers `alloy:install-scope`, `alloy:taxonomy-safety`
10. `run-account-pool-alloy.cjs` — covers `alloy:account-pool`
11. `run-prism.cjs` — covers `prism:quorum`, `prism:mcp-availability`
12. `run-oauth-rotation-prism.cjs` — covers `prism:oauth-rotation`
13. `check-trace-redaction.cjs` — covers `ci:trace-redaction`

**14 scripts covering 21 STEPS entries.** The `generate:*` and `petri:*` steps (4 entries) do NOT currently call `writeCheckResult()`. SCHEMA-03 targets the 21 STEPS entries with active callers. The REQUIREMENTS say "23 callers" meaning STEPS entries (not scripts) — the correct count appears to be 21 active caller STEPS entries, not 23 or 25. The planner should use 21 as the definitive count.

### Pitfall 4: Timer Placement in Callers
**What goes wrong:** `runtime_ms` requires a start timestamp before the check runs and an end timestamp after. Many callers have multiple early-exit paths (Java not found, JAR not found, etc.) before the actual check runs.
**Why it happens:** Timer start must be placed BEFORE the actual verification tool is invoked.
**How to avoid:** Place `const _startMs = Date.now()` immediately before the external tool invocation (`spawnSync` call or similar), not at the top of the file. All `writeCheckResult()` calls in the same invocation path then use `Date.now() - _startMs`.
**Warning signs:** `runtime_ms = 0` on every record (timer start placed before early exits, making all paths look instant).

### Pitfall 5: Multiple `writeCheckResult()` Calls Per Invocation
**What goes wrong:** Many callers have multiple exit paths that each call `writeCheckResult()` (e.g., `run-tlc.cjs` has 8+ calls — on Java not found, on JAR not found, on TLC fail, on liveness inconclusive, on pass). Each path needs `check_id` and `runtime_ms` but the timer start position differs.
**Why it happens:** The runner scripts were written with early-exit safety valves.
**How to avoid:** Define `_startMs = Date.now()` at module-level (before any `spawnSync`), and pass `runtime_ms: Date.now() - _startMs` in all `writeCheckResult()` calls. For early-exit paths (Java not found), `runtime_ms` will be near 0 which is acceptable.

### Pitfall 6: Callers Must Know Their Own `check_id`
**What goes wrong:** Callers don't know which STEPS entry they are serving. E.g., `run-tlc.cjs` is called with `MCsafety`, `MCliveness`, or `MCMCPEnv` — each maps to a different `check_id`.
**Why it happens:** The STEPS registry is in `run-formal-verify.cjs`; callers are independent scripts.
**How to avoid:** Build a local config-to-check_id map inside each multi-config runner. For example, in `run-tlc.cjs`:
```javascript
const CHECK_ID_MAP = {
  'MCsafety':       'tla:quorum-safety',
  'MCliveness':     'tla:quorum-liveness',
  'MCMCPEnv':       'tla:mcp-environment',
};
const check_id = CHECK_ID_MAP[configName] || 'tla:' + configName.toLowerCase();
```
**Warning signs:** All TLA+ checks emit the same `check_id` regardless of config.

### Pattern 2: Updated `writeCheckResult()` Validation

```javascript
// Source: bin/write-check-result.cjs (updated)
function writeCheckResult(entry) {
  // Existing required field validations (unchanged)
  if (!entry || typeof entry.tool !== 'string' || entry.tool.length === 0) {
    throw new Error('[write-check-result] tool is required');
  }
  if (!VALID_FORMALISMS.includes(entry.formalism)) {
    throw new Error('[write-check-result] formalism must be one of: ' + VALID_FORMALISMS.join(', '));
  }
  if (!VALID_RESULTS.includes(entry.result)) {
    throw new Error('[write-check-result] result must be one of: ' + VALID_RESULTS.join(', '));
  }

  // New v2.1 required field validations
  if (typeof entry.check_id !== 'string' || entry.check_id.length === 0) {
    throw new Error('[write-check-result] check_id is required (string)');
  }
  if (typeof entry.surface !== 'string' || entry.surface.length === 0) {
    throw new Error('[write-check-result] surface is required (string)');
  }
  if (typeof entry.property !== 'string' || entry.property.length === 0) {
    throw new Error('[write-check-result] property is required (string)');
  }
  if (typeof entry.runtime_ms !== 'number') {
    throw new Error('[write-check-result] runtime_ms is required (number)');
  }
  if (typeof entry.summary !== 'string' || entry.summary.length === 0) {
    throw new Error('[write-check-result] summary is required (string)');
  }

  const record = {
    tool:               entry.tool,
    formalism:          entry.formalism,
    result:             entry.result,
    timestamp:          new Date().toISOString(),
    check_id:           entry.check_id,
    surface:            entry.surface,
    property:           entry.property,
    runtime_ms:         Math.round(entry.runtime_ms),
    summary:            entry.summary,
    triage_tags:        entry.triage_tags || [],
    observation_window: entry.observation_window || undefined,
    metadata:           entry.metadata || {},
  };

  // Remove undefined fields to keep NDJSON clean
  if (record.observation_window === undefined) delete record.observation_window;

  fs.appendFileSync(NDJSON_PATH, JSON.stringify(record) + '\n', 'utf8');
}
```

3. **`triage_tags` initial population — who sets them?**
   - What we know: SCHEMA-02 says callers "receive typed interface." TRIAGE-01 (Phase v0.20-06) will read triage_tags for suspects.md.
   - What's unclear: Should callers in Phase v0.20-01 set meaningful triage_tags, or just emit empty arrays?
   - Recommendation: Phase v0.20-01 callers emit `triage_tags: []` (empty array). Triage population is Phase v0.20-06's job.

## Validation Architecture

### Test Framework

This project uses `node:test` (built-in) with `node:assert`. Test files use the `.test.cjs` suffix. The test suite is run with:

```bash
# Quick run (write-check-result tests only):
node --test bin/write-check-result.test.cjs

# Full suite:
npm test
```

The `npm test` script in `package.json` runs `node scripts/lint-isolation.js` first, then `node --test [all test files]` as a single command. The test file for `write-check-result.cjs` is `bin/write-check-result.test.cjs` and is already included in the `npm test` list.

### Wave 0 Test Scaffolding Requirements

Before implementation starts (i.e., before modifying `write-check-result.cjs`), the following test cases must be added to `bin/write-check-result.test.cjs`:

1. **`writeCheckResult throws on missing check_id`** — passes `{ tool, formalism, result }` without `check_id`; expects error matching `/check_id/`
2. **`writeCheckResult throws on missing surface`** — passes `{ tool, formalism, result, check_id }` without `surface`; expects error matching `/surface/`
3. **`writeCheckResult throws on missing property`** — passes `{ tool, formalism, result, check_id, surface }` without `property`; expects error matching `/property/`
4. **`writeCheckResult throws on missing runtime_ms`** — passes all required except `runtime_ms`; expects error matching `/runtime_ms/`
5. **`writeCheckResult throws on missing summary`** — passes all required except `summary`; expects error matching `/summary/`
6. **`v2.1 record has all required fields at top level`** — calls with all v2.1 fields; reads back NDJSON; checks `check_id`, `surface`, `property`, `runtime_ms`, `summary` are top-level keys
7. **`v2.1 record without observation_window omits the field`** — `observation_window` absent when not passed
8. **`triage_tags defaults to empty array`** — `triage_tags` is `[]` when not provided

These 8 tests must be written (failing) before `write-check-result.cjs` is modified. This is the Nyquist Wave 0 scaffolding requirement.
