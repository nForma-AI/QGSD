  <files>bin/compute-per-model-gates.cjs</files>
  <action>
Modify the main per-model loop in compute-per-model-gates.cjs (around lines 500-660) to implement three changes:

**GPROMO-01: consecutive_pass_count tracking (increment/reset)**
After the gate evaluation section (where maturity is computed from gate A/B/C results, around line 520) and BEFORE the promotion logic block:

1. Determine if the model "passes" this session. A model passes when:
   - maturity >= 1 (at least one gate passes)
   - Evidence is not regressing: evidenceReadiness.skipped OR evidenceReadiness.score >= threshold for current gate level (ADVISORY: 0, SOFT_GATE: 0.8, HARD_GATE: 2.5)
   - Model is NOT in UNSTABLE or cooling-down state (isUnstableOrCooling is false)

2. If the model passes: `model.consecutive_pass_count = (model.consecutive_pass_count || 0) + 1`
3. If the model fails ANY condition: `model.consecutive_pass_count = 0`

**Reset on demotion paths (lines ~625-660):**
In BOTH demotion blocks (SOFT_GATE->ADVISORY around line 629, HARD_GATE->SOFT_GATE around line 646), add:
```javascript
model.consecutive_pass_count = 0;
```
immediately after setting model.gate_maturity to the demoted level.

**Reset on UNSTABLE detection:**
When isUnstableOrCooling is true AND promotion is blocked, also reset:
```javascript
model.consecutive_pass_count = 0;
```
This covers Pitfall 2 from research: flip-flop detection must zero the streak.

**GPROMO-02: Gate SOFT_GATE->HARD_GATE promotion on consecutive_pass_count >= 3**
In the SOFT_GATE->HARD_GATE promotion block (line ~589-622), wrap the existing `allowHardPromo` logic with an additional check:
```javascript
if (allowHardPromo && (model.consecutive_pass_count || 0) >= 3) {
```
This means the existing maturity >= 3, evidence >= 3, and stability checks all still apply, PLUS the model needs 3+ consecutive clean sessions. Do NOT change the existing conditions — ADD the consecutive_pass_count check as an additional gate.

**GPROMO-03: Add consecutive_pass_count to changelog evidence snapshot**
In the SOFT_GATE->HARD_GATE appendChangelog call (around line 610-615), add the field:
```javascript
consecutive_pass_count: model.consecutive_pass_count,
```
to the changelog entry object, alongside the existing evidence_readiness and trigger fields.

**Important edge cases (from research pitfalls):**
- Do NOT reset consecutive_pass_count when a model promotes (promotion is a success, not a failure)
- Once a model is HARD_GATE, consecutive_pass_count continues incrementing for tracking but is not used for retention (demotion uses evidence thresholds per existing logic)
- The --dry-run flag should NOT affect consecutive_pass_count tracking (it's computed state, not a mutation). However, if --dry-run skips the registry write, the count won't persist. This is acceptable — dry-run is for testing, not production sessions.
