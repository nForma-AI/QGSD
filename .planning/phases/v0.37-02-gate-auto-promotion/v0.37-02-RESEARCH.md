# Phase v0.37-02: Gate Auto-Promotion - Research

**Researched:** 2026-03-16
**Domain:** Formal verification gate promotion, model registry state tracking
**Confidence:** HIGH

## Summary

This phase extends nForma's existing gate promotion system with session-aware tracking. Currently, models auto-promote from ADVISORY→SOFT_GATE when they achieve layer_maturity >= 1, and SOFT_GATE→HARD_GATE when layer_maturity >= 3. The change adds explicit session-level pass counting: each solve session increments `consecutive_pass_count` if a model remains in good standing (passes gate validation), and resets to 0 on any validation failure.

The auto-promotion criteria expand from purely threshold-based (layer_maturity + evidence readiness) to include consecutive validation streak: a model must achieve 3+ consecutive passes, maintain wiring >= 1.0, and maintain semantic >= 0.8 to auto-promote SOFT_GATE→HARD_GATE. Each promotion is logged to `promotion-changelog.json` during Phase 4 (report step) with the gate evidence snapshot.

**Primary recommendation:** Extend `compute-per-model-gates.cjs` to track consecutive passes per session, modify SOFT_GATE→HARD_GATE promotion logic to require consecutive_pass_count >= 3, and ensure phase 4 (report phase) of nf-solve.cjs calls the gate computation with changelog writing enabled.

## Architecture Patterns

### Current Gate Promotion Flow
The existing system operates in three layers:

1. **Gate Evaluation (per-model)**: evaluateGateA/B/C() compute binary pass/fail for each formal model
2. **Layer Maturity Scoring**: Sum of gate passes (0-3) stored in model.layer_maturity
3. **Threshold Promotion**: ADVISORY→SOFT_GATE if maturity >= 1 + evidence >= 1; SOFT_GATE→HARD_GATE if maturity >= 3 + evidence >= 3

### Phase 4 (Report) Integration
The solve process has these phases:
- Phase 1: Initialization, collect residuals
- Phase 2: Auto-close remediation (mutations)
- Phase 3: Convergence check
- Phase 4: Final report, compute gates, write changelog

Gate computation runs in Phase 4 via `node bin/compute-per-model-gates.cjs` (called from nf-solve.cjs around line 2870).

### New: Session-Level Pass Tracking
Add `consecutive_pass_count` to each model in model-registry.json:

```json
{
  "model_path": {
    "version": 1,
    "last_updated": "2026-03-16T...",
    "layer_maturity": 3,
    "gate_maturity": "SOFT_GATE",
    "consecutive_pass_count": 2,
    "source_layer": "L3",
    "requirements": [...]
  }
}
```

**Increment Logic:**
- Each solve session runs gate evaluation
- If model passes ALL gate criteria (gates A, B, C OR layer_maturity >= 1 minimum), increment consecutive_pass_count
- On ANY validation failure (demotion, gate regression, flip-flop block), reset to 0
- On successful SOFT_GATE→HARD_GATE promotion, log with evidence snapshot and current consecutive_pass_count

**Promotion Criteria (new):**
```
SOFT_GATE → HARD_GATE when:
  consecutive_pass_count >= 3 AND
  wiring >= 1.0 AND
  semantic >= 0.8
```

Wiring is the aggregate of gates A+B+C scoring. Semantic is derived from haiku-semantic-eval or gate-c (validation) scoring.

## Common Pitfalls

### Pitfall 1: Resetting consecutive_pass_count on Demotion But Not on Evidence Regression
**What goes wrong:** A model dips below evidence threshold (evidence_readiness score < 2.5), demotes HARD_GATE→SOFT_GATE, but the consecutive_pass_count doesn't reset. Next session, it artificially inflates because the session didn't pass all gates.
**Why it happens:** The increment logic isn't clear about what "pass" means — is it layer_maturity >= 1, or is it passing ALL gates?
**How to avoid:** Define "pass" explicitly: a model passes a session if it meets its CURRENT gate level (ADVISORY models need layer_maturity >= 1, SOFT_GATE models need maturity >= 1 + evidence >= 0.8, HARD_GATE needs maturity >= 3 + evidence >= 2.5).
**Warning signs:** Test a model's consecutive_pass_count history across a demotion and verify it resets.

### Pitfall 2: Not Resetting on Flip-Flop Detection
**What goes wrong:** Flip-flop detection (STAB-01) flags a model UNSTABLE and blocks promotion. But the consecutive_pass_count doesn't reset. Later, cooldown expires, and the count artificially allows promotion.
**Why it happens:** Forgetting that flip-flops indicate a model is oscillating, which is itself a "fail" for tracking pass streaks.
**How to avoid:** When a model is detected as UNSTABLE or when createUnstableEntry() is called, reset consecutive_pass_count to 0.
**Warning signs:** Models with UNSTABLE status should have consecutive_pass_count near 0 after the unstable entry is written.

### Pitfall 3: Confusing Wiring Score with Gate A Pass
**What goes wrong:** Implementation assumes "wiring >= 1.0" means gate A passes (score >= 0.8). But wiring is the aggregate of THREE gates (A, B, C), each with its own target.
**Why it happens:** The requirement says "wiring >= 1.0" which sounds like a single gate, but wiring is composite.
**How to avoid:** Clarify that wiring is (gate_a_score + gate_b_score + gate_c_score) / 3, or the sum of gate passes (0-3). The requirement likely means "sum of gate passes >= 1" (i.e., layer_maturity >= 1).
**Warning signs:** Check gate-a-grounding.json, gate-b-abstraction.json, gate-c-validation.json to understand their individual scoring; verify which aggregation the promotion criteria uses.

### Pitfall 4: Phase 4 Report Runs with --dry-run Flag
**What goes wrong:** Phase 4 runs compute-per-model-gates.cjs --dry-run (testing), which skips appendChangelog(). Auto-promotion appears to happen but never logs.
**Why it happens:** nf-solve.cjs sets --dry-run when reportOnly is true (line 2803), preventing mutations.
**How to avoid:** Separate "write registry" from "write changelog". The --dry-run should skip registry writes (model.gate_maturity updates) but still log promotions to changelog (immutable append-only log). OR, ensure nf-solve Phase 4 calls gates WITHOUT --dry-run.
**Warning signs:** promotion-changelog.json has no entries for sessions where models promoted.

### Pitfall 5: Not Testing Consecutive Count Reset on Fail Cascade
**What goes wrong:** A model passes 3 sessions, gets to consecutive_pass_count=3, promotes to HARD_GATE. Session 4, it barely fails one gate. The count resets to 0. Session 5, it passes all gates again. But the model is now HARD_GATE, and the 3-pass requirement is only for SOFT_GATE→HARD_GATE transition, not for staying in HARD_GATE.
**Why it happens:** Requirement GPROMO-02 specifies the promotion rule, not the retention rule. Unclear whether HARD_GATE models need consecutive_pass_count to stay high.
**How to avoid:** Define explicitly: once a model is HARD_GATE, consecutive_pass_count is for historical tracking only; demotion uses evidence thresholds (existing HARD_GATE→SOFT_GATE logic), not consecutive count.
**Warning signs:** Unit tests should cover: promote ADVISORY→SOFT_GATE (maturity >= 1), SOFT_GATE→HARD_GATE (consecutive >= 3), demotion (HARD_GATE→SOFT_GATE on evidence < 2.5), verify consecutive_pass_count state at each transition.

## Code Examples

### Increment consecutive_pass_count (in compute-per-model-gates.cjs)

```javascript
// After gate evaluation passes (maturity >= 1 for SOFT/HARD gates, or maturity >= 1 for ADVISORY→SOFT)
// and before promotion logic:

const modelStaysHealthy = maturity >= (model.gate_maturity === 'HARD_GATE' ? 2.5 : 1);
const noEvidenceRegression = evidenceReadiness.skipped || evidenceReadiness.score >= (model.gate_maturity === 'HARD_GATE' ? 2.5 : 0.8);

if (modelStaysHealthy && noEvidenceRegression && !isUnstableOrCooling) {
  model.consecutive_pass_count = (model.consecutive_pass_count || 0) + 1;
} else {
  model.consecutive_pass_count = 0;
}
model.last_updated = new Date().toISOString();

// Source: compute-per-model-gates.cjs lines ~520-530 (revised)
```

### Promotion criteria with consecutive_pass_count (GPROMO-02)

```javascript
// Auto-promotion: SOFT_GATE → HARD_GATE with consecutive pass requirement
if (model.gate_maturity === 'SOFT_GATE' && maturity >= 3 && (evidenceReadiness.skipped || evidenceReadiness.score >= 3)) {
  // NEW: require consecutive_pass_count >= 3
  if ((model.consecutive_pass_count || 0) >= 3 && !isUnstableOrCooling) {
    const v = validateCriteria(modelPath, model, 'HARD_GATE', checkResults, evidenceReadiness);
    if (v.valid) {
      model.gate_maturity = 'HARD_GATE';
      model.last_updated = new Date().toISOString();
      appendChangelog({
        model: modelPath,
        from_level: 'SOFT_GATE',
        to_level: 'HARD_GATE',
        timestamp: new Date().toISOString(),
        evidence_readiness: { score: evidenceReadiness.score, total: evidenceReadiness.total },
        trigger: 'auto_promotion',
        consecutive_pass_count: model.consecutive_pass_count,  // GPROMO-03: evidence snapshot
      });
      promotions.push({ model: modelPath, from: 'SOFT_GATE', to: 'HARD_GATE' });
    }
  }
}
```

## Open Questions

1. **Wiring vs. Semantic Scoring in GPROMO-02**
   - What we know: Requirement says "wiring >= 1.0 AND semantic >= 0.8"
   - What's unclear: How are these calculated? Wiring appears to be layer_maturity (0-3), but where does "1.0" come from (is it maturity >= 1, or a normalized score)?
   - Recommendation: Consult promote-gate-maturity.cjs::validateCriteria() to understand what "wiring" and "semantic" refer to. If not explicitly scored, interpret as: layer_maturity >= 1 (wiring) and gate-c/validation >= 0.8 (semantic).

2. **Consecutive Count Persistence Across Demotion Cycles**
   - What we know: STAB-01/STAB-02 cooldown resets after 3 stable sessions; consecutive_pass_count should also reset on demotion.
   - What's unclear: If a model demotes HARD_GATE→SOFT_GATE on evidence regression, does it need 3 new consecutive passes to re-promote, or does the prior consecutive_pass_count apply?
   - Recommendation: Treat consecutive_pass_count as tied to current gate level. On demotion, reset to 0. A re-promoted model must earn 3 new passes at its new level.

3. **Phase 4 Dry-Run Behavior**
   - What we know: compute-per-model-gates.cjs skips appendChangelog() when --dry-run is set.
   - What's unclear: Should Phase 4 of solve run with --dry-run (inhibiting changelog writes) or without?
   - Recommendation: Check nf-solve.cjs line 2803. If reportOnly sets --dry-run, revisit: consider separating "registry mutation" (—dry-run skips) from "immutable logging" (always log). OR, ensure Phase 4 is never reportOnly.
