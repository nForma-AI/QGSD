# Phase v0.39-02: Cycle 1 Diagnostic — Research

**Researched:** 2026-03-18
**Domain:** Diagnostic diff generation, model incompleteness messaging, quorum diagnostic injection
**Confidence:** HIGH (architecture + foundation from Phase 1), MEDIUM-HIGH (diagnostic semantics, constraint relevance ranking)

## Summary

Phase v0.39-02 implements the critical **diagnostic feedback loop** of Cycle 1. When a model fails to reproduce a bug, instead of generic "retry" guidance, users see a **targeted diagnostic diff** that identifies exactly what the model assumes vs. what the bug trace shows. This diagnostic is then injected into quorum worker prompts to guide specific model corrections (which invariants to add, which state variables are missing), replacing blind refinement with evidence-based refinement.

The phase builds on Phase v0.39-01 infrastructure:
- **Verification mode tagging** (FND-01) enables correct interpretation of traces (diagnostic vs. validation semantics)
- **ITF trace parsing** (FND-02) provides structured state sequences for comparison
- **Configurable iteration limits** (FND-03) bounds the refinement loop

This phase adds three new capabilities:

1. **Diagnostic Diff Generation (DX1-01):** When a model checker passes (doesn't capture bug), parse its state space + compare against the observed bug trace to generate "model assumes X but bug shows Y" diffs.

2. **Reframed UX (DX1-02):** Update all model-driven-fix and refinement-loop output to use evidence-based language: "model is incomplete/wrong" (bug is ground truth, oracle) instead of "bug not captured" (which sounds like the bug description is wrong).

3. **Diagnostic Injection into Quorum (DX1-03, DX1-04):** When refinement loop calls quorum workers to improve the model, include the diagnostic diff in the prompt context. Workers read the diff and propose specific corrections ("add invariant X", "add state variable Y tracking Z") instead of generic "improve the model".

**Primary recommendation:** Implement diagnostic diff generation as a three-stage pipeline: (1) parse model's state space from checker output or trace, (2) align states to observed bug trace using field-level matching, (3) extract divergences and propose corrections. Generate corrections in two forms: a **mismatch diff** (human-readable markdown) for user feedback and a **structured correction proposal** (JSON) for quorum injection.

### Pattern 1: Diagnostic Diff Generation Pipeline

**What:** Three-stage pipeline to generate diagnostic diffs:
1. Extract model's final state space from checker output
2. Compare against observed bug trace states
3. Generate mismatch report with field-level divergences

**When to use:** After model checker passes (outcome = 'INCOMPLETE' in verification-mode semantics)

**Example:**

```javascript
// Source: Proposed diagnostic pipeline in refinement-loop.cjs

const { parseITFTrace } = require('./parse-tlc-counterexample.cjs');
const { generateStateDiff, formatDiffAsMarkdown } = require('./diagnostic-diff-generator.cjs');
const { extractModelStateSpace } = require('./model-state-extractor.cjs');

/**
 * Generate diagnostic diff when model fails to reproduce bug.
 * @param {Object} checkerResult - Result from run-tlc or run-alloy
 * @param {string} bugTraceJsonPath - Path to parsed bug trace (ITF JSON)
 * @param {string} bugContext - Original bug description
 * @returns {{ mismatch_diff: string, correction_proposals: Array, trace_alignment: Object }}
 */
function generateDiagnosticFeedback(checkerResult, bugTraceJsonPath, bugContext) {
  // Stage 1: Extract model's final states from checker output
  const modelFinalStates = extractModelStateSpace(checkerResult.output);

  // Stage 2: Parse bug trace
  const bugTrace = parseITFTrace(bugTraceJsonPath);
  const bugFinalStates = bugTrace.states;

  // Stage 3: Compare state spaces
  const stateDiff = generateStateDiff(modelFinalStates, bugFinalStates, {
    fieldFilter: extractRelevantFields(bugContext)  // Focus on fields mentioned in bug
  });

  // Generate human-readable markdown
  const mismatchDiff = formatDiffAsMarkdown(stateDiff);

  // Generate structured proposals
  const proposals = generateCorrectionProposals(stateDiff, bugContext);

  return {
    mismatch_diff: mismatchDiff,
    correction_proposals: proposals,
    trace_alignment: {
      model_trace_length: modelFinalStates.length,
      bug_trace_length: bugFinalStates.length,
      first_divergence_index: stateDiff.summary.first_divergence_index,
      diverged_fields: stateDiff.summary.changed_fields
    }
  };
}
```

**Integration point:** Called by `verifyBugReproduction()` in refinement-loop.cjs when outcome is 'INCOMPLETE'. Result is stored in iteration feedback and passed to nf-prompt for quorum injection.

### Pattern 2: Correction Proposal Generation

**What:** Convert diagnostic diffs into structured proposals for model correction.

**When to use:** After diff generation, before quorum injection

**Example:**

```javascript
// Source: Proposed diagnostic-proposal-generator.cjs

/**
 * Generate structured correction proposals from diagnostic diff.
 * Proposals are specific, not generic (e.g., "add invariant Timeout" not "improve model").
 *
 * @param {Object} stateDiff - Result from generateStateDiff
 * @param {string} bugContext - Original bug description
 * @returns {Array<{type: string, target: string, reasoning: string, example: string}>}
 */
function generateCorrectionProposals(stateDiff, bugContext) {
  const proposals = [];
  const { summary, per_state_diffs } = stateDiff;

  // Analysis 1: Missing state variables
  // If divergence appears in final states but model has no corresponding field,
  // propose adding state variable
  const divergedFields = summary.changed_fields || [];
  for (const field of divergedFields) {
    // Extract field type from bug trace if mentioned
    const fieldType = inferFieldType(field, bugContext);

    proposals.push({
      type: 'add_state_variable',
      target: field,
      reasoning: `Divergence detected in ${field}: model does not track this variable`,
      example: `VARIABLE ${field}  \\\\ Type: ${fieldType || 'TBD'}`
    });
  }

  // Analysis 2: Missing invariants
  // If divergence is in a specific state, propose invariant constraining that state
  if (per_state_diffs.length > 0) {
    const firstDiff = per_state_diffs[0];
    proposals.push({
      type: 'add_invariant',
      target: `StateConstraint_${firstDiff.index}`,
      reasoning: `Divergence first occurs at state ${firstDiff.index}: propose invariant constraining transitions to this state`,
      example: `Inv_${firstDiff.index} == /\\ state=${firstDiff.index} => conditions`
    });
  }

  // Analysis 3: Temporal property
  // If trace length differs, propose eventually/always property
  if (summary.first_divergence_index !== null) {
    proposals.push({
      type: 'add_temporal_property',
      target: 'FinalStateInvariant',
      reasoning: `Model reaches different final state; propose property asserting correct final conditions`,
      example: `<>[](finalState.${divergedFields[0]} = bugValue)`
    });
  }

  return proposals;
}
```

**Integration point:** Results are stored in diagnostic context and passed to nf-prompt.cjs for quorum injection.

### Pattern 3: Refinement Loop Diagnostic Integration

**What:** Update refinement-loop.cjs to generate diagnostics on model incompleteness and inject into quorum context.

**When to use:** When `verifyBugReproduction` exhausts iterations without reproducing bug

**Example:**

```javascript
// Source: Updated refinement-loop.cjs integration

/**
 * Enhanced verifyBugReproduction with diagnostic feedback.
 * On mismatch (iteration fails), generate diagnostic and inject into quorum context.
 */
function verifyBugReproductionWithDiagnostics(modelPath, bugContext, options) {
  options = options || {};

  // ... existing loop logic ...

  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    // Run checker
    const result = runChecker(modelPath, { verification_mode: 'diagnostic' });

    // Interpret result
    const outcome = interpretGateResult(result.hasViolation, 'diagnostic');

    if (outcome === 'REPRODUCED') {
      // Success path unchanged
      return { status: 'reproduced', attempts: attempt, ... };
    }

    // Mismatch: Generate diagnostic for refinement
    if (outcome === 'INCOMPLETE' && attempt < maxAttempts) {
      const diagnostic = generateDiagnosticFeedback(
        result,
        options.bugTraceJsonPath,  // Path to parsed bug trace
        bugContext
      );

      // Store in iteration for later injection
      iteration.diagnostic = {
        mismatch_diff: diagnostic.mismatch_diff,
        correction_proposals: diagnostic.correction_proposals,
        trace_alignment: diagnostic.trace_alignment,
        attempt_number: attempt,
        max_attempts: maxAttempts
      };

      // Prepare context for next quorum call
      if (options.onDiagnosticGenerated) {
        options.onDiagnosticGenerated(diagnostic, attempt);
      }
    }
  }

  // All attempts exhausted: return diagnostic for user feedback
  return {
    status: 'not_reproduced',
    attempts: maxAttempts,
    final_diagnostic: iterations[iterations.length - 1]?.diagnostic,
    iterations
  };
}
```

**Integration point:** Quorum dispatcher reads `final_diagnostic` from refinement result and injects into nf-prompt additionalContext for next worker invocation.

### Anti-Patterns to Avoid

- **Diagnostic without state space extraction:** Don't generate diffs without parsing model's actual state space; compare apples to apples (both from traces)
- **Ignoring verification mode in diff interpretation:** Don't treat diagnostic traces as normal validation traces; respect mode tag when selecting comparison strategy
- **Generic proposals:** Don't generate "improve the model" proposals; make proposals concrete and specific ("add invariant X" with example)
- **Hardcoding field names:** Don't assume model has same field names as bug context; use flexible field matching with confidence scores
- **Blocking on perfect alignment:** Don't require 100% trace alignment before generating diagnostic; fail-open with best-effort proposals

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| **Field matching between traces** | Custom string/regex alignment | Simple fuzzy field name matching (soundex or levenshtein for typos) or user confirmation dialog | Exact matching fails when bug description uses different terminology than model state variables |
| **State space extraction** | Regex-based state capture from arbitrary checker output | Leverage ITF JSON traces (structured output); fall back to final state only if trace unavailable | Regex is fragile; ITF is standard and complete |
| **Divergence detection** | Custom tree-walk differ | json-diff-ts (already integrated in Phase 1) | Proven library handles nested objects, arrays, type mismatches; no need to re-implement |
| **Proposal ranking/prioritization** | ML model scoring proposals by importance | Heuristic ranking: count affected fields, distance to divergence, field type confidence | Simple heuristics are sufficient; ML adds complexity without clear benefit |
| **Quorum prompt injection** | Direct string concatenation of diagnostic into prompt | Structured context in nf-prompt additionalContext, let nf-prompt.cjs format for each model | Centralized formatting prevents prompt injection bugs and duplication |

**Key insight:** Phase 2 is about connecting existing Phase 1 infrastructure (trace parsing, diff generation) to refinement guidance. The hard work (parsing, comparing) is done; Phase 2 adds the guidance layer (proposals, quorum injection) on top.

## Common Pitfalls

### Pitfall 1: Diagnostic Without Ground Truth Bug Trace

**What goes wrong:** Diagnostic diff is generated comparing model trace to bug context (text), not to actual observed bug trace (structured). Result: diagnostic is imprecise ("model assumes X, bug says X is not mentioned") rather than evidence-based ("model assumes X=5, bug shows X=10").

**Why it happens:** Bug context is informal; temptation to extract expected state directly from text rather than from trace JSON.

**How to avoid:**
- DX1-01 requirement: Diagnostic must compare model trace to bug **trace**, not to bug description
- Require bug trace in JSON format (ITF) before generating diagnostic
- If bug trace unavailable, fail-open: suggest "need to capture bug as formal trace; try /nf:trace-bug-execution"
- Store bug trace alongside model in diagnostic context

**Warning signs:**
- Diagnostic proposals don't match observed bug behavior
- User says "diagnostic is wrong; I ran the bug and saw Y not X"
- Proposals suggest changes unrelated to actual bug

### Pitfall 2: Diagnostic Blocked by Verification Mode Contradiction

**What goes wrong:** Phase 1 set verification_mode='diagnostic' for refinement checker invocations. Phase 2 code assumes 'validation' semantics when interpreting traces. Result: diffs are inverted (fields that should match appear as divergent).

**Why it happens:** Easy to forget Phase 1's mode tagging contract. Naive diff comparison treats violation = failure instead of violation = success.

**How to avoid:**
- DX1-01: Verify mode tag on checker result before generating diagnostic. Only generate diagnostic if mode='diagnostic' AND outcome='INCOMPLETE'
- Document mode semantics clearly: "In diagnostic mode, trace comparison detects what model is missing; in validation mode, comparison detects what model breaks"
- Add assertion: `assert(result.verification_mode === 'diagnostic', 'expected diagnostic mode for diff generation')`

**Warning signs:**
- Diagnostic diffs are empty for models that clearly differ from bug
- Proposals suggest removing invariants instead of adding them
- Trace alignment shows divergence where none should exist

### Pitfall 3: Field Matching Produces Too Many False Positives

**What goes wrong:** Model tracks "timeout" and bug context mentions "timeout"; exact match succeeds but they're measuring different things (milliseconds vs. seconds). Diagnostic suggests "add invariant" for a field that already exists but with wrong units.

**Why it happens:** Field names are ambiguous across contexts. No semantic understanding of field meaning.

**How to avoid:**
- DX1-01: Field matching must include confidence scoring, not boolean exact match
- Produce match candidates with confidence: `{ field: "timeout", confidence: 0.8, reason: "name match + numeric field" }`
- For low-confidence matches, suggest to user: "Did you mean X?" before generating proposals
- Document assumption: "Phase 2 uses syntactic field matching; Phase 3 can refine with semantic analysis"

**Warning signs:**
- Diagnostic suggests changes to fields that already exist in model
- Proposals are obviously wrong ("add invariant for field Y that model already constrain")
- User rejects diagnostic guidance as irrelevant

### Pitfall 4: Proposals Too Generic to Guide Refinement

**What goes wrong:** Diagnostic suggests "add invariant" but doesn't specify which invariant. Quorum worker gets vague guidance, refinement tries random additions, fails again. Cycle repeats.

**Why it happens:** Temptation to avoid over-specificity. Proposals become so generic they don't guide action.

**How to avoid:**
- DX1-04 requirement: Proposals must be specific and actionable
- Each proposal includes: (1) target (invariant name, variable name), (2) reasoning (why it's needed), (3) example skeleton (what it might look like)
- Test: "Could a developer read this proposal and write code without asking for clarification?" If not, too generic
- Example good proposal: "Add state variable `phase: {IDLE, ACTIVE, DONE}` to track state machine phase"
- Example bad proposal: "Add more invariants"

**Warning signs:**
- Quorum workers ignore diagnostic proposals and retry blind
- Refinement iterations don't improve; diagnostics unchanged across attempts
- User feedback: "diagnostic tells me something is wrong but not how to fix it"

### Pitfall 5: Diagnostic Context Lost Between Iterations

**What goes wrong:** First iteration generates diagnostic, passes to quorum worker. Worker refines model. Second iteration re-runs refinement loop, generates new diagnostic, but doesn't compare to first diagnostic. Result: worker is blind to progress. Same proposals repeated; iteration stalls.

**Why it happens:** Each refinement iteration starts fresh. Diagnostics are session-scoped, not persisted across iterations.

**How to avoid:**
- DX1-03 requirement: Diagnostic context persisted in iteration history
- Store diagnostic in iteration object: `iterations[i].diagnostic = { mismatch_diff, proposals, ... }`
- On next iteration, compare new diagnostic to previous: "Divergence reduced from 5 fields to 3 fields" provides progress signal
- Pass diagnostic history to quorum worker: "Previous iteration suggested adding invariants X, Y, Z; those didn't help; try different approach"
- Test: Run refinement loop over 3 iterations on a failing model; verify each diagnostic is different and shows progress (divergence shrinks or proposals change)

**Warning signs:**
- Iteration 1 diagnostic ≈ Iteration 2 diagnostic ≈ Iteration 3 diagnostic (no progress)
- Worker receives same proposals in iteration N+1 as in iteration N
- User complains: "Diagnostics never change; loop is broken"

### Pitfall 6: Quorum Injection Overwrites Existing Context

**What goes wrong:** nf-prompt.cjs receives diagnostic in additionalContext. It appends to existing prompt context. But existing context already includes bug description + constraint extraction results. Diagnostic gets buried or contradicts earlier guidance. Worker is confused.

**Why it happens:** Multiple sources of context (bug desc, constraints, diagnostic) competing for space; no clear priority/structure.

**How to avoid:**
- DX1-03: Design clear context structure for nf-prompt injection
- Diagnostic context is a **section**, not a dump: "## Model Diagnostic Feedback" heading, separate from constraints
- Diagnostic is **latest information**: comes after constraints in the prompt so worker reads most current feedback last
- If diagnostic contradicts earlier constraint guidance, explicitly note: "Earlier iteration suggested X; diagnostic feedback now suggests Y instead"
- Document injection point in nf-prompt.cjs: where diagnostic is added relative to other context sections

**Warning signs:**
- Quorum worker ignores diagnostic, follows old constraint guidance
- Prompts are too long (context overload); diagnostic gets truncated
- Worker output mentions different fix strategy than diagnostic suggested

### Example 1: Extract Model State Space from ITF Trace

```javascript
// Source: Proposed model-state-extractor.cjs (Phase 2 new module)

const { parseITFTrace } = require('./parse-tlc-counterexample.cjs');

/**
 * Extract the final state(s) from a model's ITF trace.
 * For single-trace checkers, this is the last state.
 * For lasso traces, final state is before loop point.
 *
 * @param {string} traceJsonPath - Path to ITF JSON trace written by checker
 * @returns {Array<Object>} Array of final state(s)
 */
function extractFinalStates(traceJsonPath) {
  const trace = parseITFTrace(traceJsonPath);
  const { states, loopPoint } = trace;

  if (!states || states.length === 0) {
    return [];
  }

  // If trace is a lasso (cycle), final states are those before loop
  if (loopPoint !== null && loopPoint < states.length) {
    return states.slice(loopPoint);  // States in cycle
  }

  // Otherwise, return the last state
  return [states[states.length - 1]];
}

/**
 * Extract all reachable states that differ from initial state.
 * Useful for understanding model's full state space.
 *
 * @param {string} traceJsonPath - Path to ITF JSON trace
 * @returns {Object} { initial_state, final_states, all_states, state_count }
 */
function analyzeStateSpace(traceJsonPath) {
  const trace = parseITFTrace(traceJsonPath);
  const { states } = trace;

  if (!states || states.length === 0) {
    return {
      initial_state: null,
      final_states: [],
      all_states: [],
      state_count: 0
    };
  }

  const initialState = states[0];
  const uniqueStates = new Map();
  states.forEach((s, i) => {
    const key = JSON.stringify(s);
    if (!uniqueStates.has(key)) {
      uniqueStates.set(key, { state: s, first_index: i });
    }
  });

  return {
    initial_state: initialState,
    final_states: [states[states.length - 1]],
    all_states: Array.from(uniqueStates.values()).map(v => v.state),
    state_count: states.length
  };
}

module.exports = {
  extractFinalStates,
  analyzeStateSpace
};
```

### Example 2: Generate Structured Correction Proposals

```javascript
// Source: Proposed diagnostic-proposal-generator.cjs (Phase 2 new module)

/**
 * Generate correction proposals from state divergence.
 * Each proposal targets a specific model element that needs addition/modification.
 *
 * @param {Object} stateDiff - Result from generateStateDiff (Phase 1)
 * @param {string} bugContext - Original bug description
 * @returns {Array<{id, type, target, confidence, reasoning, example}>}
 */
function generateCorrectionProposals(stateDiff, bugContext) {
  const { summary, per_state_diffs } = stateDiff;
  const proposals = [];

  // Proposal 1: Add missing state variables
  // For each field that diverges in final state but model doesn't track
  const divergedFields = summary.changed_fields || [];
  for (let i = 0; i < divergedFields.length; i++) {
    const field = divergedFields[i];
    proposals.push({
      id: `PROP-SV-${i}`,  // State Variable proposal
      type: 'add_state_variable',
      target: field,
      confidence: 0.8,  // High confidence: field appears in bug trace
      reasoning: `Model does not track variable '${field}' which is critical to bug behavior`,
      example: `VARIABLE ${field} \\\\ Track this field across states`,
      priority: 1,  // Highest priority: missing fields are critical
      effort: 'low'  // Usually easy to add
    });
  }

  // Proposal 2: Add invariant constraining divergent state
  if (per_state_diffs.length > 0 && summary.first_divergence_index !== null) {
    const idx = summary.first_divergence_index;
    proposals.push({
      id: 'PROP-INV-1',
      type: 'add_invariant',
      target: `Inv_StateConstraint_${idx}`,
      confidence: 0.7,  // Medium confidence: state divergence detected but not understood
      reasoning: `Divergence occurs at state ${idx}; add invariant constraining valid state transitions`,
      example: `Inv_${idx} == state = ${idx} => <valid condition>`,
      priority: 2,  // Medium priority: helps constrain state space
      effort: 'medium'
    });
  }

  // Proposal 3: Add temporal property for final state correctness
  if (divergedFields.length > 0) {
    const sample_field = divergedFields[0];
    proposals.push({
      id: 'PROP-TEMP-1',
      type: 'add_temporal_property',
      target: 'FinalStateCorrectness',
      confidence: 0.6,  // Lower confidence: temporal properties require deep understanding
      reasoning: `Ensure model's final state satisfies correct termination condition`,
      example: `Prop_FinalCorrect == <>[](${sample_field} = bugValue)`,
      priority: 3,
      effort: 'high'
    });
  }

  return proposals.sort((a, b) => a.priority - b.priority);  // Sort by priority
}

module.exports = {
  generateCorrectionProposals
};
```

### Example 3: Diagnostic Injection into Refinement Loop

```javascript
// Source: Updated refinement-loop.cjs integration pattern

const { generateDiagnosticFeedback } = require('./diagnostic-diff-generator.cjs');
const { generateCorrectionProposals } = require('./diagnostic-proposal-generator.cjs');

/**
 * Enhanced iteration feedback including diagnostic diff.
 * Called when model doesn't reproduce bug, to provide targeted guidance.
 */
function formatIterationFeedbackWithDiagnostic(iteration, diagnosticContext) {
  let feedback = `Attempt ${iteration.attempt}: ${iteration.passed ? 'still passes' : 'reproduced'}\n`;
  feedback += `  Summary: ${iteration.summary}\n`;

  // Add diagnostic if available (when model fails to reproduce)
  if (diagnosticContext && diagnosticContext.mismatch_diff) {
    feedback += '\n--- Diagnostic Feedback ---\n';
    feedback += `The model is INCOMPLETE. It does not capture the bug.\n`;
    feedback += `Here's what the model assumes vs. what the bug shows:\n\n`;
    feedback += diagnosticContext.mismatch_diff;
    feedback += '\n\nTo fix this model, try:\n';

    const proposals = diagnosticContext.correction_proposals || [];
    for (let i = 0; i < Math.min(3, proposals.length); i++) {
      const p = proposals[i];
      feedback += `  ${i + 1}. [${p.type}] ${p.target}: ${p.reasoning}\n`;
      feedback += `     Example: ${p.example}\n`;
    }
  }

  return feedback;
}

/**
 * Generate diagnostic context for quorum injection.
 * Called during refinement loop when model fails to reproduce bug.
 */
function generateQuorumDiagnosticContext(checkerResult, bugTraceJsonPath, bugContext, iteration) {
  try {
    const diagnostic = generateDiagnosticFeedback(checkerResult, bugTraceJsonPath, bugContext);
    const proposals = generateCorrectionProposals(diagnostic.stateDiff, bugContext);

    return {
      status: 'incomplete_model',
      iteration_number: iteration,
      mismatch_diff: diagnostic.mismatch_diff,
      correction_proposals: proposals,
      trace_alignment: diagnostic.trace_alignment,
      guidance: `Model does not capture the bug. ${proposals.length} specific corrections proposed.`
    };
  } catch (err) {
    // Fail-open: return empty diagnostic on error
    return {
      status: 'diagnostic_failed',
      error: err.message,
      guidance: 'Could not generate diagnostic; try blind refinement'
    };
  }
}

module.exports = {
  formatIterationFeedbackWithDiagnostic,
  generateQuorumDiagnosticContext
};
```

## Open Questions

1. **How to handle field name mismatches between bug trace and model?**
   - What we know: Field names in bug trace (from TLC output) may not match variable names in model spec
   - What's unclear: Should we do fuzzy matching (soundex), user confirmation dialog, or require exact match?
   - Recommendation: Phase 2 uses fuzzy matching with confidence scores; low-confidence matches prompt user confirmation before generating proposals. Phase 3 can refine with semantic analysis if needed.

2. **Should diagnostic proposals be ranked by priority or by confidence?**
   - What we know: Different proposals have different priority (missing state variables = critical; temporal properties = nice-to-have) and different confidence (exact field match = high; heuristic guess = low)
   - What's unclear: What's the right sorting? Should low-effort-high-confidence proposals come first (most likely to help), or critical-high-confidence (most important)?
   - Recommendation: Sort by priority first (critical before optional), then by effort (low before high), then by confidence (high before low). This prioritizes "add the missing variable" over "refactor temporal properties".

3. **How many iterations before diagnostic should be considered stable?**
   - What we know: First iteration diagnostic is based on fresh trace; subsequent iterations have refined models that may show different divergences
   - What's unclear: If diagnostic changes across iterations, should worker ignore early diagnostics? Should we only inject diagnostic when stable (no change iteration N->N+1)?
   - Recommendation: Always inject latest diagnostic. Allow worker to compare diagnostics across iterations ("earlier diagnostic suggested X, current diagnostic suggests Y"). Phase 3 can add convergence heuristics if needed.

4. **Should diagnostic block refinement if trace unavailable?**
   - What we know: Diagnostic requires bug trace in ITF JSON format
   - What's unclear: If bug trace is unavailable, should we (a) fail-open and let refinement continue blind, (b) error and block, (c) attempt to infer trace from bug context?
   - Recommendation: Fail-open with warning. If bug trace unavailable, refinement continues without diagnostic guidance. Log message: "Bug trace not found; proceeding with blind refinement". Phase 2 implementation can add feature to auto-capture bug trace if needed.
