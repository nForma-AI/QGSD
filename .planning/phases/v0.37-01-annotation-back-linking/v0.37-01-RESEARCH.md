# Phase v0.37-01: Annotation Back-Linking - Research

**Researched:** 2026-03-16
**Domain:** Requirements traceability through reverse-direction scanning, proximity graph integration, test/code annotation systems
**Confidence:** HIGH

## Summary

This phase implements bidirectional requirements traceability by adding annotation support to test and code files, integrating with the proximity graph system, and modifying reverse scanners to honor these annotations. The core insight is that tests and source files should declare their requirement context directly via `@requirement` comment annotations, which then suppress false-positive orphan flags in the T->R (test-to-requirement) and C->R (code-to-requirement) scanners.

The implementation spans three parallel tracks: (1) parsing and declaring `@requirement` annotations in test and source files, (2) extending the proximity graph builder to recognize these direct declarations as high-confidence edges, and (3) modifying the reverse scanners to query proximity-index.json before flagging orphans. A new utility `bin/annotate-tests.cjs` uses the proximity graph to suggest annotations for orphaned tests, accelerating manual remediation.

**Primary recommendation:** Implement in order: proximity graph annotation parsing → C->R scanner proximity lookup → T->R scanner proximity lookup → annotate-tests.cjs tool → full test coverage. The existing extract-annotations.cjs parser provides the canonical pattern for all annotation patterns across comment types.

## Architecture Patterns

### Annotation Format and Parsing

**What:** Consistent @requirement comment annotation across file types (test, source, formal model).

**Pattern in TLA+ (formal-proximity.cjs line 45):**
```
\* @requirement TLINK-01
PropertyName == ...
```

**Pattern in Alloy:**
```
-- @requirement REQ-01
assert AssertName ...
```

**Pattern in PRISM:**
```
// @requirement PROP-01
property_definition ...
```

**Pattern in JavaScript test files (nf-solve.cjs line 2067):**
```javascript
// @requirement TLINK-01
test('...', () => { ... })
```

**Pattern in source files (bin/*.cjs) — new:**
```javascript
// Requirements: CLINK-01, CLINK-02
// or
// @requirement CLINK-01
module.exports = { ... }
```

**Key:** Single regex `/@requirement\s+([A-Z]+-\d+)/i` works across all file types. Extract-annotations.cjs already implements language-specific versions; code extraction uses generic JS pattern matching.

**Source:** extract-annotations.cjs lines 45, 93, 309; nf-solve.cjs lines 2067-2068

### Proximity Graph Integration

**What:** Direct `declares` / `declared_in` edges when annotations are found in source files.

**Edge Type:** `declares` (file → requirement) with semantic weight 0.7
- Defined in formal-proximity.cjs line 19: `declares: 0.7, declared_in: 0.7`
- Semantically: "file explicitly declares requirement via annotation"
- Cost (0.7): Strong signal but not as strong as direct formal verification (1.0)

**Parser Integration Point:** formal-proximity.cjs sourceArtifacts function
- Currently processes 12 artifact types (scope.json, requirements.json, model-registry.json, etc.)
- Add `code_files` and `test_files` artifact types that extract annotations
- Return: `{ type: 'code_file', id: 'file_path', declares: [REQ-01, REQ-02] }`

**Query Integration Point:** Scanners use formal-query.cjs reach() function
- Already used by sweepCtoR (line 1959) and sweepTtoR (line 2088)
- Example: `reach(pi, 'code_file::bin/nf-solve.cjs', 2, ['requirement'])`
- Returns: Nodes reachable in 2 hops of type 'requirement'

**Source:** formal-proximity.cjs lines 71-85 (scoring methods), 94-126 (reverse relations)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Parsing @requirement from 5+ file types | Custom regex per type | extract-annotations.cjs pattern library | Maintains consistency, handles edge cases (comments, blank lines, orphan annotations) |
| Building proximity graph from scratch | Custom graph builder | formal-proximity.cjs + artifact registry | Already processes 12 artifact sources, handles scoring, bidirectional relations, semantic weighting |
| Querying semantic similarity across files | Embedding-based search | proximity-index.json + formal-query reach() | Deterministic, fast, no external services, already integrated in scanners |
| Generating annotation suggestions | Static template tool | Proximity embedding (existing) | Embedding ranks candidates by semantic proximity; can reuse bin/proximity-embed.mjs logic |

**Key insight:** The proximity graph system was designed exactly for this use case — it captures semantic relationships between code/tests and requirements. Trying to build a parallel annotation-matching system would duplicate logic and create maintenance burden.

## Common Pitfalls

### Pitfall 1: Forgetting Bidirectional Edge Semantics
**What goes wrong:** Parser adds `declares` edge but not `declared_in` reverse edge; queries miss suppression because they look for `declared_in`.

**Why it happens:** Proximity graph requires bidirectional edges for reachability; formal-proximity.cjs line 94-126 defines REVERSE_RELS mapping. Missing reversal breaks BFS queries in both directions.

**How to avoid:** When adding a declares edge, immediately add reverse declared_in edge in the same operation. Test both directions: `reach(pi, 'code_file::...', 2, ['requirement'])` and `reach(pi, 'requirement::...', 2, ['code_file'])`.

**Warning signs:** T->R scanner still flags files with annotations; "nearest_req" lookup succeeds but suppression doesn't fire.

### Pitfall 2: Overriding Direct Annotations with Proximity Suggestions
**What goes wrong:** If file already has @requirement annotation, annotate-tests.cjs overwrites it with proximity-based suggestion (often weaker).

**Why it happens:** Annotation tool doesn't check for existing declarations before proposing.

**How to avoid:** In annotate-tests.cjs, check file content for existing @requirement patterns first. Only suggest if no annotation found. Preserve existing annotations verbatim.

**Warning signs:** Developers report losing annotations they manually added; version control shows unexpected diffs in test files.

### Pitfall 3: Strict Threshold Blocking Legitimate Edges
**What goes wrong:** Using threshold >= 0.6 blocks some valid requirement links (e.g., 0.59 score edges that are semantically clear).

**Why it happens:** Threshold is arbitrary; different artifact combinations produce different scores.

**How to avoid:** Start with 0.6 from success criteria; monitor FP/FN rate in early runs. Plan for Phase v0.37-03 (Scanner FP Tuning) to auto-adjust thresholds per scanner. Add --report-only output showing what was suppressed and why.

**Warning signs:** Legitimate requirements still flagged as orphaned; proximity embedding shows high relevance but score slightly below 0.6.

### Pitfall 4: Missing Node Keys in Proximity Index
**What goes wrong:** Parser creates edges to nodes that don't exist in pi.nodes; reach() fails silently or creates orphan edge entries.

**Why it happens:** Nodes must be explicitly registered in proximity index; parser might reference unknown file paths or requirement IDs.

**How to avoid:** Validate before adding edge: check pi.nodes[targetKey] exists. If not, log warning and skip edge. In formal-proximity.cjs, ensure artifact scanner registers nodes before creating edges.

**Warning signs:** Empty nearest_req in scanner output; console warnings about undefined node lookups.

### Pitfall 5: Comment Format Inconsistency
**What goes wrong:** Some files use `// @requirement`, others use `// Requirements:`, others use `/* @requirement */`; scanner misses some due to pattern mismatch.

**Why it happens:** JavaScript has multiple comment styles; no standard enforced.

**How to avoid:** Formalize in DESIGN doc: test files use `// @requirement ID` (single line), source files use `// Requirements: ID1, ID2` (header comment block). Enforce via pre-commit linter. Scanner accepts both patterns with OR logic.

**Warning signs:** Some tests flagged as orphaned even after manual annotation; inconsistent pattern use across repo.

## Code Examples

Verified patterns from official project sources:

### Parsing @requirement from Source Files
```javascript
// Source: extract-annotations.cjs lines 228-260 (adapted for JS)
function parseSourceAnnotations(content) {
  const lines = content.split('\n');
  const results = [];
  let pendingReqs = [];

  for (let i = 0; i < Math.min(30, lines.length); i++) {
    const line = lines[i];

    // Match annotation: // @requirement REQ-ID or // Requirements: REQ-ID, REQ-02
    const annMatch = line.match(/\/\/\s*@requirement\s+([\w-]+)/);
    const headerMatch = line.match(/\/\/\s*Requirements:\s*(.+)/);

    if (annMatch) {
      pendingReqs.push(annMatch[1]);
    } else if (headerMatch) {
      const ids = headerMatch[1].split(/[,\s]+/).map(s => s.trim()).filter(Boolean);
      pendingReqs.push(...ids);
    }
  }

  return pendingReqs;
}
```

### Adding Declares Edge to Proximity Graph
```javascript
// Source: formal-proximity.cjs lines 155-185 (nodeBuilder pattern)
// After parsing file annotations:
const fileName = 'bin/nf-solve.cjs';
const requirements = parseSourceAnnotations(fileContent);

for (const reqId of requirements) {
  const sourceKey = 'code_file::' + fileName;
  const targetKey = 'requirement::' + reqId;

  // Create bidirectional edges
  edges.push({
    source: sourceKey,
    target: targetKey,
    type: 'declares',
    weight: SEMANTIC_WEIGHTS['declares'] // 0.7
  });
  edges.push({
    source: targetKey,
    target: sourceKey,
    type: 'declared_in',
    weight: SEMANTIC_WEIGHTS['declared_in'] // 0.7
  });
}
```

### Suppressing Orphan Based on Proximity Score
```javascript
// Source: nf-solve.cjs lines 1958-1980 (sweepCtoR pattern)
const SUPPRESS_THRESHOLD = 0.6;
const untraced = [{ file: 'bin/unused.cjs' }];

try {
  const piPath = path.join(ROOT, '.planning', 'formal', 'proximity-index.json');
  const pi = JSON.parse(fs.readFileSync(piPath, 'utf8'));
  const { reach } = require('./formal-query.cjs');

  for (const item of untraced) {
    const nodeKey = 'code_file::' + item.file;
    const edges = pi.edges[nodeKey] || [];

    // Check for high-score declares edge
    let shouldSuppress = false;
    for (const edge of edges) {
      if (edge.target.startsWith('requirement::')) {
        const weight = SEMANTIC_WEIGHTS[edge.type];
        if (weight >= SUPPRESS_THRESHOLD) {
          shouldSuppress = true;
          item.suppressed_by = edge.target + ' (' + weight + ')';
          break;
        }
      }
    }

    if (!shouldSuppress) {
      // Keep in residual
    }
  }
} catch (e) {
  // fail-open: keep original orphans
}
```

// Suggested pattern based on extract-annotations.cjs + proximity usage
const suggestions = [];
for (const orphanTest of orphanTests) {
  const testKey = 'code_file::' + orphanTest.file;
  const reachable = reach(pi, testKey, 3, ['requirement']);

  // Find best match by score
  let best = null;
  let bestScore = 0;
  for (const nodes of Object.values(reachable)) {
    for (const node of nodes) {
      if (node.score > bestScore) {
        bestScore = node.score;
        best = node.key.split('::')[1];

    suggestions.push({
      file: orphanTest.file,
      suggested_requirement: best,
      confidence: bestScore,
      recommendation: `// @requirement ${best}`
    });
  }
}

## Open Questions

1. **Should annotations support multiple requirement IDs per file?**
   - What we know: extract-annotations.cjs supports multiple IDs per entity (line 54-57: `requirement_ids: [...]`)
   - What's unclear: Whether test files should map to multiple requirements (test×req matrix)
   - Recommendation: Support multiple IDs per line (`// @requirement REQ-01, REQ-02`) but start with single-ID pattern. Phase 03 (FP tuning) can revisit if many tests map to many requirements.

2. **Should annotation suggestions in annotate-tests.cjs be interactive or batch-write?**
   - What we know: extract-annotations.cjs is batch/reporting only
   - What's unclear: Should annotate-tests.cjs prompt user for each suggestion or write all at once
   - Recommendation: Start with batch suggestion (JSON output to stdout + --write flag for auto-inject). Interactive mode can be added later if UX demand justifies.

3. **How to handle test files that legitimately belong to no requirement?**
   - What we know: Current logic flags all orphan tests; Phase 03 will suppress by FP rate
   - What's unclear: Should some tests be marked "no requirement" (e.g., regression tests, integration framework tests)
   - Recommendation: Add optional `// @no-requirement` comment pattern that explicitly marks tests as infrastructure. Scanners treat this as mapped (not orphan). Document in DESIGN.md.
