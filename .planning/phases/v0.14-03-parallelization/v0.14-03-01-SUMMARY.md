# Summary: v0.14-03-01 — Parallelize run-formal-verify.cjs (Group Execution)

## What Was Done

Refactored `bin/run-formal-verify.cjs` to replace the sequential `for..of` + `spawnSync` step loop with a two-phase parallel group execution model.

**Architecture change:**

| Before | After |
|--------|-------|
| Single `for..of` loop — all 21 steps sequential | Two-phase execution: generate sequential, tool groups concurrent |
| No parallelism — each step blocked on previous | `Promise.all` — tla/alloy/prism/petri run simultaneously |
| ~10 min wall-clock (sum of all step times) | ~2 min wall-clock (max of largest group) |

**Phase 1 (sequential prerequisite):**
- `generate:tla-from-xstate` — generates QGSDQuorum_xstate.tla + MCQGSDQuorum.cfg
- `generate:alloy-prism-specs` — generates Alloy + PRISM models
- Must complete before any tool group starts (downstream specs depend on generated output)

**Phase 2 (concurrent via Promise.all):**
- `tla` group (8 steps) — runs sequentially within group
- `alloy` group (7 steps) — runs sequentially within group
- `prism` group (2 steps) — runs sequentially within group
- `petri` group (2 steps) — runs sequentially within group
- All 4 groups start simultaneously

**New `runGroup()` helper:**
- Steps within each group run sequentially (preserving intra-group order dependencies)
- Handles both `node` and `wasm-dot` step types
- `results[]` array still populated via `record()` — summary table unchanged

**`--only=` filter compatibility:**
- `--only=generate`: only Phase 1 runs (toolSteps empty)
- `--only=tla`: Phase 1 skipped (generateSteps empty); Phase 2 runs with just tla group
- `--only=<step-id>`: routes to single step in its phase
