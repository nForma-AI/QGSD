# Phase v0.33-03: Gate Stabilization - Research

**Researched:** 2026-03-10
**Domain:** Gate promotion stability, flip-flop detection, cooldown enforcement, per-model gate persistence
**Confidence:** HIGH

## Summary

This phase adds stability enforcement to the gate promotion pipeline. Currently, `compute-per-model-gates.cjs` promotes and demotes models freely based on per-run evidence scores, with no memory of prior promotion history. The promotion-changelog.json records all transitions but nothing reads it to detect instability patterns. This phase adds three capabilities: (1) scanning the changelog for flip-flop patterns and flagging unstable models, (2) enforcing a cooldown window before re-promotion, and (3) making `--write-per-model` the default in every nf-solve run.

The codebase is 100% Node.js CommonJS (`.cjs`). All gate data lives in `.planning/formal/gates/`. The promotion changelog at `.planning/formal/promotion-changelog.json` currently has 38 entries (19 per model, all ADVISORY->SOFT_GATE for 2 models). While no actual flip-flops exist yet in the data, the promotion/demotion code paths in `compute-per-model-gates.cjs` (lines 446-505) make them structurally possible -- a model promoted to SOFT_GATE could be demoted back to ADVISORY if evidence score drops, then re-promoted next run. The hysteresis thresholds (promote at >=1, demote at <0.8) reduce but do not eliminate oscillation risk.

The INTG-01 requirement is straightforward: `nf-solve.cjs` currently calls `compute-per-model-gates.cjs --aggregate --json` in two places (`getAggregateGates()` at line 2131 and `sweepPerModelGates()` at line 2269) but never passes `--write-per-model`. Adding this flag to both call sites ensures per-model gate reasons persist to `per-model-gates.json` every run.

**Primary recommendation:** Implement as 3 focused tasks: (1) flip-flop detection module that scans promotion-changelog.json for direction alternations, (2) cooldown enforcement integrated into the promotion decision in compute-per-model-gates.cjs, (3) INTG-01 flag addition in nf-solve.cjs.

### Pattern 1: Flip-Flop Detection (STAB-01)
**What:** Scan promotion-changelog.json for direction alternations per model. A "direction change" is when consecutive transitions for the same model go in opposite directions (e.g., ADVISORY->SOFT_GATE then SOFT_GATE->ADVISORY).
**When to use:** Every time per-model-gates.json is written.
**Algorithm:**
1. Group changelog entries by model path
2. For each model, walk entries chronologically
3. Track direction: promotion (level goes up) vs demotion (level goes down)
4. Count direction changes (transitions where direction flips from the previous transition)
5. If direction_changes >= 3, flag model as UNSTABLE

**Example:**
```javascript
// Source: Project codebase analysis
const LEVEL_ORDER = ['ADVISORY', 'SOFT_GATE', 'HARD_GATE'];

function countDirectionChanges(entries) {
  if (entries.length < 2) return 0;
  let changes = 0;
  let lastDirection = null; // 'up' or 'down'
  for (let i = 0; i < entries.length; i++) {
    const fromIdx = LEVEL_ORDER.indexOf(entries[i].from_level);
    const toIdx = LEVEL_ORDER.indexOf(entries[i].to_level);
    if (fromIdx < 0 || toIdx < 0 || fromIdx === toIdx) continue;
    const direction = toIdx > fromIdx ? 'up' : 'down';
    if (lastDirection !== null && direction !== lastDirection) {
      changes++;
    }
    lastDirection = direction;
  }
  return changes;
}

function detectFlipFlops(changelog, threshold) {
  threshold = threshold || 3;
  const byModel = {};
  for (const entry of changelog) {
    if (!byModel[entry.model]) byModel[entry.model] = [];
    byModel[entry.model].push(entry);
  }
  const unstable = {};
  for (const [model, entries] of Object.entries(byModel)) {
    const changes = countDirectionChanges(entries);
    if (changes >= threshold) {
      unstable[model] = { direction_changes: changes, flagged_at: new Date().toISOString() };
    }
  }
  return unstable;
}
```

### Pattern 2: Cooldown Enforcement (STAB-02)
**What:** Block re-promotion of UNSTABLE-flagged models until both conditions are met: (a) 3 consecutive solve sessions with sustained gate score, and (b) >= 1 hour wall time since flagging.
**When to use:** In the auto-promotion decision path of compute-per-model-gates.cjs (lines 446-488).
**State tracking:** Store cooldown state in per-model-gates.json alongside the model entry:
```json
{
  "stability_status": "UNSTABLE",
  "flagged_at": "2026-03-10T12:00:00Z",
  "cooldown": {
    "consecutive_stable_sessions": 0,
    "required_sessions": 3,
    "required_wall_time_ms": 3600000,
    "last_session_timestamp": null
  }
}
```

**Example:**
```javascript
// Source: Project codebase analysis
function isCooldownSatisfied(modelStability) {
  if (!modelStability || modelStability.stability_status !== 'UNSTABLE') return true;
  const cooldown = modelStability.cooldown;
  if (!cooldown) return true;

  const sessionsMet = cooldown.consecutive_stable_sessions >= cooldown.required_sessions;
  const wallTimeMet = (Date.now() - new Date(modelStability.flagged_at).getTime()) >= cooldown.required_wall_time_ms;

  return sessionsMet && wallTimeMet;
}
```

### Anti-Patterns to Avoid
- **Modifying promotion-changelog.json format:** The changelog is append-only with a 200-entry cap. STAB-01 reads it but must never modify its schema. The STAB-03 dedup guard was already added in Phase v0.33-01.
- **Blocking promotion without logging:** Every blocked promotion must produce a stderr warning with `[compute-per-model-gates]` prefix (existing TAG convention) so operators can diagnose.
- **Storing cooldown state outside per-model-gates.json:** STATE.md decision says "Outer-loop state lives in separate files... never modifies solve-state.json or promotion-changelog.json format." Per-model-gates.json is the correct location since it already contains per-model state.

## Common Pitfalls

### Pitfall 1: Counting Duplicate Entries as Alternations
**What goes wrong:** The promotion-changelog.json has many duplicate entries (same model, same transition repeated). If you count each entry as a transition, you might miscount direction changes.
**Why it happens:** Before STAB-03 dedup guard was added, every solve run could append a duplicate promotion entry.
**How to avoid:** When counting direction changes, only count entries where the direction actually changes. Same-direction consecutive entries are not alternations. The dedup guard from Phase v0.33-01 prevents future duplicates.
**Warning signs:** Test fixtures that don't include duplicate entries may mask this.

### Pitfall 2: Race Between Flip-Flop Detection and Promotion Decision
**What goes wrong:** If flip-flop detection runs after the promotion decision, a model could be promoted and then immediately flagged as UNSTABLE in the same run.
**Why it happens:** Ordering matters -- detection must precede the promotion gate check.
**How to avoid:** Load and evaluate flip-flop status BEFORE the promotion loop. The flip-flop status from the previous run's per-model-gates.json plus fresh changelog scan determines whether promotion is blocked.
**Warning signs:** A model showing both `promoted: true` and `stability_status: "UNSTABLE"` in the same output.

### Pitfall 3: Cooldown Session Counter Reset
**What goes wrong:** The consecutive_stable_sessions counter must reset to 0 if the model's gate score drops during the cooldown window. Failing to reset means the cooldown silently passes.
**Why it happens:** Only incrementing without checking current gate scores.
**How to avoid:** Each solve session checks: does this model still pass the gate it was last promoted to? If yes, increment. If no, reset to 0.
**Warning signs:** Models getting re-promoted despite ongoing instability.

### Pitfall 4: --write-per-model Doubling File Writes
**What goes wrong:** Both `getAggregateGates()` and `sweepPerModelGates()` call compute-per-model-gates.cjs. If both run in the same solve session, per-model-gates.json gets written twice.
**Why it happens:** The memoization cache (`_aggregateCache`) only caches the aggregate data, not the per-model write.
**How to avoid:** Either: (a) only add `--write-per-model` to `sweepPerModelGates()` (which runs second and has the final data), or (b) add `--write-per-model` to both but accept the double-write as harmless (second write overwrites first with identical data). Option (a) is cleaner.
**Warning signs:** Two `per-model-gates.json` writes visible in stderr output.

## Open Questions

1. **Should cooldown state persist across process restarts?**
   - What we know: per-model-gates.json is written to disk, so cooldown state survives restarts. The consecutive_stable_sessions counter needs to be read from the previous run's per-model-gates.json.
   - What's unclear: If per-model-gates.json is deleted or regenerated from scratch, cooldown state is lost. Is this acceptable?
   - Recommendation: Accept it -- regenerating from scratch implies a clean-slate intent. Document this in code comments.

2. **What constitutes a "solve session" for cooldown counting?**
   - What we know: Each nf-solve invocation is one session. solve-trend.jsonl (from Phase v0.33-01) tracks session timestamps.
   - What's unclear: Does a fast-mode run (which skips gate computation) count as a session?
   - Recommendation: Only count sessions where gate computation actually ran (i.e., non-fast-mode, non-skipped). This avoids inflating the counter with no-op runs.
