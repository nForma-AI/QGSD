## Summary

Phase v0.20-03 adds a lightweight formal verification gate to the planning workflow. After the planner creates PLAN.md files (step 8 of plan-phase.md) and before quorum review (step 8.5), a new step 8.3 runs `node bin/run-formal-verify.cjs --only=tla` and reads `formal/check-results.ndjson`. ### Recommended Implementation: Step 8.3 in plan-phase.md

Insert immediately **after step 8** (planner completes) and **before step 8.4** (envelope update):

```markdown
## 8.3. Formal Verification Gate (PLAN-01, PLAN-02, PLAN-03)

Run TLA+ formal verification before quorum. Gate is fail-open per PLAN-03.

\`\`\`bash
node /Users/jonathanborduas/.claude/qgsd/bin/gsd-tools.cjs activity-set \
  "{\"activity\":\"plan_phase\",\"sub_activity\":\"fv_gate\",\"phase\":${PHASE_NUMBER}}"
\`\`\`

\`\`\`bash
# Run TLA+ verification (fail-open — errors must not block planning)
FV_HYPOTHESES=""
FV_STATUS="skipped"

FV_EXIT=0
node bin/run-formal-verify.cjs --only=tla 2>/dev/null || FV_EXIT=$?

# Read NDJSON output regardless of exit code (fail-open)
if [ -f "formal/check-results.ndjson" ]; then
  # Extract fail entries only
  FV_FAILS=$(node -e "
    const fs = require('fs');
    const lines = fs.readFileSync('formal/check-results.ndjson','utf8').split('\n').filter(l=>l.trim());
    const fails = lines.map(l => { try { return JSON.parse(l); } catch(e) { return null; } })
      .filter(r => r && r.result === 'fail');
    if (fails.length > 0) {
      console.log(fails.map(r =>
        '- **' + r.check_id + '** (' + r.surface + '): ' + (r.summary || r.property)
      ).join('\n'));
    }
  " 2>/dev/null || echo "")

  if [ -n "$FV_FAILS" ]; then
    FV_STATUS="fail"
    FV_HYPOTHESES="$FV_FAILS"
  else
    FV_STATUS="pass"
  fi
fi
\`\`\`

**Route on FV_STATUS:**

- **`pass` or `skipped`:** Log `FV gate: no TLC failures — quorum proceeds normally`. Do NOT add FV context to quorum step.
- **`fail`:** Log `FV gate: ${N} TLC failure(s) — surfacing as hypotheses`. Set $FV_HYPOTHESES for use in step 8.5. When dispatching quorum (step 8.5), inject into review_context:

\`\`\`
**FV Gate results — TLC failures detected (treat as hypotheses, not blockers):**
${FV_HYPOTHESES}

These properties are currently failing TLA+ model checks. The planner should acknowledge whether the plans address them or accept them as known issues.
\`\`\`

If FV runner is unavailable (non-zero exit with no NDJSON output): Log `FV gate: runner unavailable — proceeding without FV context (PLAN-03 fail-open)`. Set FV_STATUS="skipped".
```

## Common Pitfalls

### Pitfall 1: Exit Code Blocking
**What goes wrong:** Developer treats non-zero exit from run-formal-verify.cjs as a fatal error and prevents quorum from running.
**Why it happens:** Natural instinct to check exit codes.
**How to avoid:** Always use `|| FV_EXIT=$?` pattern, never `set -e` or `&&` chaining. Non-zero exit means SOME steps failed, but NDJSON may still have useful data.
**Warning signs:** Any `process.exit(1)` or bash `exit 1` in the FV gate step.

### Pitfall 2: Reading Stale NDJSON
**What goes wrong:** Reading check-results.ndjson before run-formal-verify.cjs truncates and rewrites it.
**Why it happens:** run-formal-verify.cjs truncates the file at the TOP of runOnce() — before any steps run. But the gate runs AFTER planner completes (step 8), so the file is from the PREVIOUS run.
**How to avoid:** The gate RUNS the TLA+ steps first, then reads. The file is always fresh after the gate's `node bin/run-formal-verify.cjs --only=tla` call completes.
**Verification:** Check that NDJSON timestamps are recent (within last 5 minutes) relative to gate execution.

### Pitfall 3: Surfacing Inconclusive as Failures
**What goes wrong:** Developer surfaces `result=inconclusive` entries as hypotheses, creating false urgency.
**Why it happens:** Inconclusive looks like a problem.
**How to avoid:** Only filter for `result === 'fail'`. Inconclusive (e.g., liveness properties without fairness) is expected and normal.

### Pitfall 4: FV_HYPOTHESES in quorum_context when empty
**What goes wrong:** Appending empty FV_HYPOTHESES to every quorum call, adding noise.
**Why it happens:** Forgetting the conditional.
**How to avoid:** Only inject FV context when `FV_STATUS === "fail"`. The condition `if [ -n "$FV_HYPOTHESES" ]` gate.
