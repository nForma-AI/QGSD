# Phase v0.38-02: Debug Command Integration - Research

**Researched:** 2026-03-18
**Domain:** Debug command formal model consultation, quorum worker constraint injection, verdict reporting
**Confidence:** HIGH

## Summary

This phase integrates Phase v0.38-01's model discovery and constraint extraction infrastructure directly into the `/nf:debug` command. The existing debug command already has a complete quorum dispatch loop (Step A–G) with consensus voting across 4 agents and artifact persistence. Phase v0.38-02 inserts a new **Step A.5: Formal Model Consultation** that:

1. Calls `formal-scope-scan.cjs --bug-mode --run-checkers` to find matching models and verify if they reproduce the bug
2. Calls `model-constrained-fix.cjs` to extract fix constraints from reproducing models
3. Injects both the model verdict (reproduced/not-reproduced/no-model) and extracted constraints into the quorum worker prompts
4. Reports the FORMAL verdict in the NEXT STEP result table alongside each worker's recommendation
5. Tracks bugs with no model coverage in `bug-model-gaps.json` for future model refinement (Phase v0.38-03)

The implementation reuses all existing infrastructure:
- Quorum slot dispatch system (`quorum-slot-dispatch.cjs`, `call-quorum-slot.cjs`)
- Formal model registry, proximity index, and model checkers (TLC/Alloy)
- Existing debug artifact schema (`.planning/quick/quorum-debug-latest.md`)

**Primary recommendation:** Add Step A.5 as a subprocess call within the debug orchestrator's Step A (after test collection, before quorum dispatch). Build constraint-injected worker prompts that append `[FORMAL CONSTRAINTS: ...]` and `[MODEL VERDICT: ...]` blocks. Use the existing quorum consensus table format to render the FORMAL row. Validation architecture uses node:test for unit tests of prompt injection logic and integration tests against real models; Wave 0 scaffolding includes test stubs for constraint injection accuracy and verdict classification.

### Pattern 1: Step A.5 Formal Model Consultation Subprocess

**What:** After Step A collects failure context and assembles the bundle, but before Step C dispatches quorum workers, run formal model lookup and verification.

**When to use:** Always in `/nf:debug` when failure context is available and formal models exist.

**Example (pseudocode from debug.md Step A.5):**

```javascript
// Step A.5: Formal Model Consultation (new)
const { spawnSync } = require('child_process');
const formalResult = spawnSync(process.execPath, [
  path.join(__dirname, 'bin', 'formal-scope-scan.cjs'),
  '--bug-mode',
  '--run-checkers',
  '--description', failureDescription,
  '--format', 'json'
], {
  encoding: 'utf8',
  timeout: 30000,  // Max 30s for formal verification
  stdio: ['pipe', 'pipe', 'pipe']
});

let formalContext = { verdict: 'no-model', constraints: [], models: [] };
if (!formalResult.error && formalResult.status === 0) {
  try {
    const result = JSON.parse(formalResult.stdout);
    formalContext = {
      verdict: result.some(m => m.reproduced) ? 'reproduced' :
               result.length > 0 ? 'not-reproduced' : 'no-model',
      constraints: extractConstraints(result),
      models: result
    };
  } catch (e) {
    // Fail-open: parse error returns empty context
  }
}

// $FORMAL_CONTEXT now contains { verdict, constraints, models }
// Append to $BUNDLE for Step B assembly
```

**Key safety:** Fail-open on any error. If formal tools error, missing, or timeout (>30s), return empty context and proceed with debug loop. Never let formal verification block debugging.

### Pattern 2: Constraint Injection into Quorum Prompts

**What:** Extend Mode A and Mode B prompt templates in `quorum-slot-dispatch.cjs` to include a `[FORMAL CONSTRAINTS]` block.

**When to use:** When $FORMAL_CONTEXT.constraints is non-empty.

**Example template injection (Mode A):**

```markdown
You are a debugging advisor for the nForma project.

<bundle>
$BUNDLE
</bundle>

[FORMAL CONSTRAINTS]
The following constraints were extracted from formal models that might explain this failure:
- $CONSTRAINT_1
- $CONSTRAINT_2

These are safety properties (must always hold) or liveness properties (must eventually hold).
Do NOT propose fixes that violate these constraints.
[END FORMAL CONSTRAINTS]

Given this failure, answer ONLY:
root_cause: <the single most likely root cause in one sentence>
next_step: <the single best next debugging action>
confidence: HIGH | MEDIUM | LOW

Rules:
- Do NOT suggest a fix. Suggest the next investigative step only.
- Respect the formal constraints above — they represent verified properties of the system.
```

**Integration point:** In `buildModeAPrompt()` and `buildModeBPrompt()` functions in `quorum-slot-dispatch.cjs`, insert constraint block between bundle and question if constraints exist.

**When to use:** After quorum dispatch completes (Step D).

### Pattern 4: bug-model-gaps.json Persistence (DBG-04)

**What:** If formal-scope-scan returns no matching models (verdict: "no-model"), Step G appends an entry to bug-model-gaps.json for future model creation (Phase v0.38-03).

**When to use:** At end of debug loop (Step G), only if verdict was "no-model".

**Example schema:**

```json
{
  "gaps": [
    {
      "bug_id": "a1b2c3d4",
      "description": "Circuit breaker timeout failure in quorum deliberation",
      "failure_context": "$ARGUMENTS string",
      "timestamp": "2026-03-18T12:34:56Z",
      "status": "no_coverage",
      "worker_consensus_root_cause": "Guard condition on Deliberate action not enforced",
      "worker_consensus_next_step": "Add Alloy spec to capture state transition invariant"
    }
  ]
}
```

**Integration point:** In Step G (Execute or escalate), after handling consensus step, check `if (formalVerdict === 'no-model')` and append entry with deterministic bug_id (SHA-256 hash of description, 8 chars, same as Phase v0.38-01 BML-01).

### Anti-Patterns to Avoid

- **Blocking on formal verification:** If formal tools timeout or fail, proceed with debug loop without formal context. Never wait >30s for model checking.
- **Over-constraining worker reasoning:** Constraints should guide, not dictate. Workers should still suggest investigation steps; formal context is evidence, not gospel.
- **Mixing formal verdicts with worker consensus:** FORMAL row is separate from CONSENSUS row. They inform each other but don't override each other.
- **Not persisting gaps:** Bugs with no model coverage MUST be tracked (DBG-04) so Phase v0.38-03 can create models for them.

## Common Pitfalls

### Pitfall 1: Subprocess Timeout Blocks Debug Loop

**What goes wrong:** formal-scope-scan with --run-checkers can timeout waiting for TLC/Alloy (60s per model). If Step A.5 doesn't have its own timeout, a long-running model checker blocks the entire debug loop.

**Why it happens:** Model checkers can explore large state spaces. No built-in timeout in subprocess call means the orchestrator waits indefinitely.

**How to avoid:** Wrap formal-scope-scan subprocess call with a 30-second maximum timeout. On timeout, return empty formal context and continue (fail-open). Model verification is nice-to-have evidence, not required for debug consensus.

**Warning signs:** Debug loop stalls for >30 seconds before showing quorum table. Check stderr for "TLC running..." or "Alloy solver..." still active.

**Code pattern:**
```javascript
const formalPromise = new Promise((resolve) => {
  // Set hard 30s timeout
  const timer = setTimeout(() => {
    resolve({ verdict: 'no-model', constraints: [] }); // fail-open
  }, 30000);

  const result = spawnSync(process.execPath, [
    path.join(__dirname, 'bin', 'formal-scope-scan.cjs'),
    '--bug-mode', '--run-checkers',
    '--description', desc
  ], { encoding: 'utf8', timeout: 30000 });

  clearTimeout(timer);
  // ... parse result
  resolve(...);
});
```

### Pitfall 2: Constraint Block Confuses Workers with Too Much Text

**What goes wrong:** Injecting 10+ constraints into the prompt causes workers to focus on constraint compliance instead of root cause diagnosis.

**Why it happens:** Workers have bounded attention. Each constraint line consumes cognitive space.

**How to avoid:** Limit constraint injection to top 3 constraints (by confidence score from Phase v0.38-01 CEX-03). Use the rule: `if (constraints.length > 3) constraints = constraints.slice(0, 3)`.

**Warning signs:** Workers' next_step suggestions focus on "verify invariant X" instead of "check file Y for root cause". Review the constraint block size in worker prompts.

**Code pattern:**
```javascript
const topConstraints = constraints
  .sort((a, b) => (b.confidence || 0) - (a.confidence || 0))
  .slice(0, 3); // Max 3 constraints to avoid cognitive overload
```

### Pitfall 3: FORMAL Verdict Contradicts All Worker Consensus

**What goes wrong:** Model says "reproduced" (bug is valid), but all 4 workers say "user error" (bug is invalid). Confusing output.

**Why it happens:** Formal models capture low-level state invariants; workers reason about high-level architecture and symptom interpretation. Different perspectives.

**How to avoid:** Render FORMAL row as _additional evidence_, not override. In Step E commentary, note when FORMAL and CONSENSUS diverge: "**Note:** Model reproduction (FORMAL: reproduced) suggests state invariant violation, but workers attribute to user configuration. Consider: is invariant too strict, or is user config incorrect?".

**Warning signs:** FORMAL row shows "reproduced" but CONSENSUS is "no-root-cause-found" or "user-error". Not necessarily wrong, but flag for user review.

### Pitfall 4: Failing to Persist no_coverage Gaps

**What goes wrong:** Bug has no model coverage, debug reports "FORMAL: no-model", but nothing gets written to bug-model-gaps.json. Phase v0.38-03 never learns about this gap.

**Why it happens:** Step G conditional (`if verdict === 'no-model'`) is easy to forget when integrating. Code might log to stderr instead of persisting to file.

**How to avoid:** Make bug-model-gaps.json write mandatory in Step G. Add explicit check: `if (formalVerdict === 'no-model') { persistGap(...); }`. Test this explicitly (Wave 0 unit test).

**Warning signs:** No entries appear in bug-model-gaps.json after several /nf:debug calls with novel bugs. Check Step G code for missing file write.

### Example 3: FORMAL Row Rendering (following existing table pattern)

```javascript
// Source: pattern from debug.md Step E (NEXT STEP table rendering)
function renderFormalRow(verdict, modelName, constraints) {
  const confidenceMap = {
    'reproduced': 'HIGH',
    'not-reproduced': 'MEDIUM',
    'no-model': 'N/A'
  };

  let stepText = '';
  if (verdict === 'reproduced') {
    stepText = 'Model ' + modelName + ' reproduced the bug. ' +
               (constraints.length > 0 ? 'Check constraint: ' + constraints[0].text : '');
  } else if (verdict === 'not-reproduced') {
    stepText = 'Models exist but did not reproduce. Bug may be environmental.';
  } else {
    stepText = 'No formal model covers this failure.';
  }

  return {
    model: 'FORMAL',
    confidence: confidenceMap[verdict],
    nextStep: stepText
  };
}
```

### Example 4: bug-model-gaps.json Persistence (DBG-04 Step G)

```javascript
// Source: pattern adapted from Phase v0.38-01 persistent storage
const crypto = require('crypto');
const fs = require('fs');
const path = require('path');

function persistBugGap(description, failureContext, workerConsensus, verdict) {
  const bugId = crypto.createHash('sha256')
    .update(description)
    .digest('hex')
    .slice(0, 8);

  const ROOT = process.cwd();
  const gapsPath = path.join(ROOT, '.planning', 'formal', 'bug-model-gaps.json');
  let gaps = { gaps: [] };

  try {
    if (fs.existsSync(gapsPath)) {
      gaps = JSON.parse(fs.readFileSync(gapsPath, 'utf8'));
    }
  } catch (e) {
    // Fail-open: start fresh if file corrupted
  }

  const entry = {
    bug_id: bugId,
    description: description,
    failure_context: failureContext.slice(0, 500), // Truncate to 500 chars
    timestamp: new Date().toISOString(),
    status: 'no_coverage',
    worker_consensus_root_cause: workerConsensus.rootCause,
    worker_consensus_next_step: workerConsensus.nextStep
  };

  gaps.gaps.push(entry);
  fs.writeFileSync(gapsPath, JSON.stringify(gaps, null, 2), 'utf8');
}
```

## Open Questions

1. **Should Step A.5 run checkers for all matching models or only top N?**
   - What we know: formal-scope-scan.cjs already supports `--run-checkers` with max 3 models. Matching can return more than 3.
   - What's unclear: Should we prioritize by relevance score and run top 3? Or run all matches regardless of count?
   - Recommendation: Run top 3 by relevance (existing default behavior). Model checking is expensive; 3 models with high relevance is better than 10 models with lower relevance. Revisit if coverage is too narrow.

2. **How should constraint confidence be scored when multiple models provide constraints?**
   - What we know: Phase v0.38-01 CEX-03 renders constraints as plain English with implicit confidence (INVARIANT vs PROPERTY classification).
   - What's unclear: If Model A says "constraint X is critical" and Model B says "constraint X is optional", how do we weight?
   - Recommendation: Use consensus: if 2+ models produce the same constraint, score as HIGH. If only 1 model produces it, score as MEDIUM. Phase v0.38-01 can optionally add a `confidence` field to constraint output; implement in v0.38-03 if needed.

3. **Should FORMAL row always appear in NEXT STEP table, even if verdict is no-model?**
   - What we know: User should always know if formal models were consulted, even if none matched.
   - What's unclear: Including a no-model FORMAL row adds table rows; is this noise?
   - Recommendation: Always include FORMAL row (even if "no-model" verdict). It signals "we looked, found nothing" vs "we didn't check". Helps users understand debug confidence level.

4. **How should workers be told NOT to violate constraints without sounding preachy?**
   - What we know: Injecting "[FORMAL CONSTRAINTS]...[END FORMAL CONSTRAINTS]" is clear but lengthy.
   - What's unclear: Can we make it more concise while preserving clarity?
   - Recommendation: Keep current template for Phase v0.38-02. If workers consistently ignore constraints in later phases, refine template in v0.38-03 with more directive language ("You MUST respect these constraints...").
