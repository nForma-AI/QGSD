## Summary

Phase v0.27-03 builds cross-source deduplication and formal reference linking on top of the v0.27-01 foundation (debt schema, fingerprinting, ledger I/O). The work requires: (1) a Levenshtein similarity function for near-duplicate detection, (2) a dedup engine that chains fingerprint exact-match with Levenshtein fallback, (3) a merge function that preserves source_entries and resolves field conflicts, (4) formal reference auto-linking with layered resolution, and (5) schema extension to support the new `formal_ref_source` field.

## Technical Research

### 1. Levenshtein Distance Implementation

**Approach:** Inline implementation (no external dependency). The algorithm is simple (O(m*n) where m,n are string lengths) and debt titles are capped at 256 chars, making it fast enough.

**Wagner-Fischer algorithm** (dynamic programming, bottom-up):
- Space optimization: only need two rows (current and previous) -> O(min(m,n)) space
- For 256-char strings worst case: 256*256 = 65,536 operations — negligible

**Similarity score:** `1 - (levenshteinDistance(a, b) / max(a.length, b.length))`
- Returns 0.0 (completely different) to 1.0 (identical)
- Threshold check: `similarity >= 0.85` (per CONTEXT.md decision: inclusive, favor merge)

**Edge cases:**
- Empty strings: distance = length of other string, similarity = 0
- Identical strings: distance = 0, similarity = 1.0
- Case sensitivity: normalize to lowercase before comparison (titles may differ in casing across sources)

### 2. Dedup Engine Design

**Two-phase approach** (per CONTEXT.md specifics):

**Phase 1 — Fingerprint exact match (O(n)):**
- Build a Map<fingerprint, entry[]> from existing ledger entries
- For each new observation, compute fingerprint and lookup
- On match: merge into existing entry (occurrence++, last_seen update, source_entries concat)

**Phase 2 — Levenshtein near-duplicate detection (O(n^2) on remaining):**
- Only runs on entries NOT matched in Phase 1
- Compare new entry title against all existing entry titles
- If similarity >= threshold: merge as near-duplicate
- Must handle transitive merges carefully: if A merges with B, and B was already merged with C, use the merged result

**Merge candidate pairing:** For O(n^2) Levenshtein on remaining unmatched:
- With typical debt ledger sizes (10s to low 100s of entries), this is fast
- No need for optimization (e.g., length-based pre-filtering) at current scale

### 3. Merge Function Design

**Per CONTEXT.md decisions:**
- **Primary entry:** the one with higher `occurrences` count
- **Title:** primary's title preserved
- **Severity/fields:** `max(severity)` wins (pessimistic) — note: current schema doesn't have a severity field, so this applies if added later or via source_entries metadata
- **source_entries:** concatenate both arrays (no information loss)
- **formal_ref:** non-null value preserved. If both non-null, primary wins
- **formal_ref_source:** non-null value preserved, same precedence as formal_ref
- **environments:** union of both arrays (deduplicated)
- **occurrences:** sum of both
- **first_seen:** min of both
- **last_seen:** max of both
- **fingerprint:** primary's fingerprint kept (it has more observations)
- **id:** primary's id kept
- **status:** for merging entries in different states (e.g., acknowledged + open): keep the more advanced state (acknowledged > open). Per state machine: open < acknowledged < resolving < resolved

### 4. Formal Reference Linking

**Layered resolution** (per CONTEXT.md):

1. **Manual:** if `formal_ref` already set, skip — set `formal_ref_source: "manual"`
2. **Auto-detect:** keyword-match title against `.formal/requirements.json` (231 entries)
   - Tokenize debt title into keywords
   - Search requirement `text` and `id` fields for matches
   - If single match: set `formal_ref: "requirement:{id}"`, `formal_ref_source: "auto-detect"`
   - If multiple matches: use best match (most keyword overlap)
3. **Spec-inferred:** scan `.formal/spec/` module names (10 modules: account-manager, breaker, convergence, deliberation, mcp-calls, oscillation, prefilter, quorum, recruiting, tui-nav)
   - If any module name is substring of title (case-insensitive): set `formal_ref: "spec:{module}"`, `formal_ref_source: "spec-inferred"`
4. **Unmatched:** `formal_ref: null`, `formal_ref_source: null`

### 5. Schema Extension

**New field:** `formal_ref_source`
- Type: `["string", "null"]`
- Enum: `["manual", "auto-detect", "spec-inferred", null]`
- Must update: `debt.schema.json`, `validate-debt-entry.cjs` (allowedProps set + validation logic)
- Backward compatible: existing entries without this field should default to null

### 6. Integration with observe-debt-writer

The existing `writeObservationsToDebt` in `observe-debt-writer.cjs` does fingerprint-based upsert. Phase v0.27-03 needs to:
- Add Levenshtein fallback after fingerprint check (Phase 2 dedup)
- Add formal_ref resolution after upsert/creation
- Add merge reporting (counts for verbose output)

**Two approaches:**
1. **Extend observe-debt-writer.cjs** — add dedup + formal-ref logic inline
2. **Separate dedup module** — new `debt-dedup.cjs` called by observe-debt-writer after initial write

**Recommended:** Separate module (`bin/debt-dedup.cjs`) that operates on the full ledger. This keeps single-responsibility and allows dedup to be run independently (e.g., manual cleanup). The observe-debt-writer calls it after writing new entries.

Similarly, formal reference linking should be a separate module (`bin/formal-ref-linker.cjs`) called after dedup completes.

## Risk Analysis

| Risk | Severity | Mitigation |
|------|----------|------------|
| Levenshtein O(n^2) slow on large ledgers | Low | Debt ledgers expected <200 entries; 200^2 * 256 = trivial |
| Schema migration breaks existing debt.json | Medium | New field is optional (null default), additionalProperties update is backward compatible |
| Merge loses data | High | source_entries concatenation preserves all audit trail; test comprehensively |
| False positive merges at 0.85 threshold | Medium | Threshold configurable; favor merge per CONTEXT.md; verbose log shows detail |
| Transitive merge chains | Low | Process merges in single pass; merged entries removed from candidate pool |
