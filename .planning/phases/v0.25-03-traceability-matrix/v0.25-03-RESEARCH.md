# Phase v0.25-03: Traceability Matrix - Research

**Researched:** 2026-03-03
**Domain:** Automated traceability matrix generation linking requirements to formal properties
**Confidence:** HIGH

## Summary

Phase v0.25-03 builds `bin/generate-traceability-matrix.cjs` -- a generator that reads annotations (from `bin/extract-annotations.cjs`, built in v0.25-02) as the primary data source, falls back to `formal/model-registry.json` requirement arrays (from v0.25-01), and merges check results from `formal/check-results.ndjson` to produce `formal/traceability-matrix.json`. The matrix provides property-level links between requirements and formal properties with pass/fail/warn status, plus a `coverage_summary` section. The generator is then wired into `bin/run-formal-verify.cjs` as a post-verification step so the matrix always reflects the latest run.

**Primary recommendation:** The generator should use a three-layer data merge strategy: (1) `extract-annotations.cjs` output as primary source (23 files, 93 properties, 89 unique requirement IDs with property-level granularity), (2) `model-registry.json` as fallback for any models missing annotations, (3) `check-results.ndjson` for pass/fail/warn status per check. The matrix schema should be a JSON object with `metadata`, `requirements` (keyed by ID, each listing properties that verify it), `properties` (keyed by model+property, each listing requirement IDs), and `coverage_summary`.

```

                      │    ▲
                      │    │

  "generator_version": "1.0",

    "check_results": { "entry_count": 30 }

  "requirements": {
    "STOP-01": {
      "id": "STOP-01",
      "properties": [
        {
          "model_file": "formal/tla/QGSDStopHook.tla",
          "property_name": "TypeOK",
          "source": "annotation",
          "latest_result": "pass",
          "check_id": "tla:stop-hook-safety"
        }
      ]
    }
  },
  "properties": {
    "formal/tla/QGSDStopHook.tla::TypeOK": {
      "model_file": "formal/tla/QGSDStopHook.tla",
      "property_name": "TypeOK",
      "requirement_ids": ["STOP-01"],

      "latest_result": "pass",

    }

    "total_requirements": 210,
    "covered_count": 89,
    "coverage_percentage": 42.4,
    "uncovered_requirements": ["ACT-01", "ACT-02", "..."],
    "orphan_properties": []
  }
}
```

### Check Result Matching Strategy

The check-results.ndjson entries use `check_id` (e.g., `tla:quorum-safety`) and `requirement_ids` arrays, but do NOT have property-level granularity matching annotations. The matching strategy:

1. **Direct match by requirement_ids:** For each check result entry, match its `requirement_ids` to requirements in the matrix
2. **Surface/formalism match:** Group check results by formalism (tla, alloy, prism) and match to model files
3. **check_id parsing:** Extract model hints from check_id (e.g., `alloy:scoreboard-recompute` -> `formal/alloy/scoreboard-recompute.als`)
4. **Fallback:** If a check result has requirement_ids but no property-level match, attach it at the model level

### Integration with run-formal-verify.cjs

The traceability matrix generation step should be added to the STEPS array in run-formal-verify.cjs as a final step (after triage bundle generation):

```javascript
{
  tool: 'traceability', id: 'traceability:matrix',
  label: 'Generate traceability matrix (requirements <-> formal properties)',
  type: 'node', script: 'generate-traceability-matrix.cjs', args: [],
}
```

This ensures the matrix is generated after all checks complete and check-results.ndjson is fully populated. The step runs in the triage/post-processing phase.

### Fallback Logic (ANNOT-05)

```
for each model_file in model-registry.json:
  annotations = extract-annotations output for this file
  if annotations exist and have property-level data:
    use annotations (source = "annotation")
  else:
    use model-registry requirements array (source = "model-registry")
    log warning: "No annotations found for {file}, using model-registry fallback"
```

## Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| check-results.ndjson empty (no verification run yet) | Matrix generates without status data | Generate matrix with `latest_result: null` for all properties; coverage_summary still valid |
| Annotations and model-registry disagree on requirement IDs | Confusing matrix with inconsistent data | Annotations are authoritative (ANNOT-05); model-registry is fallback only; log discrepancies as warnings |
| Model files not in model-registry but have annotations | Properties discovered by annotations but no registry entry | Include them -- annotations are the primary source; log as informational |
| run-formal-verify.cjs changes between planning and execution | Integration step may not match current code | Read current STEPS array structure; add step following existing pattern |
