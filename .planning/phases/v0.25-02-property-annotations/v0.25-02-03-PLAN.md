<task type="auto">
  <name>Task 1: Build bin/extract-annotations.cjs parser</name>
  <files>bin/extract-annotations.cjs</files>
  <action>
Create `bin/extract-annotations.cjs` — a Node.js script that extracts `@requirement` annotations from all formal model files and outputs a structured JSON map.

**Input sources:**
- Read `formal/model-registry.json` to get the list of model file paths (the `models` object keys)
- For each model file, determine the formalism from file extension: `.tla` = TLA+, `.als` = Alloy, `.props` = PRISM
- Skip files that don't exist on disk (e.g., test model entries with relative paths that don't resolve)
- Skip TTrace files (any `.tla` file containing `_TTrace_` in its name)

**Parsing logic per formalism:**

**TLA+ (`.tla`):**
- Annotation pattern: `^\s*\\\*\s*@requirement\s+([\w-]+)` — matches `\* @requirement REQ-ID`
- Property pattern: `^(\w+)\s*==` — matches `PropertyName ==` at the start of a line
- Algorithm: scan lines top-to-bottom. When an annotation line is found, collect requirement IDs. When a property definition line is found, associate all collected IDs with that property name, then reset the collection. Skip lines that are neither annotations nor property definitions.
- Handle multiline: annotations can span multiple consecutive `\* @requirement` lines before a single property

**Alloy (`.als`):**
- Annotation pattern: `^--\s*@requirement\s+([\w-]+)` — matches `-- @requirement REQ-ID`
- Property pattern: `^assert\s+(\w+)\s*\{` or `^assert\s+(\w+)\s*$` — matches `assert AssertName {`
- Same accumulation algorithm as TLA+

**PRISM (`.props`):**
- Annotation pattern: `^\/\/\s*@requirement\s+([\w-]+)` — matches `// @requirement REQ-ID`
- Property pattern: `^[A-Z]` or `^[A-Z]=?\?` — matches property formula lines (P=?, R{...}=?, S=?) that start with uppercase letter
- More specifically: match lines starting with `P=?`, `R{`, `S=?`, or any PRISM property keyword
- Same accumulation algorithm as TLA+
- IMPORTANT: The annotation lines may appear before descriptive comments (e.g., `// P1: ...`). The parser should NOT treat descriptive comment lines as property definitions. Only actual formula lines (starting with P, R, S followed by special chars) count as properties.

**Output format:**
```json
{
  "formal/tla/QGSDStopHook.tla": [
    { "property": "TypeOK", "requirement_ids": ["STOP-01"] },
    { "property": "SafetyInvariant1", "requirement_ids": ["STOP-02", "SPEC-01"] }
  ],
  "formal/alloy/install-scope.als": [
    { "property": "NoConflict", "requirement_ids": ["INST-01"] }
  ],
  "formal/prism/quorum.props": [
    { "property": "P=? [ F s=1 ]", "requirement_ids": ["PRM-01", "QUORUM-04"] }
  ]
}
```

For PRISM, use the full formula line as the `property` name (since PRISM properties don't have named identifiers).

**CLI interface:**
- `node bin/extract-annotations.cjs` — output JSON to stdout
- `node bin/extract-annotations.cjs --validate` — check for unannotated properties, exit 0 if all annotated, exit 1 with report if any gaps found
- `node bin/extract-annotations.cjs --summary` — output summary counts per file
- `node bin/extract-annotations.cjs --pretty` — pretty-print JSON output

**Validation mode (`--validate`):**
For each model file, after parsing annotations, also scan for ALL property definitions (using the property patterns above). If a property exists without a preceding `@requirement` annotation, report it as unannotated. Output:
```

```
Or:
```

UNANNOTATED:
  formal/tla/TUINavigation.tla: TypeOK (no @requirement annotation)
  formal/tla/TUINavigation.tla: NoDeadlock (no @requirement annotation)
FAIL
```

**Style:** Follow existing bin/ script patterns (CommonJS, `#!/usr/bin/env node` shebang, `'use strict'`). Use only Node.js built-ins (fs, path, readline). No external dependencies.
