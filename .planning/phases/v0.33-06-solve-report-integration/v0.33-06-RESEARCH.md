# Phase v0.33-06: Solve Report Integration — Research

## Summary

Phase v0.33-06 integrates convergence analytics into the nf:solve output so users see trend sparklines, oscillation status, action items, and escalation classification without running separate tools. Two requirements drive this: INTG-02 (convergence section in solve report) and OSC-03 (Haiku escalation classifier on oscillation breaker fire).

## Existing Infrastructure

### Data Sources (built in prior phases)

1. **solve-trend.jsonl** (Phase v0.33-01 / TRACK-01): JSONL append log with per-layer residuals, gate summary, requirement count, scope_change, converged boolean per solve run.

2. **oscillation-detector.cjs** (Phase v0.33-02 / TRACK-02, OSC-01, OSC-02): Mann-Kendall trend test per layer, oscillation credit enforcement (Option C), cascade grace period. Writes `oscillation-verdicts.json` with per-layer `{ trend, z_score, oscillation_count, credits_remaining, blocked, last_direction, grace_period }`.

3. **predictive-power.cjs** (Phase v0.33-04 / PRED-01, PRED-02, TRACK-03): Bug-to-property linking, per-model recall scoring, convergence velocity estimation (exponential decay fit). Already imported in nf-solve.cjs.

4. **solve-trend-helpers.cjs** (Phase v0.33-01): `readLastTrendEntry()`, `readGateSummary()`, `appendTrendEntry()`, `LAYER_KEYS` (19 layers).

5. **gate-stability.cjs** (Phase v0.33-03): Flip-flop detection, cooldown enforcement.

### Consumers (where output must appear)

1. **solve-report.md** — Sub-skill invoked by `solve.md` orchestrator via Agent tool. Steps 6-8 generate terminal output. The convergence section is a NEW step between current Step 6 (before/after table) and Step 7 (formal verification detail).

2. **nf-solve.cjs** — Main diagnostic engine. Already calls `updateVerdicts()` and `appendTrendEntry()`. The escalation classifier hook must be wired here (or in the orchestrator solve.md).

3. **nf-circuit-breaker.js** — PreToolUse hook with Haiku classifier for GENUINE vs REFINEMENT. The new OSC-03 escalation classifier is conceptually similar but operates at the solve-layer level (not git-commit level).

## Requirement Analysis

### INTG-02: Convergence Section in Solve Report

**What must be true:** "The solve report includes a convergence section showing per-layer trend sparklines, oscillation status, and top-3 action items derived from trend data."

**Sparklines:** Unicode block characters (e.g., `▁▂▃▄▅▆▇█`) encoding the last N residual values per layer. The oscillation-detector already reads the trend window (last 20 entries). Sparklines map each value to one of 8 levels based on min/max normalization within the series.

**Oscillation status:** Already computed by `oscillation-detector.cjs` and stored in `oscillation-verdicts.json`. Need to read the verdicts file and display per-layer status: trend classification (DECREASING/STABLE/INCREASING/OSCILLATING/INSUFFICIENT_DATA), credits remaining, blocked status, grace period.

**Top-3 action items:** Derived from trend data by ranking layers by "actionability":
- Blocked layers (OSC fired) = highest priority
- INCREASING trend layers = second priority
- OSCILLATING layers = third priority
- Layers with high residual + STABLE trend = fourth priority (stalled)
Format: numbered list with layer name, current residual, trend, and recommended action.

**Implementation approach:** New module `bin/convergence-report.cjs` that:
1. Reads `solve-trend.jsonl` (last 20 entries) via `readTrendWindow()`
2. Reads `oscillation-verdicts.json`
3. Generates sparkline strings per layer
4. Ranks and selects top-3 action items
5. Exports `formatConvergenceSection(root)` returning a formatted string

The solve-report.md sub-skill calls this module (Step 6.5) after the before/after table.

### OSC-03: Haiku Escalation Classifier

**What must be true:** "When the oscillation breaker fires, a Haiku-based escalation classifier reports root cause as GENUINE_REGRESSION, MEASUREMENT_NOISE, or INSUFFICIENT_EVIDENCE with structured context (layer, delta, git diff)."

**Trigger:** When `oscillation-detector.cjs` transitions a layer to `blocked: true` (credits_remaining drops to 0).

**Classifier design:** Similar pattern to the existing Haiku reviewer in `nf-circuit-breaker.js` (which classifies GENUINE vs REFINEMENT for git commits). The new classifier:
1. Receives structured context: layer name, residual delta series, git diff of recent changes to files related to that layer
2. Calls Claude Haiku (claude-haiku-4-5-20251001) with a classification prompt
3. Returns one of: GENUINE_REGRESSION, MEASUREMENT_NOISE, INSUFFICIENT_EVIDENCE
4. Includes structured context: `{ layer, delta, git_diff_summary, classification, confidence }`

**Integration point:** Inside `updateVerdicts()` or as a post-step in `nf-solve.cjs` after verdicts update. When a layer transitions from `blocked: false` to `blocked: true`, fire the classifier.

**Implementation approach:** New module `bin/escalation-classifier.cjs` that:
1. Takes `{ layer, verdicts, trendPath, root }` as input
2. Extracts the recent delta series for the blocked layer
3. Gets a git diff summary for files related to that layer (using layer-to-file-path mapping)
4. Calls Haiku API via the unified MCP server pattern or direct HTTP
5. Returns `{ classification, confidence, context: { layer, delta, git_diff_summary } }`

Since MCP tool calls are prohibited from sub-skills (per R3.2), the classifier should use direct HTTP to the Anthropic API (same pattern as `nf-circuit-breaker.js` Haiku calls).

## Architecture Decision

**Option A: Inline in solve-report.md** — Add the convergence section directly to the solve-report sub-skill markdown. Simple but makes the sub-skill longer and harder to test.

**Option B: Separate module (recommended)** — Create `bin/convergence-report.cjs` for the sparkline/trend rendering and `bin/escalation-classifier.cjs` for the Haiku classification. The solve-report.md sub-skill calls these via `node bin/convergence-report.cjs --json` and the solve orchestrator triggers escalation classification when needed.

Recommend **Option B** for consistency with the project pattern (solve-trend-helpers.cjs, oscillation-detector.cjs, gate-stability.cjs, predictive-power.cjs all follow this extract-as-module pattern).

## Sparkline Rendering

Unicode sparkline characters: `▁▂▃▄▅▆▇█` (8 levels, 0-7).

For a series of values [v1, v2, ..., vN]:
1. Find min and max of the series
2. If min == max, all values map to `▄` (middle)
3. Otherwise, map each value: `index = Math.round((v - min) / (max - min) * 7)`
4. Concatenate corresponding characters

Example: residuals [5, 4, 3, 2, 1, 0, 0, 0] -> `█▇▅▃▂▁▁▁`

For missing values (-1 in the JSONL), use `·` (middle dot) to indicate skipped.

## Test Strategy

### convergence-report.cjs tests
- Sparkline generation: empty series, single value, increasing series, decreasing series, constant series, series with -1 gaps
- Action item ranking: blocked layer priority, INCREASING priority, OSCILLATING priority
- Full section formatting: with verdicts data, without verdicts file, with empty trend

### escalation-classifier.cjs tests
- Classification prompt construction: correct layer, delta, diff context
- Response parsing: valid classification, malformed response (defaults to INSUFFICIENT_EVIDENCE)
- Trigger detection: layer transitioning to blocked, layer already blocked (no re-trigger)

## Validation Architecture

### Test Framework
- Framework: Node built-in test runner (`node --test`)
- Quick run: `node --test bin/convergence-report.test.cjs bin/escalation-classifier.test.cjs`
- Full suite: `node --test bin/*.test.cjs`

### Per-Task Test Types
- convergence-report.cjs: unit tests (pure function sparkline + ranking logic)
- escalation-classifier.cjs: unit tests (prompt construction, response parsing) + mock integration test (HTTP stub)
- solve-report.md integration: smoke test (run `node bin/convergence-report.cjs --json` with sample data)

### Wave 0 Requirements
- Test scaffold files must exist before implementation: `bin/convergence-report.test.cjs`, `bin/escalation-classifier.test.cjs`

## Risk Assessment

- **Low risk:** Sparkline rendering is pure computation with no side effects
- **Medium risk:** Haiku API call in escalation classifier — needs timeout handling and fail-open behavior (if Haiku is unavailable, default to INSUFFICIENT_EVIDENCE)
- **Low risk:** Integration into solve-report.md — additive change to an existing display-only sub-skill
