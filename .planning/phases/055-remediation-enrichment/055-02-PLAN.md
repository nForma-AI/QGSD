<objective>
Add --enrich-recipes mode to formal-test-sync.cjs that uses coderlm findTests/peek to populate recipe JSON sidecars with observed test patterns, then wire this into solve-remediate section 3b so dispatched stubs contain real assert patterns instead of generic boilerplate.

Purpose: Test stubs are generated with recipes as context. Currently recipes contain structural info (formal property, source files, import hint) but no knowledge of how existing tests in the project actually assert similar behavior. Enriching recipes with observed patterns (assert calls, test setup sequences, describe blocks from findTests results) gives the stub-implementing executor agent concrete patterns to follow.

Output: formal-test-sync.cjs --enrich-recipes flag that augments existing recipe JSON files with observed_test_patterns field; solve-remediate section 3b runs enrichment after stub generation phase.

2. Add a new `enrichRecipesWithTestPatterns()` async function after the existing `generateStubs()` function. This function:
   - Reads all `*.stub.recipe.json` files from `stubsDir`
   - For each recipe that has non-empty `source_files`:
     - Calls `adapter.findTests(sourceFile)` for each source file (up to first 3).
       **IMPORTANT:** `findTests()` accepts a FILE PATH (absolute or relative to project root), NOT a module name or symbol. Pass the path as-is from `recipe.source_files[i]`. If a path is a module name (no slashes, no `.cjs`/`.js` extension), skip it with a warning to stderr.
     - For each test file returned by findTests (up to 5 per source file), calls `adapter.peek(testFile, 1, 100)` to get the first 100 lines.
       **Why 100 lines instead of 50:** Many test files have long `describe` blocks and `beforeEach` setup before any assertions appear. 50 lines frequently captures only preamble with no assert patterns. 100 lines reliably captures the first describe block and initial assertions.
       As an alternative two-pass strategy if 100 lines still yields no assert patterns: call `adapter.peek(testFile, 1, 30)` for setup patterns, then `adapter.peek(testFile, 31, 150)` for assertion patterns and merge both results.
     - Extracts assert patterns from peeked content: lines matching `/assert\./`, `/expect\(/`, `/describe\(/`, `/beforeEach\(/` via simple regex
     - **Idempotency:** Skips recipes where `observed_test_patterns.assert_patterns` is already a non-empty array (not where `test_files.length > 0` — test_files being populated does not guarantee assert_patterns were extracted). This allows partial runs to be re-run successfully.
     - Writes `observed_test_patterns` field into the recipe JSON:
       ```json
       {
         "observed_test_patterns": {
           "test_files": ["path/to/test.cjs"],
           "assert_patterns": ["assert.strictEqual(result, expected)", "assert.ok(mod.functionName)"],
           "setup_patterns": ["const mod = require('../src/foo.cjs');"]
         }
       }
       ```
   - All errors are caught and logged to stderr (CADP-02 fail-open pattern)
   - Returns `{ enriched: N, skipped: M, errors: K }`

The function signature:

async function enrichRecipesWithTestPatterns(adapter) {
  // ...
}

3. Wire the entry point so that `main()` remains synchronous and is called BEFORE the async enrichRecipes tail. Do NOT wrap `main()` inside the async IIFE — doing so changes `process.exit()` timing and breaks existing synchronous test assertions. Use this pattern at the bottom of the file:

```javascript
// Synchronous main — must run (and call process.exit on error) before any async work.
main();

// Async enrichment tail — only runs if --enrich-recipes flag is set.
// main() has already completed by this point.
if (enrichRecipes) {
  (async () => {
    const { createAdapter } = require('./coderlm-adapter.cjs');
    const adapter = createAdapter({ enabled: true });
    // Health check — skip if coderlm unavailable (fail-open)
    const health = adapter.healthSync();
    if (health.error) {
      process.stderr.write(TAG + ' --enrich-recipes: coderlm unavailable (' + health.error + '), skipping\n');
      return;
    }
    try {
      const result = await enrichRecipesWithTestPatterns(adapter);
      process.stderr.write(TAG + ' --enrich-recipes: enriched=' + result.enriched + ' skipped=' + result.skipped + ' errors=' + result.errors + '\n');
    } catch (e) {
      process.stderr.write(TAG + ' --enrich-recipes failed (non-fatal): ' + e.message + '\n');
    }
  })().catch(e => {
    process.stderr.write(TAG + ' --enrich-recipes fatal: ' + e.message + '\n');
  });
}
```

**Important constraints:**
- The `--enrich-recipes` path is ONLY entered when the flag is explicitly set — existing behavior when flag absent is 100% unchanged
- `main()` is called synchronously FIRST, then the async IIFE runs as a tail — never wrap `main()` inside the IIFE
- healthSync() is called synchronously inside the IIFE before any async work to check availability (fail-open)
- createAdapter() uses `{ enabled: true }` matching the pattern from nf-solve.cjs
- `findTests()` receives a FILE PATH — if a source_files entry has no path separator and no known extension, skip it with stderr warning rather than passing an unusable symbol name
- `peek()` is called with lines 1–100 to reliably capture both setup and assertion sections
- If peek(1, 100) returns content with zero assert pattern matches, apply two-pass strategy: peek(1, 30) for setup + peek(31, 150) for assertions
- Idempotency check: skip when `observed_test_patterns.assert_patterns` is a non-empty array (NOT when `test_files.length > 0` — partial enrichment must be re-runnable)
- assert pattern extraction is simple regex (no AST parsing): lines containing `.assert`, `assert.`, `expect(`, `beforeEach(`, `describe(`
- Maximum 3 source files queried per recipe, maximum 5 test files peeked per source file (avoid runaway queries)
