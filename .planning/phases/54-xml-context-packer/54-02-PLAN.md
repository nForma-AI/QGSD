<objective>
Create `context-packer.cjs` as the single entry point for Repowise context packing. It orchestrates all Repowise intelligence modules (skeleton, hotspot, co-change) into a unified XML output. In this phase, skeleton/hotspot/co-change render as empty self-closing placeholders (`<skeleton available="false"/>`, etc.) since their implementations come in Phases 55-57. The module provides both a CLI interface and a programmatic `packContext()` API for use by other modules.

Purpose: Establish the orchestration layer that all future Repowise modules feed into. Without context-packer.cjs, there is no single entry point for Repowise context delivery.
Output: bin/repowise/context-packer.cjs, bin/repowise/context-packer.test.cjs, updated package.json test:ci script.
</objective>

  <name>Task 1: Create context-packer.cjs CLI + programmatic entry point (PACK-03)</name>
  <files>bin/repowise/context-packer.cjs</files>
  <action>
Create `bin/repowise/context-packer.cjs` following the pattern of `bin/design-impact.cjs` (CLI flags, project root resolution, JSON output mode):

1. `#!/usr/bin/env node` shebang and `'use strict'`
2. Module header: `// bin/repowise/context-packer.cjs — Single entry point for Repowise context packing`
3. Imports: `fs`, `path`, `{ packFile }` from `./pack-file.cjs`
4. Project root resolution (same pattern as design-impact.cjs):
   - `--project-root=` arg → `PROJECT_ROOT` env → `path.resolve(__dirname, '../..')`
5. CLI flag parsing:
   - `--files=path1,path2` — comma-separated file paths to pack
   - `--stdin` — read file paths from stdin (one per line)
   - `--json` — output structured JSON instead of XML
   - `--project-root=/path` — override project root
   - `--help` — print usage
6. `packContext({ files, projectRoot, signals })` — the core programmatic API:
   - Parameters:
     - `files`: `Array<{ filePath: string, content: string, lang?: string }>` — files to pack
     - `projectRoot`: `string` — project root for relative path resolution
     - `signals`: `{ skeleton?: string, hotspot?: string, cochange?: string }` — pre-rendered signal sections (default: empty placeholders)
   - Returns: `{ xml: string, json: object }` — both XML and JSON representations
   - XML format:
     ```
     <repowise>
       <skeleton available="false"/>
       <hotspot available="false"/>
       <cochange available="false"/>
       <files>
         <file path="src/foo.js" lang="js">...escaped content...</file>
         <file path="src/bar.ts" lang="ts">...escaped content...</file>
       </files>
     </repowise>
     ```
   - JSON format:
     ```json
     {
       "repowise": {
         "skeleton": { "available": false, "data": null },
         "hotspot": { "available": false, "data": null },
         "cochange": { "available": false, "data": null },
         "files": [
           { "path": "src/foo.js", "lang": "js", "content": "...raw (unescaped) content..." }
         ]
       }
     }
     ```
   - When `signals.skeleton` is provided (future), it replaces the placeholder: `<skeleton available="true">...signal xml...</skeleton>` (same for hotspot, cochange)
7. `resolveFileList(filesArg, stdinMode, projectRoot)` — helper to resolve file paths:
   - If `--files` provided: split by comma, resolve each relative to projectRoot
   - If `--stdin`: read from stdin, split by newlines, filter empty lines
   - Read each file's content via `fs.readFileSync(path.resolve(projectRoot, f), 'utf8')`
   - Return `Array<{ filePath, content }>`
8. CLI main function:
   - Parse flags
   - Resolve file list
   - Call `packContext()`
   - Print XML to stdout (or JSON if `--json`)
   - Exit 0 on success, 1 on error (error message to stderr)
   - On missing files arg (no `--files` and no `--stdin`): print help and exit 1
9. Exports: `module.exports = { packContext, resolveFileList };`
10. If called directly (not required), run CLI main function (use `require.main === module` guard)

The `available="false"` attribute pattern is deliberate: it tells LLM consumers that these sections exist but are not yet populated, and future phases will set `available="true"` with actual data.

5. **JSON format**: same input produces `json` with `repowise` key containing `skeleton`, `hotspot`, `cochange`, `files` keys
