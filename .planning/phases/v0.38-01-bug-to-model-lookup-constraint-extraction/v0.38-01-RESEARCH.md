# Phase v0.38-01: Bug-to-Model Lookup & Constraint Extraction - Research

**Researched:** 2026-03-18
**Domain:** Formal verification model discovery and constraint extraction for debugging
**Confidence:** HIGH

## Summary

This phase implements the foundational infrastructure for model-driven debugging: finding which formal models (TLA+/Alloy specs) are relevant to a reported bug, and extracting human-readable constraint guidance from those models. The existing codebase has 200+ formal models with rich metadata (model-registry.json), a working proximity-index for semantic lookup, and proven model checkers (TLC/Alloy runners). This phase reuses all of that and adds two focused layers: (1) a `--bug-mode` flag to `formal-scope-scan.cjs` for bug-specific model matching, and (2) a new `model-constrained-fix.cjs` tool that parses INVARIANT/PROPERTY/assertion blocks from TLA+ and Alloy specs and translates them to plain-English constraints.

**Primary recommendation:** Extend existing formal-scope-scan.cjs infrastructure with bug-mode matching; add lightweight custom parsers for TLA+/Alloy constraint extraction; avoid external parsing libraries (no TLA+ AST parser exists; keep regex-based extraction simple and maintainable). Dependencies: `fast-xml-parser@5.4.1` (Alloy XML instance parsing) and `acorn@8.11.0` (JavaScript code-to-model mapping). Test using node:test framework (already in use across codebase). Validation architecture: unit tests for parsers, integration tests against real specs, wave 0 scaffold for constraint extraction accuracy.

- **Constraint translation to English:** Reuse existing quorum dispatch with Haiku (already available, integrated).
- **Proximity graph traversal:** Existing proximity-index.json structure is ready; layer 2 of formal-scope-scan.cjs already implements graph walk.

### Pattern 1: Bug-to-Model Lookup via Semantic Matching

**What:** Extend formal-scope-scan.cjs with `--bug-mode` flag. Instead of source-file-based matching only, add three matching strategies layered by signal strength:

1. **File overlap:** If bug mentions specific file (e.g., "hooks/nf-stop.js"), match models with that file in source_files globs (existing Layer 1).
2. **Semantic/concept matching:** Bug description "timeout failure" matches models tagged with concepts `timeout`, `failure`, `liveness` in model-registry.json.
3. **Pattern matching:** Common error signatures (e.g., "active=TRUE but disabled=TRUE") matched against model descriptions and requirement tags.

**When to use:** Every `/nf:debug` call (implicit); explicit use via `node bin/formal-scope-scan.cjs --bug-mode --description "..."`.

### Pattern 2: Constraint Extraction from Specs

**What:** Parse TLA+/Alloy specs to extract INVARIANT/PROPERTY/assertion definitions, normalize to structured format, and link back to requirements via @requirement annotations.

**When to use:** After model lookup succeeds; feeds into constraint-to-English translation.

### Anti-Patterns to Avoid

- **Hand-rolling TLA+ AST parser:** TLA+ has no standard grammar tool; regex is sufficient and maintainable.
- **Calling Haiku synchronously in lookup loop:** Batch constraint translation. Call Haiku once per model, not once per invariant.
- **Parsing specs without @requirement markers:** Enforce traceability upfront; use model-registry as source of truth for which specs are in-scope.
- **Model reproduction without timeout:** BML-02 requires 60s timeout per checker. If model doesn't converge, fail-open gracefully (mark as "timeout").
- **Returning all invariants to developer:** Overwhelming. Rank by relevance (frequency in traces); surface top 3-5 only. Let developers ask for more.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| XML parsing (Alloy output) | Regex-based SAT instance parser | fast-xml-parser | Handles nested elements, attributes, escaping; proven in production |
| JavaScript scope analysis (code-to-model mapping) | Custom import tracer | acorn + walk | ESLint standard; acorn is lightweight, well-maintained |
| TLA+ spec parsing | Hand-written recursive descent | Regex + @requirement markers | TLA+ has no standard grammar; specs are highly structured; 95% regex coverage avoids parser generator dependency |
| Model checking execution | New wrapper around TLC/Alloy | Refactor existing run-tlc.cjs/run-alloy.cjs | Runners are battle-tested, proven to handle Java spawning, heap management, timeout recovery |
| Constraint-to-English translation | Rule-based template engine | Haiku (existing quorum dispatch) | Haiku produces higher-quality summaries; already available; batching handles latency |
| Proximity graph traversal | BFS implementation | Existing proximity-index.json + layer 2 of formal-scope-scan.cjs | Index is pre-built, semantically rich; leverage existing work |

**Key insight:** Model-driven debugging's value comes from integrating formal and natural language (LLM translation), not building new model checkers. Reuse existing checkers, add lightweight metadata layers (lookup, constraint extraction), connect to existing LLM infra (Haiku). Avoid NIH traps around parsing/AST.

## Common Pitfalls

### Pitfall 1: False Confidence from Symptom Reproduction

**What goes wrong:** Model reproduces a bug's symptom (execution reaches error state) but doesn't capture the mechanism (why the error is reachable). Developer fixes based on model assumptions, test passes, but fails in production with different trigger.

**Why it happens:** Model is more abstract than code. Code timing, concurrency, external I/O—these are often over-simplified in TLA+/Alloy.

**How to avoid:**
- In Phase 1: Document that model reproduction is hypothesis, not fact. Flag with `confidence: "provisional"` until verified by developer.
- In Phase 2 (debug integration): Require explicit mechanism-verification step. Developer compares model-claimed root cause to actual code path.

**Warning signs:**
- Model reproduces bug in 60s but developer reports "same issue in prod after fix"
- Model and code disagree on ordering of events
- Model lacks representation of a key external factor (network latency, clock skew)

### Pitfall 2: State Space Explosion on On-Demand Checking

**What goes wrong:** Model checker (TLC) doesn't converge within 60-second timeout. BML-02 requires `max 3, 60s timeout each`. Large models (500+ state variables) can exhaust heap or timeout.

**Why it happens:** BML-02 runs model checkers in `/nf:debug` response path (interactive). TLC uses exhaustive state enumeration. Large models explode combinatorially.

**How to avoid:**
- Phase 1: Pre-filter models by state-space size. Use existing `model-complexity-profile.json` to exclude HIGH-risk models from --bug-mode results.
- Phase 1: Impose timeout strictly. If timeout, report `status: "timeout"` and gracefully degrade.
- Phase 2+: For large models, switch to symbolic solver (APALACHE) instead of enumeration (TLC).
- Phase 2: Add `--quick-verify` mode that pre-constrains state space using ASSUME statements derived from bug trace.

**Warning signs:**
- `/nf:debug` hangs beyond 60s
- TLC process hits Java heap limit
- Model reproduces easily in isolation but times out in --bug-mode pipeline

### Pitfall 3: Over-Constraint in Fix Space

**What goes wrong:** CEX-01/02 extract too many invariants. Constraint list says "X AND Y AND Z must hold". Developer tries to satisfy all three; it's impossible with available code changes.

**Why it happens:** Models capture all safety properties. Not all are relevant to the bug. Returning all 20 invariants exhausts the developer.

**How to avoid:**
- CEX-03: Rank constraints by relevance (frequency in failing traces, causality distance from violated predicate).
- CEX-03: Surface top 3-5 constraints only. Distinguish "hard constraints" (safety invariants that MUST hold) from "soft hints" (patterns that help but aren't necessary).
- CEX-03: Assign confidence scores. Confidence depends on how directly invariant relates to failure.
- Phase 2: Developer-facing output explicitly marks which constraints are must-have vs. nice-to-have.

**Warning signs:**
- Constraint summary has 15+ items
- Developer reports "I can't satisfy all of these, which one can I ignore?"

### Pitfall 4: Refined Models That Don't Converge

**What goes wrong:** Phase 3 (B→F solve layer) auto-refines models when bugs aren't covered. Refinement loop spirals and never terminates.

**Why it happens:** Models are coupled via shared requirements. Refining one invariant can destabilize others.

**How to avoid:**
- Phase 3 (future): Cap auto-refinement to 3 iterations. After 3, escalate to human review.
- Phase 3: Track model version history (epoch counter). Detect loops by comparing TLA+ traces.

**Warning signs:** (Phase 3+)
- Model version epoch > 3
- Same invariant refined multiple times across epochs

### Pitfall 5: Cross-Model Regression Cascades

**What goes wrong:** Fix verified against Model A passes. Model A's invariant depends on Model B's property. Fix breaks B, leading to cascade failures across model graph.

**Why it happens:** Models share requirements. Requirement DAG is not a tree; it's a DAG with shared nodes.

**How to avoid:**
- Phase 1: Document transitive dependencies in model-registry.
- Phase 2 (pre-verification): Before shipping a fix, run verification against all models with shared requirements, not just the one that reproduced the bug.

**Warning signs:**
- After shipping fix for bug covered by Model A, new failures appear in systems covered by Models B, C, D.

### Pitfall 6: B→F Layer Noise from Test Flakiness

**What goes wrong:** Test passes 80% of the time, fails 20%. B→F layer routes it to "bugs formal models should explain". But it's not a real bug—it's a flaky test.

**Why it happens:** Flaky tests are indistinguishable from real bugs to automated analysis.

**How to avoid:**
- Phase 3 (future): Before routing a bug to B→F layer, apply stability filter: require bug to appear in >= 2 consecutive commits OR pass rate < 50%.
- Phase 3: B→F handler includes explicit check for flakiness.

**Warning signs:**
- B→F layer grows unbounded
- Most entries in bug-model-gaps.json are eventually resolved without model changes

### Pitfall 7: Debug Performance Impact

**What goes wrong:** Model consultation (lookup + constraint extraction + Haiku translation) adds 5-10 seconds per `/nf:debug` invocation. Developers skip model guidance and solve manually.

**Why it happens:** Haiku is network I/O (2-5s latency). Model lookup queries proximity-index.json. Constraint extraction parses specs linearly.

**How to avoid:**
- Phase 1: Cache lookup results in session memory. Clear only on new failure context.
- Phase 1: Pre-parse all specs once at startup (batch operation). Store constraint structures in JSON.
- Phase 1: Batch Haiku calls. Instead of 5 calls, collect 5 constraints, call once with batch prompt.
- Phase 2: Measure `/nf:debug` latency (baseline before, after integration). Target: add < 3 seconds.

**Warning signs:**
- `/nf:debug` response time > 10s
- Developer feedback: "I just skip the formal part, it's too slow"

## Code Examples

Verified patterns from Phase 1 infrastructure:

### Pattern: Semantic Matching in Bug-to-Model Lookup

Extend existing `formal-scope-scan.cjs` Layer 2 (proximity enrichment) with semantic scoring:

```javascript
// Layer 2 concept matching for --bug-mode
function scoreConceptMatch(bugDescription, modelMetadata) {
  const tokens = bugDescription.toLowerCase().split(/\s+/).filter(t => t.length > 2);
  let score = 0;

  // Check model description, requirements, concepts
  if (modelMetadata.description) {
    for (const modelToken of modelMetadata.description.toLowerCase().split(/\s+/)) {
      if (tokens.includes(modelToken)) score += 0.1;
    }
  }

  // Check requirement categories (e.g., "DELIB" from "DELIB-01")
  for (const req of (modelMetadata.requirements || [])) {
    const category = req.split('-')[0].toLowerCase();
    if (tokens.some(t => category.includes(t) || t.includes(category))) {
      score += 0.2;
    }
  }

  return Math.min(score, 1.0);
}
```

### Pattern: TLA+ Constraint Extraction via Regex

```javascript
// Extract INVARIANT blocks from TLA+ spec
function extractInvariants(specContent) {
  const results = [];
  const regex = /^([A-Z]\w*)\s*==\s*(.+?)(?=\n(?:[A-Z]|\n|---|----|\*\)))/gms;

  let match;
  while ((match = regex.exec(specContent)) !== null) {
    const [, name, formal] = match;
    // Extract @requirement marker from preceding comment (200 chars before)
    const prefix = specContent.slice(Math.max(0, match.index - 200), match.index);
    const reqMatch = prefix.match(/@requirement\s+(\w+-\d+)/i);

    results.push({
      name,
      formal: formal.trim(),
      requirement_id: reqMatch ? reqMatch[1] : null,
      confidence: reqMatch ? 0.95 : 0.7,
    });
  }

  return results;
}
```

### Pattern: Alloy XML Parsing with fast-xml-parser

```javascript
// Parse Alloy SAT solver XML output
const { XMLParser } = require('fast-xml-parser');

function parseAlloyInstance(xmlContent) {
  const parser = new XMLParser({ ignoreAttributes: false });
  const instance = parser.parse(xmlContent);

  // Extract signatures and field values
  const sigs = {};
  if (instance.instance && instance.instance.sig) {
    const sigArray = Array.isArray(instance.instance.sig)
      ? instance.instance.sig
      : [instance.instance.sig];

    for (const sig of sigArray) {
      sigs[sig['@_type']] = {
        atoms: sig.atom ? (Array.isArray(sig.atom) ? sig.atom : [sig.atom]) : [],
        tuples: sig.tuple ? (Array.isArray(sig.tuple) ? sig.tuple : [sig.tuple]) : [],
      };
    }
  }

  return sigs;
}
```

## Open Questions

1. **Should BML-02 support APALACHE (symbolic solver) in Phase 1?**
   - Recommendation: **Defer to Phase v0.38-02+.** Phase 1 uses proven TLC/Alloy infrastructure. APALACHE can be added as performance optimization after basic flow is working.

2. **How to handle specs lacking @requirement markers?**
   - Recommendation: **Gracefully degrade.** Extract constraints without markers; confidence score 0.7 (vs. 0.95 with marker). Flag as "provisional". Encourage adding markers during Phase 2 hardening.

3. **Should constraint-to-English batch across multiple models?**
   - Recommendation: **Batch within formalism, not across.** One Haiku call for all TLA+ constraints from Model A, separate call for Alloy constraints from Model B. Avoids syntax confusion; achieves batching efficiency.

4. **How should B→F layer distinguish "model gap" from "model mismatch"?**
   - Recommendation: **Defer to Phase 2.** Phase 1 returns lookup + constraints. Phase 2 adds mechanism verification. Phase 3 uses Phase 2 verdict to classify gaps vs. mismatches.

5. **Should proximity-index.json be regenerated when model-registry.json changes?**
   - Recommendation: **Add rebuild command** (`bin/rebuild-proximity-index.cjs`). Call during Phase 1 testing if registry is updated. No automatic rebuild needed for now; codebase is stable.
