# Phase v0.27-05: Solve P->F Integration - Research

**Researched:** 2026-03-04
**Domain:** Production-to-Formal residual layer for debt remediation
**Confidence:** HIGH

## Summary

The solve command needs a new residual layer that reads debt entries from `.formal/debt.json`, compares production measurements against formal model thresholds, and dispatches remediation workflows. This phase closes the feedback loop by automating the transition from acknowledged debt to formal model updates or investigation flags.

Key insight: P->F integration reuses the existing solve architecture (7-layer residual computation, sweep/detail pattern) and formal_ref linking infrastructure (v0.27-03), adding a thin orchestration layer that:
1. Filters for `acknowledged` status entries only
2. Resolves formal_ref links to actual thresholds
3. Detects divergence (production metric vs formal expected value)
4. Dispatches via `/qgsd:quick` for parameter updates OR flags for investigation
5. Freezes entries in `resolving` status during solve cycles

**Primary recommendation:** Implement P->F as a new `sweepPtoF()` function in `bin/qgsd-solve.cjs`, following the existing sweep pattern. Reuse `bin/formal-ref-linker.cjs` for resolution. Dispatch `/qgsd:quick` tasks inline during solve execution (not as background jobs). Count residual as number of divergent entries, not depth of remediation attempts.

## User Constraints (from CONTEXT.md)

### Locked Decisions

1. **Threshold comparison logic**
   - P->F residual layer reads `.formal/debt.json` for entries with status `acknowledged`
   - Each entry's `formal_ref` links to a requirement or spec parameter in `.formal/requirements.json` or `.formal/spec/`
   - For drift entries: compare production metric (from `meta`) against formal expected value (from formal model)
   - Residual count = number of acknowledged entries where production diverges from formal parameter values
   - Entries without `formal_ref` (null) are skipped — they need formal linking first (v0.27-03 handles this)

2. **Remediation dispatch strategy**
   - Two-track dispatch per success criteria SC-5:
     - **Parameter updates** (threshold/limit drift): Auto-dispatch `/qgsd:quick` task to update the formal model parameter to match production reality
     - **Regressions** (correctness invariant violations): Flag for investigation — do NOT auto-remediate, create investigation entry
   - Heuristic: drift entries with `issue_type: "drift"` → parameter update track; entries with `issue_type: "issue"` → investigation track
   - Dispatch is logged in debt entry metadata for audit trail

3. **Freeze semantics during solve**
   - Per-entry freeze via existing `status: "resolving"` in debt state machine (v0.27-01)
   - When solve begins processing an entry: transition status `acknowledged` → `resolving`
   - observe-debt-writer (v0.27-02) already respects state machine — `resolving` entries cannot be overwritten by new observations
   - After remediation completes: transition `resolving` → `resolved`
   - If remediation fails: transition back to `acknowledged` (re-eligible for next solve cycle)

4. **Regression vs parameter update heuristic**
   - `issue_type: "drift"` + `formal_ref` points to a numeric threshold/limit → parameter update (auto-dispatch)
   - `issue_type: "issue"` OR `formal_ref` points to a correctness invariant → investigation (flag only)
   - Unknown/ambiguous: default to investigation track (conservative — never auto-update invariants)
   - Heuristic is deterministic and testable — no ML or fuzzy logic

### Pattern 1: Residual Computation (7-layer sweep)

**What:** Each layer (R->F, F->T, C->F, T->C, F->C, R->D, D->C, P->F) computes a residual count and detail object. Residual is aggregated at the top level.

**When to use:** For any consistency gap that needs automated detection and optional remediation.

**Example:**

```javascript
// Source: bin/qgsd-solve.cjs (existing sweepFtoT pattern)
function sweepFtoT() {
  // Load formal invariants
  const invariants = loadFormalInvariants();
  // Load test coverage
  const testCoverage = loadTestCoverage();
  // Find uncovered invariants
  const uncovered = invariants.filter(inv => !testCoverage.has(inv.id));
  return {
    residual: uncovered.length,
    detail: {
      uncovered_invariants: uncovered.map(inv => ({ id: inv.id, type: inv.type }))
    }
  };
}
```

For P->F, the pattern is identical:

```javascript
function sweepPtoF() {
  const ledgerPath = path.join(ROOT, '.formal/debt.json');
  const ledger = readDebtLedger(ledgerPath);

  // Filter for acknowledged entries only
  const acknowledged = ledger.debt_entries.filter(e => e.status === 'acknowledged');

  // Resolve formal refs and detect divergence
  const { entries: linked } = linkFormalRefs(acknowledged);
  const divergent = linked.filter(e => e.formal_ref && isDivergent(e));

  return {
    residual: divergent.length,
    detail: {
      divergent_entries: divergent.map(e => ({
        id: e.id,
        formal_ref: e.formal_ref,
        measured: e.meta?.measured_value,
        expected: extractFormalExpected(e.formal_ref)
      }))
    }
  };
}
```

### Pattern 2: Formal Reference Resolution

**What:** Layered resolution (manual > auto-detect > spec-inferred > null) for linking debt entries to formal model locations.

**When to use:** Whenever you need to map an entry to its formal specification.

**Example:**

```javascript
// Source: bin/formal-ref-linker.cjs (existing)
const { entries: linked, linkedCount } = linkFormalRefs(debtEntries, {
  requirementsPath: '.formal/requirements.json',
  specDir: '.formal/spec'

// linked entries now have formal_ref and formal_ref_source populated
// formal_ref format: "requirement:DEBT-01" or "spec:safety" or null
```

### Pattern 3: Entry State Transition with Freeze

**What:** Transition entries through state machine (acknowledged → resolving → resolved) to freeze them during solve cycles.

**When to use:** To prevent concurrent modifications of entries being remediated.

**Example:**

```javascript
// Source: bin/debt-state-machine.cjs (existing)
// Entry starts: status: "acknowledged"

// sweepPtoF identifies divergence
const divergent = [...];

// autoClosePtoF freezes entries before remediation
for (const entry of divergent) {
  const result = transitionDebtEntry(entry, 'resolving');
  if (result.success) {
    // Entry is now frozen; observe will skip overwriting it
    // Remediation proceeds
    spawnTool('qgsd:quick', [...]);
    // After remediation completes (success or fail):
    if (remediationSucceeded) {
      transitionDebtEntry(entry, 'resolved');
    } else {
      transitionDebtEntry(entry, 'acknowledged'); // Re-eligible for next solve
    }
  }
}
```

### Anti-Patterns to Avoid

- **Processing open entries:** P->F should skip entries with status="open" (locked decision PF-03). Only acknowledged entries are eligible. Unlinked entries (formal_ref=null) are also skipped — they need human triage first.
- **Merging parameter updates with investigation:** The heuristic (issue_type="drift" vs "issue") must be binary and deterministic. No fuzzy logic or ML-based classification. Ambiguous cases default to investigation (conservative).
- **Async remediation without tracking:** If `/qgsd:quick` dispatch becomes async, you must track which entries are "remediation in-progress" to prevent re-triggering. Current blocking dispatch avoids this.
- **Updating invariant thresholds automatically:** Only numeric thresholds (e.g., timeouts, limits) should auto-update. Correctness invariants must be flagged for investigation, never auto-fixed.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Atomic read/write of debt ledger | Custom JSON file handling | `bin/debt-ledger.cjs` (readDebtLedger, writeDebtLedger) | Already implemented with temp file + rename pattern; handles crashes safely |
| State machine transitions | Manual if-else for valid transitions | `bin/debt-state-machine.cjs` (transitionDebtEntry, canTransition) | Already tested (34 tests); enforces locked transitions; prevents invalid states |
| Formal reference linking | Keyword matching from scratch | `bin/formal-ref-linker.cjs` (linkFormalRefs) | Already implemented with 3 layers (manual > auto-detect > spec-inferred); handles null refs gracefully |

| Task dispatch to /qgsd:quick | HTTP calls, REST APIs, background jobs | `spawnSync()` via spawnTool() helper (existing in qgsd-solve.cjs) | Matches all existing sweep closures; blocking is sufficient for single solve cycle |

**Key insight:** The P->F layer is thin orchestration over existing infrastructure. Resist the urge to add new generic abstractions. Reuse the established solve sweep pattern, debt ledger, and state machine instead.

## Common Pitfalls

### Pitfall 1: Comparing Against Wrong Threshold

**What goes wrong:** You extract the formal parameter value from the wrong source (e.g., requirements.json text instead of spec model file), leading to false divergence detections or missed regressions.

**Why it happens:** Formal parameters live in multiple places:
- `.formal/spec/safety/thresholds.json` (authoritative numeric values)
- `.formal/requirements.json` (human text describing the parameter, not numeric)
- `.formal/tla/MCsafety.cfg` (TLA+ model constants)
- Production code constants (code values)

Mixing sources causes misalignment.

**How to avoid:**
- Establish a canonical parameter source in `.formal/spec/` directory
- `formal_ref` should point specifically to that source (e.g., "spec:safety/MCsafety.cfg:MaxDeliberation")
- Create a helper `extractFormalExpected(formal_ref)` that always reads from the canonical location
- Document the mapping in `.formal/parameter-mapping.json` (phase 5 discretion)

**Warning signs:**
- Divergence detections that don't match actual production behavior
- Parameters in requirements.json that don't have numeric values
- Multiple code paths extracting thresholds (inconsistency)

### Pitfall 2: Auto-Updating Invariants Instead of Flagging Investigation

**What goes wrong:** An entry with `issue_type: "issue"` (indicating a correctness problem) gets auto-dispatched to `/qgsd:quick`, which changes the formal model. The model is now wrong.

**Why it happens:** The heuristic isn't enforced strictly. Code assumes "if divergence detected, update formal model." But invariants should never be auto-updated — only parameters.

**How to avoid:**
- Make the track decision **explicit and early** in `autoClosePtoF()`:
  ```javascript
  const isParameter = entry.issue_type === 'drift' && isNumericThreshold(entry.formal_ref);
  const isInvariant = entry.issue_type === 'issue' || !isNumericThreshold(entry.formal_ref);
  if (isInvariant) {
    entry.meta.investigation_notes = `Production diverged from formal invariant ${entry.formal_ref}. Manual review required.`;
    // DO NOT dispatch /qgsd:quick
  } else {
    // Dispatch parameter update
  }
  ```
- Add a gate: "if (isInvariant) { throw Error(...) }" so auto-dispatch is impossible
- Document the invariant vs parameter distinction in comments

**Warning signs:**
- Formal model parameters that don't have numeric values
- Investigation flags being ignored (no follow-up actions)
- Entries dispatched with "update formal invariant" in the quick task title

### Pitfall 3: Not Freezing Entries During Solve Cycle

**What goes wrong:** While solve is processing an entry in `resolving` status, a concurrent `/qgsd:observe` run writes a new observation to the same entry, overwriting the state transition and causing the entry to escape the solve cycle.

**Why it happens:** The solve and observe paths run independently. If they modify the same entry simultaneously, the last writer wins (race condition).

**How to avoid:**
- Ensure `observe-debt-writer.cjs` checks `if (entry.status === 'resolving') { skip overwrite }` (this is already implemented in v0.27-02)
- Test this explicitly: observe run while solve is mid-cycle should skip the entry
- Add a comment in sweepPtoF: "Entries transitioned to 'resolving' are safe from concurrent observe overwrites"

**Warning signs:**
- Entries bouncing between acknowledged/resolving states
- Observations during solve appearing to disappear
- Solve completing but entries still in "resolving" state (state transition failed)

### Pitfall 4: Unlinked Entries Causing Silent Skips

**What goes wrong:** An acknowledged entry has `formal_ref: null` (never linked by formal-ref-linker.cjs). The sweepPtoF() skips it silently. User doesn't know it's waiting for manual linking.

**Why it happens:** The locked decision says "entries without formal_ref are skipped" (correct), but there's no visibility into the skipped count. Over time, the backlog of unlinked entries grows unnoticed.

**How to avoid:**
- Include in the sweep detail: count of "skipped (unlinked)" entries
- Return both divergent and skipped in the residual detail:
  ```javascript
  return {
    residual: divergent.length,
    detail: {
      divergent_entries: [...],
      skipped_unlinked: unlinked.length,
      skipped_unlinked_ids: unlinked.map(e => e.id)
    }
  };
  ```
- Report "N entries skipped (waiting for formal link)" in the solve summary
- Consider flagging unlinked entries for triage in a future phase

**Warning signs:**
- Divergent entry count stays zero even though observe is running
- Manual `formal_ref` assignments have no effect (entry still skipped)
- Unlinked entries piling up in debt.json

### Pitfall 5: Heuristic Ambiguity Causing Wrong Track Selection

**What goes wrong:** An entry has both `issue_type: "drift"` AND `formal_ref` pointing to an invariant (e.g., "spec:safety/invariant-consistency"). The heuristic says "drift → parameter update," so it dispatches /qgsd:quick to update the invariant. Wrong.

**Why it happens:** The heuristic isn't checking `isNumericThreshold(formal_ref)`. It only checks `issue_type`.

**How to avoid:**
- Make the heuristic binary:
  1. First check: does `formal_ref` point to a numeric parameter? (e.g., "spec:safety/MCsafety.cfg:MaxDeliberation" or "requirement:PERF-01")
  2. Second check: is `issue_type === 'drift'`?
  3. Only if BOTH are true → parameter update track
  4. Otherwise → investigation track (default safe)
- Add a helper `isNumericThreshold(formal_ref)` that scans the spec file to confirm the target is numeric
- Test the heuristic with mixed cases (drift + invariant, issue + parameter, etc.)

**Warning signs:**
- Quick tasks with titles like "Update safety invariant to match production"
- Invariants changing unexpectedly between solve runs
- Entries with issue_type="issue" getting auto-dispatched

### Dispatching Remediation Tasks

```javascript
// Source: bin/qgsd-solve.cjs (existing sweepRtoD pattern)
function autoClosePtoF(residual) {
  if (residual.p_to_f.residual === 0) return { actions_taken: [] };

  const actions = [];
  const divergent = residual.p_to_f.detail.divergent_entries || [];

  for (const entry of divergent) {
    const isParameter = entry.issue_type === 'drift' && isNumericThreshold(entry.formal_ref);

    if (isParameter) {
      // Parameter update track: dispatch /qgsd:quick
      const result = spawnTool('qgsd:quick', [
        `Update ${entry.formal_ref} to match production`,
        `Production measurement: ${entry.meta.measured_value}`,
        `Current formal: ${entry.formal_expected}`
      ]);

      if (result.ok) {
        actions.push(`Dispatched parameter update for ${entry.id}`);
        transitionDebtEntry(entry, 'resolved'); // Mark as resolved after dispatch
      } else {
        actions.push(`Failed to dispatch parameter update for ${entry.id}`);
        transitionDebtEntry(entry, 'acknowledged'); // Re-eligible for next cycle
      }
    } else {
      // Investigation track: flag for manual review
      actions.push(`Flagged ${entry.id} for investigation (potential regression)`);
      entry.meta = entry.meta || {};
      entry.meta.investigation_notes = `Production diverged from ${entry.formal_ref}. Requires manual review.`;
    }
  }

  return { actions_taken: actions, entries_processed: divergent.length };
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual formal model updates based on production issues | Automated P->F residual detection + targeted /qgsd:quick dispatch | v0.27-05 | Closes the feedback loop; formal model stays in sync with production reality without human intervention for parameters |

**Deprecated/outdated:**
- **Manual formal model parameter updates:** Replaced by automated `/qgsd:quick` dispatch when P->F detects divergence. Humans now review the dispatch tasks, not update parameters directly.
- **Direct requirement-to-code links:** Replaced by debt entries with formal_ref links. More flexible, supports informal debt (GitHub issues, Sentry errors) that don't map to requirements.

## Open Questions

1. **How to extract formal expected values from spec files?**
   - What we know: Entries have `formal_ref` strings like "spec:safety/MCsafety.cfg:MaxDeliberation"
   - What's unclear: The exact file structure and parameter naming in `.formal/spec/` directories
   - Recommendation: Create a helper `extractFormalExpected(formal_ref, specDir)` that:
     - Parses the formal_ref string to extract module and parameter key
     - Loads the corresponding spec file (e.g., `.formal/spec/safety/MCsafety.cfg`)
     - Extracts the parameter value (may need JSON, YAML, or custom parsing depending on spec format)
     - Returns the value or null if not found (fail-open)
   - This is phase 5 discretion; propose implementation during planning

2. **Exact `/qgsd:quick` prompt templates for parameter updates?**
   - What we know: We want to dispatch a quick task with a template like "Update [param] to [value]"
   - What's unclear: What parameters should the quick task receive? Just the formal_ref and measured value, or also rationale?
   - Recommendation: Minimal template during phase 5, extendable later:
     ```
     Update formal parameter ${formal_ref} to ${measured_value}
     (Production measurement: ${production_value}, Current formal: ${formal_expected})
     ```
   - This gives the planner (human) enough context to decide yes/no

3. **Batch processing vs one-by-one dispatch?**
   - What we know: Solve currently dispatches sequentially (one dispatch at a time via spawnSync)
   - What's unclear: Should P->F dispatch all divergent entries in a single quick task, or one task per entry?
   - Recommendation: One task per entry (fine-grained). Easier to reason about, easier to retry if one fails, easier to audit. Batch dispatch can be added in a future phase if performance requires it.

4. **Output format for solve P->F residual summary?**
   - What we know: Existing solve output includes tables per layer (R->F, F->T, etc.)
   - What's unclear: How should P->F summary look? Just count, or also include dispatch details?
   - Recommendation: Follow existing pattern:
     ```
     P->F Residual: 5 divergent entries
     ├─ Parameter updates: 3 dispatched
     ├─ Investigations: 2 flagged
     └─ Skipped (unlinked): 1
     ```
   - Verbose mode (-v) adds details per entry
