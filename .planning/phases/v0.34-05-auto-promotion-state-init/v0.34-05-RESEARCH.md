### Pattern 2: Clean Session Evaluation (VERIFY SEMANTIC SCORE READING)

**What:** `checkCleanSession()` evaluates THREE conditions: wiring scores, semantic scores, and formal checks.

**Current Implementation** (nf-solve.cjs line 3616-3660):
```javascript
function checkCleanSession() {
  const GATE_FILES = {
    A: { file: 'gate-a-grounding.json', wiringKey: 'wiring_evidence_score' },
    B: { file: 'gate-b-abstraction.json', wiringKey: 'wiring_purpose_score' },
    C: { file: 'gate-c-validation.json', wiringKey: 'wiring_coverage_score' },
  };

  const gatesDir = path.join(ROOT, '.planning', 'formal', 'gates');
  const wiring = {};
  const semantic = {};

  for (const [label, cfg] of Object.entries(GATE_FILES)) {
    try {
      const gateData = JSON.parse(fs.readFileSync(path.join(gatesDir, cfg.file), 'utf8'));
      wiring[label] = gateData[cfg.wiringKey] != null ? gateData[cfg.wiringKey] : 0;
      semantic[label] = gateData.semantic_score != null ? gateData.semantic_score : 0;
    } catch (_) {
      wiring[label] = 0;
      semantic[label] = 0;
    }
  }

  // Check formal results — no counterexamples
  let formalPass = true;
  try {
    const checkResultsPath = path.join(ROOT, '.planning', 'formal', 'check-results.ndjson');
    if (fs.existsSync(checkResultsPath)) {
      const lines = fs.readFileSync(checkResultsPath, 'utf8').trim().split('\n').filter(Boolean);
      for (const line of lines) {
        try {
          const entry = JSON.parse(line);
          if (entry.result === 'counterexample') {
            formalPass = false;
            break;
          }
        } catch (_) { /* skip malformed lines */ }
      }
    }
  } catch (_) { /* fail-open */ }

  const wiringClean = wiring.A >= 1.0 && wiring.B >= 1.0 && wiring.C >= 1.0;
  const semanticClean = semantic.A >= 0.8 && semantic.B >= 0.8 && semantic.C >= 0.8;
  const isClean = wiringClean && semanticClean && formalPass;

  process.stderr.write(
    TAG + ' Clean session check: wiring=[A:' + wiring.A + ',B:' + wiring.B + ',C:' + wiring.C +
    '] semantic=[A:' + semantic.A + ',B:' + semantic.B + ',C:' + semantic.C +
    '] formal=' + (formalPass ? 'pass' : 'fail') + ' -> ' + (isClean ? 'CLEAN' : 'NOT_CLEAN') + '\n'
  );

  return isClean;
}
```

**Status:** ✓ Implementation is correct but depends on v0.34-04 completing semantic_score enrichment. The code reads `semantic_score` from gate files and defaults to 0 if missing.

**Verification needed:**
1. Gate files from v0.34-04 actually contain `semantic_score` fields after compute-semantic-scores.cjs runs
2. Threshold values (wiring >= 1.0, semantic >= 0.8) are as intended by requirements

### Anti-Patterns to Avoid

- **Don't:** Hardcode consecutive_clean_sessions in preflight(). **DO:** Initialize to 0 and let checkCleanSession() increment it.
- **Don't:** Skip semantic_score check if field is missing from gate. **DO:** Default to 0 (which fails the >= 0.8 threshold correctly).
- **Don't:** Promote models without 3 consecutive clean sessions recorded. **DO:** Require consecutive_clean_sessions >= 3 in solve-state.json before auto-promotion eligibility.
- **Don't:** Reset consecutive_clean_sessions on cooldown violations. **DO:** Only reset on semantic/wiring regression (isClean == false).

## Common Pitfalls

### Pitfall 1: Missing semantic_score Silently Fails Threshold

**What goes wrong:** If v0.34-04 semantic scoring pipeline does NOT run (e.g., if compute-semantic-scores.cjs is skipped), gate files lack semantic_score fields. checkCleanSession() defaults to 0, so semanticClean becomes false (0 < 0.8), and consecutive_clean_sessions resets to 0 forever.

**Why it happens:** Soft dependency: semantic_score field is optional if not enriched. No error is raised; the session simply fails to be clean.

**How to avoid:**
- Verify v0.34-04 wires compute-semantic-scores.cjs in the solve pipeline
- Add diagnostic logging: "Loaded semantic_score from gates" vs "defaulted to 0"
- Document that auto-promotion requires semantic scoring pipeline (v0.34-04)

**Warning signs:**
- `consecutive_clean_sessions` stays at 0 across multiple solve runs
- "semantic=[A:0, B:0, C:0]" consistently logged in checkCleanSession output
- No promotions occur despite wiring and formal checks passing

### Pitfall 2: State Corruption on Crash

**What goes wrong:** If nf-solve.cjs crashes after incrementing consecutive_clean_sessions but before writing solve-state.json, the counter is lost. On next run, it resets to 0 (from previous file), breaking the sequence.

**Why it happens:** State write happens after promotion attempt; if solve exits abnormally, write is skipped.

**How to avoid:**
- Write solve-state.json early and atomically (current code does this at line 3821)
- Use try-catch with fail-open for state writes (current code does this)
- Test recovery: simulate crash, verify next run reads prior state correctly

**Warning signs:**
- Repeated solve runs show consecutive_clean_sessions not incrementing consistently
- Promotion logs show "clean sessions: 2" then "clean sessions: 0" on next run
- solve-state.json has old timestamps (was not updated)

### Pitfall 3: Flip-Flop Models Block All Promotions

**What goes wrong:** If a model shows flip-flop behavior (promote/demote/promote pattern), gate-stability.cjs flags it. The model is skipped even when consecutive_clean_sessions >= 3.

**Why it happens:** Flip-flop detection prevents unstable models from being promoted, which is correct. But if the wrong threshold is set (DEFAULT_FLIP_FLOP_THRESHOLD = 3 direction changes), legitimate models with 2 ups and 1 down are incorrectly promoted.

**How to avoid:**
- Verify DEFAULT_FLIP_FLOP_THRESHOLD = 3 in gate-stability.cjs (currently correct)
- Understand direction change counting: A→B→C→A has 2 changes, A→B→A has 1 change
- Test with real promotion-changelog.json to ensure flip-flop detection works as intended

**Warning signs:**
- Logs show "SKIP flip-flop: model X" even when manual verification shows it's stable
- Promotion results show total_eligible > 0 but all models skipped for flip-flop
- Check promotion-changelog.json and count: if 3+ direction alternations, it's flagged correctly

### Pitfall 4: Cooldown Prevents Expected Promotions

**What goes wrong:** If a SOFT_GATE model was previously flagged as UNSTABLE, it enters cooldown. Even with consecutive_clean_sessions >= 3, the model is skipped because isCooldownSatisfied() returns false.

**Why it happens:** Cooldown requires BOTH consecutive_stable_sessions >= 3 AND wall time >= 1 hour. If only one is satisfied, the model waits.

**How to avoid:**
- Check gate-stability.cjs isCooldownSatisfied() at line 106-117
- Verify cooldown object structure in per-model-gates.json has both fields
- Test with controlled timestamps: set flagged_at to 1+ hours ago, increment consecutive_stable_sessions

**Warning signs:**
- Logs show "SKIP cooldown: model X" despite clean sessions >= 3
- Model was promoted, then demoted, and is now waiting for re-promotion
- Check per-model-gates.json per-model stability field for cooldown state
