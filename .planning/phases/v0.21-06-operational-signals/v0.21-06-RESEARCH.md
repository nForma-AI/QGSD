## SIG-01: Coverage Gap Detection

### Approach
1. Parse TLC output to extract enumerated states (TLC `-dump` flag or parse "distinct states found" output)
2. Parse conformance traces to extract observed states (from XState replay)
3. Compute set difference: TLC states not seen in traces = coverage gaps
4. Write `formal/coverage-gaps.md` as a structured test backlog

### Key Challenge
TLC state representation is TLA+ model-level (s=0, s=1, s=2 for quorum), while conformance traces use XState event/state names (IDLE, COLLECTING_VOTES, DELIBERATING, DECIDED). Need a state-name mapping layer.

### Design Decision
Use the state mapping that already exists in the TLA+ specs (commented state map in each spec, e.g., `s=0 : COLLECTING_VOTES` in quorum.pm). Extract this mapping programmatically or hardcode the known mappings per spec.

Simpler approach: run TLC with `-dump dot` to get reachable states, then compare against the set of states observed in conformance traces. The gap is "states TLC can reach that we've never seen in production." This is a structural coverage metric.

## SIG-02: Roadmap Petri Net

### Approach
1. Read ROADMAP.md, parse phase entries (regex: `### Phase vN.NN-MM: Name`)
2. Parse `Depends on:` field for each phase
3. Build Petri net: phases are transitions, completion tokens enable downstream phases
4. Compute critical path: longest path through the dependency graph (or phases with most downstream dependents)
5. Add `--roadmap` flag to existing `generate-petri-net.cjs` or create new module

### Design Decision
Extend `generate-petri-net.cjs` with `--roadmap` flag. When set, build a dependency-based Petri net instead of the quorum token-passing net. Export a new pure function `buildRoadmapDot(phases)` alongside existing `buildDot`.

### Critical Path Algorithm
Since roadmap dependencies form a DAG, use topological sort + longest-path-from-source algorithm. Each phase is weight 1 (or weight = plan count). Critical path = sequence of phases where delay cascades most.

## SIG-03: PRISM Priority Ranking

### Approach
1. Read check-results.ndjson for PRISM entries (formalism=prism)
2. For each failure mode: extract P(failure) from PRISM result
3. Assign impact scores based on check_id (e.g., prism:quorum failures have high impact on all quorum-dependent phases)
4. Compute `P(failure) x impact` ranking
5. Output as formatted text for injection into plan-phase.md quorum context

### Impact Assignment
Use a simple ordinal scale:
- `prism:quorum` → impact=10 (affects every planning decision)
- `prism:mcp-availability` → impact=7 (affects quorum slot availability)
- Other PRISM models → impact=5 (default)

### Key Design
The module reads check-results.ndjson (not live PRISM runs), so it's fast and doesn't need Java/PRISM. It's a post-hoc analysis tool.

## SIG-04: PRISM Consensus Probability Gate

### Approach
1. Before each quorum round, read current scoreboard availability rates
2. Plug rates into mcp-availability.pm model (or use cached results from check-results.ndjson)
3. Compute P(consensus_reached) = product of P(available) for majority of slots
4. If P(consensus) < 0.70 threshold, defer round with structured warning
5. This is a gate function, not a CLI tool -- it returns true/false + reason

### Key Challenge
Running PRISM live before every quorum round is too slow (PRISM takes 2-5s). Instead, use a closed-form approximation: given per-slot availability rates, compute the probability of having >= threshold available slots using the binomial distribution (each slot independent).

### Closed-Form Calculation
P(consensus) = P(>=k slots available out of n) = sum_{i=k}^{n} C(n,i) * p^i * (1-p)^(n-i)
where k = min quorum size, n = total slots, p = average slot availability rate.

For heterogeneous rates, use Poisson binomial distribution (exact computation via recursive formula or approximation).

## Pitfalls

1. **TLC state dump format varies by version** -- parse carefully, test with actual TLC output
2. **Petri net rendering** -- reuse existing @hpcc-js/wasm-graphviz pattern from generate-petri-net.cjs
3. **PRISM round-trip latency** -- SIG-04 MUST use closed-form math, not live PRISM invocation
4. **Scoreboard may be empty** -- all tools must handle missing/empty scoreboard gracefully (fail-open)
5. **ROADMAP.md format** -- parse defensively; phase headers may have varying formats
