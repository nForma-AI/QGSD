# Phase v0.38-03: Model Refinement Workflow - Research

**Researched:** 2026-03-18
**Domain:** Formal model refinement loop, bug-context-biased spec generation, 6-phase prescriptive cycle orchestrator
**Confidence:** HIGH

## Summary

This phase delivers the ability for nForma to create or refine formal models when a bug has no matching model or existing models fail to reproduce the failure. It builds on Phase v0.38-01's model discovery (`formal-scope-scan.cjs --bug-mode`), constraint extraction (`model-constrained-fix.cjs`), and model checker infrastructure (`run-tlc.cjs`, `run-alloy.cjs`), plus Phase v0.38-02's debug integration that populates `bug-model-gaps.json` with coverage gaps.

The phase has three deliverables: (1) a `--bug-context` flag for `close-formal-gaps.md` that biases spec generation toward capturing a specific failure mode, (2) a model refinement verification loop that confirms the new/refined model actually FAILS (reproduces the bug) and iterates up to 2 additional times if the model passes, and (3) a `model-driven-fix.md` workflow orchestrator that runs the full 6-phase prescriptive cycle end-to-end (discovery, reproduction, refinement, constraint extraction, constrained fix, pre-verification).

All three deliverables reuse existing infrastructure extensively. `close-formal-gaps.md` already generates specs and runs checkers — `--bug-context` adds a bias mechanism to its Step 5 (Generate Model). The refinement loop wraps the checker execution (Step 6) with pass/fail inversion logic. `model-driven-fix.md` composes existing commands (`formal-scope-scan.cjs`, `close-formal-gaps.md`, `model-constrained-fix.cjs`) into a sequential orchestrator workflow, following the same pattern as `debug.md`.

**Primary recommendation:** Implement in two plans: Plan 01 adds `--bug-context` to `close-formal-gaps.md` and the refinement verification loop (MRF-01 + MRF-02), Plan 02 creates `model-driven-fix.md` as a workflow orchestrator composing all existing tools (MRF-03). Wave 1 for both since Plan 02 depends on Plan 01.

## Architecture Patterns

### Pattern 1: Bug Context Injection into Spec Generation

**What:** `--bug-context` normalizes input (file or inline text) to plain text, then injects it into close-formal-gaps.md's Step 5 (Generate Model) as additional prompt context that biases invariant/assertion generation toward capturing the described failure.

**When to use:** When a bug has no matching formal model or existing models fail to reproduce.

**Integration point in close-formal-gaps.md:**
```
Step 5 — Generate the Formal Model
  Current: generates spec from requirement text + module context
  New: if $BUG_CONTEXT is set, append to generation prompt:
    "This model must capture the following failure mode: {$BUG_CONTEXT}
     Generate invariants/assertions that would FAIL when this bug is present."
```

### Pattern 2: Inverted Verification Loop (Refinement)

**What:** Normal model checking treats "no errors" as success. For bug reproduction, the logic is inverted: "error found" means the model captures the bug (success), "no errors" means the model fails to reproduce (retry needed).

**When to use:** After generating a model that should capture a known bug.

**Loop structure:**
```
for attempt in 1..3:
  run_checker(model)
  if checker_found_error:
    # Model reproduces the bug — SUCCESS
    return { status: 'reproduced', attempt, counterexample }
  else:
    # Model passes — does NOT capture the failure
    if attempt < 3:
      display "still passes — retry {attempt} of 2"
      refine_model(model, checker_output, bug_context)
    else:
      return { status: 'not_reproduced', attempts: 3 }
```

### Pattern 3: Prescriptive 6-Phase Orchestrator

**What:** `model-driven-fix.md` follows the same orchestrator pattern as `debug.md` — a markdown workflow with numbered steps that call existing tools.

**Phases:**
```
Phase 1 — Discovery
  formal-scope-scan.cjs --bug-mode --description "$BUG_DESC" --files "$AFFECTED_FILES"
  → existing_models[], coverage_status

Phase 2 — Reproduction
  if existing_models:
    formal-scope-scan.cjs --bug-mode --run-checkers --description "$BUG_DESC"
    → reproduction_status (reproduced/not_reproduced)
  else:
    → skip to Phase 3

Phase 3 — Refinement (MRF-01 + MRF-02)
  close-formal-gaps.md --bug-context "$BUG_CONTEXT" [--formalism=X]
  → new/refined model with inverted verification loop
  → reproduced_model_path

Phase 4 — Constraint Extraction
  model-constrained-fix.cjs --spec "$reproduced_model_path" --max-constraints 3
  → constraints[] (English summaries)

Phase 5 — Constrained Fix
  Apply constraints as guidance for code fix
  (orchestrator presents constraints, developer or quorum applies fix)

Phase 6 — Pre-Verification
  Run ONLY the new/refined model against the fixed code
  → verify model NOW PASSES (bug is fixed)
```

### Anti-Patterns to Avoid
- **Running all models in pre-verification:** User decision explicitly limits pre-verification to the new/refined model only. Cross-model regression is deferred.
- **Treating model pass as success in refinement:** In the refinement loop, a passing model is a FAILURE — it means the model doesn't capture the bug.
- **Generating models without bug context:** If `--bug-context` is provided, it MUST bias the spec generation. A generic model that doesn't account for the failure mode will likely pass (not reproduce).

## Common Pitfalls

### Pitfall 1: Inverted Verification Logic Confusion
**What goes wrong:** Developer implements refinement loop treating "no errors" as success (normal verification semantics) instead of treating "error found" as success (bug reproduction semantics).
**Why it happens:** Model checking convention is that errors are bad. In refinement, errors mean the model captures the bug.
**How to avoid:** Name the function clearly (e.g., `verifyBugReproduction`) and document the inverted semantics. Test both paths explicitly.
**Warning signs:** Refinement loop exits on first attempt with "reproduced" when the model actually passes.

### Pitfall 2: Bug Context Not Reaching Spec Generation
**What goes wrong:** `--bug-context` is parsed and normalized but not actually injected into the spec generation prompt in Step 5.
**Why it happens:** close-formal-gaps.md has multiple code paths (TLA+, Alloy, PRISM, Petri). Bug context must be injected in ALL formalism-specific generation paths.
**How to avoid:** Add bug context injection at the shared prompt construction level (before formalism dispatch), not in each formalism handler.
**Warning signs:** Generated spec has no reference to the failure mode described in `--bug-context`.

### Pitfall 3: Refinement Loop Modifying Existing Models
**What goes wrong:** The refinement loop modifies an existing working model instead of creating a new one or a refined copy.
**Why it happens:** close-formal-gaps.md constraint says "NEVER modify existing formal models — only create new ones."
**How to avoid:** Refinement iterations should create versioned copies (e.g., `Model_v2.tla`) or new models with distinct names.
**Warning signs:** Existing model checker tests start failing after refinement runs.

### Pitfall 4: Pre-Verification Running Wrong Model
**What goes wrong:** Phase 6 (pre-verification) runs all models or the original model instead of the new/refined model.
**Why it happens:** Existing `run-formal-verify.cjs` defaults to running all discovered models.
**How to avoid:** Use `run-tlc.cjs` / `run-alloy.cjs` directly with the specific model path, not the batch verification tool.
**Warning signs:** Pre-verification passes trivially because it ran the original (non-reproducing) model.

## Code Examples

### Bug Context Normalization

```javascript
// Normalize --bug-context: inline text or file path
function normalizeBugContext(value) {
  if (!value || typeof value !== 'string') return '';
  const trimmed = value.trim();
  // Auto-detect: if it's a valid file path, read contents
  if (fs.existsSync(trimmed)) {
    return fs.readFileSync(trimmed, 'utf-8').trim();
  }
  // Otherwise treat as inline text
  return trimmed;
}
```

### Inverted Verification Loop

```javascript
// Refinement loop: verify model FAILS (reproduces bug)
async function verifyBugReproduction(modelPath, bugContext, maxAttempts) {
  maxAttempts = maxAttempts || 3;
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    const result = await runChecker(modelPath);
    if (result.hasError) {
      // Model found a violation — it reproduces the bug
      return { status: 'reproduced', attempt, counterexample: result.trace };
    }
    // Model passes — does NOT capture the failure
    if (attempt < maxAttempts) {
      console.log(`still passes — retry ${attempt} of ${maxAttempts - 1}`);
      modelPath = await refineModel(modelPath, result.output, bugContext);
    }
  }
  return { status: 'not_reproduced', attempts: maxAttempts };
}
```

No new formal invariants are needed for the agent-loop module, but the refinement loop's bounded nature should be validated in tests.

## Open Questions

1. **Refinement strategy between iterations**
   - What we know: Between failed reproduction attempts, the model needs to be refined to better capture the failure.
   - What's unclear: The exact refinement prompt — should it include the full checker output, just the state space summary, or the specific states explored?
   - Recommendation: Include checker summary (state count, depth) and the original bug context. The planner should specify the exact prompt template. Start with summary + bug context; if reproduction rates are low, escalate to full checker output.

2. **Model naming for refined versions**
   - What we know: close-formal-gaps.md constraint says "NEVER modify existing formal models."
   - What's unclear: Naming convention for refined models (suffix `_v2`, `_bugfix`, timestamp?).
   - Recommendation: Use `_bugN` suffix where N is the bug_id from bug-model-gaps.json (e.g., `NFDispatch_bug3a2f.tla`). This links the model to the specific bug and avoids collisions.
