# Phase v0.38-03: Model Refinement Workflow - Context

**Gathered:** 2026-03-18
**Status:** Ready for planning

<domain>
## Phase Boundary

When a bug has no matching formal model or the model fails to reproduce, nForma can create or refine a model that captures the failure mode through a prescriptive 6-phase cycle (discovery, reproduction, refinement, constraint extraction, constrained fix, pre-verification). This phase delivers `model-driven-fix.md` orchestrator, `--bug-context` flag for `close-formal-gaps.md`, and a refinement verification loop.

</domain>

<decisions>
## Implementation Decisions

### Refinement iteration feedback
- Show summary verdicts by default between refinement iterations: `reproduced` / `still passes` / `retry 1 of 2` with a short reason
- Full model checker output is opt-in via verbose flag (not shown by default)
- Pointer to full checker details included in summary for debugging

### Bug context input format
- `--bug-context` accepts inline text (string) or file path (auto-detected via file existence check)
- Normalized internally to plain text before biasing spec generation
- No structured JSON schema required — maintainer workflow speed over schema authoring

### Model formalism selection
- Auto-select formalism (TLA+/Alloy/PRISM) based on bug type and module characteristics
- Override flag `--formalism=TLA+|alloy|prism` available for explicit selection
- Matches existing `close-formal-gaps` heuristic selection pattern

### Pre-verification scope
- Run only the new/refined model during pre-verification phase
- Broader regression testing across all module models is deferred to cross-model work
- Keeps refinement loop fast and phase boundaries clear

### Claude's Discretion
- Exact heuristic for formalism auto-selection (TLA+ vs Alloy vs PRISM mapping)
- Internal text normalization approach for `--bug-context` file vs inline
- Progress display format within the 6-phase cycle
- Error messages and retry prompts between refinement iterations

</decisions>

<specifics>
## Specific Ideas

- Refinement loop verifies the new model actually FAILS (reproduces the bug) — a passing model means the model doesn't capture the failure mode and needs iteration
- Up to 2 additional refinement iterations if model incorrectly passes (total 3 attempts)
- Follows existing fail-open, auto-select-with-override semantic pattern used throughout nForma
- `close-formal-gaps.md` already has spec generation infrastructure — `--bug-context` biases this existing pipeline

</specifics>
