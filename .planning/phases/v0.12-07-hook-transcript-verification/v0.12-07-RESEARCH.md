## Summary

Phase v0.12-07 formally verifies four invariants in the `getCurrentTurnLines` and `wasSlotCalledSuccessfully` algorithms in `hooks/qgsd-stop.js` using Alloy 6. The algorithm reads a JSONL transcript, scans backward to find the last human-authored user message (distinguishing it from tool_result-only user messages), then scans forward across all entries in the current turn to pair tool_use blocks with their tool_result responses and count successful (non-error) results. The four invariants to verify are: `BoundaryCorrect` (the backward scan correctly identifies the last human message), `PairingUnique` (each tool_use_id matches at most one tool_result), `NoDuplicateCounting` (no tool_result is counted more than once toward successCount), and `CeilingEnforced` (successCount never exceeds minSize).

The project pattern for this class of work is fully established by v0.12-03 through v0.12-06: a handwritten Alloy 6 `.als` spec models the algorithm as abstract sigs and predicates/assertions; a Node.js CJS runner (`bin/run-transcript-alloy.cjs`) invokes the Alloy JAR headless using `spawnSync`; a `node:test` test file (`bin/run-transcript-alloy.test.cjs`) covers error paths without requiring Java. Three plans follow the wave structure: Wave 0 RED stubs (Plan 01), Alloy spec authoring (Plan 02), runner implementation + GREEN tests (Plan 03).

The key modeling challenge for this phase is representing an ordered JSONL sequence in Alloy 6, which uses sets and relations by default (no built-in sequence order). The project's existing specs use integer-indexed ordering or `seq` (the Alloy built-in sequence). For transcript-scan, the natural model is a linear ordered sequence of `Entry` atoms with a `next` relation or integer `idx` field. The `wasSlotCalledSuccessfully` algorithm's two-pass structure (collect IDs → match results) maps cleanly to relational Alloy: a relation `paired: ToolUse -> lone ToolResult` captures uniqueness; a set `counted: set ToolResult` captures non-double-counting.

**Primary recommendation:** Model the transcript as an ordered sequence using an integer `idx` field on `Entry` (not Alloy's `seq` builtin, which has scope limitations in Alloy 6). Use `fact` constraints to enforce ordering uniqueness and contiguity. Write the four predicates as `assert` + `check` following the scoreboard-recompute.als and availability-parsing.als pattern exactly.

### Pattern 1: Alloy 6 Ordered Sequence via Integer Index

**What:** Represent the transcript as a set of `Entry` atoms each with an integer `idx` field. Facts enforce uniqueness and contiguity. Scanning direction (backward for boundary, forward for pairing) is expressed using `<`, `>` comparisons on `idx`.

**When to use:** Any algorithm that reads a JSONL file in order. The `idx` approach avoids Alloy's `seq` builtin which has worse bounded scope in Alloy 6.

**Example:**
```alloy
-- Source: pattern derived from scoreboard-recompute.als (indexed sets)
-- and project Alloy 6 conventions

module transcript_scan

abstract sig Entry {
  idx: one Int
}

sig HumanMessage extends Entry {}
sig ToolUse extends Entry {
  useId: one Id
}
sig ToolResult extends Entry {
  resultId: one Id,
  isError: one Bool
}

abstract sig Id {}
abstract sig Bool {}
one sig True, False extends Bool {}

-- All entries have unique indices (no two entries share the same slot)
fact UniqueIndices {
  all disj e1, e2: Entry | e1.idx != e2.idx
}

-- Contiguous: no gaps in index sequence (0..N-1)
fact ContiguousIndices {
  Entry.idx = { i: Int | 0 <= i and i < #Entry }
}
```

### Pattern 2: Backward Scan for Last Human Message Boundary

**What:** `BoundaryCorrect` — the last human message boundary is the highest-indexed `HumanMessage` entry. Everything from that index onward is the "current turn." No tool_result-only user entry is treated as the boundary.

**When to use:** Modeling `getCurrentTurnLines` in qgsd-stop.js.

**Example:**
```alloy
-- Source: mirrors getCurrentTurnLines logic in hooks/qgsd-stop.js lines 58-72

-- The boundary index: the maximum idx among all HumanMessage entries
fun boundaryIdx : Int {
  max[HumanMessage.idx]
}

-- BoundaryCorrect: boundary is the LAST human message (highest idx)
-- No HumanMessage after the boundary exists
pred BoundaryCorrect {
  no h: HumanMessage | h.idx > boundaryIdx
}

assert BoundaryCorrectCheck {
  -- If at least one HumanMessage exists, boundary points to the last one
  some HumanMessage => BoundaryCorrect
}
```

### Pattern 3: Pairing Uniqueness (tool_use → tool_result)

**What:** Each `ToolUse` atom pairs with at most one `ToolResult` via matching `Id`. Each `ToolResult` pairs with at most one `ToolUse`. This is a functional mapping in both directions — no fan-out, no fan-in.

**When to use:** Modeling the ID-matching logic in `wasSlotCalledSuccessfully` (lines 199-239 of qgsd-stop.js).

**Example:**
```alloy
-- Source: mirrors wasSlotCalledSuccessfully two-pass ID matching

-- A pairing relates each ToolUse to at most one ToolResult with matching ID
-- and each ToolResult to at most one ToolUse
pred PairingUnique {
  -- Each tool_use_id matches at most one tool_result
  all tu: ToolUse | lone tr: ToolResult | tr.resultId = tu.useId
  -- Each tool_result matches at most one tool_use (no fan-in)
  all tr: ToolResult | lone tu: ToolUse | tr.resultId = tu.useId
}

assert PairingUniqueCheck {
  PairingUnique
}
```

### Pattern 4: No Double-Counting

**What:** `NoDuplicateCounting` — the successCount is the cardinality of the set of non-error `ToolResult` atoms that are paired (have a matching ToolUse). Each such result is counted exactly once. Since it is a set cardinality, duplicates cannot exist by construction.

**When to use:** Modeling the `successCount++` loop in qgsd-stop.js line 451.

**Example:**
```alloy
-- Successful results: paired non-error ToolResult entries in the current turn
fun successfulResults : set ToolResult {
  { tr: ToolResult | tr.isError = False and (some tu: ToolUse | tr.resultId = tu.useId) }
}

pred NoDuplicateCounting {
  -- Set cardinality = each result counted exactly once
  -- (Alloy sets are by definition duplicate-free)
  #successfulResults = #successfulResults  -- tautological but explicit contract
}

assert NoDuplicateCountingCheck {
  NoDuplicateCounting
}
```

### Pattern 5: Ceiling Enforcement

**What:** `CeilingEnforced` — `successCount` (the number of successful results) never exceeds `minSize`. In the spec, model `minSize` as a one-sig with an integer field.

**When to use:** Modeling the early-exit ceiling check (`if (successCount >= minSize) break`) at qgsd-stop.js line 452.

**Example:**
```alloy
one sig Config {
  minSize: one Int
}

-- successCount is the count of successful non-error paired results
fun successCount : Int {
  #successfulResults
}

pred CeilingEnforced {
  successCount <= Config.minSize
}

assert CeilingEnforcedCheck {
  -- If minSize is positive, successCount does not overflow it
  Config.minSize > 0 => CeilingEnforced
}
```

### Anti-Patterns to Avoid

- **Using Alloy `seq` for the ordered sequence:** `seq` in Alloy 6 has bounded scope that competes with other scope allocations. Use integer `idx` fields with `fact` constraints instead.
- **Modeling tool_use_id and tool_result_id as string-valued:** Alloy does not have strings. Use abstract sig `Id` atoms — the pairing relation is purely structural (same atom = same ID).
- **Asserting `NoDuplicateCounting` via counting twice:** The Alloy set is already duplicate-free; the assertion should verify the count equals the cardinality of the set, not re-enumerate.
- **Running the runner without JAVA_HOME gate:** All Alloy runners in this project are gated on JAVA_HOME being set AND valid Java >=17 being found. `npm test` must pass even when Java is not installed (tests skip the Java-dependent paths).
- **Putting `run-transcript-alloy.test.cjs` in package.json `scripts.test` before the file exists:** The Wave 0 plan creates the file first (even if stubbed), then Plan 03 makes it GREEN. The package.json `scripts.test` entry must be added by Plan 01 after the stub file is created.

## Common Pitfalls

### Pitfall 1: Alloy 6 Integer Scope Overflow

**What goes wrong:** Alloy 6 uses bounded integers (bitwidth). With `idx` fields counting up to N entries, the bitwidth must be large enough to represent N-1. If `check ... for 5 Entry, 4 Int` is used (4-bit = [-8..7]), expressions involving `#Entry` (up to 5) and arithmetic on idx values fit. But if the scope is raised without raising bitwidth, Alloy will produce spurious counterexamples from integer overflow.

**Why it happens:** Alloy 6 integer arithmetic is modular (bitwidth-bounded). `sum` and cardinality expressions over large sets overflow silently.

**How to avoid:** Match scope declarations to bitwidth: `check BoundaryCorrect for 5 Entry, 4 Int` allows idx values 0..4 within [-8..7]. Follow the pattern from scoreboard-recompute.als (`7 Int` for 7-bit integers when scores reach ±7).

**Warning signs:** Alloy finds a counterexample immediately with a trivially small structure. Inspect the counterexample — if idx values are negative or wrap around, increase bitwidth.

### Pitfall 2: Confusing Alloy's `lone` with `one`

**What goes wrong:** `lone tr: ToolResult | ...` means "at most one" (zero or one). `one tr: ToolResult | ...` means "exactly one." For `PairingUnique`, a tool_use may have no matching tool_result (e.g., result not yet received). Using `one` instead of `lone` would make unpaired tool_use entries violate the predicate spuriously.

**Why it happens:** The English "each tool_use_id matches at most one tool_result" maps to `lone`, not `one`.

**How to avoid:** Use `lone` for all pairing predicates. Reserve `one` only for exactly-one cardinality constraints in sigs and facts.

### Pitfall 3: HumanMessage vs. ToolResult User Entries

**What goes wrong:** In the JSONL transcript, both human messages and tool_result batches have `type: "user"`. The algorithm distinguishes them via `isHumanMessage` (checks for text content blocks vs. tool_result-only content). In the Alloy model, failing to distinguish these types leads to a `BoundaryCorrect` predicate that cannot be falsified — any user entry would qualify as a boundary.

**Why it happens:** The JS type system does not help — both use `type: "user"`. The Alloy model must explicitly represent this distinction as separate sig types: `HumanMessage extends Entry` vs. `ToolResult extends Entry`.

**How to avoid:** Make `HumanMessage` and `ToolResult` disjoint sig subtypes of `Entry`. Never conflate them in the model. The sig hierarchy is specified exactly in the success criteria: `Entry`, `ToolUse extends Entry`, `ToolResult extends Entry`, `HumanMessage extends Entry`.

### Pitfall 4: npm test Adding Runner Before Test File Exists

**What goes wrong:** If `run-transcript-alloy.test.cjs` is added to `package.json scripts.test` before the file is created, `npm test` will fail with a "module not found" error on all subsequent runs.

**Why it happens:** Plan 01 creates the test stub (Wave 0 RED). Plan 03 implements the runner. The package.json update must happen in Plan 01 (when the stub exists) or Plan 03 (when both files exist). Never in between.

**How to avoid:** Plan 01 creates both `bin/run-transcript-alloy.test.cjs` AND updates `package.json scripts.test` to include it. This matches the run-audit-alloy pattern: the test file was created in v0.12-06 Plan 01 and added to scripts.test at the same time.

### Pitfall 5: Alloy `fun` vs `pred` for Computed Values

**What goes wrong:** `fun successCount : Int { #successfulResults }` returns an integer. If this is mistakenly defined as a `pred`, it becomes a boolean and `#` (cardinality) cannot be used directly in assertions.

**Why it happens:** Alloy's `fun` and `pred` have different return semantics. `fun` returns a value (set, relation, integer). `pred` returns truth.

**How to avoid:** Use `fun` for computed values (`boundaryIdx`, `successCount`, `successfulResults`). Use `pred` for boolean conditions. Use `assert` + `check` for the four invariants that must hold universally.

## Open Questions

1. **Alloy 6 ordered sequence: `idx` field vs. `util/ordering`**
   - What we know: Alloy 6 ships `util/ordering` module which provides a total order on a sig. Previous project specs use integer fields (scoreboard-recompute uses set iteration, not ordering).
   - What's unclear: Whether `open util/ordering[Entry]` is cleaner than an integer `idx` field for the transcript model.
   - Recommendation: Use integer `idx` field. The `util/ordering` module adds verbosity and the integer approach is already established in this project. If scope limits become an issue, `util/ordering` can be evaluated then.

2. **Scope for `check` statements**
   - What we know: scoreboard-recompute uses `5 Round, 5 Model, 7 VoteCode, 7 Int`. Transcript scan needs Entry (5 entries covers most real scenarios), Id (5 IDs), Bool (2 values), Int (4 bits = [-8..7] covers indices 0..4 and small counts).
   - What's unclear: Whether 5 entries is sufficient to catch boundary bugs or if 8 entries would expose more edge cases.
   - Recommendation: Use `5 Entry, 5 Id, 2 Bool, 4 Int` as baseline. The planner should document the scope rationale in Plan 02.

3. **CeilingEnforced: modeling the `minSize` break**
   - What we know: In the JS code, the loop breaks when `successCount >= minSize`. The Alloy model cannot directly model imperative break — it must model the invariant as a property of the final state.
   - What's unclear: Whether `CeilingEnforced` should assert `successCount <= Config.minSize` (strong invariant) or `successCount = min[#successfulResults, Config.minSize]` (exact match).
   - Recommendation: Assert `successCount <= Config.minSize` with `Config.minSize > 0` as a precondition. This is the weaker but more useful invariant — it verifies the ceiling is never violated regardless of input.
