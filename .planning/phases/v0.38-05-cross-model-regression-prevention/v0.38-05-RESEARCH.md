# Phase v0.38-05: Cross-Model Regression Prevention - Research

**Researched:** 2026-03-18
**Domain:** Formal verification pipeline — cross-model regression detection
**Confidence:** HIGH

## Summary

Phase v0.38-05 is the final phase of milestone v0.38, tying together all prior phases into a complete formal-model-backed verification pipeline. The core deliverable is extending the model-driven-fix workflow (Phase 5) to verify fixes against proximity-neighbor models before declaring a fix complete, adding scoped verification to run-formal-verify.cjs, and enriching bug-model-gaps.json with post-fix verification results.

The implementation builds on three existing infrastructure pieces: (1) proximity-index.json — a graph database with nodes keyed as `type::id` (e.g., `formal_module::account-manager`) connected by typed edges (`owned_by`, `described_by`, etc.), already populated by `/nf:proximity`; (2) run-formal-verify.cjs — the master verification runner that discovers models dynamically from `.planning/formal/{tla,alloy,prism,petri}/` and supports `--only=` filtering by tool type; (3) bug-model-gaps.json — currently an empty entries array with schema version 1.0, populated by Phase v0.38-02's debug integration (Step G gap persistence).

**Primary recommendation:** Implement a neighbor-resolution utility function that traverses proximity-index.json's graph to find 2-hop formal_module neighbors, then wire it into model-driven-fix.md Phase 5 and run-formal-verify.cjs's new `--scope` flag. Keep the architecture modular: neighbor resolution is a pure function, scoped verification is a CLI flag, and post-fix results are a JSON schema extension.

### Pattern 1: Graph BFS for 2-hop Neighbor Resolution
**What:** Breadth-first search from a formal_module node, following edges through intermediate nodes (concepts, code files) to find other formal_module nodes reachable within 2 hops.
**When to use:** When model-driven-fix Phase 5 needs to identify which neighbor models to verify.
**Key detail:** proximity-index.json nodes are keyed as `type::id`. Edges have `to` (target node key), `rel` (relationship type), and `source` fields. A 2-hop path from `formal_module::A` might go through `concept::X` to reach `formal_module::B`.

```javascript
// Pseudo-code for 2-hop neighbor resolution
function resolveNeighbors(index, startModuleId, maxHops = 2, maxNeighbors = 10) {
  const startKey = `formal_module::${startModuleId}`;
  const visited = new Set([startKey]);
  let frontier = [startKey];
  const neighbors = [];

  for (let hop = 0; hop < maxHops; hop++) {
    const nextFrontier = [];
    for (const nodeKey of frontier) {
      const node = index.nodes[nodeKey];
      if (!node || !node.edges) continue;
      for (const edge of node.edges) {
        if (!visited.has(edge.to)) {
          visited.add(edge.to);
          nextFrontier.push(edge.to);
          if (edge.to.startsWith('formal_module::')) {
            neighbors.push({ id: edge.to.replace('formal_module::', ''), hop: hop + 1 });
          }
        }
      }
    }
    frontier = nextFrontier;
  }

  // Also check reverse edges (nodes pointing TO startKey)
  // ... (bidirectional traversal needed since edges are directional)

  // Cap at maxNeighbors, sorted by proximity score if available
  return neighbors.slice(0, maxNeighbors);
}
```

### Pattern 2: Scoped Model Filtering in run-formal-verify.cjs
**What:** Filter the dynamically discovered STEPS array to only include models whose id matches the `--scope` list.
**When to use:** When `--scope=model1,model2` is passed to run-formal-verify.cjs.
**Key detail:** Step ids are formatted as `tla:configname`, `alloy:specname`, `prism:modelname`. The scope flag should accept model names without the tool prefix for convenience, matching against the id suffix.

```javascript
// In run-formal-verify.cjs CLI filter section
const scopeArg = argv.find(a => a.startsWith('--scope='));
const scope = scopeArg ? scopeArg.split('=')[1].split(',') : null;

// Filter: if --scope given, only include steps whose model name matches
if (scope) {
  steps = steps.filter(s => {
    const modelName = s.id.split(':')[1]; // e.g., "mcsafety" from "tla:mcsafety"
    return scope.some(m => modelName === m.toLowerCase() || s.id === m.toLowerCase());
  });
}
```

### Anti-Patterns to Avoid
- **Running full suite when only scoped verification needed:** `--scope` flag exists precisely to avoid 36+ step verification when only 3-5 neighbor models need checking.
- **Blocking fix on neighbor regressions by default:** User decision locks fail-open pattern. Only `--strict` mode blocks.
- **Hardcoding neighbor list:** Always resolve from proximity-index.json at runtime. The graph changes as models are added.
- **Deep traversal beyond 2 hops:** Success criteria specifies exactly 2-hop. Deeper traversal explodes the neighbor set without proportional value.

## Common Pitfalls

### Pitfall 1: Unidirectional Edge Traversal
**What goes wrong:** proximity-index.json edges are directional (A->B doesn't imply B->A). A naive BFS following only outgoing edges will miss neighbors reachable via reverse edges.
**Why it happens:** Graph structures often assume bidirectional traversal but store edges unidirectionally.
**How to avoid:** Build a reverse edge index at load time: for each edge `A->B`, also index `B->A` for traversal purposes. Only formal_module nodes at the end of traversal count as neighbors.
**Warning signs:** Neighbor count is suspiciously low (0-1) for well-connected models.

### Pitfall 2: Scope Flag vs Only Flag Confusion
**What goes wrong:** `--only=tla` filters by tool TYPE; `--scope=model1,model2` filters by model NAME. Mixing semantics or allowing both simultaneously creates ambiguous behavior.
**Why it happens:** Both are filter flags on the same step list.
**How to avoid:** Define clear precedence: `--scope` takes priority over `--only`. If both present, filter by scope first (model names), then by only (tool type). Document this interaction.
**Warning signs:** User passes `--scope=MCsafety --only=alloy` and gets empty results.

### Pitfall 3: Missing proximity-index.json
**What goes wrong:** If `/nf:proximity` hasn't been run or the file is missing, neighbor resolution fails.
**Why it happens:** proximity-index.json is generated by a separate workflow, not guaranteed to exist.
**How to avoid:** Fail-open: if file missing or malformed, log warning, skip neighbor verification, set `neighbor_models_pass: null` (inconclusive). Suggest running `/nf:proximity` to populate.
**Warning signs:** post_fix_verification always shows 0 neighbors.

### Pitfall 4: Stale Proximity Data
**What goes wrong:** proximity-index.json was generated before new models were added. New models won't appear as neighbors.
**Why it happens:** The graph is a point-in-time snapshot, not live-updated.
**How to avoid:** Check `generated` timestamp in proximity-index.json. If older than 7 days, emit a warning suggesting re-run of `/nf:proximity`. Do NOT auto-regenerate (expensive operation).
**Warning signs:** Known related models don't appear in neighbor lists.

## Code Examples

### Neighbor Resolution Entry Point
```javascript
#!/usr/bin/env node
'use strict';
// bin/resolve-proximity-neighbors.cjs
// Resolves 2-hop formal_module neighbors from proximity-index.json
//
// Usage:
//   node bin/resolve-proximity-neighbors.cjs --model=account-manager [--max-hops=2] [--max-neighbors=10]
//   node bin/resolve-proximity-neighbors.cjs --model=oscillation --format=json
//
// Output: JSON array of { id, hop_distance, path_description }

const fs = require('fs');
const path = require('path');

const DEFAULT_MAX_HOPS = 2;
const DEFAULT_MAX_NEIGHBORS = 10;

function loadProximityIndex(root) {
  const indexPath = path.join(root, '.planning', 'formal', 'proximity-index.json');
  if (!fs.existsSync(indexPath)) return null;
  return JSON.parse(fs.readFileSync(indexPath, 'utf8'));
}

function buildReverseEdges(nodes) {
  const reverse = {}; // nodeKey -> [{ from, rel, source }]
  for (const [key, node] of Object.entries(nodes)) {
    if (!node.edges) continue;
    for (const edge of node.edges) {
      if (!reverse[edge.to]) reverse[edge.to] = [];
      reverse[edge.to].push({ from: key, rel: edge.rel, source: edge.source });
    }
  }
  return reverse;
}
```

## Open Questions

1. **Proximity score calculation**
   - What we know: User decision says "take the 10 with highest proximity score" when capping
   - What's unclear: proximity-index.json edges don't have explicit numeric scores — they have relationship types
   - Recommendation: Use hop distance as primary sort (1-hop before 2-hop), then edge count as tiebreaker (more shared connections = higher implicit proximity). This is Claude's discretion area per CONTEXT.md.

2. **Bidirectional edge traversal completeness**
   - What we know: Edges are stored directionally in the JSON
   - What's unclear: Whether all meaningful relationships are captured in both directions by `/nf:proximity`
   - Recommendation: Build reverse index at load time (documented above). This ensures full traversal coverage regardless of edge direction in source data.
