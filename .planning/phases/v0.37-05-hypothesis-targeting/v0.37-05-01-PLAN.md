<objective>
Add hypothesis-driven priority weighting to solve-wave-dag.cjs so that layers with recent hypothesis transitions (UNMEASURABLE -> CONFIRMED/VIOLATED) execute first within their wave.

Purpose: Currently computeWaves orders layers purely by dependency topology. This plan adds a secondary sort key (priority weight) so layers that have recent hypothesis evidence get remediated first, creating a feedback loop between formal verification and remediation dispatch.

Output: bin/hypothesis-layer-map.cjs (mapping + transition detection + weight computation), bin/hypothesis-layer-map.test.cjs (unit tests), extended bin/solve-wave-dag.cjs with priority-weighted intra-wave sorting, bin/solve-wave-dag.test.cjs (DAG priority tests).
</objective>

  <action>
Create `bin/hypothesis-layer-map.cjs` as a Node.js module with three exported functions:

**1. `mapSourceToLayer(sourceModel)`** — Map hypothesis source_model paths to DAG layer keys:
- Extract the TLA+ filename from the source_model path (e.g., `NFDeliberation.tla` -> `deliberation`)
- Use a keyword-to-layer mapping table:
  ```
  deliberation -> [deliberation-revision]  (not a DAG layer, skip)
  quorum -> []  (no direct DAG layer mapping)
  oscillation -> [c_to_r]  (oscillation detection is correlation-to-residual)
  safety -> [hazard_model]
  session -> [git_heatmap]  (session persistence uses git)
  solve -> [l1_to_l3]  (solve convergence maps to L1->L3)
  recruit -> [per_model_gates]  (recruiting maps to per-model gates)
  ```
- Return array of matching layer keys (may be empty for unmappable models)
- Use case-insensitive filename matching

**2. `loadHypothesisTransitions(root, previousMeasurementsPath)`** — Detect transitions:
- Read current measurements from `.planning/formal/evidence/hypothesis-measurements.json`
- Read previous measurements from `previousMeasurementsPath` (default: `.planning/formal/evidence/hypothesis-measurements.prev.json`)
- For each hypothesis present in both: compare verdict
- A "transition" is when verdict changes from UNMEASURABLE to CONFIRMED or VIOLATED
- Return array of `{ assumption_name, source_model, from_verdict, to_verdict, layer_keys }` for each transition found
- Fail-open: return empty array if either file missing

**3. `computeLayerPriorityWeights(transitions)`** — Convert transitions to weights:
- For each transition, add +1 to each mapped layer key
- Return object `{ [layerKey]: weightDelta }` (only keys with delta > 0)
- Multiple transitions mapping to the same layer are additive

**CLI mode (if require.main):**
- `node bin/hypothesis-layer-map.cjs` — print transitions and computed weights
- `node bin/hypothesis-layer-map.cjs --json` — JSON output

**Module exports:**
```javascript
module.exports = { mapSourceToLayer, loadHypothesisTransitions, computeLayerPriorityWeights };
```

function computeWaves(residualVector, priorityWeights = {})

**Changes to wave computation:**

After the topological sort assigns wave numbers (existing logic unchanged), add a secondary sort step within each wave group:

1. In the "Group by wave number" block (around line 98-102), after collecting layers into `waveGroups`:
2. For each wave group, sort layers by `(priorityWeights[layer] || 0)` descending, with alphabetical as tiebreaker (preserving current deterministic sort behavior when no weights are provided)
3. The existing `.sort()` on line 109 becomes: `.sort((a, b) => (priorityWeights[b] || 0) - (priorityWeights[a] || 0) || a.localeCompare(b))`

**This means:**
- Layers with higher priority weight appear first in the wave's layers array
- When parallel execution picks from the array in order, higher-priority layers start first
- When all weights are 0 (no hypothesis transitions), behavior is identical to current alphabetical sort
- The dependency topology (which wave a layer belongs to) is NEVER changed by priority weights -- only intra-wave ordering changes
