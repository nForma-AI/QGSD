## Summary

Phase v0.12-05 requires two new TLA+ specifications: `QGSDDeliberation.tla` (models the R3 deliberation + R3.6 improvement iteration protocol) and `QGSDPreFilter.tla` (models the R4 pre-filter protocol auto-resolution soundness and termination). Both feed into `MCdeliberation.cfg` and `MCprefilter.cfg`, which TLC verifies. A runner `bin/run-protocol-tlc.cjs` invokes TLC for both configs, is gated on `JAVA_HOME`, and is tested via `bin/run-protocol-tlc.test.cjs` with error-path tests. This follows the exact same 3-plan pattern established by v0.12-04.

The protocols being modeled live entirely in CLAUDE.md R3 and R4. They are finite-round bounded loops with explicit caps, making them excellent candidates for TLA+ bounded termination proofs. The key modeling challenge for GAP-2 is capturing the interaction between the deliberation counter and the improvement iteration counter: they are separate variables, but the combined bound is `deliberationRound + improvementIteration ≤ 20`. The regression rule (APPROVE→BLOCK restarts the blocker resolution path) must also be modeled — it does NOT reset the counters, it transitions `voteState` to a BLOCK phase and resumes deliberation. For GAP-6, the challenge is encoding "auto-resolved iff all models agree + same answer" as a deterministic predicate with the 3-round cap.

The existing TLA+ infrastructure and runner/test pattern from v0.12-04 (`QGSDOscillation.tla`, `MCoscillation.cfg`, `run-oscillation-tlc.cjs`, `run-oscillation-tlc.test.cjs`) provides exact templates for all four deliverables. This phase is a direct structural clone of v0.12-04's three-plan pattern.

**Primary recommendation:** Follow the v0.12-04 three-plan structure exactly: Plan 01 = Wave 0 RED stubs for `bin/run-protocol-tlc.test.cjs` + add to npm test; Plan 02 = author `QGSDDeliberation.tla` + `MCdeliberation.cfg` + `QGSDPreFilter.tla` + `MCprefilter.cfg`; Plan 03 = implement `bin/run-protocol-tlc.cjs` and turn tests GREEN.

### Pattern 1: Deliberation + Improvement Iteration Spec (QGSDDeliberation.tla)

**What:** Model the R3 deliberation loop (max 10 rounds) and R3.6 improvement iteration loop (max 10 iterations) as concurrent counters with a combined bound. The key behavioral distinction:
- `deliberationRound` advances when any model votes BLOCK and the round re-runs under R3.3
- `improvementIteration` advances when consensus is reached but models propose improvements under R3.6 — this is a NEW quorum round on the revised plan
- Regression (APPROVE→BLOCK during R3.6) transitions `voteState` back to "BLOCKING" without resetting counters; deliberationRound then advances normally

**State variables:**
- `deliberationRound` — Nat: current R3.3 deliberation round (0..10)
- `improvementIteration` — Nat: current R3.6 improvement iteration (0..10)
- `voteState` — One of: "COLLECTING", "BLOCKING", "IMPROVING", "CONSENSUS", "ESCALATED"
  - `COLLECTING`: initial vote collection (R3.2)
  - `BLOCKING`: BLOCK received, deliberation in progress (R3.3)
  - `IMPROVING`: APPROVE+improvement proposed, iteration in progress (R3.6)
  - `CONSENSUS`: all available models APPROVE, no improvements — terminal
  - `ESCALATED`: 10 deliberation rounds exhausted — terminal (R3.4)

**Key invariants (GAP-2):**
```tla
(* Combined round bound: deliberationRound + improvementIteration <= 20 *)
TotalRoundsBounded ==
    deliberationRound + improvementIteration <= 20

(* Deliberation counter monotonically increases *)
DeliberationMonotone ==
    [][deliberationRound' >= deliberationRound]_vars

(* Improvement counter monotonically increases *)
ImprovementMonotone ==
    [][improvementIteration' >= improvementIteration]_vars

(* Regression: APPROVE→BLOCK during improvement transitions back to BLOCKING *)
RegressionHandled ==
    voteState = "IMPROVING" =>
        \/ voteState' = "IMPROVING"        \* improvement continues
        \/ voteState' = "CONSENSUS"        \* improvement accepted
        \/ voteState' = "BLOCKING"         \* regression: new BLOCK
        \/ voteState' = "ESCALATED"        \* round cap hit
```

**Key liveness (GAP-2):**
```tla
(* Every behavior eventually reaches CONSENSUS or ESCALATED *)
ProtocolTerminates ==
    <>(voteState = "CONSENSUS" \/ voteState = "ESCALATED")
```

**Spec fairness:** `WF_vars(Deliberate)` and `WF_vars(Improve)` — both counters advance whenever enabled. The combined bound (deliberationRound + improvementIteration ≤ 20) ensures the algorithm terminates.

### Pattern 2: Pre-Filter Protocol Spec (QGSDPreFilter.tla)

**What:** Model the R4 pre-filter protocol: each candidate question is evaluated by available models; if all agree (CONSENSUS-READY + same answer), it is auto-resolved. Otherwise, up to 3 deliberation rounds run. The protocol terminates at ≤ 3 rounds.

**State variables:**
- `filterRound` — Nat: current pre-filter deliberation round (0..3)
- `modelAgreement` — BOOLEAN: whether all available models returned CONSENSUS-READY with same answer
- `autoResolved` — BOOLEAN: whether the question was auto-resolved
- `filterPhase` — One of: "EVALUATING", "AUTO_RESOLVED", "USER_NEEDED"

**Key invariants (GAP-6):**
```tla
(* Auto-resolution soundness: auto-resolved iff all models agree + same answer *)
AutoResolutionSound ==
    autoResolved <=> modelAgreement

(* Pre-filter terminates: at most 3 rounds *)
FilterRoundsBounded ==
    filterRound <= 3

(* filterPhase monotonically advances: never goes backwards *)
FilterPhaseConsistent ==
    filterPhase = "AUTO_RESOLVED" => autoResolved = TRUE
```

**Key liveness (GAP-6):**
```tla
(* Every behavior eventually exits EVALUATING *)
PreFilterTerminates ==
    <>(filterPhase = "AUTO_RESOLVED" \/ filterPhase = "USER_NEEDED")
```

**Spec fairness:** `WF_vars(AutoResolve)` and `WF_vars(EscalateToUser)` — protocol terminates via one of two paths.

### Anti-Patterns to Avoid

- **Resetting counters on regression:** The APPROVE→BLOCK transition (R3.6 regression) does NOT reset `deliberationRound` or `improvementIteration`. It transitions `voteState` to `"BLOCKING"` and the existing deliberation counter continues advancing. The combined bound still applies.
- **Conflating deliberation rounds with improvement iterations:** These are distinct counters. `deliberationRound` tracks R3.3 "re-vote" rounds. `improvementIteration` tracks R3.6 "revised plan + new quorum" rounds. They can both be advancing in the same behavior.
- **Using auto-workers for liveness specs:** Both `MCdeliberation` and `MCprefilter` have liveness properties. Use `-workers 1` for both to avoid the TLC multi-worker liveness bug (established in v0.12-04 via MCoscillation precedent).
- **Modeling agent identities in deliberation spec:** The deliberation spec does not need to model which specific model voted what. Abstract the vote outcome as a nondeterministic parameter (any model may BLOCK, any may APPROVE with improvements). This keeps the state space small.
- **SYMMETRY on voteState domain:** The `voteState` domain is an ordered enum, not a symmetric set. Do not apply SYMMETRY.

## Common Pitfalls

### Pitfall 1: Mismodeling Regression as Counter Reset

**What goes wrong:** The APPROVE→BLOCK transition during R3.6 (regression) may be modeled as resetting `deliberationRound` to 0, making the combined bound vacuously trivially satisfied (counters never grow large enough to matter).
**Why it happens:** The regression rule says "treated as a new blocker" — this means a new BLOCK state, NOT a new quorum from scratch.
**How to avoid:** Model regression as `voteState' = "BLOCKING"` while keeping `deliberationRound` at its current value. The deliberationRound then increments on the next Deliberate action. The `TotalRoundsBounded` invariant catches any implementation that would allow the combined total to exceed 20.
**Warning signs:** TLC enumerates very few states (< 100) for the deliberation spec — likely the counters are resetting and preventing full exploration.

### Pitfall 2: Both Configs Need -workers 1 (Liveness)

**What goes wrong:** Using `-workers auto` for `MCdeliberation` or `MCprefilter` causes spurious liveness violations from the TLC multi-worker liveness bug.
**Why it happens:** Both specs have liveness properties (`ProtocolTerminates`, `PreFilterTerminates`). TLC v1.8.0 has a known multi-worker liveness checking bug.
**How to avoid:** In `run-protocol-tlc.cjs`, use `const workers = '1'` unconditionally (both configs have liveness). This differs from `run-oscillation-tlc.cjs` which uses `'1'` only for MCoscillation.
**Warning signs:** TLC reports a counterexample for `ProtocolTerminates` showing an infinite stuttering behavior that shouldn't exist — this is the multi-worker false positive.

### Pitfall 3: Auto-Resolution Predicate Direction

**What goes wrong:** `AutoResolutionSound` is an iff (`<=>`) predicate: `autoResolved <=> modelAgreement`. Modeling it as only one direction (`modelAgreement => autoResolved`) misses the converse: you should never auto-resolve when models disagree.
**Why it happens:** The iff is easy to forget — it looks symmetric but the CLAUDE.md R4 spec is precise: "all available models agree CONSENSUS-READY + same answer" is the ONLY condition for auto-resolution.
**How to avoid:** Model `AutoResolutionSound` as a strict biconditional in both the invariant and the `AutoResolve` action's precondition. The action `AutoResolve` must require `modelAgreement = TRUE`.
**Warning signs:** TLC finds no violations even when you inject a state where `autoResolved = TRUE /\ modelAgreement = FALSE` — the biconditional enforcement is missing.

### Pitfall 4: State Space Size for Deliberation Spec

**What goes wrong:** The deliberation spec has two integer counters (0..10 each) and a 5-value enum — state space is 11 * 11 * 5 = 605 reachable combinations. This is small and TLC handles it instantly. However, if additional agent-level modeling is added (e.g., per-model vote variables), the state space explodes multiplicatively.
**Why it happens:** Over-specifying agent identities when the spec only needs to track aggregate vote outcomes.
**How to avoid:** Abstract all agent behavior as nondeterministic choices: `\E n \in 0..MaxAgents : CollectVotes(n)`. Do not model individual agent variables. Keep vars to `{deliberationRound, improvementIteration, voteState}` as specified in the success criteria.
**Warning signs:** TLC reports > 10,000 states for MCdeliberation — indicates over-specification.

### Pitfall 5: npm test Script Gap

**What goes wrong:** `bin/run-protocol-tlc.test.cjs` is created but not added to the `npm test` script in `package.json`.
**Why it happens:** The test script must be manually appended; there is no auto-discovery.
**How to avoid:** Plan 01 MUST add `bin/run-protocol-tlc.test.cjs` to `package.json` scripts.test as part of the Wave 0 stub task. Verify with `node -e "const p = require('./package.json'); console.log(p.scripts.test.includes('run-protocol-tlc'))"`.
**Warning signs:** `npm test` passes but `node --test bin/run-protocol-tlc.test.cjs` reveals failures not reported by npm.

### QGSDDeliberation.tla — Full Skeleton

```tla
---- MODULE QGSDDeliberation ----
(*
 * formal/tla/QGSDDeliberation.tla
 * Handwritten — not generated from XState.
 * Source: CLAUDE.md R3 (deliberation loop) and R3.6 (improvement iteration loop)
 *
 * Models the R3 quorum deliberation protocol:
 *   - R3.3: BLOCK triggers deliberation; up to 10 rounds
 *   - R3.6: APPROVE+improvement triggers iteration; up to 10 iterations
 *   - Combined bound: deliberationRound + improvementIteration <= 20
 *   - Regression: APPROVE→BLOCK during R3.6 resumes deliberation without counter reset
 *   - Escalation: deliberationRound >= MaxDeliberationRounds => ESCALATED
 *
 * State vars:
 *   deliberationRound    -- Nat: R3.3 deliberation rounds (0..10)
 *   improvementIteration -- Nat: R3.6 improvement iterations (0..10)
 *   voteState            -- Protocol phase
 *)
EXTENDS Naturals, TLC

CONSTANTS
    MaxDeliberationRounds,    \* Maximum R3.3 deliberation rounds (= 10)
    MaxImprovementIterations  \* Maximum R3.6 improvement iterations (= 10)

ASSUME MaxDeliberationRounds \in Nat /\ MaxDeliberationRounds > 0
ASSUME MaxImprovementIterations \in Nat /\ MaxImprovementIterations > 0

VoteStates == {"COLLECTING", "BLOCKING", "IMPROVING", "CONSENSUS", "ESCALATED"}

VARIABLES deliberationRound, improvementIteration, voteState

vars == <<deliberationRound, improvementIteration, voteState>>

TypeOK ==
    /\ deliberationRound    \in 0..MaxDeliberationRounds
    /\ improvementIteration \in 0..MaxImprovementIterations
    /\ voteState            \in VoteStates

Init ==
    /\ deliberationRound    = 0
    /\ improvementIteration = 0
    /\ voteState            = "COLLECTING"

(* ── Actions ────────────────────────────────────────────────────────────────── *)

(* ReceiveApprove: all available models APPROVE with no improvements → CONSENSUS *)
ReceiveApprove ==
    /\ voteState \in {"COLLECTING", "BLOCKING"}
    /\ voteState' = "CONSENSUS"
    /\ UNCHANGED <<deliberationRound, improvementIteration>>

(* ReceiveBlock: any model BLOCKs → BLOCKING; advance deliberation round *)
ReceiveBlock ==
    /\ voteState \in {"COLLECTING", "BLOCKING"}
    /\ deliberationRound < MaxDeliberationRounds
    /\ deliberationRound' = deliberationRound + 1
    /\ voteState'         = "BLOCKING"
    /\ UNCHANGED improvementIteration

(* ReceiveApproveWithImprovement: all APPROVE but some propose improvements → IMPROVING *)
ReceiveApproveWithImprovement ==
    /\ voteState \in {"COLLECTING", "BLOCKING"}
    /\ improvementIteration < MaxImprovementIterations
    /\ improvementIteration' = improvementIteration + 1
    /\ voteState'            = "IMPROVING"
    /\ UNCHANGED deliberationRound

(* ImprovementAccepted: next round after improvement — no new blocks or improvements *)
ImprovementAccepted ==
    /\ voteState = "IMPROVING"
    /\ voteState' = "CONSENSUS"
    /\ UNCHANGED <<deliberationRound, improvementIteration>>

(* ImprovementRegression: APPROVE→BLOCK during R3.6 — resume deliberation *)
ImprovementRegression ==
    /\ voteState = "IMPROVING"
    /\ deliberationRound < MaxDeliberationRounds
    /\ deliberationRound' = deliberationRound + 1
    /\ voteState'         = "BLOCKING"
    /\ UNCHANGED improvementIteration

(* Escalate: deliberation round cap exhausted → ESCALATED *)
Escalate ==
    /\ voteState = "BLOCKING"
    /\ deliberationRound >= MaxDeliberationRounds
    /\ voteState' = "ESCALATED"
    /\ UNCHANGED <<deliberationRound, improvementIteration>>

Next ==
    \/ ReceiveApprove
    \/ ReceiveBlock
    \/ ReceiveApproveWithImprovement
    \/ ImprovementAccepted
    \/ ImprovementRegression
    \/ Escalate

(* ── Invariants (GAP-2) ──────────────────────────────────────────────────────── *)

(* Combined round bound: never exceeds 20 total rounds *)
TotalRoundsBounded ==
    deliberationRound + improvementIteration <= MaxDeliberationRounds + MaxImprovementIterations

(* Deliberation counter is monotonically non-decreasing *)
DeliberationMonotone ==
    [][deliberationRound' >= deliberationRound]_vars

(* Improvement counter is monotonically non-decreasing *)
ImprovementMonotone ==
    [][improvementIteration' >= improvementIteration]_vars

(* ── Liveness (GAP-2) ────────────────────────────────────────────────────────── *)

(* Every behavior eventually reaches CONSENSUS or ESCALATED *)
ProtocolTerminates ==
    <>(voteState = "CONSENSUS" \/ voteState = "ESCALATED")

(* ── Full specification ──────────────────────────────────────────────────────── *)
Spec == Init /\ [][Next]_vars
        /\ WF_vars(Escalate)
        /\ WF_vars(ReceiveApprove)
        /\ WF_vars(ImprovementAccepted)

====
```

### QGSDPreFilter.tla — Full Skeleton

```tla
---- MODULE QGSDPreFilter ----
(*
 * formal/tla/QGSDPreFilter.tla
 * Handwritten — not generated from XState.
 * Source: CLAUDE.md R4 (pre-filter protocol)
 *
 * Models the R4 pre-filter protocol:
 *   - Each candidate question is evaluated by available models
 *   - If all models return CONSENSUS-READY + same answer → auto-resolved
 *   - Otherwise → up to 3 deliberation rounds (R4 table: "3 deliberation rounds")
 *   - After 3 rounds without consensus → USER_NEEDED (presented to user)
 *
 * State vars:
 *   filterRound    -- Nat: current deliberation round (0..3)
 *   modelAgreement -- BOOLEAN: all models CONSENSUS-READY + same answer
 *   autoResolved   -- BOOLEAN: question auto-resolved (not presented to user)
 *   filterPhase    -- Protocol phase
 *)
EXTENDS Naturals, TLC

CONSTANTS
    MaxFilterRounds  \* Maximum R4 deliberation rounds (= 3)

ASSUME MaxFilterRounds \in Nat /\ MaxFilterRounds > 0

FilterPhases == {"EVALUATING", "AUTO_RESOLVED", "USER_NEEDED"}

VARIABLES filterRound, modelAgreement, autoResolved, filterPhase

vars == <<filterRound, modelAgreement, autoResolved, filterPhase>>

TypeOK ==
    /\ filterRound    \in 0..MaxFilterRounds
    /\ modelAgreement \in BOOLEAN
    /\ autoResolved   \in BOOLEAN
    /\ filterPhase    \in FilterPhases

Init ==
    /\ filterRound    = 0
    /\ modelAgreement = FALSE
    /\ autoResolved   = FALSE
    /\ filterPhase    = "EVALUATING"

(* ── Actions ────────────────────────────────────────────────────────────────── *)

(* AutoResolve: all models CONSENSUS-READY + same answer → auto-resolved *)
(* R4 invariant: autoResolved can only be TRUE when modelAgreement is TRUE *)
AutoResolve ==
    /\ filterPhase    = "EVALUATING"
    /\ modelAgreement = TRUE
    /\ autoResolved'  = TRUE
    /\ filterPhase'   = "AUTO_RESOLVED"
    /\ UNCHANGED <<filterRound, modelAgreement>>

(* Disagree: models diverge or USER-INPUT-NEEDED → run deliberation round *)
Disagree ==
    /\ filterPhase    = "EVALUATING"
    /\ modelAgreement = FALSE
    /\ filterRound < MaxFilterRounds
    /\ filterRound'   = filterRound + 1
    /\ UNCHANGED <<modelAgreement, autoResolved, filterPhase>>

(* SetAgreement: nondeterministically set modelAgreement (models respond) *)
SetAgreement(agree) ==
    /\ filterPhase    = "EVALUATING"
    /\ modelAgreement' = agree
    /\ UNCHANGED <<filterRound, autoResolved, filterPhase>>

(* EscalateToUser: max rounds reached without agreement → USER_NEEDED *)
EscalateToUser ==
    /\ filterPhase  = "EVALUATING"
    /\ filterRound >= MaxFilterRounds
    /\ filterPhase' = "USER_NEEDED"
    /\ UNCHANGED <<filterRound, modelAgreement, autoResolved>>

Next ==
    \/ AutoResolve
    \/ Disagree
    \/ \E agree \in BOOLEAN : SetAgreement(agree)
    \/ EscalateToUser

(* ── Invariants (GAP-6) ──────────────────────────────────────────────────────── *)

(* Auto-resolution soundness: auto-resolved IFF all models agree + same answer *)
AutoResolutionSound ==
    autoResolved <=> modelAgreement

(* Pre-filter never runs more than MaxFilterRounds deliberation rounds *)
FilterRoundsBounded ==
    filterRound <= MaxFilterRounds

(* AUTO_RESOLVED implies autoResolved=TRUE and modelAgreement=TRUE *)
AutoResolvedPhaseConsistent ==
    filterPhase = "AUTO_RESOLVED" => (autoResolved = TRUE /\ modelAgreement = TRUE)
```

## Open Questions

1. **Whether QGSDDeliberation.tla should reference or extend QGSDQuorum.tla**
   - What we know: `QGSDQuorum.tla` has a `MaxDeliberation` constant and `deliberationRounds` variable. The new spec has `MaxDeliberationRounds` and `deliberationRound` (singular). They model overlapping concepts.
   - What's unclear: Whether the planner should extend or replace QGSDQuorum.tla's deliberation model.
   - Recommendation: Keep them separate. QGSDQuorum.tla is GENERATED from the XState machine; QGSDDeliberation.tla is handwritten against CLAUDE.md R3. They have different levels of abstraction. The two specs can coexist — QGSDDeliberation provides the R3.6 improvement iteration that QGSDQuorum.tla cannot express.

2. **AutoResolutionSound invariant holds transitionally in QGSDPreFilter.tla**
   - What we know: `AutoResolutionSound` asserts `autoResolved <=> modelAgreement`. In the Init state, both are FALSE — biconditional holds. After AutoResolve fires, both are TRUE — holds. After Disagree+EscalateToUser, both are FALSE — holds.
   - What's unclear: Whether the invariant can be violated during the `SetAgreement(agree)` action before `AutoResolve` fires (i.e., `modelAgreement = TRUE` but `autoResolved = FALSE`).
   - Recommendation: The invariant will be transiently violated in the intermediate state where `modelAgreement = TRUE` but `AutoResolve` has not yet fired. This is an expected TLC finding. To fix: either (a) weaken the invariant to `autoResolved => modelAgreement` (one-directional), or (b) make `SetAgreement(TRUE)` and `AutoResolve` atomic. Option (b) is cleaner — combine them into a single `EvaluateAndResolve(agree)` action.

3. **Workers=1 performance for MCdeliberation**
   - What we know: MCdeliberation has a very small state space (11 * 11 * 5 = 605 max states). `-workers 1` will be fast.
   - What's unclear: Whether the TLC multi-worker liveness bug actually affects specs this small.
   - Recommendation: Use `-workers 1` anyway — the performance cost is negligible and consistency with the established pattern (MCoscillation uses `-workers 1` for liveness) is more important than marginal speedup.

## CLAUDE.md Protocol Reference (GAP-2 and GAP-6 Source)

### R3.3 — Deliberation Rules (GAP-2 source)

From CLAUDE.md:
```
| All available models APPROVE | CONSENSUS reached. Deliver output. |
| Any model BLOCK | BLOCK state. Run deliberation (up to 10 rounds total). |
| Any model REVIEW-NEEDED | Run deliberation round to resolve. |
```
"Deliberation: share all positions with all models simultaneously. Re-vote. Repeat until consensus or 10 rounds exhausted."

### R3.4 — Escalation (GAP-2 terminal state)

"If 10 rounds complete without consensus, Claude MUST escalate to the user."

### R3.6 — Iterative Improvement Protocol (GAP-2 source)

```
IF CONSENSUS is reached but one or more models propose specific, actionable improvements:
1. Claude MUST incorporate improvements into a revised plan iteration.
2. Claude MUST present the revised plan to a new QUORUM round.
3. This process MAY repeat up to 10 total iterations.
4. Claude MUST stop iterating when no member proposes further improvements OR 10 iterations complete.
Regression: IF refinement causes any model to switch from APPROVE to BLOCK, treat as new BLOCKER.
```

### R4 — Pre-Filter Protocol (GAP-6 source)

```
| All available models agree CONSENSUS-READY + same answer | Record as assumption. Remove from user-facing list. |
| Any model returns USER-INPUT-NEEDED, OR conflicting CONSENSUS-READY answers | Run R3.3 deliberation (up to 3 rounds). |
| No consensus after 3 deliberation rounds | Mark for user presentation. |
```
