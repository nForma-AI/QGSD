# Phase v0.33-01: Time Series Foundation - Research

**Researched:** 2026-03-10
**Domain:** Cross-session solve history persistence, data quality (deduplication), scope-growth tagging
**Confidence:** HIGH

## Summary

This phase establishes the data foundation for all downstream convergence analysis (phases v0.33-02 through v0.33-06). The core deliverable is a JSONL append log (`solve-trend.jsonl`) written at the end of every `nf-solve` run, capturing per-layer residuals, iteration count, convergence status, gate scores, and requirement count. Two data quality fixes are also in scope: deduplicating existing promotion-changelog.json entries and adding a write-time guard to prevent future duplicates.

The codebase is 100% Node.js (CommonJS `.cjs`), uses `node:test` + `node:assert/strict` for testing, and follows a fail-open error handling pattern throughout. All solve-related data lives in `.planning/formal/`. The solve pipeline is orchestrated by `bin/nf-solve.cjs` which already writes `solve-state.json` and session summaries at the end of each run -- the JSONL append hooks into the same location. The promotion changelog is written by `appendChangelog()` in `bin/compute-per-model-gates.cjs` with no dedup guard -- this is the function that needs the write-time guard.

The existing `promotion-changelog.json` has 200 entries with 1088 duplicate pairs (same model, same transition, within 5-minute window). Most are `ADVISORY -> SOFT_GATE` transitions for the same 2 models repeated across solve runs. A one-time cleanup script plus the write-time guard will address STAB-03.

**Primary recommendation:** Implement as 3 focused tasks: (1) JSONL schema + append function + integration into `main()`, (2) scope-growth detection via requirement count delta, (3) changelog dedup cleanup + write-time guard.

### Pattern 1: JSONL Append (TRACK-01)

**What:** After writing `solve-state.json` in `main()`, append a single JSON line to `solve-trend.jsonl` with the full snapshot.
**When to use:** End of every `nf-solve` run (both normal and `--report-only`).
**Schema:**
```javascript
// solve-trend.jsonl entry schema
{
  "timestamp": "2026-03-10T12:00:00.000Z",   // ISO 8601
  "iteration_count": 3,                       // number of iterations in this run
  "converged": false,                          // did the solver converge?
  "total_residual": 34,                        // sum of forward layers
  "reverse_discovery_total": 336,              // sum of reverse layers
  "layer_total": 14,                           // sum of alignment layers
  "heatmap_total": 6306,                       // git heatmap residual
  "per_layer": {
    "r_to_f": 11,    "f_to_t": 0,    "c_to_f": 0,
    "t_to_c": 0,     "f_to_c": 0,    "r_to_d": 0,
    "d_to_c": 22,    "p_to_f": 0,
    "c_to_r": 135,   "t_to_r": 176,  "d_to_r": 25,
    "l1_to_l2": 8,   "l2_to_l3": 4,  "l3_to_tc": 2,
    "per_model_gates": 0,
    "git_heatmap": 6306,
    "git_history": 39,
    "formal_lint": 0,
    "hazard_model": 9
  },
  "gate_summary": {
    "a": { "score": 0.208, "target_met": false },
    "b": { "score": 0.0,   "target_met": false },
    "c": { "score": 0.0,   "target_met": false }
  },
  "requirement_count": 335,                    // for TRACK-04 scope-growth detection
  "scope_change": null                         // null on first entry; "SCOPE_GROWTH" | "STABLE" | "SCOPE_REDUCTION" on subsequent
}
```

**Implementation approach:**
```javascript
// In main(), after the solveState write block (~line 3494):
function appendTrendEntry(finalResidual, converged, iterations) {
  try {
    const trendPath = path.join(ROOT, '.planning', 'formal', 'solve-trend.jsonl');

    // Read requirement count from requirements.json
    let reqCount = 0;
    try {
      const reqData = JSON.parse(fs.readFileSync(
        path.join(ROOT, '.planning', 'formal', 'requirements.json'), 'utf8'
      ));
      reqCount = Array.isArray(reqData.requirements) ? reqData.requirements.length : 0;
    } catch (_) { /* fail-open */ }

    // Read gate scores
    const gateSummary = readGateSummary();

    // Determine scope change by reading last entry
    let scopeChange = null;
    try {
      const lines = fs.readFileSync(trendPath, 'utf8').trim().split('\n');
      if (lines.length > 0 && lines[lines.length - 1]) {
        const prev = JSON.parse(lines[lines.length - 1]);
        if (prev.requirement_count !== undefined) {
          if (reqCount > prev.requirement_count) scopeChange = 'SCOPE_GROWTH';
          else if (reqCount < prev.requirement_count) scopeChange = 'SCOPE_REDUCTION';
          else scopeChange = 'STABLE';
        }
      }
    } catch (_) { /* first entry or read error -- leave null */ }

    // Build per_layer map from residual object
    const perLayer = {};
    const layerKeys = [
      'r_to_f', 'f_to_t', 'c_to_f', 't_to_c', 'f_to_c', 'r_to_d', 'd_to_c', 'p_to_f',
      'c_to_r', 't_to_r', 'd_to_r',
      'l1_to_l2', 'l2_to_l3', 'l3_to_tc',
      'per_model_gates', 'git_heatmap', 'git_history', 'formal_lint', 'hazard_model'
    ];
    for (const k of layerKeys) {
      perLayer[k] = finalResidual[k] ? finalResidual[k].residual : -1;
    }

    const entry = {
      timestamp: new Date().toISOString(),
      iteration_count: iterations.length,
      converged,
      total_residual: finalResidual.total,
      reverse_discovery_total: finalResidual.reverse_discovery_total || 0,
      layer_total: finalResidual.layer_total || 0,
      heatmap_total: finalResidual.heatmap_total || 0,
      per_layer: perLayer,
      gate_summary: gateSummary,
      requirement_count: reqCount,
      scope_change: scopeChange,
    };

    fs.appendFileSync(trendPath, JSON.stringify(entry) + '\n');
  } catch (e) {
    process.stderr.write(TAG + ' WARNING: could not append to solve-trend.jsonl: ' + e.message + '\n');
  }
}
```

### Pattern 2: Scope-Growth Detection (TRACK-04)

**What:** Compare requirement count in current entry vs. last JSONL entry. If requirement count increased AND total residual increased, tag as `SCOPE_GROWTH`.
**When to use:** Automatically during JSONL append.
**Key insight:** The `scope_change` field in the JSONL entry is computed by comparing `requirement_count` from the previous entry. Downstream phases (TRACK-02 trend detection) will use this tag to exclude scope-growth entries from regression analysis.

**Decision:** Track requirement count from `requirements.json` (335 entries currently). The count changes when `bin/aggregate-requirements.cjs` runs after requirement files are modified.

### Pattern 3: Write-Time Dedup Guard (STAB-03)

**What:** Before appending to `promotion-changelog.json`, check if the last N entries contain the same (model, from_level, to_level) within a 5-minute window. Skip the write if a duplicate exists.
**Where:** Modify `appendChangelog()` in `bin/compute-per-model-gates.cjs` (~line 58).
**Implementation:**
```javascript
function appendChangelog(entry) {
  if (DRY_RUN_FLAG) return;
  try {
    let changelog = [];
    if (fs.existsSync(CHANGELOG_PATH)) {
      try {
        changelog = JSON.parse(fs.readFileSync(CHANGELOG_PATH, 'utf8'));
        if (!Array.isArray(changelog)) changelog = [];
      } catch (_) {
        changelog = [];
      }
    }

    // STAB-03: Write-time dedup guard -- skip if same transition within 5 minutes
    const DEDUP_WINDOW_MS = 5 * 60 * 1000;
    const entryTime = new Date(entry.timestamp).getTime();
    const isDuplicate = changelog.some(existing => {
      if (existing.model !== entry.model) return false;
      if (existing.from_level !== entry.from_level) return false;
      if (existing.to_level !== entry.to_level) return false;
      const existingTime = new Date(existing.timestamp).getTime();
      return Math.abs(entryTime - existingTime) < DEDUP_WINDOW_MS;
    });

    if (isDuplicate) {
      process.stderr.write(TAG + ' DEDUP: skipping duplicate changelog entry for ' + entry.model + '\n');
      return;
    }

    changelog.push(entry);
    // Retention: trim from front to keep last 200 entries
    if (changelog.length > CHANGELOG_MAX_ENTRIES) {
      changelog = changelog.slice(changelog.length - CHANGELOG_MAX_ENTRIES);
    }
    fs.writeFileSync(CHANGELOG_PATH, JSON.stringify(changelog, null, 2) + '\n');
  } catch (e) {
    process.stderr.write(TAG + ' WARNING: changelog write failed: ' + e.message + '\n');
  }
}
```

### Anti-Patterns to Avoid
- **Reading entire JSONL into memory for every append:** Only read the last line (or last N lines) when computing scope_change. Use `fs.appendFileSync` for writes.
- **Modifying solve-state.json format:** Per STATE.md decision, outer-loop state lives in separate files. Never add trend fields to solve-state.json.
- **Using JSON array file instead of JSONL:** JSON arrays require parsing the entire file to append. JSONL supports atomic line appends.
- **Checking ALL changelog entries for dedup:** Only check entries within the time window. The `some()` approach is fine for 200 entries but use a reverse scan for efficiency.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSONL format | Custom serialization | `JSON.stringify(obj) + '\n'` per line | Standard format, trivially parseable |
| File locking for concurrent writes | Lock manager | Not needed -- nf-solve runs synchronously, one at a time | Process is single-threaded and sequential |
| Time window comparison | Duration library | Simple `Math.abs(t1 - t2) < WINDOW_MS` | 5-minute windows don't need timezone/DST handling |
| Gate score reading | Custom gate parser | Read the 3 existing gate JSON files directly | Already well-structured with score + target_met fields |

**Key insight:** This phase is pure file I/O with simple data transformations. No external libraries needed.

## Common Pitfalls

### Pitfall 1: JSONL Partial Write Corruption
**What goes wrong:** Process crash mid-write leaves incomplete JSON line, breaking subsequent reads.
**Why it happens:** `fs.appendFileSync` is not truly atomic on all filesystems.
**How to avoid:** When reading JSONL, wrap each line parse in try/catch and skip malformed lines. The append function should write the complete string in a single call.
**Warning signs:** `SyntaxError: Unexpected end of JSON input` when reading trend file.

### Pitfall 2: Skipped Layers in Fast Mode
**What goes wrong:** `--fast` mode sets several layers to `{ residual: -1, detail: { skipped: true } }`. If JSONL records `-1`, downstream trend analysis interprets it as improvement.
**Why it happens:** Fast mode skips `t_to_c`, `f_to_c`, and all layer alignment sweeps.
**How to avoid:** Record the raw residual value (including -1). Downstream phases (TRACK-02) must treat -1 as "not measured", not zero.
**Warning signs:** Anomalous residual drops in trend data correlating with fast-mode runs.

### Pitfall 3: Changelog Dedup Scan Performance
**What goes wrong:** Scanning all 200 entries with string comparison on every write.
**Why it happens:** The current file can have up to 200 entries.
**How to avoid:** 200 entries with 3 string comparisons each is trivially fast. No optimization needed. However, scan from the end (most recent entries) for earlier short-circuit.
**Warning signs:** N/A -- not a real concern at this scale.

### Pitfall 4: Gate Score Field Names Vary
**What goes wrong:** Gate A uses `grounding_score`, Gate B uses `gate_b_score`, Gate C uses `gate_c_score`. Inconsistent naming.
**Why it happens:** Different scripts wrote different gates at different times.
**How to avoid:** The `readGateSummary()` helper must handle all three field names explicitly.
**Warning signs:** `undefined` gate scores in JSONL entries.

### Pitfall 5: Requirement Count Source
**What goes wrong:** Reading requirement count from wrong source (e.g., REQUIREMENTS.md instead of requirements.json).
**Why it happens:** REQUIREMENTS.md has 17 milestone requirements, but requirements.json has 335 aggregated requirements.
**How to avoid:** Always use `requirements.json` (the aggregated envelope with 335 entries) as the canonical count source.
**Warning signs:** Requirement count in JSONL entry is much smaller than expected.

### Example 1: Reading Gate Summary

```javascript
// Read gate scores from the 3 gate JSON files
function readGateSummary() {
  const gatesDir = path.join(ROOT, '.planning', 'formal', 'gates');
  const summary = { a: null, b: null, c: null };
  try {
    const a = JSON.parse(fs.readFileSync(path.join(gatesDir, 'gate-a-grounding.json'), 'utf8'));
    summary.a = { score: a.grounding_score || 0, target_met: !!a.target_met };
  } catch (_) { /* fail-open */ }
  try {
    const b = JSON.parse(fs.readFileSync(path.join(gatesDir, 'gate-b-abstraction.json'), 'utf8'));
    summary.b = { score: b.gate_b_score || 0, target_met: !!b.target_met };
  } catch (_) { /* fail-open */ }
  try {
    const c = JSON.parse(fs.readFileSync(path.join(gatesDir, 'gate-c-validation.json'), 'utf8'));
    summary.c = { score: c.gate_c_score || 0, target_met: !!c.target_met };
  } catch (_) { /* fail-open */ }
  return summary;
}
```

```javascript
function readLastTrendEntry(trendPath) {
  try {
    const content = fs.readFileSync(trendPath, 'utf8').trim();
    if (!content) return null;
    const lines = content.split('\n');
    for (let i = lines.length - 1; i >= 0; i--) {
      try {
        return JSON.parse(lines[i]);
      } catch (_) { continue; } // skip malformed lines
    }
  } catch (_) { /* file doesn't exist yet */ }

}
```

### Example 3: One-Time Changelog Dedup Script
```javascript
// bin/dedup-changelog.cjs -- one-time cleanup
const fs = require('fs');
const path = require('path');

const CHANGELOG_PATH = path.join(__dirname, '..', '.planning', 'formal', 'promotion-changelog.json');
const DEDUP_WINDOW_MS = 5 * 60 * 1000;

function dedup(changelog) {
  const kept = [];
  for (const entry of changelog) {
    const isDup = kept.some(existing =>
      existing.model === entry.model &&
      existing.from_level === entry.from_level &&
      existing.to_level === entry.to_level &&
      Math.abs(new Date(existing.timestamp) - new Date(entry.timestamp)) < DEDUP_WINDOW_MS
    );
    if (!isDup) kept.push(entry);
  }
  return kept;
}

if (require.main === module) {
  const raw = JSON.parse(fs.readFileSync(CHANGELOG_PATH, 'utf8'));
  const cleaned = dedup(raw);
  console.log(`Before: ${raw.length}, After: ${cleaned.length}, Removed: ${raw.length - cleaned.length}`);
  fs.writeFileSync(CHANGELOG_PATH, JSON.stringify(cleaned, null, 2) + '\n');
}

module.exports = { dedup };
```

## Open Questions

1. **JSONL file size management**
   - What we know: At ~500 bytes per entry and 1-5 solve runs per day, the file will grow ~1KB/day. Over a year, ~365KB.
   - What's unclear: Whether to add retention/rotation.
   - Recommendation: No retention needed for v0.33. File stays small. Future milestone can add rotation if needed.

2. **Should solve-trend.jsonl be gitignored or committed?**
   - What we know: `solve-state.json` is committed (it's in git status as modified). Session summaries are in `.planning/formal/solve-sessions/`.
   - What's unclear: Whether trend data should persist across clones.
   - Recommendation: Commit it (same as solve-state.json). Trend data is valuable across sessions and clone boundaries.

3. **Fast-mode entries in trend data**
   - What we know: Fast mode skips several layers, recording -1 for their residuals.
   - What's unclear: Should fast-mode runs be recorded in trend data at all?
   - Recommendation: Record them with a `fast_mode: true` field. Downstream analysis (Phase 2) can filter if needed.
