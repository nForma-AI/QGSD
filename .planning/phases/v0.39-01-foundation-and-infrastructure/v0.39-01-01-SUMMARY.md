# Phase v0.39-01 Plan 01: Verification Mode & Config Persistence Summary

Verification mode tagging for model checker invocations and configurable iteration limits via .planning/config.json.

## Overview

This plan implements two complementary foundational pieces for the v0.39 dual-cycle formal reasoning architecture:

1. **Verification Mode Tagging (Task 1):** Enables Cycle 1 (diagnostic, inverted semantics) and Cycle 2 (validation, normal semantics) to coexist without mode contradiction. Model checker invocations now carry a `verification_mode` metadata tag that downstream consumers can read and interpret.

2. **Config-Driven Iteration Limits (Task 2):** Replaces hardcoded iteration defaults with a persisted configuration mechanism, allowing iteration bounds to be controlled globally via .planning/config.json or overridden per-invocation.

## Tasks Completed

### Task 1: Verification Mode Tagging Module and Checker Updates

**Files created:**
- `bin/verification-mode.cjs` — Mode envelope and gate interpretation logic
- `bin/verification-mode.test.cjs` — 13 tests covering truth table and E2E metadata emission

**Files modified:**
- `bin/run-tlc.cjs` — Accepts `--verification-mode=diagnostic|validation` CLI arg, passes through to result metadata
- `bin/run-alloy.cjs` — Same pattern as run-tlc.cjs
- `bin/refinement-loop.cjs` — Invokes checkers with `--verification-mode=diagnostic`, calls interpretGateResult to classify outcomes

**Exports from verification-mode.cjs:**
- `MODES` — Object literal `{ DIAGNOSTIC: 'diagnostic', VALIDATION: 'validation' }`
- `createVerificationEnvelope(options)` — Validates and returns envelope with verification_mode defaulting to 'validation'
- `interpretGateResult(hasViolation, verification_mode)` — Four-cell truth table:
  - `diagnostic + violation → 'REPRODUCED'` (model captures bug)
  - `diagnostic + no violation → 'INCOMPLETE'` (model missing bug)
  - `validation + violation → 'FAILED'` (invariant broken)
  - `validation + no violation → 'PASSED'` (consequence model valid)

**Tests (13):**
- createVerificationEnvelope defaults to 'validation'
- createVerificationEnvelope passes through 'diagnostic'
- createVerificationEnvelope throws on invalid mode
- interpretGateResult diagnostic + violation = REPRODUCED
- interpretGateResult diagnostic + no violation = INCOMPLETE
- interpretGateResult validation + violation = FAILED
- interpretGateResult validation + no violation = PASSED
- interpretGateResult throws on unknown mode
- MODES contains expected values
- E2E: run-tlc accepts --verification-mode and includes in metadata
- E2E: run-alloy accepts --verification-mode and includes in metadata
- Consumer test: interpretGateResult semantics match truth table
- createVerificationEnvelope preserves other options

**Result:** All 13 tests pass (1786ms). Verification mode metadata now flows through checker invocations and can be consumed by gate interpretation logic.

## Key Decisions

1. **Verification mode defaults to 'validation'** — Ensures backward compatibility for existing checker invocations that don't specify a mode.

2. **Config read fails open** — If .planning/config.json is missing or malformed, getMaxIterations returns 3 instead of throwing. This prevents cascading failures in edge cases.

3. **Atomic config updates** — updateMaxIterations reads, modifies, writes with 2-space JSON indentation and trailing newline to match project conventions.

4. **Session directory naming** — Uses crypto.randomBytes(8).toString('hex') for 16-hex collision resistance suitable for downstream Cycle 2 artifact isolation.
