Create bin/verification-mode.cjs with these exports:

1. `MODES` — Object literal `{ DIAGNOSTIC: 'diagnostic', VALIDATION: 'validation' }`.

2. `createVerificationEnvelope(options)` — Accepts `{ verification_mode, modelPath, config, bugContext?, dumpTrace? }`. Returns the same object with `verification_mode` defaulting to `'validation'` if not provided. Validates mode is one of MODES values; throws if invalid non-null value supplied.

3. `interpretGateResult(hasViolation, verification_mode)` — Implements the four-cell truth table:
   - diagnostic + violation = `'REPRODUCED'` (model captures bug — success)
   - diagnostic + no violation = `'INCOMPLETE'` (model missing the bug — failure)
   - validation + violation = `'FAILED'` (invariant broken)
   - validation + no violation = `'PASSED'` (consequence model valid)
   Throws on unknown mode.

Update bin/run-tlc.cjs: In the function/CLI section that builds the TLC command args, accept an optional `--verification-mode=diagnostic|validation` CLI arg. Pass it through to the result object written by `writeCheckResult()` as `metadata.verification_mode`. Do NOT change TLC invocation logic itself — this is purely metadata on the result envelope. Default to `'validation'` when not specified, preserving backward compatibility.

Update bin/run-alloy.cjs: Same pattern — accept optional `--verification-mode` CLI arg, pass through to result metadata.

Update bin/refinement-loop.cjs:
- The existing `verifyBugReproduction` function already uses inverted semantics. Add `--verification-mode=diagnostic` to the CLI args of its checker invocations (the `execFileSync` calls to run-tlc/run-alloy). This makes the implicit inversion explicit without changing any logic.
- Import `interpretGateResult` from `./verification-mode.cjs` and call it from `verifyBugReproduction` to classify the checker outcome (reproduced vs incomplete) instead of relying solely on raw pass/fail. This ensures the mode tag actually changes downstream behavior.
