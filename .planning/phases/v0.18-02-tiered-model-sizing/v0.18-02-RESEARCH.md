## Summary

Phase v0.18-02 implements tiered model sizing for plan-phase sub-agents — a cost optimization where the researcher and plan-checker tasks run on haiku by default (15-20x cost reduction vs sonnet) while the planner itself remains on opus/sonnet. This is controlled by two new flat config keys: `model_tier_planner` and `model_tier_worker`, which override the existing per-profile model defaults without requiring schema changes.

The implementation is lightweight: extend config-loader.js with validation for the new tier keys, update gsd-tools.cjs to add a tier lookup layer in resolveModelInternal(), and wire those tier values into the Task() spawn calls in plan-phase.md. No npm dependencies, no database, no breaking changes — just flat config keys merged via the existing shallow-merge pattern.

**Primary recommendation:** Add `model_tier_planner` and `model_tier_worker` to DEFAULT_CONFIG as flat string keys. Update resolveModelInternal() to check tier keys before profile lookup. Map agent types to tier categories ('planner' vs 'worker'). Validate tier values in config-loader.js as strings matching 'haiku' | 'sonnet' | 'opus'. Test with config-loader.test.js covering all tier combinations.

### Pattern 1: Flat Config Key Schema

**What:** Add `model_tier_planner` and `model_tier_worker` as flat string keys to DEFAULT_CONFIG. These are NOT nested — they sit at the root level alongside `circuit_breaker`, `model_preferences`, `fail_mode`, etc.

**Why flat?** config-loader.js uses shallow merge: `{ ...DEFAULT_CONFIG, ...global, ...project }`. Nested objects from project config silently replace the global object; flat keys are preserved correctly. Documented decision in STATE.md.

**When to use:** User wants to override the default model tier for all researcher/plan-checker agents without changing the model_profile.

**Example in ~/.claude/qgsd.json:**
```json
{
  "model_tier_planner": "sonnet",
  "model_tier_worker": "sonnet",
  "fail_mode": "open"
}
```

This causes the planner task to use sonnet (not opus) and researcher/checker tasks to use sonnet (not haiku).

### Pattern 2: Tier Lookup in resolveModelInternal()

**What:** Extend gsd-tools.cjs resolveModelInternal(cwd, agentType) with a tier check before profile lookup.

**When to use:** Mapping agent type names to cost tiers (planner vs worker).

```javascript
// Source: gsd-tools.cjs (modified for TIER-02)
function resolveModelInternal(cwd, agentType) {
  const config = loadConfig(cwd);

  // 1. Check per-agent override first (existing behavior)
  const override = config.model_overrides?.[agentType];
  if (override) {
    return override === 'opus' ? 'inherit' : override;
  }

  // 2. NEW: Check tier keys
  const agentToTier = {
    'gsd-planner': 'planner',
    'gsd-roadmapper': 'planner',
    'gsd-phase-researcher': 'worker',
    'gsd-plan-checker': 'worker',
    'gsd-executor': 'worker',
    // ... other agents map to 'worker' by default
  };
  const tier = agentToTier[agentType] || 'worker';
  const tierKey = 'model_tier_' + tier; // e.g., 'model_tier_worker'
  const tierModel = config[tierKey];
  if (tierModel && ['haiku', 'sonnet', 'opus'].includes(tierModel)) {
    return tierModel;
  }

  // 3. Fall back to profile (existing behavior)
  const profile = config.model_profile || 'balanced';
  const agentModels = MODEL_PROFILES[agentType];
  if (!agentModels) return 'sonnet';
  const resolved = agentModels[profile] || agentModels['balanced'] || 'sonnet';
  return resolved === 'opus' ? 'inherit' : resolved;
}
```

**Lookup order:**
1. Per-agent override (e.g., `model_overrides.gsd-phase-researcher`)
2. Tier model (e.g., `model_tier_worker` for 'worker' agents)
3. Profile model (existing MODEL_PROFILES lookup)
4. Default: 'sonnet'

### Pattern 3: Agent → Tier Mapping

**What:** Define which agents fall into 'planner' vs 'worker' tiers.

**Planner tier:** High-value decisions requiring full reasoning capacity
- `gsd-planner`
- `gsd-roadmapper`

**Worker tier:** Repetitive, lower-risk sub-tasks that benefit from speed/cost
- `gsd-phase-researcher`
- `gsd-plan-checker`
- `gsd-executor`
- `gsd-verifier`
- `gsd-codebase-mapper`
- etc.

### Anti-Patterns to Avoid

- **Using nested config for tier keys:** Shallow merge will cause project values to replace global tier object entirely. Use flat keys: `model_tier_planner` not `model_tiers.planner`.
- **Forgetting install sync:** Changes to config-loader.js MUST be synced to `hooks/dist/config-loader.js` before running `node bin/install.js --claude --global`. The installed copy at `~/.claude/hooks/config-loader.js` is what actually runs in hook context.
- **Skipping tier validation:** Adding tier keys to DEFAULT_CONFIG without validation in validateConfig() allows typos ('hakik', 'snoet') to silently fail. Validate in config-loader.js with a whitelist.
- **Hardcoding tier in plan-phase.md:** Do NOT add tier mapping logic to the plan-phase.md markdown instructions. The tier resolution happens in gsd-tools.cjs `init` command, which populates the init JSON. plan-phase.md just uses `{researcher_model}` etc. from the init JSON.

## Common Pitfalls

### Pitfall 1: Shallow Merge Loses Nested Tier Objects

**What goes wrong:** User creates project config with `model_tiers: { planner: "haiku", worker: "sonnet" }`. The project object replaces the global tier object entirely, losing the global tier settings.

**Why it happens:** config-loader.js uses shallow spread merge: `{ ...DEFAULT_CONFIG, ...project }`. Nested objects are replaced, not merged.

**How to avoid:** Use flat keys: `model_tier_planner: "haiku"` and `model_tier_worker: "sonnet"` at the root level. Flat keys are preserved correctly in shallow merge.

**Warning signs:** `grep -n "model_tiers\|tier_config" qgsd.json` returns nested objects; config validation should reject these with a stderr warning "Use flat keys model_tier_planner and model_tier_worker instead".

### Pitfall 2: Typo in Tier Model Name Silent Failure

**What goes wrong:** User sets `model_tier_worker: "hakiu"` (typo). No error is raised; resolveModelInternal() doesn't recognize 'hakiu', so it falls back to profile defaults silently. User thinks haiku is running, but it's not.

**Why it happens:** No validation of tier model values. The code checks `if (tierModel && ['haiku', 'sonnet', 'opus'].includes(tierModel))` — if tierModel doesn't match, it just falls through to the next lookup.

**How to avoid:** Validate tier values in config-loader.js validateConfig() with explicit whitelist. If tier key is present but value is not in ['haiku', 'sonnet', 'opus'], emit stderr warning and remove the key (fallback to profile).

**Warning signs:** User reports "still seeing sonnet tokens" when they set tier to haiku; check ~/.claude/qgsd.json for typos.

### Pitfall 3: Forgetting Install Sync After config-loader.js Edit

**What goes wrong:** Developer modifies hooks/config-loader.js locally but doesn't sync to hooks/dist/config-loader.js. Runs `npm test` locally (hooks/ is in CWD, test uses local file). Tests pass. Then runs `node bin/install.js --claude --global`. The global hook at ~/.claude/hooks/config-loader.js is NOT updated because it's synced from dist/, not src/.

**Why it happens:** CLAUDE.md Rule: "edits to hook source files MUST sync to hooks/dist/ first, then run install.js". Many developers forget the sync step.

**How to avoid:** Before running install.js, run: `cp hooks/config-loader.js hooks/dist/config-loader.js`. Test file should verify install.js syncs correctly. PLAN.md tasks must include: `cp ... && node bin/install.js --claude --global` as one atomic command.

**Warning signs:** `diff hooks/config-loader.js hooks/dist/config-loader.js` shows differences; ~/.claude/hooks/config-loader.js is stale (older file timestamp than hooks/config-loader.js).

### Pitfall 4: Agent Type Not in Tier Mapping

**What goes wrong:** A new agent type (e.g., 'qgsd-custom-agent') is added to gsd-tools.cjs. The agent is not mapped in the agentToTier object. resolveModelInternal() defaults that agent to 'worker' tier, but the intent was for it to use 'planner' tier.

**Why it happens:** The tier mapping is implicit in resolveModelInternal(); if you add a new agent, you must add it to agentToTier.

**How to avoid:** Document the agent → tier mapping in a comment above agentToTier in gsd-tools.cjs. When adding a new agent type, check the mapping. If the agent is high-value (decision-making), add it to 'planner'; if it's repetitive (research, checks), add to 'worker'.

**Warning signs:** New agent runs slower than expected (using wrong tier); check gsd-tools.cjs resolveModelInternal() to confirm agent is in agentToTier.

## Open Questions

1. **Agent → Tier Mapping Stability**
   - What we know: v0.18-02 maps ~10 agents to tiers; mapping is static in agentToTier object.
   - What's unclear: If new agent types are added frequently, should the mapping be externalized to a config file or constants?
   - Recommendation: Keep inline for now (10 agents is manageable). If agent count grows >20, extract to a TIER_MAPPING constant in gsd-tools.cjs. No action needed for v0.18-02.

2. **Model Inheritance vs Explicit 'opus'**
   - What we know: resolveModelInternal() returns 'inherit' when model is 'opus', which tells Task() to use the parent Claude's model.
   - What's unclear: Should tier keys support 'inherit' or only explicit models?
   - Recommendation: Support only explicit models ('haiku', 'sonnet', 'opus') in tier keys. Tier keys are for cost defaults, not inheritance. If a user wants inheritance, they use per-agent override: `model_overrides.gsd-phase-researcher: "opus"`.

3. **Tier Key Naming: Singular vs Plural**
   - What we know: v0.18-02 uses `model_tier_planner` (singular agent type).
   - What's unclear: Should it be `model_tiers_planner` (plural) or `tier_planner` (shorter)?
   - Recommendation: Stick with `model_tier_[category]` for consistency with existing `model_preferences` and `model_overrides`. Matches the "model_" namespace.
