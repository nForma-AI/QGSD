# Phase v0.39-03: Cycle 2 Solution Simulation — Research

**Researched:** 2026-03-18
**Domain:** Fix intent normalization, consequence model generation, three-gate convergence verification, simulation loop UX, dependency failure resilience
**Confidence:** HIGH (architecture + formal spec), MEDIUM-HIGH (implementation patterns, UX details)

## Summary

Phase v0.39-03 implements **Cycle 2: Solution Simulation**, the second half of dual-cycle formal reasoning. In Cycle 1 (v0.39-02), bugs validate and refine models. In Cycle 2, models validate fix ideas before code is touched.

The phase accepts fix ideas in multiple forms (natural language, constraint sets, code sketches) and normalizes them to formal model mutations. A consequence model then simulates the post-fix system state. Three automated gates check convergence: (1) original invariants still hold, (2) bug no longer triggers (inverted check passes), (3) no 2-hop neighbor regressions. Write-once resolution semantics ensure a convergence verdict—once reached—cannot be silently reverted. When external dependencies (model checkers, quorum slots) become unavailable, all accumulated state is preserved without corruption.

The phase builds on Phase v0.39-01 (FND-01: verification mode tagging, FND-02: ITF trace parsing, FND-03: configurable iteration limits) and Phase v0.39-02 (DX1-01 through DX1-04: diagnostic pipelines, diagnostic injection). This phase adds four new capabilities:

1. **Fix Intent Normalization (SIM-01):** Accept natural language, constraints, or code sketches; normalize to formal model mutations in a standard format.

2. **Consequence Model Generation (SIM-02):** Given reproducing model + normalized fix mutations, generate a new consequence model representing the post-fix system.

3. **Three-Gate Convergence Check (SIM-03):** Automated convergence verification runs three gates: invariants hold + bug resolved + no neighbor regressions. Write-once resolution (ResolvedAtWriteOnce formal invariant).

4. **Simulation Loop UX (SIM-04):** Phase 4.5 integration in model-driven-fix.md with iteration display, per-gate pass/fail status, convergence progress, escalation after max iterations. Dependency unavailability (HaikuUnavailableNoCorruption) must not corrupt logs or state.

**Primary recommendation:** Implement the solution simulation as a four-stage pipeline: (1) Parse fix intent into normalized mutation format (constraint sets, variable additions, transition modifications), (2) Generate consequence model by applying mutations to reproducing model, (3) Run three convergence gates in sequence with write-once outcome persistence, (4) Display iteration progress with per-gate verdicts and escalate when convergence exhausted or dependencies fail. Use formal module ResolvedAtWriteOnce (write-once semantics) to prevent silent resolution reversal. Use HaikuUnavailableNoCorruption (dependency failure safety) to preserve state when external services become unavailable.

### Anti-Patterns to Avoid

- **Consequence model without mutation record:** Don't generate consequence model without recording which mutations were applied; lose ability to debug why consequence differs from expectation
- **Ignoring write-once semantics:** Don't rewrite verdict log; once a convergence verdict is written, appending is the only valid operation
- **Silent dependency failures:** Don't suppress HaikuUnavailableNoCorruption errors (when quorum/model-checker becomes unavailable); escalate to user immediately
- **Convergence reversal:** Don't allow a converged fix to become unconverged in a later iteration; write-once verdict persists regardless of subsequent re-runs
- **Lost context between iterations:** Don't regenerate consequence model from scratch each iteration; reuse accumulated mutations and build incrementally
- **Vague escalation messages:** Don't escalate without explaining which gate failed and why; provide evidence from gate verdict log

The hard problems (model checking, constraint extraction) are solved; Phase 3 adds the workflow orchestration and formal guarantees around convergence.

## Common Pitfalls

### Pitfall 1: Consequence Model Without Clear Mutation Traceability

**What goes wrong:** Consequence model is generated and gates run. Gate 2 fails (bug still present). But which mutation caused the failure? No record of which mutations were applied. Result: can't debug consequence model; user is blind to what changed.

**Why it happens:** Emphasis on "generate consequence model"; focus on output spec rather than input mutations.

**How to avoid:**
- SIM-02 requirement: Every consequence model write must be accompanied by mutation record
- Store normalized-mutations.json in same directory as consequence-model.tla
- Format: `[{ iteration, type, target, content, applied: true/false }, ...]`
- Gate failure message includes: "Failed mutation: X (type: add_invariant, target: Y)"

**Warning signs:**
- Consequence model exists but mutations file missing or empty
- User asks "which mutations did you try?" and you can't answer
- Gate failure message is vague ("model failed") instead of specific ("mutation to add timeout variable failed")

### Pitfall 2: Convergence Verdict Reversal (Write-Once Violation)

**What goes wrong:** Iteration 1 runs gates; all pass; converged=true written to verdict log. User runs simulation again. Iteration 2 discovers the consequence model has a subtle flaw. Gates fail. Code tries to update verdict log: converged=false. This violates ResolvedAtWriteOnce formal invariant. System detects violation; blocks fix.

**Why it happens:** Assumption that verdicts are mutable; each run overwrites previous results.

**How to avoid:**
- SIM-03 requirement: Gate verdicts are write-once; verified by ResolvedAtWriteOnce formal invariant
- Implementation: APPEND to verdict log, never overwrite
- Code: `verdicts.push(newVerdict); writeFile(verdictLogPath, verdicts)` — append only
- Test: "Can I re-run convergence gates and add a 4th iteration?" YES. "Can 4th iteration show converged=false if 3rd was converged=true?" NO.
- Detect violation: if gate runner tries to write same iteration index twice with different converged status, raise error

**Warning signs:**
- Verdict log has duplicate iteration numbers (e.g., iteration 1 appears twice with different results)
- Code treats verdict log like a mutable database (read, modify, write-back)
- User reports: "I re-ran simulation and now it says it didn't converge, even though first run succeeded"

### Pitfall 3: Silent Dependency Failure (HaikuUnavailableNoCorruption Violation)

**What goes wrong:** During gate 2 verification, the model checker becomes unavailable (timeout, quota exceeded, network error). Code catches the error but continues: "checker unavailable, assuming pass". Later, a partial verdict is written and session state is partially deleted. HaikuUnavailableNoCorruption formal invariant is violated.

**Why it happens:** Temptation to fail-open gracefully; assumption that it's safe to skip a gate if dependency is unavailable.

**How to avoid:**
- SIM-04 requirement: Dependency unavailability must preserve all state; formal guarantee via HaikuUnavailableNoCorruption
- Implementation: When dependency becomes unavailable (model checker timeout, quorum slot unreachable):
  - STOP gate verification (do not mark as "pass" or "fail")
  - DO NOT write verdict to log
  - DO NOT delete session state or consequence model
  - Escalate to user: "Model checker unavailable; simulation paused. Try again once checker recovers."
- Test: "If checker times out during gate 2, is verdict log written?" NO. "Is consequence model deleted?" NO. "Is sessionId state preserved?" YES.

**Warning signs:**
- Verdict log has partial entries (e.g., gate 1 result but no gate 2)
- Session state (consequence model, mutations) is missing after dependency failure
- User re-runs simulation and can't find previous session; "it lost my work"

### Pitfall 4: Intent Normalization Too Ambitious (Over-Parsing)

**What goes wrong:** Intent normalizer tries to be too clever. User writes: "When timeout occurs, retry once". Normalizer extracts: "add invariant Timeout", "add action Retry", "add transition from Wait to Retry". User never mentioned those specific model changes. Consequence model is confusing; gates fail; user is frustrated.

**Why it happens:** Desire to infer detailed model mutations from high-level intent; assumption that NLP/ML can bridge the gap.

**How to avoid:**
- SIM-01 requirement: Intent normalization is rule-based, not inference-based
- Three input channels only: (1) Constraint syntax ("if X then Y"), (2) Variable syntax ("add variable Z"), (3) Code syntax ("```code block```")
- For ambiguous intents, ask user for confirmation: "Did you mean: add invariant 'timeout implies retry'?"
- Don't infer: "User said 'timeout'; they want a timeout variable" — ask: "Did you want to add a timeout variable named X?"
- Test: "If user input is ambiguous, does normalizer ask for clarification?" YES. "Does it guess?" NO.

**Warning signs:**
- Normalized mutations don't match user's stated intent
- User says: "That's not what I meant"
- Consequence model has mutations user never requested

### Pitfall 5: Missing Iteration Diagnostics (No Progress Signal)

**What goes wrong:** Iteration 1 fails gate 2 (bug still present). Iteration 2 runs but gates fail again with same error. No diagnostic shown to user comparing iteration 1 and 2. Are they the same failure? Did mutation help at all? User has no signal and gives up.

**Why it happens:** Focus on gate verdicts; forget to generate diagnostic comparing iteration to iteration.

**How to avoid:**
- SIM-04 requirement: Simulation loop displays per-gate progress and iteration comparison
- Generate diagnostic between consecutive iterations: "Iteration 2 vs. Iteration 1: gate 1 still passes (good), gate 2 still fails on same trace (no progress)"
- Display table: Attempt | Gate1 | Gate2 | Gate3 | Notes
- Store in iteration history: `iterations[i].diagnostic = { previous_iteration_diff, progress_signal }`
- Test: "Can user see that gate 2 failed in iteration 1 and 2?" YES. "Does table show which gates changed?" YES.

**Warning signs:**
- Simulation runs max iterations but user doesn't see progress or why it's failing
- Gate verdicts are identical across iterations (no mutations actually changed outcome)
- User asks: "Did anything change from iteration 1 to 2?" and you can't answer from UX

### Pitfall 6: Neighbor Regression Detection Too Strict (False Positives)

**What goes wrong:** Gate 3 runs neighbor regressions; a neighbor model has an unrelated violation (not caused by consequence model). Gate 3 fails. User is blocked from using the fix even though the specific bug is resolved. Feedback: "Your fix broke something unrelated" (wrong).

**Why it happens:** Gate 3 assumes any neighbor violation = regression caused by fix.

**How to avoid:**
- SIM-03 requirement: Regression detection must distinguish (1) regressions caused by consequence model vs. (2) pre-existing neighbor issues
- Baseline: Run neighbor checks against BOTH reproducing model and consequence model
- Compare results: regression = violation in consequence run that wasn't in reproducing run
- Regression report: "Model X previously passed; now fails" (caused by fix) vs. "Model X always fails; unrelated"
- Test: "If neighbor model X is broken before fix, does gate 3 flag it as regression?" NO. "If X passes before and fails after fix?" YES.

**Warning signs:**
- Gate 3 failures are on models unrelated to the bug
- User says: "That model failure has nothing to do with my fix"
- High false-positive rate on gate 3 neighbor checks

### Example 1: Normalize Fix Intent from Natural Language

```javascript
// Source: Proposed intent-normalizer.cjs (Phase 3 new module)

/**
 * Normalize user fix intent into structured mutations.
 * Supports three input channels: constraints, variables, code.
 * @param {string} intentText - User-provided fix idea
 * @returns {{ mutations: Array, confidence: number, ambiguities: Array }}
 */
function normalizeFixIntent(intentText) {
  const mutations = [];
  const ambiguities = [];

  // Channel 1: Constraint patterns
  // "if X then Y", "when X, Y must hold", "X unless Y"
  const constraintPattern = /(?:if|when)\s+([^,]+?)(?:then|,?)\s+([^.;]+?)(?:\.|;|$)/gi;
  let match;
  while ((match = constraintPattern.exec(intentText)) !== null) {
    mutations.push({
      type: 'add_invariant',
      target: `UserConstraint_${mutations.length}`,
      content: `${match[1]} => ${match[2]}`,
      reasoning: `User constraint: "${match[0].trim()}"`
    });
  }

  // Channel 2: Variable/state modifications
  // "add timeout variable", "track state X", "initialize field Y"
  const varPattern = /(?:add|set|track|initialize)\s+(?:state\s+)?(?:variable|field)?\s*(\w+)/gi;
  const varMatches = new Set();
  while ((match = varPattern.exec(intentText)) !== null) {
    if (!varMatches.has(match[1].toLowerCase())) {
      varMatches.add(match[1].toLowerCase());
      mutations.push({
        type: 'add_state_variable',
        target: match[1],
        content: `VARIABLE ${match[1]}`,
        reasoning: `User requested to track: ${match[1]}`
      });
    }
  }

  // Channel 3: Code snippets
  // ```code``` or indented code block
  const codePattern = /```([^`]+?)```/g;
  while ((match = codePattern.exec(intentText)) !== null) {
    mutations.push({
      type: 'code_modification',
      target: 'post_fix_code',
      content: match[1].trim(),
      reasoning: 'User-provided code snippet'
    });
  }

  // Detect ambiguity markers
  if (/maybe|might|could|possibly|perhaps/i.test(intentText)) {
    ambiguities.push('Intent contains hedging language; consider confirming specifics');
  }
  if (/or\s+(?:else|alternatively)/i.test(intentText)) {
    ambiguities.push('Intent presents multiple options; select one for simulation');
  }

  const confidence = mutations.length > 0 ? Math.min(1.0, mutations.length / 3) : 0;

  return { mutations, confidence, ambiguities };
}

// Example usage:
const intent = "When timeout occurs, must retry once. Track retryCount variable.";
const { mutations, confidence, ambiguities } = normalizeFixIntent(intent);
// Result:
// mutations: [
//   { type: 'add_invariant', target: 'UserConstraint_0', content: 'timeout => retry once', ... },
//   { type: 'add_state_variable', target: 'retryCount', content: 'VARIABLE retryCount', ... }
// ]
// confidence: 0.67
// ambiguities: []
```

 * Enforces ResolvedAtWriteOnce formal invariant: once converged verdict is written, it cannot be reverted.
 */
async function runConvergenceGates(consequenceModelPath, reproducingModelPath, bugTrace, sessionId) {
  const sessionDir = path.join(process.cwd(), '.planning/formal/cycle2-simulations', sessionId);
  const verdictLogPath = path.join(sessionDir, 'gate-verdicts.json');

  // Load existing verdicts (write-once: only append)
  const verdictHistory = fs.existsSync(verdictLogPath)
    ? JSON.parse(fs.readFileSync(verdictLogPath, 'utf-8'))
    : [];

  const iterationNumber = verdictHistory.length + 1;

  // Check for prior convergence (ResolvedAtWriteOnce)
  const priorConverged = verdictHistory.some(v => v.converged === true);
  if (priorConverged) {
    console.log('⚠ Prior convergence verdict found. Current iteration cannot revert this.');
  }

  // Run three gates (gate implementations call external checkers)
  console.log(`[Gate 1/3] Checking original invariants in consequence model...`);
  const gate1 = await runGate1(consequenceModelPath);

  console.log(`[Gate 2/3] Checking if bug is resolved...`);
  const gate2 = await runGate2(consequenceModelPath, bugTrace);

  console.log(`[Gate 3/3] Checking neighbor models...`);
  const gate3 = await runGate3(consequenceModelPath, reproducingModelPath);

  // Write-once verdict (append only, never overwrite)
  const verdict = {
    iteration: iterationNumber,
    timestamp: new Date().toISOString(),
    gate1: { passed: gate1.passed, details: gate1.details },
    gate2: { passed: gate2.passed, details: gate2.details },
    gate3: { passed: gate3.passed, regressions: gate3.regressions },
    converged: gate1.passed && gate2.passed && gate3.passed,
    mutable: false  // Formal guarantee: once written, cannot change
  };

  // APPEND to verdict history (ResolvedAtWriteOnce semantics)
  verdictHistory.push(verdict);
  fs.writeFileSync(verdictLogPath, JSON.stringify(verdictHistory, null, 2));

  // Detect write-once violation: if a prior iteration was converged, block reversal
  if (priorConverged && !verdict.converged) {
    console.log(`\n✗ WRITE-ONCE VIOLATION DETECTED`);
    console.log(`  Prior iteration: converged = true`);
    console.log(`  Current iteration: converged = false`);
    console.log(`  ResolvedAtWriteOnce formal invariant: VIOLATED`);
    throw new Error('Cannot revert convergence verdict; write-once semantics broken');
  }

  return verdict;
}

module.exports = { runConvergenceGates };

## Open Questions

1. **Consequence model mutation format standardization**
   - What we know: Phase 1–2 use constraint extraction from existing TLA+/Alloy specs; Phase 3 needs to apply mutations to specs
   - What's unclear: Should mutation format be generic (add_invariant, add_variable, code_modification) or spec-specific (TLA+ VARIABLE syntax vs. Alloy sig)?
   - Recommendation: Generic format with spec-specific applicators; one applyMutation function per formalism (applyTlaMutation, applyAlloyMutation)

2. **Intent parser ambiguity resolution UX**
   - What we know: Intent normalization should ask user for confirmation on ambiguous inputs
   - What's unclear: Should confirmation be blocking (dialog) or non-blocking (list of proposals user can select from)?
   - Recommendation: Non-blocking list of proposals; show "Did you mean one of these?" with examples; let user select or provide clarification

3. **Neighbor regression detection baseline**
   - What we know: Gate 3 should detect regressions (violation in consequence that wasn't in reproducing model)
   - What's unclear: Should we baseline against reproducing model or against original spec? If the reproducing model itself has unrelated violations, how do we distinguish regression from pre-existing issue?
   - Recommendation: Baseline against reproducing model state; store both reproducing and consequence gate results; regression = difference in verdict

4. **Consequence model cleanup strategy**
   - What we know: Phase 3 generates consequence models in cycle2-simulations/{sessionId}/ directory
   - What's unclear: Should consequence models be retained after simulation completes (for audit trail) or deleted to save space (if convergence achieved)?
   - Recommendation: Retain on disk but not in memory; mark successful convergence in metadata so UI can hide converged artifacts by default

5. **Write-once verdict reversion error handling**
   - What we know: ResolvedAtWriteOnce invariant forbids convergence reversion
   - What's unclear: If reversion is attempted (prior iteration converged, current iteration not), should we (a) raise error + block, (b) log warning + continue, (c) auto-promote to converged=true?
   - Recommendation: (a) Raise error + block; formal guarantee violation is critical; force user to understand why reversion occurred
