# Phase v0.33-05: TLA+ Meta-Verification - Research

**Researched:** 2026-03-10
**Domain:** TLA+ formal modeling, TLC model checking, liveness/safety temporal properties
**Confidence:** HIGH

## Summary

This phase creates a new TLA+ specification `NFSolveConvergence.tla` that formally models the outer nf:solve loop with cross-session state, the Option C oscillation blocking rule, and gate maturity transitions. The model must be verified by TLC for both safety (no layer oscillates more than once) and liveness (the solve loop eventually converges) with explicit fairness declarations that pass the existing liveness-fairness CI lint.

The project already has extensive TLA+ infrastructure: 40+ models in `.planning/formal/tla/`, multiple TLC runner scripts (`run-tlc.cjs`, `run-oscillation-tlc.cjs`), a liveness-fairness CI lint (`check-liveness-fairness.cjs`), and a well-established pattern for fairness declarations in `.planning/formal/spec/*/invariants.md`. The new model builds on concepts from two existing specs: `NFSolveOrchestrator.tla` (models the intra-session convergence loop with bounded iterations, phase ordering, and termination) and `NFOscillation.tla` (models the run-collapse oscillation detection algorithm). However, the new model operates at a different abstraction level -- it models cross-session behavior where each "step" is an entire solve session, not an intra-session phase transition.

The key modeling challenge is capturing the Option C credit system from `bin/oscillation-detector.cjs`: each layer starts with 1 oscillation credit, a residual increase after a prior decrease consumes a credit, and a second oscillation blocks automated remediation. The cascade-aware grace period (OSC-02) from the dependency DAG adds complexity but is essential for the liveness proof -- without it, the model could deadlock when upstream fixes legitimately increase downstream residuals.

**Primary recommendation:** Model N layers (parameterized, default 3-4 for tractable state space) with bounded history of last N sessions, per-layer oscillation credits, the Option C blocking rule, and weak fairness on the "make progress" action. Use the existing `run-oscillation-tlc.cjs` runner pattern (extend its config map to include the new spec) rather than creating a new runner script.

### Pattern 1: State Variables for Cross-Session Solve Loop

The model abstracts N layers and a bounded history window of the last H sessions. Key state variables:

```tla
CONSTANTS
    Layers,          \* Set of abstract layer identifiers, e.g. {L1, L2, L3}
    MaxSessions,     \* Bounded history window (e.g., 5)
    MaxResidual      \* Upper bound on residual values (e.g., 3)

VARIABLES
    session,                \* Current session number (0..MaxSessions)
    residual,               \* [Layers -> 0..MaxResidual] per-layer residual
    prevResidual,           \* [Layers -> -1..MaxResidual] previous session residual (-1 = no history)
    oscillation_count,      \* [Layers -> 0..2] per-layer oscillation count
    blocked,                \* [Layers -> BOOLEAN] per-layer Option C block status
    gate_maturity,          \* [Layers -> {"ADVISORY","SOFT_GATE","HARD_GATE"}] gate level
    converged,              \* BOOLEAN: all layers converged
    graceActive             \* [Layers -> BOOLEAN] cascade-aware grace period
```

**Why this structure:** Mirrors the actual data stored in `oscillation-verdicts.json` (oscillation_count, blocked, credits_remaining) and `solve-trend.jsonl` (per_layer residuals). The bounded history avoids infinite state spaces.

### Pattern 2: Option C Credit System Actions

```tla
\* RunSession(layer): a solve session processes one layer
\* Non-deterministically assigns new residual (TLC explores all values)
RunSession ==
    /\ session < MaxSessions
    /\ \E l \in Layers :
        /\ ~blocked[l]   \* Option C: blocked layers cannot be remediated
        /\ \E newResidual \in 0..MaxResidual :
            /\ residual' = [residual EXCEPT ![l] = newResidual]
            /\ prevResidual' = [prevResidual EXCEPT ![l] = residual[l]]
            \* Oscillation detection: increase after decrease
            /\ IF prevResidual[l] >= 0
                  /\ residual[l] < prevResidual[l]  \* had a decrease
                  /\ newResidual > residual[l]       \* now increasing
                  /\ ~graceActive[l]                 \* no cascade grace
               THEN oscillation_count' = [oscillation_count EXCEPT ![l] = @ + 1]
               ELSE UNCHANGED oscillation_count
            \* Block on second oscillation
            /\ blocked' = [blocked EXCEPT ![l] = oscillation_count'[l] > 1]
```

**Key insight:** The Option C rule is: `oscillation_count[l] <= 1` for safety (no more than 1 oscillation). On the second oscillation, `blocked` becomes TRUE and the layer exits automated remediation.

### Pattern 3: Gate Maturity Transitions

Gate maturity models the promotion/demotion cycle from `gate-stability.cjs`:

```tla
GateTransition ==
    \E l \in Layers :
        /\ gate_maturity[l] \in {"ADVISORY", "SOFT_GATE"}
        /\ gate_maturity' = [gate_maturity EXCEPT ![l] =
            CASE gate_maturity[l] = "ADVISORY" -> "SOFT_GATE"
              [] gate_maturity[l] = "SOFT_GATE" -> "HARD_GATE"
              [] OTHER -> gate_maturity[l]]
        /\ UNCHANGED <<session, residual, prevResidual,
                       oscillation_count, blocked, converged, graceActive>>
```

Gate maturity only advances forward in this model (matching the cooldown enforcement from STAB-02 which prevents premature re-promotion).

### Pattern 4: Safety Property (FV-02)

```tla
\* Safety: no layer oscillates more than once
\* @requirement FV-02
OscillationBounded ==
    \A l \in Layers : oscillation_count[l] <= 1
```

This is a state invariant (checked in every reachable state). It must hold because the Option C blocking rule prevents further remediation after `oscillation_count` reaches 1, so no action can increase it past 1 (the blocked layer is excluded from RunSession).

**Important:** The invariant says `<= 1`, meaning at most one oscillation is allowed. The second oscillation triggers the block BEFORE incrementing past 1 in the refined model. The requirement text says "no layer oscillates more than once" which maps to `oscillation_count[layer] <= 1`.

### Pattern 5: Liveness Property (FV-03)

```tla
\* Liveness: solve loop eventually converges
\* @requirement FV-03
EventualConvergence == <>(converged = TRUE)
```

This requires fairness assumptions. Under weak fairness on the enabling actions:
- `WF_vars(RunSession)` -- if a non-blocked layer exists with residual > 0, a session eventually runs
- `WF_vars(CheckConvergence)` -- convergence check eventually fires after a session

The liveness proof relies on:
1. The state space is finite (bounded sessions, bounded residuals, bounded layers)
2. Option C ensures layers that oscillate are blocked, reducing the problem
3. Eventually all non-blocked layers either reach residual 0 or are stable
4. Once all layers are converged or blocked, `converged` becomes TRUE

### Pattern 6: Fairness Declarations for CI Lint

The `check-liveness-fairness.cjs` CI lint scans `MC*.cfg` files for PROPERTY lines, then checks `.planning/formal/spec/{surface}/invariants.md` for matching `## PropertyName` sections. The new spec needs:

1. A `.cfg` file with `PROPERTY EventualConvergence` (and any other liveness properties)
2. A `solve-convergence/invariants.md` file with `## EventualConvergence` section documenting WF/SF rationale
3. The surface name derived from the config name: `MCsolve-convergence` -> surface `solve-convergence`

The `detectLivenessProperties()` function strips the `MC` prefix and lowercases to get the surface name, then looks for `invariants.md` in that subdirectory.

### Anti-Patterns to Avoid
- **Modeling all 19 layers:** Creates state space explosion. Abstract to 3-4 layers with one upstream-downstream pair.
- **Unbounded session history:** Use bounded `MaxSessions` constant (3-5 for model checking).
- **Forgetting cascade grace period:** Without modeling the grace period, the liveness proof fails because downstream layers would be blocked by legitimate cascade effects.
- **Using `-workers auto` for liveness:** TLC v1.8.0 has a known multi-worker liveness bug. Always use `-workers 1`.

## Common Pitfalls

### Pitfall 1: State Space Explosion
**What goes wrong:** Too many layers or too large MaxResidual causes TLC to run for hours or exhaust memory.
**Why it happens:** TLC explores every reachable state. With N layers, M max residual, and H history, state space is O(M^N * 3^N * 2^N * H).
**How to avoid:** Start with `Layers = {L1, L2, L3}`, `MaxSessions = 5`, `MaxResidual = 3`. These give tractable state spaces (~10K-100K states). The existing `quick-235` task already fixed state space explosions in other models.
**Warning signs:** TLC runtime > 120 seconds triggers `timeout-risk` triage tag.

### Pitfall 2: Liveness Without Fairness
**What goes wrong:** TLC reports liveness property violation because stuttering steps (doing nothing) are allowed.
**Why it happens:** Without fairness, TLC can find a behavior where the system stops making progress forever (infinite stuttering).
**How to avoid:** Declare `WF_vars(action)` for all actions that must eventually fire. The Spec definition must include fairness conjuncts.
**Warning signs:** `check-liveness-fairness.cjs` reports `inconclusive` for the new config.

### Pitfall 3: Oscillation Count Off-by-One
**What goes wrong:** The invariant `oscillation_count[l] <= 1` fails because the blocking rule fires too late (after incrementing to 2).
**Why it happens:** The block must be checked AFTER incrementing the counter. If the increment and block check are in the same action, the counter reaches 2 momentarily.
**How to avoid:** Either (a) block when `oscillation_count` reaches 1 (before it can reach 2), or (b) make the invariant `oscillation_count[l] <= 2` and instead check `blocked[l] = TRUE` when count >= 2. Approach (a) matches the requirement text exactly.

### Pitfall 4: Missing invariants.md Section Headers
**What goes wrong:** The CI liveness-fairness lint reports `inconclusive` even though fairness is declared in the .tla file.
**Why it happens:** `detectLivenessProperties()` looks for `## PropertyName` headers in `invariants.md`. If the section header doesn't exactly match the PROPERTY name in the .cfg file, the lint fails.
**How to avoid:** Ensure the invariants.md section header exactly matches: `## EventualConvergence` for `PROPERTY EventualConvergence`.

### Pitfall 5: Config Name Must Match Surface Directory
**What goes wrong:** The liveness lint cannot find the invariants.md file.
**Why it happens:** `detectLivenessProperties()` strips `MC` prefix and lowercases to derive the surface name. `MCsolve-convergence` -> `solve-convergence`. The invariants.md must be at `.planning/formal/spec/solve-convergence/invariants.md`.
**How to avoid:** Name the config `MCsolve-convergence.cfg` and the spec directory `solve-convergence/`.

### Example 3: Fairness Declarations (invariants.md)

```markdown
# Liveness Fairness Declarations: solve-convergence

**Spec source:** `formal/tla/NFSolveConvergence.tla`
**Config:** `formal/tla/MCsolve-convergence.cfg`

## EventualConvergence

**Property:** `EventualConvergence == <>(converged = TRUE)`
**Config line:** `PROPERTY EventualConvergence` (MCsolve-convergence.cfg)
**Fairness assumption:** WF_vars on 2 actions: RunSession, CheckConvergence
**Realism rationale:** RunSession models a single nf:solve invocation processing
a non-blocked layer. By weak fairness, if a non-blocked layer with residual > 0
exists, a solve session must eventually run on it -- models the user or CI
invoking nf:solve periodically. CheckConvergence fires after each session to
evaluate whether all layers have converged (residual = 0) or are blocked.
Together, these ensure the loop eventually terminates: layers either converge
to zero residual or are blocked by Option C after oscillating.

**Source:** `formal/tla/NFSolveConvergence.tla`
```

## Open Questions

1. **Cascade grace period modeling granularity**
   - What we know: `oscillation-detector.cjs` uses a LAYER_DEPS DAG for grace periods. The model needs a `DownstreamOf` constant.
   - What's unclear: Should the model use a full DAG or simplify to a single upstream-downstream pair?
   - Recommendation: Use a parameterized `DownstreamOf` constant with the simplest non-trivial DAG (L1 -> L2 -> L3) for the default config. This captures the cascade effect without exploding state space.

2. **Gate maturity interaction with convergence**
   - What we know: Gate maturity transitions (ADVISORY -> SOFT_GATE -> HARD_GATE) are modeled in `gate-stability.cjs` with cooldown enforcement.
   - What's unclear: Does gate maturity affect the convergence definition? FV-01 says "gate maturity transitions" but FV-02/FV-03 don't reference gates.
   - Recommendation: Model gate transitions as a separate action that can interleave with solve sessions. Convergence is defined purely on residuals and oscillation counts, not gate levels. Gate maturity is modeled for completeness (FV-01) but not part of the safety/liveness properties.

3. **Model registry integration**
   - What we know: `model-registry.json` tracks all formal models with requirements, gate_maturity, etc.
   - What's unclear: Should the new model be manually registered or auto-registered?
   - Recommendation: Manually add to model-registry.json with `requirements: ["FV-01", "FV-02", "FV-03"]` and `gate_maturity: "ADVISORY"` initially.
