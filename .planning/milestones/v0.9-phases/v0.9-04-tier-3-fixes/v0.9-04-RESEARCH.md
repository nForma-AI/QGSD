## Summary

This phase addresses three independent defects discovered in the QGSD codebase. The fixes are surgical: add guard notes at Task spawn points in two workflow files, confirm Gemini TOML conversion correctness for the missing quorum commands, and fix `parseInt()` calls in `gsd-tools.cjs` that strip the decimal part from phase numbers like `"2.1"`.

**FIX-01 / FIX-02 (Skill tool guards):** Claude Code's Skill tool is a command-line invocation mechanism that can be misused in subagent spawn contexts. The four `Task(...)` spawn points in `plan-phase.md` and the one in `discuss-phase.md` currently have no guard text warning Claude not to invoke a Skill (e.g., `mcp__gemini-cli-1__gemini`) as a way to "spawn" the sub-agent. Adding a single comment line before each Task block is sufficient.

**FIX-03 (Gemini TOML correctness):** The installer's `convertClaudeToGeminiToml()` function uses `JSON.stringify(body)` to produce the TOML `prompt = "..."` value. Verification confirms that `@iarna/toml` (the TOML parser used by Gemini CLI 0.28.2) successfully parses these outputs for all 38 QGSD command files including `quorum.md` (24KB prompt) and `quorum-test.md`. The actual gap is that `quorum.md` and `quorum-test.md` are missing from the Gemini command installation — they exist in `/Users/jonathanborduas/code/QGSD/commands/qgsd/` but are absent from `~/.gemini/commands/gsd/`. The fix is to re-run `node bin/install.js --gemini --global` after ensuring both files convert correctly. Since verification already shows they do convert correctly, the fix is primarily running the install.

**FIX-04 (Decimal phase parsing):** Multiple `parseInt(phase, 10)` calls in `gsd-tools.cjs` strip the decimal part from phases like `"2.1"` or `"06.1"`. The affected subcommands are `validate consistency` (lines ~3639, ~3645-3648), `validate health` (lines ~3791, ~3874, ~3882), and potentially edge cases in `phase complete`. The fix is to replace `parseInt(p, 10)` with a decimal-preserving extraction: use `parseFloat(p)` or the existing `normalizePhaseName()` function in the comparison logic.

**Primary recommendation:** One PLAN.md addressing all four fixes in a single execution wave. The guard text is a two-line edit per spawn point; the Gemini install is one bash command; the parseInt fixes are targeted regex replacements in gsd-tools.cjs.

### FIX-04: Decimal Phase Parsing

**The bug pattern:** JavaScript's `parseInt("2.1", 10)` returns `2`. Any code comparing a decimal phase number string after passing through `parseInt` loses the decimal part.

**Root cause locations in `~/.claude/qgsd/bin/gsd-tools.cjs`:**

```
Line ~3639 (validate consistency):
  const unpadded = String(parseInt(p, 10));  // "06.1" → "6"

Line ~3645-3648 (validate consistency, integer-only filter):
  .map(p => parseInt(p, 10))  // This is OK — explicitly integers only

Line ~3791 (validate health, STATE.md refs check):
  const normalizedRef = String(parseInt(ref, 10)).padStart(2, '0');  // "2.1" → "02"

Line ~3874 (validate health, ROADMAP vs disk):
  const padded = String(parseInt(p, 10)).padStart(2, '0');  // "2.1" → "02"

Line ~3882 (validate health, disk vs ROADMAP):
  const unpadded = String(parseInt(p, 10));  // "06.1" → "6"
```

**Fix pattern for comparison sites:**

```javascript
// BEFORE (broken for decimals):
const unpadded = String(parseInt(p, 10));

// AFTER (decimal-preserving):
// Use normalizePhaseName() which already handles "06.1" correctly:
const normalized = normalizePhaseName(p);
// OR for un-padded comparison:
const unpadded = p.includes('.')
  ? `${parseInt(p, 10)}.${p.split('.')[1]}`  // "06.1" → "6.1"
  : String(parseInt(p, 10));                  // "06" → "6"
```

**Why `normalizePhaseName()` is the right tool:**

```javascript
// Existing function in gsd-tools.cjs:
function normalizePhaseName(phase) {
  const msp = parseMilestonePhaseId(phase);
  if (msp) return msp.full;
  const match = phase.match(/^(\d+(?:\.\d+)?)/);
  if (!match) return phase;
  const num = match[1];
  const parts = num.split('.');
  const padded = parts[0].padStart(2, '0');
  return parts.length > 1 ? `${padded}.${parts[1]}` : padded;
}
// "2.1" → "02.1"  ✓
// "6.1" → "06.1"  ✓
// "2"   → "02"    ✓
```

**Specific fix strategy per site:**

| Site | Current code | Fix |
|------|-------------|-----|
| L3639 `validate consistency` | `String(parseInt(p, 10))` | Extract decimal-aware: `normalizePhaseName(p)` or strip-pad manually |
| L3791 `validate health` STATE ref | `String(parseInt(ref, 10)).padStart(2,'0')` | Use `normalizePhaseName(ref)` instead |
| L3874 `validate health` ROADMAP vs disk | `String(parseInt(p, 10)).padStart(2,'0')` | Use `normalizePhaseName(p)` |
| L3882 `validate health` disk vs ROADMAP | `String(parseInt(p, 10))` | Use decimal-preserving extraction |

**Note on installed vs source files:** `~/.claude/qgsd/bin/gsd-tools.cjs` is the installed copy. The source is `get-shit-done/bin/gsd-tools.cjs` in the QGSD repo. After fixing the source, run `node bin/install.js --claude --global` to sync.

## Common Pitfalls

### Pitfall 1: Editing the installed copy instead of the source
**What goes wrong:** Editing `~/.claude/qgsd/bin/gsd-tools.cjs` directly. Changes are overwritten on next install.
**How to avoid:** Always edit `get-shit-done/bin/gsd-tools.cjs` in the QGSD source repo, then run `node bin/install.js --claude --global` to sync.
**Warning signs:** Changes work in one session but disappear after `/qgsd:update` or reinstall.

### Pitfall 2: Guard text breaking workflow parsing
**What goes wrong:** If the guard note is placed INSIDE a code block or in a position that disrupts the markdown structure, the orchestrating Claude may misparse the workflow.
**How to avoid:** Place guard notes OUTSIDE code blocks, as a prose paragraph or blockquote immediately before the ```` ```Task( ``` ```` block.
**Warning signs:** Workflow orchestrator fails to spawn sub-agents or spawns wrong agent type.

### Pitfall 3: Missing Gemini reinstall after quorum.md install
**What goes wrong:** quorum.md and quorum-test.md get added to `~/.gemini/commands/gsd/` but with stale path references (old install prefix).
**How to avoid:** Always run `node bin/install.js --gemini --global` from the QGSD repo root — the installer handles path replacement.
**Warning signs:** `/gsd:quorum` in Gemini fails to load or references wrong paths.

### Pitfall 4: parseInt fix breaking integer-only filter
**What goes wrong:** Line ~3645-3648 in `validate consistency` explicitly filters for integer-only phases: `.filter(p => !p.includes('.')).map(p => parseInt(p, 10))`. This is CORRECT behavior — decimal phases should not be in the "sequential integers" check. Do not change this block.
**How to avoid:** Only fix the comparison sites (lines ~3639, ~3791, ~3874, ~3882), not the integer-only sequential check.
