# Plan v0.21-04-04 Summary: Phase Spec Generator (SPEC-04)

## What Was Built

A CLI tool that reads `must_haves: truths:` from `*-PLAN.md` YAML frontmatter and translates them into a per-phase TLA+ scratch spec with INVARIANT/PROPERTY stubs. Closes the loop between planning requirements and formal verification.

**SPEC-04: Plan truths → machine-checkable TLA+ contracts**

### Artifacts Created

| File | Purpose |
|------|---------|
| `bin/generate-phase-spec.cjs` | CLI tool: reads *-PLAN.md frontmatter, classifies truths (INVARIANT/PROPERTY), emits scratch TLA+ spec |
| `bin/generate-phase-spec.test.cjs` | 8 unit tests: classifyTruth, parsePlanFrontmatter, generatePhaseSpec, CLI integration |
| `.formal/tla/scratch/.gitkeep` | Ensures .formal/tla/scratch/ directory is tracked in git |

### Key Exports

```js
module.exports = { generatePhaseSpec, classifyTruth, parsePlanFrontmatter };
```

- `parsePlanFrontmatter(content)` — extracts phase + truths from *-PLAN.md YAML frontmatter (NOT task-envelope.json)
- `classifyTruth(truth)` — returns 'PROPERTY' for liveness keywords, 'INVARIANT' for safety keywords
- `generatePhaseSpec({ phase, truths })` — produces TLA+ module with INVARIANT/PROPERTY stubs

| "progress guaranteed after quorum" | PROPERTY (liveness) |

## Decisions

- **Data source is *-PLAN.md frontmatter**: task-envelope.json has empty truths at planning time. The plan's YAML frontmatter is the correct source (confirmed: 26 truths from v0.21-04 plans, 0 from task-envelope.json).
- **No YAML dependency**: manual parser handles the `must_haves: truths:` sub-key structure to avoid adding dependencies.
- **Output in .formal/tla/scratch/**: scratch specs are timestamped per-run and NOT committed to git. The .gitkeep ensures the directory exists.
