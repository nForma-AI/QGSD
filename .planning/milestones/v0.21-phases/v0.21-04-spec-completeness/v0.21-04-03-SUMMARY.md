## Facts Verified by Alloy

- **NoEmptySelection**: `all c : Config | (#c.availableSlots > 0) implies (#c.selectedSlots > 0)` — no zero-agent polling
- **HighRiskFullFanOut**: `all c : Config | (c.riskLevel = High) implies (#c.selectedSlots >= #c.availableSlots)` — high-risk uses all available slots
- **SoloModeSingleSlot**: `all c : Config | (c.soloMode = True) implies (#c.selectedSlots = 1)` — solo mode single-slot enforcement
- **AllRulesHold**: checked assertion — no counterexample found within scope 5 Config, 5 Slot, 4 int

## Deviation

**HighRiskFullFanOut uses `>= #c.availableSlots` instead of `min[#c.availableSlots, c.maxSize]`**

The plan specified `min[#c.availableSlots, c.maxSize]` for the HighRiskFullFanOut fact. Alloy 6.2.0 CLI exec mode does not support the `min[a, b]` syntax — it resolves as a bad join type error. Per the plan's fallback: the over-approximation `#c.selectedSlots >= #c.availableSlots` is used instead. The `selectedSlots in availableSlots` Config constraint (line 13) bounds selection to available slots only, so this over-approximation is equivalent when `availableSlots.size <= maxSize`. The formal verification is sound for the standard quorum composition case.
