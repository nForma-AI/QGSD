## Summary

This phase establishes the foundation for QGSD's Production Feedback Loop by implementing a debt ledger (`.formal/debt.json`) with validated schema, state machine enforcement, and deterministic fingerprinting for both issues and drifts. The debt ledger tracks observations from production sources and acts as the single source of truth for what production reality has surfaced as problems.

The critical work spans five areas: (1) schema design and validation using JSON Schema + Node.js runtime validation, (2) state machine enforcement for debt status transitions (open → acknowledged → resolving → resolved), (3) retention policies for archived resolved entries, (4) hierarchical fingerprinting for issues (exception type → function → message hash), and (5) parameter-key fingerprinting for drifts. All patterns follow the project's established conventions: CommonJS utilities, node:test framework for testing, crypto module for hashing, and fail-open error handling.

**Primary recommendation:** Use JSON Schema (draft-7) for validation schema definition with explicit runtime validation via simple `validateDebtEntry()` and `validateDebtLedger()` functions (not AJV — keep dependencies minimal). Implement state machine as a simple guard function using switch statements (not a separate state machine library). Use crypto.createHash('sha256') for fingerprinting, matching existing project patterns. Write comprehensive unit tests covering schema validation, state transitions, fingerprint stability, and retention logic before implementation tasks begin.

## Architecture Patterns

### Pattern 1: Schema Validation
**What:** Validate debt entries against JSON Schema at write time. Reject invalid entries before persistence.

**When to use:** Every write operation to debt.json (add, update); every read operation before processing.

**Implementation approach:**
```javascript
// bin/validate-debt-entry.cjs (NEW)
function validateDebtEntry(entry) {
  const errors = [];

  // Check required fields
  if (!entry.id || typeof entry.id !== 'string') {
    errors.push('id required (string)');
  }
  if (!entry.fingerprint || typeof entry.fingerprint !== 'string') {
    errors.push('fingerprint required (string)');
  }
  if (!entry.title || typeof entry.title !== 'string' || entry.title.length < 1) {
    errors.push('title required (non-empty string)');
  }
  if (typeof entry.occurrences !== 'number' || entry.occurrences < 1) {
    errors.push('occurrences required (integer >= 1)');
  }

  // Check status enum
  if (!['open', 'acknowledged', 'resolving', 'resolved'].includes(entry.status)) {
    errors.push('status must be one of: open, acknowledged, resolving, resolved');
  }

  // Check timestamps are valid ISO8601
  if (!isValidISO8601(entry.first_seen)) errors.push('first_seen must be ISO8601');
  if (!isValidISO8601(entry.last_seen)) errors.push('last_seen must be ISO8601');
  if (new Date(entry.last_seen) < new Date(entry.first_seen)) {
    errors.push('last_seen must be >= first_seen');
  }

  // Check environments array
  if (!Array.isArray(entry.environments) || entry.environments.length === 0) {
    errors.push('environments required (non-empty array)');
  }
  const validEnvs = ['production', 'staging', 'development', 'test', 'local'];
  for (const env of entry.environments) {
    if (!validEnvs.includes(env)) errors.push(`invalid environment: ${env}`);
  }

  // Check source_entries array
  if (!Array.isArray(entry.source_entries) || entry.source_entries.length === 0) {
    errors.push('source_entries required (non-empty array)');
  }
  for (const se of (entry.source_entries || [])) {
    if (!se.source_type || !se.source_id || !se.observed_at) {
      errors.push('each source_entry must have: source_type, source_id, observed_at');
    }
  }

  return errors.length === 0 ? true : errors;
}

function validateDebtLedger(ledger) {
  const errors = [];
  if (ledger.schema_version !== '1') {
    errors.push('schema_version must be "1"');
  }
  if (!Array.isArray(ledger.debt_entries)) {
    errors.push('debt_entries must be an array');
  }
  for (let i = 0; i < (ledger.debt_entries || []).length; i++) {
    const entryErrors = validateDebtEntry(ledger.debt_entries[i]);
    if (entryErrors !== true) {
      errors.push(`debt_entries[${i}]: ${entryErrors.join('; ')}`);
    }
  }
  return errors.length === 0 ? true : errors;
}

module.exports = { validateDebtEntry, validateDebtLedger };
```

**Pattern reference:** Matches project's validation approach in aggregate-requirements.cjs and validate-traces.cjs.

### Pattern 2: State Machine - Debt Status Transitions
**What:** Enforce allowed state transitions (open → acknowledged → resolving → resolved). Reject invalid transitions (e.g., resolved → open).

**When to use:** Any operation that attempts to change a debt entry's status field.

**Implementation approach:**
```javascript
// bin/debt-state-machine.cjs (NEW)
// Simple state transition guard — no external state machine library needed.

const ALLOWED_TRANSITIONS = {
  'open': ['acknowledged'],
  'acknowledged': ['resolving', 'open'],  // can revert to open for clarification
  'resolving': ['resolved'],
  'resolved': []  // terminal state — no outbound transitions
};

function canTransition(fromStatus, toStatus) {
  // Reject transition to same status (no-op)
  if (fromStatus === toStatus) return false;

  // Check if transition is in allowed list
  const allowed = ALLOWED_TRANSITIONS[fromStatus] || [];
  return allowed.includes(toStatus);
}

function transitionDebtEntry(entry, newStatus) {
  const errors = [];

  if (!['open', 'acknowledged', 'resolving', 'resolved'].includes(newStatus)) {
    errors.push(`Invalid target status: ${newStatus}`);
  }

  if (!canTransition(entry.status, newStatus)) {
    errors.push(`Transition not allowed: ${entry.status} -> ${newStatus}`);
  }

  if (errors.length > 0) {
    return { success: false, errors };
  }

  // Apply transition
  const updated = { ...entry, status: newStatus };
  if (newStatus === 'resolved') {
    updated.resolved_at = new Date().toISOString();
  }

  return { success: true, entry: updated };
}

module.exports = { canTransition, transitionDebtEntry, ALLOWED_TRANSITIONS };
```

**Anti-Patterns to Avoid:**
- **Reopening resolved debt:** State machine prevents transitions out of 'resolved' state. Resolved is terminal.
- **Skipping acknowledgement:** Code must not allow direct open → resolving. Human triage (acknowledgement) is intentional gate.
- **Status stored as number enum:** Keep status as string literal (open|acknowledged|resolving|resolved) for readability and schema clarity.

### Pattern 3: Fingerprinting - Issues
**What:** Generate stable, deterministic fingerprint for error/exception debt. Same error input → same fingerprint.

**When to use:** When a debt entry is created from an issue source (GitHub, Sentry, bash error).

**Hierarchical strategy:**
1. Extract exception type (e.g., "TypeError", "RuntimeError")
2. Extract function/method name where error occurred
3. Hash error message text (ignore timestamp, stack traces vary)
4. Combine: `sha256(type + ':' + functionName + ':' + msgHash)`

**Implementation approach:**
```javascript
// bin/fingerprint-issue.cjs (NEW)
const crypto = require('crypto');

function hashMessage(msg) {
  // Hash just the error message, stripped of timestamps/line numbers
  const normalized = (msg || '')
    .replace(/\d{4}-\d{2}-\d{2}T[\d:.Z]+/g, 'TIMESTAMP')
    .replace(/:\d+/g, ':LINE')
    .toLowerCase()
    .trim();
  return crypto.createHash('sha256').update(normalized).digest('hex').slice(0, 16);
}

function fingerprintIssue(issue) {
  // issue = { exception_type?, function_name?, message, ... }
  const exceptionType = (issue.exception_type || 'unknown').toLowerCase();
  const functionName = (issue.function_name || 'unknown').toLowerCase().replace(/[^a-z0-9_]/g, '_');
  const msgHash = hashMessage(issue.message);

  const combined = `${exceptionType}:${functionName}:${msgHash}`;
  const fingerprint = crypto.createHash('sha256').update(combined).digest('hex').slice(0, 16);

  return fingerprint;
}

// Test: same input always produces same output
const fp1 = fingerprintIssue({
  exception_type: 'TypeError',
  function_name: 'parseConfig',
  message: 'Cannot read property "id" of undefined'
});
const fp2 = fingerprintIssue({
  exception_type: 'TypeError',
  function_name: 'parseConfig',
  message: 'Cannot read property "id" of undefined'
});
console.assert(fp1 === fp2, 'Fingerprints must be identical for same input');

module.exports = { fingerprintIssue, hashMessage };
```

**Stability guarantee:** Fingerprints are deterministic (no random components, no timestamps). Running the same issue through fingerprinting 1000 times yields the same fingerprint.

### Pattern 4: Fingerprinting - Drifts
**What:** Generate stable fingerprint for production drift entries. Key is formal parameter reference.

**When to use:** When a debt entry represents a divergence between formal model parameter and production measurement.

**Implementation approach:**
```javascript
// bin/fingerprint-drift.cjs (NEW)
const crypto = require('crypto');

function fingerprintDrift(drift) {
  // drift = { formal_parameter_key, ... }
  // formal_parameter_key = "MCsafety.cfg:MaxDeliberation" or "model.prism:lambda_rate"

  if (!drift.formal_parameter_key) {
    throw new Error('Drift fingerprinting requires formal_parameter_key');
  }

  // Deterministic: just hash the parameter key directly
  const fingerprint = crypto.createHash('sha256')
    .update(drift.formal_parameter_key)
    .digest('hex')
    .slice(0, 16);

  return fingerprint;
}

// Test: identical parameter keys produce identical fingerprints
const fp1 = fingerprintDrift({ formal_parameter_key: 'MCsafety.cfg:MaxDeliberation' });
const fp2 = fingerprintDrift({ formal_parameter_key: 'MCsafety.cfg:MaxDeliberation' });
console.assert(fp1 === fp2, 'Fingerprints must match for identical parameter keys');

// Different parameters produce different fingerprints
const fp3 = fingerprintDrift({ formal_parameter_key: 'MCsafety.cfg:MinQuorum' });
console.assert(fp1 !== fp3, 'Different parameters must produce different fingerprints');

module.exports = { fingerprintDrift };
```

**Key property:** Fingerprint depends only on the parameter key, not on the measured value or timestamp. This ensures cross-run stability.

### Pattern 5: Retention Policy
**What:** Archive resolved debt entries older than a configurable age (default 90 days) from active ledger.

**When to use:** During a periodic cleanup or observe run. Entries are moved to `.formal/debt-archive.jsonl` (one entry per line).

**Implementation approach:**
```javascript
// bin/debt-retention.cjs (NEW)
const fs = require('fs');
const path = require('path');

function applyRetentionPolicy(ledger, maxAgeDays = 90) {
  const now = new Date();
  const cutoffDate = new Date(now.getTime() - maxAgeDays * 24 * 60 * 60 * 1000);

  const active = [];
  const archived = [];

  for (const entry of (ledger.debt_entries || [])) {
    const resolvedDate = new Date(entry.resolved_at || entry.last_seen);

    // Keep: not resolved, or resolved but recent
    if (entry.status !== 'resolved' || resolvedDate > cutoffDate) {
      active.push(entry);
    } else {
      // Archive: resolved and older than maxAgeDays
      archived.push(entry);
    }
  }

  return { active, archived };
}

function writeArchive(archivedEntries, archivePath) {
  // Write JSONL: one entry per line, no array wrapper
  const lines = archivedEntries.map(e => JSON.stringify(e));
  fs.appendFileSync(archivePath, lines.join('\n') + '\n', 'utf8');
}

module.exports = { applyRetentionPolicy, writeArchive };
```

**Archive format:** `.formal/debt-archive.jsonl` — JSONL (JSON Lines), one entry per line, no outer array.

**Cleanup trigger:** Can run:
- On demand: `bin/debt-retention.cjs`
- During observe: After writing new observations
- Scheduled: Nightly or weekly

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON Schema validation | Custom recursive type checker | Simple function-based validation (like aggregate-requirements.cjs) | Recursion complexity, edge cases with nested objects, type coercion traps |
| State machine | Custom state tracker with flags | Simple ALLOWED_TRANSITIONS lookup table + canTransition guard | State explosion (conflicting flags), hard to audit transitions, less testable |
| Deterministic hashing | Custom hash algorithm or salt-based IDs | crypto.createHash('sha256') | Hash collision risks, non-reproducibility, security issues |
| Duplicate detection | Custom string distance algorithm | For Phase 3: use Levenshtein via simple implementation (not npm pkg) | Wheel reinvention; Levenshtein is well-studied; can implement in ~50 lines |
| File I/O orchestration | Manual fs.writeFileSync + try/catch scattered | Centralized utility (debt-write.cjs, debt-read.cjs) | Fail-open consistency, atomic writes, backup handling |

**Key insight:** All of these look deceptively simple (especially state machines and fingerprinting) but have subtle failure modes. Pre-built solutions add external dependencies that conflict with the project's minimal-dependency ethos. Instead, the project pattern is: define the data structure precisely, implement slim validators/guards, and test exhaustively.

## Common Pitfalls

### Pitfall 1: Fingerprint Collisions from Normalization
**What goes wrong:** Two different errors normalize to the same fingerprint because message hashing strips too much context. Example: "Cannot read X of undefined" and "Cannot read Y of undefined" both hash to same value if function name is same.

**Why it happens:** Aggressive normalization (removing variable names, line numbers) to achieve stability.

**How to avoid:** Include function name AND exception type in the combined hash input. Test with realistic error messages from actual Sentry/GitHub data to verify separation.

**Warning signs:** Multiple distinct errors are getting deduplicated together during Phase 3 testing. Fingerprint hash value appears in multiple debt entries with different error messages.

**Remediation:** Increase hash granularity — include more context (class name, module path) before hashing message.

### Pitfall 2: Timestamps Causing Instability
**What goes wrong:** Fingerprint includes timestamps or line numbers, causing identical errors to get different fingerprints on second occurrence.

**Why it happens:** Copy-pasting raw error message instead of normalizing it.

**How to avoid:** Normalize before hashing: remove ISO8601 timestamps, replace line numbers with `LINE`, convert paths to relative.

**Warning signs:** Same error produces different fingerprints in successive runs. `git diff .formal/debt.json` shows duplicated entries with only timestamp differences.

**Remediation:** Write a normalization test: run the same error message through fingerprinting 5 times, assert all fingerprints are identical.

### Pitfall 3: State Machine Allows Invalid Transitions
**What goes wrong:** Code allows transition from resolved → open (reopening), violating the intent that resolved is terminal.

**Why it happens:** Forgot to enforce terminal state, or added "revert to open for re-triage" without updating guard logic.

**How to avoid:** Write explicit test cases for invalid transitions. Test harness should verify:
- `canTransition('resolved', 'open')` returns false
- `canTransition('acknowledged', 'acknowledged')` returns false (no-op)
- `canTransition('open', 'acknowledged')` returns true

**Warning signs:** A resolved debt entry suddenly becomes open again in production. Tests don't cover state transitions.

**Remediation:** Unit tests for all 16 state pairs (4 states × 4 possible targets), and one integration test: load real debt.json, attempt 20 random transitions, verify all invalid ones are rejected.

### Pitfall 4: Retention Policy Deletes Recent Entries
**What goes wrong:** Entries with `last_seen` very recent are incorrectly archived because `resolved_at` is missing or misinterpreted.

**Why it happens:** Using wrong timestamp field (`created_at` vs `last_seen` vs `resolved_at`), or off-by-one date calculation.

**How to avoid:** Always use `resolved_at` for archival cutoff, and only archive entries where `status === 'resolved'`. Test with entries that have `last_seen` = now but `status = open` — these should never be archived.

**Warning signs:** `.formal/debt-archive.jsonl` contains entries with `last_seen` less than 90 days old. Debt.json shrinks unexpectedly.

**Remediation:** Log each entry's decision (keep vs archive) with timestamps. Add unit test: create entry with `resolved_at` = 1 day ago, run retention with 90-day policy, verify it's archived. Create entry with `resolved_at` = 120 days ago but `status = open`, verify it's kept.

### Pitfall 5: Source Entries Array Becomes Empty
**What goes wrong:** A debt entry has `source_entries: []` after an update, violating the non-empty constraint.

**Why it happens:** Deduplication logic during Phase 3 merges entries but forgets to preserve source_entries from both sources.

**How to avoid:** Validation rejects any entry with `source_entries.length === 0`. Tests verify that merging two entries combines their source_entries arrays (preserves both sources).

**Warning signs:** Validation errors during read: "source_entries required (non-empty array)". Debt entries lose track of which observation triggered them.

**Remediation:** Ensure update path always preserves source_entries. If merging, use array union.

### Pitfall 6: Fingerprint Collision Between Issues and Drifts
**What goes wrong:** An issue fingerprint collides with a drift fingerprint, causing cross-type deduplication.

**Why it happens:** Both use SHA-256 sliced to 16 chars; no type prefix in fingerprint.

**How to avoid:** Prefix fingerprints with type: `issue:abc123` vs `drift:def456`. Or use separate namespaces (issue IDs start with `iss-`, drift IDs start with `drft-`).

**Warning signs:** Phase 3 tests show an issue and a drift being merged together.

**Remediation:** Add type prefix to fingerprint string before storing in debt entry. Update validation schema to reflect prefix pattern.

### State Machine - Debt Transitions
```javascript
// Source: Lightweight guard approach (not xstate, consistent with project philosophy)
// Location: bin/debt-state-machine.cjs (NEW)

const ALLOWED_TRANSITIONS = {
  'open': ['acknowledged'],
  'acknowledged': ['resolving', 'open'],
  'resolving': ['resolved'],
  'resolved': []
};

function canTransition(from, to) {
  if (from === to) return false;  // no-op transitions not allowed
  return (ALLOWED_TRANSITIONS[from] || []).includes(to);
}

function transitionDebtEntry(entry, newStatus) {
  if (!canTransition(entry.status, newStatus)) {
    return {
      success: false,
      error: `Cannot transition: ${entry.status} -> ${newStatus}`
    };
  }
  return {
    success: true,
```

### Retention Policy Application
```javascript
// Source: Fail-open pattern from project's archive utilities
// Location: bin/debt-retention.cjs (NEW)

const fs = require('fs');

function applyRetentionPolicy(ledger, maxAgeDays = 90) {
  const now = new Date();
  const cutoff = new Date(now.getTime() - maxAgeDays * 24 * 60 * 60 * 1000);

  const active = [];
  const archived = [];

  for (const entry of (ledger.debt_entries || [])) {
    // Only archive if: status is resolved AND resolved_at is older than cutoff
    const resolvedAt = new Date(entry.resolved_at || '2099-01-01');

    if (entry.status === 'resolved' && resolvedAt < cutoff) {
      archived.push(entry);
    } else {
      active.push(entry);
    }
  }

  return { active, archived };
}

module.exports = { applyRetentionPolicy };
```

### Ledger Read/Write Operations
```javascript
// Source: Project pattern from bin/aggregate-requirements.cjs
// Location: bin/debt-ledger.cjs (NEW)

const fs = require('fs');
const path = require('path');

function readDebtLedger(ledgerPath) {
  try {
    if (!fs.existsSync(ledgerPath)) {
      return {
        schema_version: '1',
        created_at: new Date().toISOString(),
        last_updated: new Date().toISOString(),
        debt_entries: []
      };
    }
    return JSON.parse(fs.readFileSync(ledgerPath, 'utf8'));
  } catch (err) {
    // Fail-open: corrupt file returns empty ledger
    console.error(`[debt-ledger] Failed to read ${ledgerPath}: ${err.message}`);
    return {
      schema_version: '1',
      created_at: new Date().toISOString(),
      last_updated: new Date().toISOString(),
      debt_entries: []
    };
  }
}

function writeDebtLedger(ledgerPath, ledger) {
  const dir = path.dirname(ledgerPath);
  if (!fs.existsSync(dir)) {
    fs.mkdirSync(dir, { recursive: true });
  }

  // Atomic write: temp file → rename
  const tmpPath = ledgerPath + '.tmp';
  const content = JSON.stringify(ledger, null, 2);
  fs.writeFileSync(tmpPath, content, 'utf8');
  fs.renameSync(tmpPath, ledgerPath);
}

module.exports = { readDebtLedger, writeDebtLedger };
```

## Open Questions

1. **Should fingerprint be deterministic UUID or hash?**
   - What we know: Project uses UUIDs in activity tracking, SHA-256 hashes in requirements aggregation
   - What's unclear: Is a UUID v4 sufficient, or should we use deterministic hash (stable across runs)?
   - Recommendation: Use deterministic SHA-256 hash slice (16 hex chars) as fingerprint. This ensures that the same issue observed again produces the exact same fingerprint, enabling later deduplication. UUIDs are random and wouldn't work for this.

2. **Should formal_ref field be required or optional?**
   - What we know: Phase 1 creates debt entries from raw observations (GitHub/Sentry); Phase 3 (dedup) links to formal models
   - What's unclear: Can Phase 1 entries have null formal_ref, or must they be pre-linked?
   - Recommendation: Make formal_ref optional (null by default). Phase 1 writes observations as-is. Phase 3 enriches with formal_ref during dedup.

3. **How to avoid fingerprint collisions between issues and drifts?**
   - What we know: Both use SHA-256(...)
   - What's unclear: Namespace collision risk if two different issue/drift pairs hash to same 16-char hex string
   - Recommendation: Use type-prefixed fingerprint IDs: `issue:abc123` or `drift:def456`. Or separate the ID from the fingerprint (store separately in schema). Adds clarity.

4. **Should debt.json be one file or split by environment/source?**
   - What we know: Project uses single aggregated files (requirements.json, model-registry.json)
   - What's unclear: Performance impact of single 10MB+ debt.json; querying/updating individual entries
   - Recommendation: Single `.formal/debt.json` for v0.27-01. If performance becomes issue in Phase 3+ (with thousands of entries), split by year/month (debt-2026-03.json). For now, keep it simple.

5. **Retention policy: when should archival happen?**
   - What we know: Entries accumulate over time; need cleanup
   - What's unclear: Should archival be automatic (every observe run), manual, or scheduled (cron)?
   - Recommendation: Provide the function, but don't auto-invoke it yet. Phase 2 observe can call it optionally. Default: manual (`bin/debt-retention.cjs`), triggered by user or CI nightly job. Add config entry for max_age.
