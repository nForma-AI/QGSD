# Phase v0.27-01 Plan 01: Debt Schema & Validation Summary

**One-liner:** JSON Schema draft-07 validation and 4-state debt status machine with deterministic fingerprinting support.

## Objective Completion

Established the data contract for QGSD's debt ledger with:
1. **Debt schema definition** (.formal/debt.schema.json) covering id, fingerprint, title, occurrences, timestamps, environments, status, formal_ref, and source_entries
2. **Runtime validation** (bin/validate-debt-entry.cjs) enforcing schema constraints at write time
3. **State machine enforcement** (bin/debt-state-machine.cjs) preventing invalid status transitions

## Tasks Completed

### Task 1: Debt schema definition and runtime validation (TDD)

**RED Phase:** Created comprehensive test suite in `bin/validate-debt-entry.test.cjs` with 36 test cases covering:
- Valid entries with all required fields pass validation
- All required fields individually tested for missing/invalid scenarios
- Status enum validation (open, acknowledged, resolving, resolved)
- ISO8601 timestamp validation and ordering (last_seen >= first_seen)
- Environment enum validation (production, staging, development, test, local)
- Source_entries array validation with required fields (source_type, source_id, observed_at)
- Formal_ref optional field (string or null)
- Additional properties rejection (additionalProperties: false)
- Ledger-level validation (schema_version="1", debt_entries array, entry validation chaining)

**GREEN Phase:** Implemented validation module with two functions:
- `validateDebtEntry(entry)` — Returns true on valid, or array of error strings on invalid
- `validateDebtLedger(ledger)` — Validates container structure and chains to entry validation

**Schema File:** Created `.formal/debt.schema.json` with:
- UUID v4 id pattern: `^[a-z0-9]{8}-[a-z0-9]{4}-[a-z0-9]{4}-[a-z0-9]{4}-[a-z0-9]{12}$`
- Fingerprint pattern: `^[a-z0-9]{16,64}$` (supports deterministic hashes)
- Status enum: open, acknowledged, resolving, resolved
- Environments enum: production, staging, development, test, local
- Source types enum: github, sentry, sentry-feedback, prometheus, grafana, logstash, bash
- source_entries array with minItems: 1 (non-empty constraint)
- formal_ref optional (string or null)

**Tests:** All 36 validation tests pass ✔

- Includes schema file, validation module, and test file

### Task 2: Debt state machine enforcement (TDD)

**RED Phase:** Created test suite in `bin/debt-state-machine.test.cjs` with 34 test cases covering:
- Valid forward transitions: open→acknowledged, acknowledged→resolving, resolving→resolved
- Revert transition: acknowledged→open (allowed for clarification)
- Invalid backward transitions: resolved→* (all rejected)
- Invalid skip transitions: open→resolving, open→resolved, acknowledged→resolved (all rejected)
- No-op transitions: state→state (all 4 rejected)
- Terminal state verification: resolved has no outbound transitions (empty array)
- Result structure: success/error objects with correct fields
- resolved_at timestamp: added when transitioning to resolved, not for intermediate states
- Invalid target status rejection
- Complete state matrix coverage (16 pairs: 4 states × 4 targets)

**GREEN Phase:** Implemented state machine module with three exports:
- `ALLOWED_TRANSITIONS` constant — Lookup table defining valid target states per source state
- `canTransition(fromStatus, toStatus)` — Guard function returning boolean
- `transitionDebtEntry(entry, newStatus)` — Returns {success: boolean, entry?: object, error?: string}

**Transition Rules:**
```
open → [acknowledged]
acknowledged → [resolving, open]
resolving → [resolved]
resolved → [] (terminal)
```

**Tests:** All 34 state machine tests pass ✔

3. **Collision avoidance:** Type prefixes (issue: vs drift:) deferred to FP-01/FP-02 fingerprinting phase
4. **Ledger structure:** Single `.formal/debt.json` file — sufficient for initial phase
5. **Retention trigger:** Function provided but not auto-invoked — Phase 2 observe can call optionally
