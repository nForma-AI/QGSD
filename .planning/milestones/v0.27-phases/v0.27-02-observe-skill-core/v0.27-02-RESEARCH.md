# Phase v0.27-02: Observe Skill Core - Research

**Researched:** 2026-03-04
**Domain:** Parallel multi-source issue fetching, debt ledger integration, output table rendering, source pluggability architecture
**Confidence:** HIGH

## Summary

This phase transforms the existing `/qgsd:triage` command into `/qgsd:observe` — a production feedback system that fetches from configured sources (GitHub, Sentry, bash) in parallel, deduplicates using stable fingerprints from v0.27-01, segregates issues into two output tables (Issues for discrete events, Drifts for quantitative divergences), and persists observations to the debt ledger for human triage and solve integration.

The critical work spans five areas: (1) adapting existing triage command architecture to add debt ledger integration, (2) implementing pluggable source handler pattern that allows new sources with only config + handler function (no orchestrator changes), (3) dual-table rendering distinguishing discrete events (Issues) from quantitative divergences (Drifts), (4) YAML frontmatter config in `.planning/observe-sources.md` backward-compatible with existing triage sources, and (5) fail-open behavior where one source timeout/failure does not block others.

The observe skill reuses the existing triage parallel fetch architecture (dispatch agents per source, collect results, deduplicate, render) and builds on v0.27-01's debt ledger, fingerprinting, and state machine foundation. No new external dependencies required — all work uses project's established patterns: Task for parallel dispatch, CommonJS for utilities, node:test for testing, and JSON for persistence.

**Primary recommendation:** Adapt existing triage.md command specification incrementally: (1) add YAML frontmatter config support to read observe-sources.md with backward compatibility to triage-sources.md, (2) extend the issue schema to include `issue_type` ("issue" vs "drift") to distinguish tables, (3) implement a source handler registry pattern where each source type (github, sentry, bash, prometheus, grafana, logstash) exports a single async fetch function matching a standard interface, (4) after collect, call debt ledger's write function to persist observations, (5) render two tables (Issues | Drifts) sorted by severity within each table. This allows v0.27-03 (cross-source dedup) to layer on top without changing observe orchestrator, and v0.27-04 (Prometheus/Grafana/Logstash) to add handlers without touching the observe command itself.

## Architecture Patterns

### Parallel Source Dispatch (Adapted from Triage)

**What:** Dispatch one agent per configured source in parallel using Task, wait for all to complete, collect results with Promise.allSettled to ensure one timeout doesn't block others.

**When to use:** Observe command Step 4 after loading sources config and showing dispatch header.

**Example:**
```javascript
// Adapted from existing triage.md Step 4
const sourceHandlers = sources.map(source => {
  return Task(subagent_type="general-purpose", {
    source_config: source,
    since_override: sinceDateOrNull,
    limit_override: limitOrNull,
    handler_instructions: getHandlerFor(source.type),
    return_json: true
  });
});

// Use Promise.allSettled to collect all results, including failures
const results = await Promise.all(sourceHandlers);
// results[0].status === "fulfilled" || "rejected"
// results[0].value === JSON output from handler
```

**Pattern notes:**
- Task subagents return **structured JSON only** (not prose) — match existing triage output format
- Each source handler is independent; failure to parse one source does not corrupt parsing of others
- Collected JSON is combined in observe orchestrator (not in subagent), keeping orchestrator logic centralized

### Source Handler Registry Pattern

**What:** Each source type (github, sentry, bash, prometheus, grafana, logstash) exports a single async function `handler(sourceConfig, options) → { source_label, source_type, status, issues[], error? }` matching a standard interface. The observe command builds a registry at startup and dispatches by looking up handler for source.type.

**When to use:** Observe integration enables addition of new sources without touching the observe orchestrator — v0.27-04 adds prometheus/grafana/logstash by adding handler functions and config docs.

**Example:**
```javascript
// handlers/github-handler.js (conceptual)
async function handleGitHub(sourceConfig, { sinceOverride, limitOverride }) {
  // Source config: { type: "github", label: "...", repo: "...", filter: {...} }
  // Returns: { source_label, source_type, status: "ok"|"error", issues: [...], error?: string }

  const repo = sourceConfig.repo || detectRepoFromGit();
  const issues = await gh.issue.list(repo, sourceConfig.filter, limitOverride);
  return {
    source_label: sourceConfig.label,
    source_type: "github",
    status: "ok",
    issues: issues.map(i => ({
      id: `gh-${i.number}`,
      title: i.title,
      severity: classifySeverity(i.labels),
      created_at: i.createdAt,
      age: formatAge(i.createdAt),
      meta: formatMeta(i)
    }))
  };
}

module.exports = { handleGitHub };
```

**Pluggability proof:**
- v0.27-04 adds prometheus source → create `handlers/prometheus-handler.js`, register in handler map, document in observe-sources.md example
- No changes to observe orchestrator required
- Handler failures (timeout, API error) are caught by Promise.allSettled; orchestrator collects partial results

### Dual-Table Rendering: Issues vs Drifts

**What:** After deduplication, split results into two categories:
- **Issues:** Discrete time-stamped events (errors, warnings, bugs, feedback). Rendered as severity-sorted table.
- **Drifts:** Quantitative divergences from formal parameters (threshold exceeded, metric out of bounds). Rendered as separate severity-sorted table.

**When to use:** Observe Step 6 (render output) after all sources collected and deduplicated.

**Example output:**
```

 QGSD ► OBSERVE: 6 issues, 2 drifts across 3 source(s)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌────────────────────────────── ISSUES ───────────────────────────────┐
│ #  │ Title                                    │ Source  │ Sev │ Age  │
├─────────────────────────────────────────────────────────────────────┤
│  1 │ TypeError: cannot read 'id' of undefined │ Sentry  │ err │ 5m   │
│  2 │ Login fails on Safari iOS [+GitHub]     │ Sentry  │ bug │ 2h   │

│  1 │ maxDeliberation (MCsafety.cfg)        │ 5      │ 12     │ err  │
│  2 │ avgLatencyP95 (Prometheus)            │ <100ms │ 245ms  │ wrn  │
└─────────────────────────────────────────────────────────────────────┘
```

**Classification logic:**
- **Issue:** `source.type ∈ {github, sentry, sentry-feedback, bash}` OR manually tagged `issue_type: "issue"` in handler output
- **Drift:** `source.type ∈ {prometheus, grafana, logstash}` AND includes `formal_parameter_key` field OR manually tagged `issue_type: "drift"`
- Severity within each table: error > bug > warning > info
- Cross-table display: Issues first (discrete signals), Drifts second (quantitative signals)

**Why separate tables:**
- **Semantic difference:** Issues require human investigation (did this bug happen? yes/no). Drifts require comparison (formal threshold vs production value — has it drifted?).
- **Different actions:** Issue → `/qgsd:debug` (investigate root cause). Drift → Compare formal vs actual, consider parameter update.
- **Prevents conflation:** Mixing them makes it unclear which action to take for each row.

### Fail-Open Handler Dispatch

**What:** Use Promise.allSettled instead of Promise.all; parse and return partial results when some sources fail (timeout, network error, API limit).

**When to use:** Observe Step 4 after dispatching handlers; Step 5 after collecting results.

**Example:**
```javascript
// After dispatching all source handlers
const results = await Promise.allSettled(sourceHandlers);

// Step 5: Collect results, preserving both successes and errors
const allIssues = [];
const errors = {};

for (let i = 0; i < results.length; i++) {
  const result = results[i];

  if (result.status === "fulfilled") {
    const data = result.value; // JSON from handler
    if (data.status === "ok") {
      allIssues.push(...data.issues);
    } else {
      errors[data.source_label] = data.error || "Unknown error";
    }
  } else if (result.status === "rejected") {
    errors[sources[i].label] = `Timeout or exception: ${result.reason.message}`;
  }
}

// Continue: deduplicate allIssues, render table, and show errors list
// Errors are shown at bottom of output but don't block the observe command
```

**Timeout configuration:**
```javascript
// Each handler gets a timeout; wrapped in Promise.race
async function dispatchSourceWithTimeout(handler, timeoutSeconds) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error(`Timeout after ${timeoutSeconds}s`)), timeoutSeconds * 1000)
  );
  return Promise.race([handler, timeout]);
}
```

**Observe output when partial success:**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 QGSD ► OBSERVE: 4 issues across 3 source(s); 1 source failed
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[issues table...]

Errors from sources (1 failed, did not block others):
  ✗ Sentry: Timeout after 10s (endpoint unreachable?)
```

### Debt Persistence: Write After Observe

**What:** After rendering output, call `writeDebtLedger(path, ledger)` to persist new/updated observations from all sources.

**When to use:** Observe Step 6b (before showing routing prompt) — after table render, before user input.

**Example:**
```javascript
// Import from v0.27-01 debt module
const { readDebtLedger, writeDebtLedger } = require('./bin/debt-ledger.cjs');
const { createDebtEntry } = require('./bin/create-debt-entry.cjs'); // Helper from v0.27-01

// After rendering issues and drifts tables
const ledger = readDebtLedger('.formal/debt.json');

// For each issue/drift returned from sources:
for (const issue of allIssues) {
  // Create or update debt entry
  const entry = createDebtEntry({
    title: issue.title,
    source_type: issue.source_type,
    source_id: issue.id,
    observed_at: issue.created_at,
    fingerprint: computeFingerprint(issue),  // Reuse from v0.27-01
    issue_type: issue.issue_type
  });

  // Upsert into ledger (check by fingerprint)
  const existing = ledger.debt_entries.find(e => e.fingerprint === entry.fingerprint);
  if (existing) {
    existing.occurrences++;
    existing.last_seen = entry.last_seen;
    existing.source_entries.push(entry.source_entries[0]);
  } else {
    ledger.debt_entries.push(entry);
  }
}

// Atomic write
writeDebtLedger('.formal/debt.json', ledger);
```

**Key invariant:** Every observation from every source produces a debt entry. Observe runs are idempotent because fingerprints are deterministic — rerunning observe with the same issues updates `occurrences` and `last_seen`, not duplicates.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Promise coordination (some sources fail, some succeed) | Custom async loop with try-catch blocks | Promise.allSettled | Handles partial failure elegantly; try-catch makes sequencing or error handling logic complex |
| Source pluggability framework | Ad-hoc if statements for each source type | Handler registry (export handler per type, map at runtime) | Prevents observe orchestrator from becoming monolithic; new sources (v0.27-04) add handlers, not modify observe |
| Fingerprint computation for issues/drifts | Hand-craft hash logic per source | Reuse v0.27-01 fingerprinting functions (issue fingerprint, drift fingerprint) | Fingerprints must be deterministic and consistent across runs; rolling your own creates collisions/splits (Pitfall 3 in PITFALLS.md) |
| Deduplication before aggregation | Manual loop comparing URLs or titles | Jaccard similarity (existing triage.md pattern) with configurable threshold | Naïve dedup (URL match only) misses duplicates from different sources; similarity matching is proved to work (existing triage command) |
| Debt ledger write operations | Inline file manipulation | debt-ledger.cjs API (readDebtLedger, writeDebtLedger) | Atomic writes + fail-open behavior already implemented in v0.27-01; ensures no data loss on crash |

**Key insight:** Observe is an orchestration layer, not a collection of utilities. The hard work (fingerprinting, dedup, state machine, I/O) is already done in v0.27-01. Observe glues those pieces together and adds dual-table rendering.

## Common Pitfalls

### Pitfall 1: Conflating Issue and Drift Semantics

**What goes wrong:** A "drift" (formal parameter threshold exceeded) is rendered in the same table as a "bug" (TypeError occurred). User sees "error: TypeError" and "error: max_deliberation > 5" side-by-side and treats them identically — investigating the TypeError when they should be comparing the deliberation time to the formal model.

**Why it happens:** Single-table rendering makes the semantic difference invisible. "Error severity" is the only distinguisher; the user must read metadata to figure out which action to take.

**How to avoid:** Separate tables by issue_type. Issues table (discrete events) comes first. Drifts table (measured divergences) comes second. Each table has severity-sorted ordering **within itself**. Planner consumes RESEARCH.md and mandates two separate tables in task descriptions.

**Warning signs:**
- User is confused about what action to take on a drift row
- Documentation shows a drift mixed with issue examples
- Rendering code has nested conditionals trying to decide "is this really a drift or an issue?"

### Pitfall 2: Observe Doesn't Write to Debt Ledger, Only Reads

**What goes wrong:** Observe fetches issues and displays them, but doesn't persist them to `.formal/debt.json`. The debt ledger stays empty. Solve command has nothing to work on (DEBT-06 requirement unmet). Users run observe, see issues, triage them manually, then run solve expecting it to have picked up the acknowledged debt — but solve sees empty ledger and finds nothing to do.

**Why it happens:** Observe author assumes the debt ledger is populated separately (e.g., by solve command or a background job), but the requirement (DEBT-06) is explicit: "observe writes new or updated debt entries after each run."

**How to avoid:** Observe Step 6b (after rendering tables, before prompting for user action) calls `writeDebtLedger()` with all collected observations. This is **mandatory**, not optional. Tests must verify that running observe with 5 sources creates debt entries.

**Warning signs:**
- `.formal/debt.json` is empty or unchanged after observe runs multiple times
- Solve command reports "0 acknowledged debt items; nothing to do"
- Requirements DEBT-06 is marked incomplete

### Pitfall 3: Fingerprint Instability Across Runs

**What goes wrong:** An issue is observed, fingerprinted as `fp-abc123`, stored in debt ledger. Next observe run, the same issue is fingerprinted differently (e.g., source ID changed, or fingerprint logic drifted), producing `fp-xyz789`. The debt ledger now has two entries for the same bug, incrementing occurrences in the wrong place.

**Why it happens:** Fingerprinting logic is duplicated in observe code instead of reusing v0.27-01's deterministic functions. Or the observe code calls the right function but passes parameters in the wrong order, changing the hash.

**How to avoid:**
1. Import and call v0.27-01's fingerprinting functions directly:
   ```javascript
   const { fingerprintIssue, fingerprintDrift } = require('./bin/debt-fingerprinting.cjs');
   ```
2. **Never compute fingerprints inline** — always go through the imported functions
3. Tests must verify: `fpA = fingerprintIssue(issue); fpB = fingerprintIssue(issue); assert(fpA === fpB);`

**Warning signs:**
- Debt ledger grows faster than expected (duplicates being created)
- Same issue appears in multiple debt entries with different fingerprints
- Occurrences count doesn't match number of times observe was run with same issue present

### Pitfall 4: Source Handler Timeout Not Enforced

**What goes wrong:** Sentry endpoint is slow (takes 30s to respond). Observe dispatch waits for Sentry to finish before continuing. The entire observe command hangs for 30+ seconds. User Ctrl+C to interrupt, loses partial results from GitHub and bash sources that completed quickly.

**Why it happens:** Promise.allSettled is not used; code uses Promise.all or a simple await without timeout wrapper. A single slow source blocks all others.

**How to avoid:**
1. Wrap each handler dispatch in a timeout:
   ```javascript
   const timeoutSeconds = source.timeout || defaultTimeout;
   const handler = dispatchSourceWithTimeout(agent, timeoutSeconds);
   ```
2. Use Promise.allSettled, not Promise.all
3. Test with simulated timeout (mock source that delays for timeout+1 second); verify that other sources complete and are returned

**Warning signs:**
- Observe command takes >30s to complete when expected <5s
- Removing one source from config makes observe fast again
- User reports: "Observe hangs waiting for Sentry"

### Pitfall 5: Config Schema Not Validated

**What goes wrong:** User edits `.planning/observe-sources.md` with invalid YAML (missing required fields, typo in enum value). Observe command loads config silently, skips the invalid source, and the user doesn't realize the source is disabled. They think observe is broken and debug the wrong thing.

**Why it happens:** Config is parsed as YAML but not validated against schema. Invalid entries are skipped with no warning or error message.

**How to avoid:**
1. Define schema in `.formal/observe-sources.schema.json` (JSON Schema draft-7)
2. Before Step 2 completes (load sources config), validate all sources against schema:
   ```javascript
   const schema = readJSON('.formal/observe-sources.schema.json');
   for (const source of sources) {
     const errors = validateAgainstSchema(source, schema);
     if (errors.length > 0) {
       console.error(`Invalid source '${source.label}': ${errors.join('; ')}`);
       process.exit(1); // Fail fast; don't skip sources silently
     }
   }
   ```
3. Display validation result before fetching:
   ```
   Loaded 4 sources from observe-sources.md; 4 valid, 0 invalid.
   Fetching...
   ```

**Warning signs:**
- Observe silently skips a source without mentioning it in the output
- User has to debug by checking if source config is syntactically valid
- Config errors are not caught until runtime, after other sources have been dispatched

## Code Examples

### Handler Interface (GitHub Example)

This is the function signature that all source handlers must implement.

```javascript
// handlers/github-handler.js (or inline in observe orchestrator)

/**
 * GitHub source handler — returns standardized issue schema
 * @param {object} sourceConfig - { type: "github", label: "...", repo: "...", filter: {...} }
 * @param {object} options - { sinceOverride, limitOverride }
 * @returns {Promise<object>} - { source_label, source_type, status, issues[], error? }
 *
 * Interface contract:
 * - Always returns an object (never throws)
 * - status: "ok" if fetch succeeded, "error" if it failed
 * - If status: "ok", issues[] is populated (may be empty)
 * - If status: "error", error field contains human-readable message
 * - Each issue conforms to standard schema: id, title, severity, url, age, created_at, meta
 */
async function handleGitHub(sourceConfig, options) {
  const repo = sourceConfig.repo || await detectRepoFromGit();
  const since = options.sinceOverride || sourceConfig.filter?.since || '7d';
  const limit = options.limitOverride || sourceConfig.filter?.limit || 10;

  try {
    // Use gh CLI (already available in project)
    const issues = await gh.issue.list(repo, {
      state: sourceConfig.filter?.state || 'open',
      labels: sourceConfig.filter?.labels || [],
      limit
    });

    // Apply since filter
    const cutoff = parseDuration(since);
    const filtered = issues.filter(i => new Date(i.createdAt) > cutoff);

    return {
      source_label: sourceConfig.label,
      source_type: "github",
      status: "ok",
      issues: filtered.map(issue => ({
        id: `gh-${issue.number}`,
        title: issue.title,
        severity: classifySeverityFromLabels(issue.labels),
        url: issue.url,
        age: formatAge(issue.createdAt),
        created_at: issue.createdAt,
        meta: `#${issue.number} · ${issue.assignees?.length || 0} assignee(s)`,
        source_type: "github",
        issue_type: "issue"  // Inferred for github type
      }))
    };
  } catch (err) {
    // Fail-open: return error status, don't throw
    return {
      source_label: sourceConfig.label,
      source_type: "github",
      status: "error",
      error: `GitHub fetch failed: ${err.message}`,
      issues: []
    };
  }
}

module.exports = { handleGitHub };
```

**What the planner expects from this:**
- Every handler exports a function named `handle<SourceType>` (e.g., handleSentry, handlePrometheus)
- Function signature matches the interface above
- Handler is responsible for its own error handling and timeout prevention
- Output schema is consistent across all handlers (source_label, source_type, status, issues[], error)
- Handlers are called once per source, results collected by Promise.allSettled

### Dual-Table Rendering

```javascript
// Observe Step 6: Render two tables

function renderTables(allIssues, allDrifts) {
  // Split collected items into two categories
  const issues = allIssues.filter(item => item.issue_type === 'issue');
  const drifts = allIssues.filter(item => item.issue_type === 'drift');

  // Sort each by severity (error > bug > warning > info) then by age
  const severity_order = { error: 0, bug: 1, warning: 2, info: 3 };
  issues.sort((a, b) => {
    const sev_cmp = severity_order[a.severity] - severity_order[b.severity];
    if (sev_cmp !== 0) return sev_cmp;
    return new Date(b.created_at) - new Date(a.created_at);
  });

  drifts.sort((a, b) => {
    const sev_cmp = severity_order[a.severity] - severity_order[b.severity];
    if (sev_cmp !== 0) return sev_cmp;
    return b.divergence_magnitude - a.divergence_magnitude;
  });

  // Render issues table
  console.log(`
┌────────────────────────────── ISSUES ───────────────────────────────┐
│ #  │ Title                                    │ Source  │ Sev │ Age  │
├─────────────────────────────────────────────────────────────────────┤
  `);

  for (let i = 0; i < issues.length; i++) {
    const issue = issues[i];
    console.log(`│ ${i+1:>2} │ ${truncate(issue.title, 40)} │ ${issue.source_label.padEnd(7)} │ ${issue.severity.padEnd(3)} │ ${issue.age} │`);
  }

  console.log(`└─────────────────────────────────────────────────────────────────────┘`);

  // Render drifts table (if any)
  if (drifts.length > 0) {
    console.log(`
┌────────────────────────────── DRIFTS ───────────────────────────────┐
│ #  │ Parameter                              │ Formal │ Actual │ Sev  │
├─────────────────────────────────────────────────────────────────────┤
    `);

    for (let i = 0; i < drifts.length; i++) {
      const drift = drifts[i];
      console.log(`│ ${i+1:>2} │ ${drift.formal_parameter_key.padEnd(36)} │ ${drift.formal_value.padEnd(6)} │ ${drift.actual_value.padEnd(6)} │ ${drift.severity.padEnd(4)} │`);
    }

    console.log(`└─────────────────────────────────────────────────────────────────────┘`);
  }
}
```

### Backward Compatibility: Config Loading

```javascript
// Observe Step 2: Load and validate config with backward compatibility

function loadObserveConfig(configPath) {
  // Try observe-sources.md first (v0.27-02+), fall back to triage-sources.md
  let configFile = configPath || '.planning/observe-sources.md';
  if (!fs.existsSync(configFile)) {
    configFile = '.planning/triage-sources.md';
  }

  if (!fs.existsSync(configFile)) {
    console.error(`No observe sources configured.`);
    console.error(`Create .planning/observe-sources.md`);
    process.exit(1);
  }

  // Parse YAML frontmatter
  const content = fs.readFileSync(configFile, 'utf8');
  const match = content.match(/^---\n([\s\S]*?)\n---/);
  const frontmatter = YAML.parse(match ? match[1] : '');

  // Extract sources, applying defaults
  let sources = frontmatter.sources || frontmatter.observe_config?.sources || [];

  // Backward compatibility: infer issue_type if not specified
  sources = sources.map(source => {
    if (!source.issue_type) {
      if (['github', 'sentry', 'sentry-feedback', 'bash'].includes(source.type)) {
        source.issue_type = 'issue';
      } else if (['prometheus', 'grafana', 'logstash'].includes(source.type)) {
        source.issue_type = 'drift';
      }
    }

    // Apply defaults
    source.timeout = source.timeout ?? (frontmatter.observe_config?.default_timeout ?? 10);
    source.fail_open = source.fail_open ?? (frontmatter.observe_config?.fail_open_default ?? true);

    return source;
  });

  return { sources, configFile };
}
```

## Open Questions

1. **Should observe support filtering by issue_type (--issues-only, --drifts-only)?**
   - What we know: OBS-06 specifies two separate tables; no requirement for filtering
   - What's unclear: Users may want to run `/qgsd:observe --issues-only` to skip drift checking for speed
   - Recommendation: Not required for v0.27-02 (out of scope per success criteria). Add as future enhancement if users request it. For now, always render both tables.

2. **How does observe handle sources that return both issues AND drifts (e.g., Prometheus)?**
   - What we know: Prometheus source can emit both metric values (drifts) and alerts (issues)
   - What's unclear: Should one handler return both types, or should we have separate prometheus and prometheus-drifts sources?
   - Recommendation: One handler per source type (prometheus-handler.js). Handler's output array can include both issue_type: "issue" and issue_type: "drift" items. Rendering step separates them into two tables.

3. **Is the debt ledger write idempotent if observe is run twice in rapid succession with same data?**
   - What we know: Fingerprints are deterministic; ledger write uses atomic file operations
   - What's unclear: If user runs observe twice with identical source data, does the second run overwrite the first, or is it idempotent (no duplicate entries)?
   - Recommendation: Second run should be idempotent — same fingerprints match existing entries, occurrences and last_seen are updated, no new entries created. This requires the "upsert by fingerprint" logic in debt persistence code (see Code Examples: Debt Persistence).
