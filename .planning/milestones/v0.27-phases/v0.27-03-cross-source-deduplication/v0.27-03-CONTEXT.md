<decisions>
## Implementation Decisions

### Near-duplicate merge behavior
- Auto-merge silently during observe — no user prompts or review flags
- Entry with higher occurrence count is kept as primary (per success criteria)
- source_entries from both entries are preserved in the merged record
- Field conflict resolution: max(severity) wins (pessimistic), primary's title kept, non-null formal_ref preserved
- Audit trail maintained via source_entries array — no information lost

### Formal reference linking
- Layered resolution order:
  1. Manual assignment — if formal_ref already present in debt entry, preserve it
  2. Auto-detect — keyword match against .formal/requirements.json (214 reqs) via title matching
  3. Spec inference — if no requirements match, scan .formal/spec/ module names for overlap
  4. Unmatched fallback — set formal_ref: null
- Add formal_ref_source field ("manual" | "auto-detect" | "spec-inferred" | null) for traceability
- Unmatched entries are future triage candidates, not errors

### Dedup output reporting
- Default output extends existing observe pattern: "Merged K near-duplicates (Levenshtein >= 0.85)"
- Summary counts only by default (no log spam)
- --verbose flag shows per-merge detail: which entries merged, which fields won, source_entries count
- Consistent with v0.27-02 "Wrote N new, updated M existing" reporting style

### Similarity threshold configuration
- Global config in .planning/observe-sources.md YAML frontmatter (same location as observe config)
- Default 0.85 inclusive (>= threshold triggers merge)
- Per-source threshold override optional but not required for this phase
- At boundary: favor merge over split (near-duplicates at 0.85 are more costly to manage separately)

### Claude's Discretion
- Exact Levenshtein implementation (library vs inline)
- Internal data structures for merge candidate pairing
- Order of operations during dedup pass (fingerprint exact match first, then Levenshtein on remaining)
- How to handle merge of entries already in non-open states (e.g., acknowledged + open)

</decisions>

- Fingerprint exact-match dedup should run first (O(1) lookup), Levenshtein only on remaining unmatched entries (expensive O(n^2))
- formal_ref_source field enables future dashboard filtering: "show me all auto-detected formal links" vs "show me unlinked debt"
- Merge log in --verbose mode should be parseable (structured, not prose) for potential future automation
