## Summary

Phase v0.10-08 is a targeted bug fix for a single critical defect in `runAutoUpdateCheck()` (`bin/manage-agents.cjs` line 1445). The bug was identified in the v0.10 milestone audit and confirmed in `v0.10-05-VERIFICATION.md` as a `gaps_found` critical item for PLCY-03.

The defect: `getUpdateStatuses()` (from `update-agents.cjs`) returns a JavaScript `Map` keyed by CLI binary basename (`codex`, `gemini`, `opencode`, `copilot`, `ccr`). But `runAutoUpdateCheck()` iterates over `autoSlots` (which are `agent_config` slot names like `codex-1`, `gemini-1`) and reads `statuses[slot]` — bracket notation on a `Map` always returns `undefined`. The result is that every auto-update check logs `SKIP`, making the warning banner in `listAgents()` impossible to trigger from this code path.

The fix has two components: (1) switch to `statuses.get(binName)`, and (2) correctly resolve the binary name for each slot by cross-referencing `providers.json` — the same approach used correctly by `listAgents()` at line 477-479.

The regression guard is a unit test that demonstrates RED behavior with `statuses[slot]` on a real `Map` and GREEN behavior with `statuses.get(binName)`. This test is added to `bin/manage-agents.test.cjs` and uses the `_pure` export pattern established for this codebase.

The final deliverable is an updated `v0.10-05-VERIFICATION.md` that clears the `gaps_found` / critical bug flag and updates PLCY-03's status from `unsatisfied` to `satisfied` (or `human_needed` if any TTY flows remain unconfirmed).

**Primary recommendation:** Two-plan execution: Plan 01 adds RED test stubs (Wave 0), Plan 02 implements the fix (GREEN) and updates `v0.10-05-VERIFICATION.md`. Test baseline is 122 passing tests.

### Pattern 1: The Bug — Map Bracket Notation

`getUpdateStatuses()` in `update-agents.cjs` returns `new Map(results)` where each entry is `[binName, { current, latest, status }]`. A JavaScript `Map` does NOT support bracket notation for key lookup. `statuses['codex-1']` is always `undefined`, just as `(new Map([['a', 1]]))['a']` is `undefined`.

**Bug location:** `bin/manage-agents.cjs` line 1445

```javascript
// BROKEN — statuses is a Map; bracket access always undefined
const statusEntry = statuses && statuses[slot];  // slot = 'codex-1', 'gemini-1', etc.
// statusEntry is ALWAYS undefined → status = 'SKIP' → banner never fires
```

**Contrast with correct usage** — `listAgents()` at line 477-479:

```javascript
// Source: bin/manage-agents.cjs lines 477-479 (CORRECT)
const binName = path.basename(p.cli);     // 'codex', 'gemini', 'opencode', 'copilot', 'ccr'
const upd = updateStatuses.get(binName);  // correct Map.get() call
```

### Pattern 2: Slot → Binary Name Mapping

The `agent_config` in `qgsd.json` uses slot names as keys (e.g. `codex-1`, `gemini-1`). The `Map` returned by `getUpdateStatuses()` is keyed by CLI binary basename (e.g. `codex`, `gemini`). The mapping requires cross-referencing `providers.json`.

**Providers.json structure** (key facts):
- Each provider record has `name` (slot name, e.g. `codex-1`) and `cli` (full path, e.g. `/opt/homebrew/bin/codex`)
- `path.basename(provider.cli)` gives the Map key: `codex`, `gemini`, `opencode`, `copilot`, `ccr`
- Multiple slot names can share the same binary: `claude-1`, `claude-2`, `claude-3`, `claude-4`, `claude-5`, `claude-6` all have `cli: "/opt/homebrew/bin/ccr"` → binary name `ccr`

**Fix approach** — build a slot-to-binName lookup map inside `runAutoUpdateCheck()`:

```javascript
// Source pattern from listAgents() lines 398-401 + 477-479
// Build provider lookup
let providerMap = {};
try {
  const pdata = readProvidersJson();
  for (const p of (pdata.providers || [])) providerMap[p.name] = p;
} catch (_) {}

// In the per-slot loop:
for (const slot of autoSlots) {
  const p = providerMap[slot];
  const binName = p && p.cli ? path.basename(p.cli) : null;
  const statusEntry = binName ? statuses.get(binName) : undefined;
  // ... rest of status branching
}
```

**Edge cases to handle:**
- Slot has no provider record in `providers.json` → `binName` is null → `statusEntry` is undefined → log SKIP (unchanged behavior, but now correctly labeled as "no provider record for slot")
- Multiple slots share the same binary (`ccr`) → each slot correctly gets the same update status, logged separately — this is correct behavior (each slot is logged independently)
- `providers.json` read fails → `providerMap = {}` → all slots log SKIP (fail-open, consistent with existing fail-open design)

### Anti-Patterns to Avoid

- **Testing `statuses[slot]` is broken with `assert.ok(!statusEntry)` instead of `assert.strictEqual(statusEntry, undefined)`:** The strictEqual form is more precise for the regression guard
- **Logging the detail as the provider `status` string directly:** The provider Map uses `status: 'update-available'` (hyphenated) but `runAutoUpdateCheck()` logs NDJSON with `status: 'UPDATE_AVAILABLE'` (uppercase). The translation logic in the `if/else if` chain must be preserved
- **Assuming `slot` === `binName`:** They differ for ALL CCR-routed slots (`claude-1` through `claude-6` all use `ccr` binary). Always look up via `providers.json`
- **Using `statuses.get(slot)` without the provider lookup:** `slot` is `codex-1`, Map key is `codex` — they never match directly

## Common Pitfalls

### Pitfall 1: Fixing `statuses.get()` but still keying on slot name
**What goes wrong:** Change `statuses[slot]` to `statuses.get(slot)` — this passes linting but still returns `undefined` because the Map is keyed by binary name, not slot name
**Why it happens:** The two-part fix (switch API + switch key) is easy to partially apply
**How to avoid:** The regression test must verify the FULL path: inject a Map with key `'codex'` (binary), verify the function produces `UPDATE_AVAILABLE` for slot `codex-1` — not just verify that `Map.get()` is called
**Warning signs:** Test passes with `statuses.get(slot)` instead of `statuses.get(binName)` — means the test is not strict enough

### Pitfall 2: `providers.json` lookup fails silently, all slots SKIP
**What goes wrong:** `readProvidersJson()` throws (file not found / parse error) and the catch block silently continues, leaving `providerMap = {}`, all `binName` lookups return null, all slots log SKIP
**Why it happens:** The function is designed fail-open; but a bad `providers.json` would mask real update availability
**How to avoid:** This is acceptable fail-open behavior (consistent with existing design). Document it clearly in the fix comment. Don't change the fail-open contract.
**Warning signs:** After fix, all slots still log SKIP → check providers.json path and content

### Pitfall 3: Logging the wrong slot name in NDJSON
**What goes wrong:** The NDJSON log entry records `binName` (`codex`) instead of `slot` (`codex-1`)
**Why it happens:** Copy-paste error when refactoring to use `binName` in the Map lookup
**How to avoid:** Keep `slot` as the NDJSON `slot` field — only use `binName` for the Map lookup, not for logging. The `listAgents()` warning banner reads `e.slot` to display slot names, so these must remain slot names

### Pitfall 4: Dependency injection changes `runAutoUpdateCheck` signature in a way that breaks the entry point
**What goes wrong:** Adding `getStatusesFn` parameter and the entry point call `await runAutoUpdateCheck()` breaks because it now always calls the real `getUpdateStatuses`
**Why it happens:** The default parameter `getStatusesFn = getUpdateStatuses` ensures backwards compatibility
**How to avoid:** Use default parameter syntax — the production entry point call `await runAutoUpdateCheck()` is unchanged; tests pass `mockGetStatuses` explicitly

### Pitfall 5: Updating VERIFICATION.md status before the fix is confirmed
**What goes wrong:** Changing PLCY-03 to `satisfied` in `v0.10-05-VERIFICATION.md` before the fix is committed and test is GREEN
**Why it happens:** Documentation and implementation done in the same task
**How to avoid:** Write VERIFICATION.md update in the same plan step that commits the fix, after the test turns GREEN
