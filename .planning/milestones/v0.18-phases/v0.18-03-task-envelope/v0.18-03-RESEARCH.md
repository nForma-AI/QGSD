# Phase v0.18-03: Task Envelope - Research

**Researched:** 2026-02-27
**Domain:** Structured JSON sidecar schema for research → plan → quorum handoff
**Confidence:** HIGH

---

## Summary

Phase v0.18-03 implements the task envelope — a structured JSON artifact (`task-envelope.json`) written to `.planning/phases/<phase>/` that carries compact handoff data across the research → plan → quorum pipeline. The envelope accumulates information at each stage: the researcher sets `objective`, `constraints`, `risk_level`, and `target_files`; the planner appends `plan_path` and `key_decisions`; the quorum orchestrator logs the final verdict. The feature is config-gated (`task_envelope.enabled` in qgsd.json, default: true) and fails open when absent or malformed.

This is foundational infrastructure for Phases v0.18-04 (Adaptive Fan-Out) and v0.18-04's tiered models feature. The envelope's `risk_level` field becomes the data source for adaptive worker scaling and model tier selection downstream.

**Primary recommendation:** Implement in two tracks: (1) lightweight `bin/task-envelope.cjs` CLI tool that initializes, updates, and validates the envelope schema; (2) wire envelope init/update calls into `plan-phase.md` workflow after researcher and planner steps; (3) extend config-loader.js with `task_envelope.enabled` validation. No new npm dependencies. Reuse existing atomic file-write and config patterns.

## User Constraints (from v0.18 REQUIREMENTS.md)

### Locked Decisions

1. **Task envelope is the structured handoff mechanism** — not lossy summarization, not full-text context passing. Structured extraction solves the token problem without losing detail.
2. **Schema version 1** — single top-level version field for forward compatibility
3. **Envelope is a sidecar at phase level** — `.planning/phases/<phase>/task-envelope.json`, not nested in STATE.md or polluting config files
4. **Config-gated with fail-open** — absent envelope causes graceful fallback (no quorum blocks); malformed envelope is silently skipped
5. **Risk level set by researcher** — not computed by planner; researcher's domain analysis informs risk tier (ENV-01)

### Claude's Discretion

1. **Risk level classification thresholds** — researcher determines `low | medium | high`; RESEARCH.md discovery process surfaces risk factors; classification rules are researcher's choice
2. **Key decisions extraction** — planner extracts from PLAN.md; structure is flexible (array of strings vs objects with rationale)
3. **Envelope update strategy** — atomic replace vs incremental merge; atomic (new tmpPath + renameSync) matches existing scoreboard pattern
4. **Nested schema depth** — whether envelope sections (research, plan, quorum) are flattened or nested; design favors flat top level (schema_version, phase, risk_level) + nested sub-objects (research: {}, plan: {}, quorum: {})

### Deferred Ideas (OUT OF SCOPE)

1. **ENV-05**: Envelope diff mode for deliberation round optimization — deferred to v0.19
2. **ENV-06**: Benched slot warm-up suppression — deferred to v0.19
3. **Per-slot envelope metadata** — not needed for Phase v0.18-03; forward-compatible in schema v1

### Pattern 1: Atomic Envelope Write

**What:** Write envelope JSON to disk atomically. A crash mid-write should not corrupt the file or leave a partial state.

**Example (task-envelope.cjs):**
```javascript
// Source: bin/gsd-tools.cjs pattern for scoreboard writes
const tmpPath = path.join(dir, `task-envelope-${Date.now()}.tmp.json`);
const targetPath = path.join(dir, 'task-envelope.json');
fs.writeFileSync(tmpPath, JSON.stringify(envelope, null, 2), 'utf8');
fs.renameSync(tmpPath, targetPath);  // atomic on POSIX
```

**When to use:** After research completes (init), after planning completes (update).

**Why:** Same as scoreboard — multiple processes may read concurrently; atomic rename prevents torn writes.

### Pattern 2: Schema Validation on Write

**What:** Envelope must conform to a fixed schema. Invalid fields are logged and skipped (partial envelope accepted if rest is valid).

**Example schema (task-envelope.cjs):**
```json
{
  "schema_version": "1",           // required, must be "1"
  "phase": "v0.18-03",             // required, matches phase_number from STATE.md
  "created_at": "2026-02-27T...",  // ISO 8601, auto-generated
  "risk_level": "low|medium|high", // required for ENV-03, set by researcher
  "research": {
    "objective": "string",         // required ENV-01, from phase goal
    "constraints": "string",       // required ENV-01, from REQUIREMENTS.md
    "target_files": ["path/..."],  // required ENV-01, from RESEARCH.md analysis
    "confidence": "HIGH|MEDIUM|LOW",  // optional, from RESEARCH.md metadata
    "summary": "string"            // optional, 3-5 sentence distillation
  },
  "plan": {
    "plan_path": "path/v0.18-03-01-PLAN.md",  // required ENV-02
    "key_decisions": ["decision 1", "decision 2"],  // required ENV-02
    "wave_count": 3,               // optional, from PLAN.md structure
    "estimated_duration_min": 45   // optional, from PLAN.md must_haves
  },
  "quorum": {
    "worker_count": 4,             // set by quorum.md after dispatch
    "rounds": 1,                   // set after consensus
    "verdict": "APPROVE|BLOCK|FLAG",  // final verdict
    "token_est_total": 12400       // optional, from token tracking (v0.18-01)
  }
}
```

**Validation rules:**
- `schema_version` must be string "1" (forward compat)
- `phase` must match `v\d+\.\d+-\d{2}` regex (same as phase detection)
- `risk_level` must be one of: "low", "medium", "high" (case-sensitive)
- `created_at` auto-generates if missing (ISO 8601)
- All other fields: type-check on write; log to stderr if invalid; accept partial envelope (don't reject entirely)

**When to use:** Every init and update call.

### Pattern 3: Config-Gated Feature Disable

**What:** When `task_envelope.enabled: false` in qgsd.json, envelope operations are skipped silently.

**How (plan-phase.md):**
```bash
ENVELOPE_ENABLED=$(node ~/.claude/qgsd/bin/gsd-tools.cjs config-get task_envelope.enabled)
if [ "$ENVELOPE_ENABLED" = "true" ]; then
  node bin/task-envelope.cjs init --phase $PHASE_NUM --objective "..." --constraints "..."
fi
```

**Why:** Allows rollback / testing / old workflows to continue without envelope overhead.

### Pattern 4: Envelope Update Strategy (Incremental, Not Replace)

**What:** When plan-phase.md calls `update --section plan`, the existing `research` section is preserved; the `plan` section is added/replaced.

**Example (task-envelope.cjs update logic):**
```javascript
// Read existing envelope
const existing = fs.existsSync(targetPath)
  ? JSON.parse(fs.readFileSync(targetPath, 'utf8'))
  : { schema_version: "1", phase, created_at: new Date().toISOString() };

// Merge new section
existing.plan = {
  plan_path: planPath,
  key_decisions: decisions,
  wave_count: waveCount
};

// Write atomically
const tmpPath = `${targetPath}.tmp.json`;
fs.writeFileSync(tmpPath, JSON.stringify(existing, null, 2), 'utf8');
fs.renameSync(tmpPath, targetPath);
```

**Why:** research section is immutable after init; plan section is appended; quorum section added last. Prevents data loss if workflow restarts.

### Pattern 5: Envelope Context Injection (Into Quorum YAML)

**What:** plan-phase.md passes `envelope_path` to the quorum dispatcher via the context bundle, similar to how it passes `artifact_path` for PLAN.md review.

**Example (plan-phase.md quorum step):**
```markdown
envelope_path: .planning/phases/v0.18-03/task-envelope.json
artifact_path: .planning/phases/v0.18-03/v0.18-03-01-PLAN.md
```

**How quorum.md uses it:**
```bash
ENVELOPE_PATH=$(echo "$CONTEXT" | grep '^envelope_path:' | cut -d' ' -f2-)
if [ -f "$ENVELOPE_PATH" ]; then
  RISK_LEVEL=$(node -e "console.log(JSON.parse(fs.readFileSync('$ENVELOPE_PATH')).risk_level)")
  echo "Envelope risk_level: $RISK_LEVEL"
else
  echo "Envelope not found — proceeding with default risk (medium)"
  RISK_LEVEL="medium"
fi
```

**Why:** Matches existing artifact_path pattern; fails gracefully if envelope missing.

### Anti-Patterns to Avoid

- **Envelope replace (not merge)**: Setting `envelope = { plan: newPlan }` loses research section. **DO:** Read existing, merge new section, write back.
- **Config nested object for task_envelope**: Setting `task_envelope: { enabled: true }` in project config silently loses global `task_envelope.enabled` due to shallow merge in config-loader.js. **DO:** Use flat key `task_envelope.enabled`.
- **Synchronous init without tmpPath atomicity**: Writing directly to `task-envelope.json` risks corruption if process crashes mid-write. **DO:** Use `tmpPath + fs.renameSync` pattern.
- **Failing quorum if envelope missing**: quorum.md should log absence and continue with default risk level. **DO:** fail-open per R6 policy.
- **Including full PLAN.md in envelope**: defeats the purpose (token reduction). **DO:** store path only, not content.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Atomic JSON file writes | Custom file locking; setImmediate+callback pattern | tmpPath + fs.renameSync (existing pattern in scoreboard) | Race condition between check/write is eliminated by POSIX atomic rename; simpler code; already proven in production |
| Config validation for new keys | Custom type checker in each file | config-loader.js validateConfig() function (extend existing) | Centralized validation reduces bugs; integrates with existing shallow-merge layer; error messages go to stderr |
| Risk level determination | Heuristic scoring algorithm | Researcher explicit classification in RESEARCH.md metadata | Researcher understands the domain; heuristics are fragile and domain-specific; explicit makes policy visible |
| Envelope schema versioning | Major.minor semver, migration code | Single string version field "1" | v0.18 is first version; no migration needed; string "1" is sufficient for forward compat; complexity not justified yet |
| File path escaping in envelope | Custom shell quote function | JSON.stringify (handles all escaping) | JSON serialization is standard; shell-safe by design |

**Key insight:** The envelope is a simple JSON artifact. The complexity is in wiring it into plan-phase.md and quorum.md workflows, not in building a custom envelope system. Use existing patterns (atomic writes, config validation) and focus on correct integration points.

## Common Pitfalls

### Pitfall 1: Envelope Overwritten Instead of Updated

**What goes wrong:** plan-phase.md init creates envelope with research section. Then planner calls `update --section plan` but calls `init` instead. The envelope is overwritten, losing the research section. Plan execution fails because envelope_path points to empty/partial file.

**Why it happens:** Init and update look similar; easy to confuse which to call when. Grep for `task-envelope.cjs` in plan-phase.md shows wrong pattern.

**How to avoid:** Explicitly name the commands: `task-envelope.cjs init` (create, research step) vs `task-envelope.cjs update --section plan` (append, planner step). Document in plan-phase.md comments which step calls which.

**Warning signs:** Envelope exists but `plan` field is missing even after planner completes; quorum.md log says "envelope not found" even after planning step.

### Pitfall 2: Config Shallow Merge Eats `task_envelope` Nested Object

**What goes wrong:** User sets `task_envelope: { enabled: false }` in project `.claude/qgsd.json` intending to disable envelope. But config-loader.js shallow merge loses global `task_envelope` entirely. Plan-phase.md checks `config.task_envelope.enabled` and crashes (undefined property).

**Why it happens:** config-loader.js uses `{ ...DEFAULT_CONFIG, ...project }`. If project sets a key at the root level, it completely replaces the global value for that key. Nested objects are particularly vulnerable.

**How to avoid:** Use flat key: `task_envelope.enabled: false` (not nested). Document in qgsd.json comments: "flat keys required due to shallow merge."

**Warning signs:** Error message "cannot read property 'enabled' of undefined"; user's qgsd.json has nested `task_envelope: { ... }` object.

### Pitfall 3: Envelope Missing Causes Quorum to Block

**What goes wrong:** plan-phase.md has a bug and doesn't call `task-envelope.cjs init`. Envelope doesn't exist. Quorum dispatch reads `envelope_path: .planning/phases/v0.18-03/task-envelope.json` from plan-phase.md and tries to parse risk_level. File not found error propagates up and quorum blocks.

**Why it happens:** quorum.md wasn't written with fail-open behavior in mind. It assumes envelope exists because plan-phase.md was supposed to create it.

**How to avoid:** quorum.md checks file existence before reading: `test -f "$ENVELOPE_PATH" && ...` or `[ -f "$ENVELOPE_PATH" ] || RISK_LEVEL="medium"`. Emit log line either way so executor can see the decision.

**Warning signs:** Quorum dispatch fails with "file not found"; envelope_path is correct but file is missing; plan-phase.md research step is missing envelope init call.

### Pitfall 4: Risk Level Invalid Value Causes Silent Fallback

**What goes wrong:** Researcher sets `risk_level: "CRITICAL"` (capitalized, or typo). Envelope is written but field value is invalid. quorum.md reads it, risk_level check fails (not "low"/"medium"/"high"), and quorum silently falls back to default. Executor doesn't realize risk was miscategorized.

**Why it happens:** task-envelope.cjs doesn't validate risk_level on write (too strict); quorum.md doesn't log the fallback reason (too silent).

**How to avoid:** (1) task-envelope.cjs validates risk_level strictly on init; logs warnings to stderr if invalid but still writes (partial envelope accepted). (2) quorum.md logs both the read value and the computed effective_size, so executor sees "envelope risk_level=CRITICAL (invalid) → using default medium".

**Warning signs:** Executor sees "Envelope risk_level: CRITICAL" in quorum pre-flight logs but never mentions why it used "medium" instead.

### Pitfall 5: Envelope Path Is Relative, Breaks in Subshell

**What goes wrong:** plan-phase.md passes `envelope_path: .planning/phases/v0.18-03/task-envelope.json` (relative path). quorum.md is dispatched from a subshell with `cwd` set to a different directory. File lookup fails.

**Why it happens:** plan-phase.md context is relative to plan-phase.md's cwd; quorum context is passed as YAML; subshell may cwd to a temp directory.

**How to avoid:** plan-phase.md passes absolute path: `envelope_path: $PWD/.planning/phases/v0.18-03/task-envelope.json`. quorum.md always treats envelope_path as absolute (or relative to repo root, explicitly documented).

**Warning signs:** quorum.md says "envelope not found" but file exists; `ls .planning/phases/v0.18-03/task-envelope.json` works from main shell but quorum subshell can't see it.

## Code Examples

### Example 1: task-envelope.cjs init Command

**Source:** Design pattern from bin/gsd-tools.cjs (atomic writes) + bin/update-scoreboard.cjs (JSON schema)

```javascript
#!/usr/bin/env node
// bin/task-envelope.cjs
// Initialize task envelope after research completes.
// Usage: node bin/task-envelope.cjs init --phase v0.18-03 --objective "..." --risk-level low --target-files [...]

const fs = require('fs');
const path = require('path');

const args = process.argv.slice(2);
const command = args[0];

if (command === 'init') {
  const phaseIndex = args.indexOf('--phase');
  const phase = phaseIndex >= 0 ? args[phaseIndex + 1] : null;
  const objectiveIndex = args.indexOf('--objective');
  const objective = objectiveIndex >= 0 ? args[objectiveIndex + 1] : '';
  const riskIndex = args.indexOf('--risk-level');
  const riskLevel = riskIndex >= 0 ? args[riskIndex + 1] : 'medium';

  if (!phase) {
    process.stderr.write('[task-envelope] ERROR: --phase required\n');
    process.exit(1);
  }

  // Validate risk_level
  if (!['low', 'medium', 'high'].includes(riskLevel)) {
    process.stderr.write(`[task-envelope] WARNING: invalid risk_level "${riskLevel}"; using "medium"\n`);
  }

  const envelope = {
    schema_version: '1',
    phase,
    created_at: new Date().toISOString(),
    risk_level: ['low', 'medium', 'high'].includes(riskLevel) ? riskLevel : 'medium',
    research: {
      objective,
      constraints: '',
      target_files: [],
      confidence: 'HIGH'
    }
  };

  const phaseDir = `.planning/phases/${phase}-*`;
  // In real code, use glob or find; simplified here
  const dir = fs.readdirSync('.planning/phases/').find(d => d.startsWith(phase));
  if (!dir) {
    process.stderr.write(`[task-envelope] ERROR: phase directory not found\n`);
    process.exit(1);
  }

  const targetPath = path.join('.planning/phases', dir, 'task-envelope.json');
  const tmpPath = `${targetPath}.${Date.now()}.tmp`;

  try {
    fs.writeFileSync(tmpPath, JSON.stringify(envelope, null, 2), 'utf8');
    fs.renameSync(tmpPath, targetPath);
    console.log(`[task-envelope] initialized: ${targetPath}`);
  } catch (e) {
    process.stderr.write(`[task-envelope] ERROR writing envelope: ${e.message}\n`);
    process.exit(1);
  }
} else {
  process.stderr.write('[task-envelope] unknown command\n');
  process.exit(1);
}
```

### Example 3: Envelope Schema Validation (task-envelope.test.cjs)

**Source:** config-loader.test.js pattern

```javascript
// bin/task-envelope.test.cjs
const assert = require('assert/strict');
const { validateEnvelope, ENVELOPE_SCHEMA } = require('./task-envelope.cjs');

describe('Envelope Validation', () => {
  it('accepts valid envelope with required fields', () => {
    const envelope = {
      schema_version: '1',
      phase: 'v0.18-03',
      created_at: '2026-02-27T12:00:00Z',
      risk_level: 'medium',
      research: {
        objective: 'test',
        constraints: 'test',
        target_files: []
      }
    };
    const result = validateEnvelope(envelope);
    assert(result.valid === true);
  });

  it('rejects invalid risk_level', () => {
    const envelope = {
      schema_version: '1',
      phase: 'v0.18-03',
      risk_level: 'CRITICAL'  // invalid
    };
    const result = validateEnvelope(envelope);
    assert(result.valid === false);
    assert(result.errors.includes('risk_level'));
  });

  it('preserves research section on plan update', () => {
    const existing = {
      schema_version: '1',
      phase: 'v0.18-03',
      research: { objective: 'test' }
    };
    const update = { plan_path: 'path/to/plan.md' };
    const merged = { ...existing, plan: update };
    assert(merged.research.objective === 'test');
    assert(merged.plan.plan_path === 'path/to/plan.md');
  });
});
```

   - Recommendation: Ship v0.18-03 with 3-tier; Phase v0.18-04 can assess if finer granularity is needed. Current mapping: `low → 2 workers, medium → 3, high → max_size`.

2. **Envelope location: phase-level vs plan-level sidecar?**
   - What we know: Phase-level (`.planning/phases/<phase>/task-envelope.json`) enables one envelope per phase with multiple plans; plan-level (`.planning/phases/<phase>/v0.18-03-01-task-envelope.json`) enables per-plan envelope
   - What's unclear: Do multi-plan phases need separate envelopes for each plan?
   - Recommendation: Ship phase-level for v0.18-03. If multi-plan phases need plan-specific risk levels, Phase v0.18-05 can nest envelopes per plan.

3. **Config key naming: `task_envelope.enabled` vs `envelope.enabled`?**
   - What we know: Flat keys required due to shallow merge; `task_envelope.enabled` is explicit (matches requirement ENV-04)
   - What's unclear: Whether future enhancements (envelope compression, caching) justify a broader `envelope.*` namespace
   - Recommendation: Use `task_envelope.enabled` (narrowly scoped to task envelope feature); if v0.19 adds envelope enhancements, they can nest under same key without schema migration.
