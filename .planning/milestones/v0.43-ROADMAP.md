# Roadmap: nForma v0.43 — CodeGen Benchmark Track

**Created:** 2026-04-27
**Milestone:** v0.43
**Profile:** cli
**Depth:** standard
**Global phase range:** 59-64

## Overview

6 phases, 22 requirements. This milestone adds industry-standard code generation benchmarks (HumanEval: 164 problems, MBPP: 500 sanitized problems) to nForma's benchmark suite as a new `codegen` track, creating a 3-axis benchmark matrix: code generation (HumanEval/MBPP) + autonomous debugging (nf:debug) + formal alignment (nf:solve). The harness is multi-language from day one (Python + JavaScript execution) with a language-agnostic problem schema designed for future MultiPL-E translations. Standard pass@k scoring (pass@1, pass@10, pass@100) is separate from the existing quorum consensus pass@k. A CI smoke gate (5 problems) runs on every PR.

## Phase Details

### Phase 59: Problem Schema & Fixture Datasets
**Goal**: Define a language-agnostic problem schema and ship HumanEval (164) and MBPP (500 sanitized) as normalized fixture files ready for the execution harness
**Depends on**: Nothing (foundation phase)
**Requirements**: SCH-01, SCH-02, FIX-01, FIX-02, FIX-03
**Success Criteria** (what must be TRUE):
  1. Every problem has a unique ID, language tag, prompt, canonical solution, test cases (as executable assertions), difficulty hint, and metadata
  2. HumanEval fixtures contain all 164 problems with original docstring prompts + reference solutions + test function bodies
  3. MBPP fixtures contain 500 sanitized problems with task descriptions + reference solutions + test assertions
  4. A smoke subset of 5 problems (3 HumanEval + 2 MBPP) is identified and tagged for CI
  5. Problem schema is JSON-compatible and does not encode any language-specific execution semantics — language routing is derived from the `language` field at execution time
**Plans**: 2 plans

Plans:
- [ ] 59-01: Problem schema definition + HumanEval fixture normalization — SCH-01, SCH-02, FIX-01, FIX-02
- [ ] 59-02: MBPP fixture normalization + smoke subset selection + schema validation tests — FIX-03

### Phase 60: Multi-Language Execution Harness
**Goal**: Execute generated code in sandboxed subprocesses (python3, node) with timeout/memory limits and deterministic pass/fail verdicts from test cases
**Depends on**: Phase 59 (problem schema defines test cases and language tag)
**Requirements**: EXE-01, EXE-02, EXE-03, EXE-04, EXE-05
**Success Criteria** (what must be TRUE):
  1. User can execute a Python solution against its test cases via `python3` subprocess and get a deterministic pass/fail verdict
  2. User can execute a JavaScript solution against its test cases via `node` subprocess and get a deterministic pass/fail verdict
  3. Each execution is isolated in a temp directory with a configurable timeout (default 30s) and memory limit
  4. Execution harness returns structured result: `{ passed: bool, stdout, stderr, exit_code, timed_out, error }`
  5. Adding a new language requires only a new executor module (implementing the same interface) + registering it in the executor registry — no changes to runner or scorer
**Plans**: 2 plans

Plans:
- [ ] 60-01: Executor interface + registry + Python executor + temp-dir isolation — EXE-01, EXE-02, EXE-03, EXE-04
- [ ] 60-02: JavaScript executor + executor registry wiring + multi-language validation tests — EXE-05

### Phase 61: Codegen Runner & Scorer
**Goal**: Run LLM code generation for each problem, execute generated solutions, compute standard pass@k scores, and track baselines
**Depends on**: Phase 60 (execution harness), reuses CLI-based LLM invocation pattern from `nf-debug-runner.cjs`
**Requirements**: RUN-01, RUN-02, SCOR-01, SCOR-02, SCOR-03, BASE-01
**Success Criteria** (what must be TRUE):
  1. User can invoke `node bin/nf-benchmark-codegen.cjs --dataset humaneval --track full --json` to run all 164 HumanEval problems through LLM code generation and execution
  2. LLM invocation reuses the CLI-based `callClaude()` pattern from `nf-debug-runner.cjs` — no new SDK dependency
  3. Runner produces per-problem results: `{ problem_id, passed, generated_code, execution_result, latency_ms }`
  4. Scorer computes pass@1 (single sample), pass@10, and pass@100 using the standard unbiased estimator: `pass@k = 1 - C(n-c, k) / C(n, k)` where n = total samples, c = correct
  5. Baseline file tracks pass@1/pass@10/pass@100 per dataset with auto-advancement (new baseline written when score improves)
  6. `--track smoke` runs only the 5-problem smoke subset in under 2 minutes
**Plans**: 2 plans

Plans:
- [ ] 61-01: Codegen runner (LLM invocation + per-problem execution loop + JSON output) — RUN-01, RUN-02
- [ ] 61-02: pass@k scorer + baseline tracking + auto-advancement — SCOR-01, SCOR-02, SCOR-03, BASE-01

### Phase 62: CI Smoke Gate & Baseline Automation
**Goal**: Wire the codegen smoke gate into CI alongside existing benchmark gates, with baseline comparison and auto-advancement
**Depends on**: Phase 61 (runner + scorer + baselines)
**Requirements**: CI-04, CI-05, GATE-01
**Success Criteria** (what must be TRUE):
  1. Every PR runs the 5-problem codegen smoke gate (no API key required for structural validation; API key required for LLM smoke)
  2. Structural validation verifies all fixture files parse correctly, schema is valid, and all reference solutions pass their own test cases
  3. LLM smoke gate (when ANTHROPIC_API_KEY present) runs 5 problems through the full pipeline and checks pass_rate against smoke baseline
  4. `check-benchmark-gate.cjs` extended to support codegen baseline format (pass@1 threshold)
  5. `update-benchmark-baseline.sh` extended to advance codegen baselines
**Plans**: 2 plans

Plans:
- [ ] 62-01: CI workflow job + structural validation + smoke LLM gate — CI-04, CI-05
- [ ] 62-02: Gate script extension + baseline auto-advancement wiring — GATE-01

### Phase 63: Benchmark Matrix Integration

  3. README or docs updated to document the 3-axis benchmark matrix and how to run each track
