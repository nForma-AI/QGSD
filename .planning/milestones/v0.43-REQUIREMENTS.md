# Requirements: nForma v0.43 — CodeGen Benchmark Track

**Defined:** 2026-04-27
**Core Value:** Industry-standard code generation benchmarks enable comparability with published results from other AI coding tools

## Baseline Requirements

*Included from nForma baseline defaults (profile: cli). Cross-cutting quality gates.*

### UX Heuristics
- [ ] **UX-01**: Every user-initiated action produces immediate feedback (loading/disabled state) and completion feedback (result message, navigation change, or visible state update)
- [ ] **UX-02**: Destructive actions (delete, reset, remove, overwrite) require explicit confirmation or provide undo within a reasonable window
- [ ] **UX-03**: Error messages are human-readable, explain what went wrong, and suggest a next step or recovery action

### Security
- [ ] **SEC-01**: Pre-commit hook runs secret scanning (e.g., Gitleaks) to block commits containing API keys, tokens, passwords, or credentials
- [ ] **SEC-02**: CI pipeline runs deep secret scanning (e.g., TruffleHog) across full repo history on every PR
- [ ] **SEC-03**: All external input (user input, API request bodies, query parameters, file uploads) is validated and sanitized at system boundaries before processing
- [ ] **SEC-04**: Dependencies are scanned for known vulnerabilities in CI (e.g., npm audit, Dependabot, Snyx) and critical/high findings block merge

### Reliability
- [ ] **REL-01**: Failures in external services (APIs, databases, third-party SDKs) are caught and handled gracefully — the application degrades functionality rather than crashing
- [ ] **REL-02**: Long-running operations (file uploads, data processing, API calls >2s) show progress indication and can be cancelled by the user

### Observability
- [ ] **OBS-01**: CLI tools exit with appropriate codes (0 = success, non-zero = failure) and write errors to stderr, output to stdout

### CI/CD
- [ ] **CI-01**: Automated test suite runs on every pull request and merge to main is blocked when tests fail
- [ ] **CI-02**: Linting and formatting checks run in CI and block merge on violations
- [ ] **CI-03**: Type checking (if applicable to the language) runs in CI and blocks merge on type errors

## Milestone v0.43 Requirements

### Problem Schema
- [ ] **SCH-01**: Every problem has a unique ID, language tag, prompt (natural language + function signature), canonical solution, test cases (as executable assertions), difficulty hint, and metadata (source dataset, original ID) — all in a single JSON object
- [ ] **SCH-02**: Problem schema is JSON-compatible and does not encode any language-specific execution semantics — language routing is derived from the `language` field at execution time, making the schema portable to any target language

### Fixture Datasets
- [ ] **FIX-01**: HumanEval fixtures contain all 164 problems with original docstring prompts, function signatures, reference solutions, and test function bodies normalized to the problem schema
- [ ] **FIX-02**: MBPP fixtures contain 500 sanitized problems with task descriptions, reference solutions, and test assertions normalized to the problem schema
- [ ] **FIX-03**: A smoke subset of 5 problems (3 HumanEval + 2 MBPP) is identified and tagged in the fixture data for CI smoke gates

### Execution Harness
- [ ] **EXE-01**: User can execute a Python solution against its test cases via `python3` subprocess and get a deterministic pass/fail verdict
- [ ] **EXE-02**: User can execute a JavaScript solution against its test cases via `node` subprocess and get a deterministic pass/fail verdict
- [ ] **EXE-03**: Each execution is isolated in a temp directory with a configurable timeout (default 30s) and memory limit — no shared state between executions
- [ ] **EXE-04**: Execution harness returns structured result: `{ passed: bool, stdout, stderr, exit_code, timed_out, error }`
- [ ] **EXE-05**: Adding a new language requires only a new executor module (implementing the same interface) + registering it in the executor registry — no changes to runner, scorer, or CI

### Codegen Runner
- [ ] **RUN-01**: User can invoke `node bin/nf-benchmark-codegen.cjs --dataset humaneval --track full --json` to run all 164 HumanEval problems through LLM code generation and execution, receiving per-problem results and aggregate scores
- [ ] **RUN-02**: LLM invocation reuses the CLI-based `callClaude()` pattern from `nf-debug-runner.cjs` — prompt is the problem's docstring/signature, response is the generated code, no new SDK dependency introduced

### Scoring
- [ ] **SCOR-01**: Scorer computes pass@1 (single sample per problem) as the fraction of problems where the generated solution passes all test cases
- [ ] **SCOR-02**: Scorer computes pass@k (k=10, 100) using the standard unbiased estimator: `pass@k = 1 - C(n-c, k) / C(n, k)` where n = total samples, c = correct — separate from the existing quorum consensus pass@k
- [ ] **SCOR-03**: Runner produces per-problem results: `{ problem_id, passed, generated_code, execution_result, latency_ms }` and aggregate: `{ dataset, track, total, passed, pass_rate, pass_at_1, pass_at_10, pass_at_100 }`

### Baseline Tracking
- [ ] **BASE-01**: Baseline file (`benchmarks/codegen/baseline.json`) tracks pass@1/pass@10/pass@100 per dataset with auto-advancement — new baseline written when score improves, matching the pattern used by `solve-baseline.json` and `debug/baseline.json`

### CI Integration
- [ ] **CI-04**: Every PR runs a codegen structural validation gate (no API key required) that verifies all fixture files parse correctly, schema is valid, and all reference solutions pass their own test cases
- [ ] **CI-05**: Every PR with ANTHROPIC_API_KEY available runs the 5-problem codegen smoke gate through the full LLM pipeline and checks pass_rate against the smoke baseline (matching the pattern in `benchmark-gate.yml`)

### Gate Wiring
- [ ] **GATE-01**: `check-benchmark-gate.cjs` and `update-benchmark-baseline.sh` are extended to support the codegen baseline format (pass@1 threshold) so the existing gate infrastructure works for all three tracks

### Benchmark Matrix
- [ ] **MAT-01**: User can run `node bin/nf-benchmark-matrix.cjs --json` and see a unified report with all three tracks: codegen (pass@1/pass@10), debug (fix rate), solve (residual reduction)
- [ ] **MAT-02**: Matrix report shows per-track baseline comparison with pass/improve/regress status

### Documentation
- [ ] **DOC-01**: README or docs updated to document the 3-axis benchmark matrix (codegen + debug + solve), how to run each track, and how to interpret the unified report

### Multi-Language Extensibility
- [ ] **MULT-01**: At least 10 HumanEval problems have JavaScript translations in the fixture format, runnable through the same harness without runner/scorer/CI code changes
- [ ] **MULT-02**: A translation guide documents the process for creating new language translations so future MultiPL-E translations can be added mechanically
- [ ] **MULT-03**: Full milestone audit completed — all requirements traced to phases, all phases verified, ROADMAP.md and MILESTONES.md updated

## Future Requirements

### Extended Datasets
- **FIX-04**: APPS dataset integration (5,000 problems across 5 difficulty levels)
- **FIX-05**: CodeContests dataset integration (competitive programming problems with hidden tests)

- **SCOR-05**: Execution-based semantic equivalence (mutation testing — generated code is semantically equivalent if it passes mutated test variants)
- **SCOR-06**: Temporal score tracking with trend analysis (pass@1 over time per model/dataset)

### Language Coverage

- **MULT-05**: Type-aware execution harness for TypeScript (typecheck before execute)
- **MULT-06**: Rust executor with `cargo test` integration

### Benchmark Enhancements
- **RUN-03**: Multi-model comparison mode — run same problem set across multiple models and produce comparison table
- **RUN-04**: Temperature/sampling parameter sweeps for pass@k sensitivity analysis
- **BASE-02**: Baseline regression alerts with Mann-Kendall trend detection (reuses oscillation-detector.cjs pattern)

## Out of Scope

| Feature | Reason |
|---------|--------|
| Changes to nf:solve or nf:debug benchmark internals | Existing tracks are stable; new track must not regress them |
| Real-time leaderboard or web UI | CLI-only output matches existing benchmark pattern; web visualization is a separate project |
| Training or fine-tuning models | Benchmark measures model capability; model improvement is out of scope |
| Docker-based sandboxing | Subprocess isolation in temp dir is sufficient for benchmark execution; Docker adds CI complexity |
| HumanEval+ / MBPP+ (test augmentation) | Can be added as FIX-07 later; initial milestone ships canonical datasets only |
