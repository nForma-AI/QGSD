# Phase v0.29-03: Reasoning Enrichment and Gate B Abstraction - Research

**Researched:** 2026-03-06
**Domain:** FMEA hazard analysis applied to operational state machines + Gate B traceability enforcement
**Confidence:** HIGH

## Summary

Phase v0.29-03 builds Layer 3 (Reasoning) on top of the Layer 2 (Semantics) artifacts completed in v0.29-02. The core deliverables are three new CLI scripts: (1) a hazard model that applies FMEA scoring to every state and transition in the observed FSM, (2) a failure mode catalog that enumerates concrete failure modes per state/transition, and (3) a risk heatmap that ranks transitions by combining FMEA severity with L1 coverage gap data. Gate B then enforces traceability: every L3 artifact must have `derived_from` links back to L2 sources, with orphaned hazards reported as failures. GATE-04 adds progressive maturity tracking so models can be promoted from ADVISORY to SOFT_GATE to HARD_GATE.

The key technical insight is that the L2 foundation is small and well-structured: 4 states (IDLE, COLLECTING_VOTES, DELIBERATING, DECIDED), 16 state-event pairs in observed-fsm.json, 151 invariants in invariant-catalog.json, 10 mismatches in mismatch-register.jsonl, and 567 assumptions in assumption-register.json. FMEA analysis on this scale is straightforward -- the challenge is mapping the industrial FMEA methodology (designed for physical components) to software state machine concepts. Severity maps to impact of a state/transition failure on the quorum workflow. Occurrence maps to observed frequency (from trace-corpus-stats.json). Detection maps to existing test/formal verification coverage (from failure-taxonomy.json and check-results.ndjson).

**Primary recommendation:** Build the hazard model and failure mode catalog together (they share the FMEA scoring logic), then build the risk heatmap (which combines FMEA output with L1 coverage gaps), then Gate B + GATE-04 gate maturity (which cross-cuts all L3 artifacts).

### Pattern 2: FMEA Scoring for Software State Machines
**What:** Map traditional FMEA dimensions (Severity, Occurrence, Detection) to software state machine concepts. Each dimension is scored 1-10.
**When to use:** RSN-01 hazard model, RSN-02 failure mode catalog.
**Mapping:**

```
SEVERITY (1-10): Impact if this state/transition fails
  10: System crash / data loss / silent corruption
   8: Quorum decision incorrect (wrong PASS/BLOCK verdict)
   6: Quorum stalled (no decision reached, requires manual intervention)
   4: Degraded operation (fallback path used, slower but correct)
   2: Cosmetic / logging gap (no functional impact)
   1: No impact

OCCURRENCE (1-10): How often this state/transition is exercised
  10: Observed in >80% of sessions (always happens)
   8: Observed in 50-80% of sessions
   6: Observed in 20-50% of sessions
   4: Observed in 5-20% of sessions
   2: Observed in <5% of sessions
   1: Never observed (theoretical only)

DETECTION (1-10): How well current testing catches a failure HERE
  10: No test coverage, no formal model, no monitoring (undetectable)
   8: Only runtime conformance events (passive detection)
   6: Unit tests exist but no formal model
   4: Formal model exists but not TLC-checked successfully
   2: Formal model exists AND TLC-checked AND unit-tested
   1: Full coverage: formal model + TLC pass + unit tests + integration tests

RPN = Severity x Occurrence x Detection (range: 1-1000)
```

**Occurrence computation from observed-fsm.json:**
```javascript
// Source: derived from observed-fsm.json structure
function computeOccurrence(stateEventPair, observedFsm, traceStats) {
  const totalSessions = traceStats.sessions.length; // 349
  // Count sessions containing this transition
  const transitionCount = observedFsm.observed_transitions[stateEventPair.from]?.[stateEventPair.event]?.count || 0;
  const ratio = transitionCount / totalSessions;
  if (ratio > 0.8) return 10;
  if (ratio > 0.5) return 8;
  if (ratio > 0.2) return 6;
  if (ratio > 0.05) return 4;
  if (ratio > 0) return 2;
  return 1;
}
```

**Detection computation from check-results and test coverage:**
```javascript
// Source: derived from failure-taxonomy.json and unit-test-coverage.json
function computeDetection(stateOrTransition, failureTaxonomy, testCoverage) {
  const hasFormalism = failureTaxonomy.categories.logic_violation
    .some(f => f.requirement_ids.some(r => relatedRequirements(stateOrTransition).includes(r)));
  const hasTests = testCoverage[stateOrTransition.related_script] !== undefined;

  if (hasFormalism && hasTests) return 2;  // well-covered
  if (hasFormalism || hasTests) return 4;  // partially covered
  return 8;  // poorly covered
}
```

### Pattern 4: Gate B Traceability Verification
**What:** Gate B scans all L3 artifacts and verifies every entry has at least one valid `derived_from` link to an L2 source. Orphaned entries (no links) are failures.
**When to use:** GATE-02 implementation.
**Algorithm:**
```
1. Load all L3 reasoning artifacts (hazard-model.json, failure-mode-catalog.json, risk-heatmap.json)
2. Load all L2 semantics artifacts (observed-fsm.json, invariant-catalog.json, mismatch-register.jsonl, assumption-register.json)
3. For each L3 entry:
   a. Check derived_from array exists and is non-empty
   b. For each derived_from link, verify the referenced L2 artifact and ref path exist
   c. Mark entry as "grounded" (all links valid) or "orphaned" (any link broken)
4. Compute gate_b_score = grounded_entries / total_entries
5. Report orphaned hazards with details
```

### Pattern 5: Gate Maturity Promotion (GATE-04)
**What:** Each model in model-registry.json has a `gate_maturity` level that can be promoted.
**When to use:** GATE-04 implementation.
**Promotion criteria:**
```
ADVISORY (default):
  - Model exists in registry
  - No gate enforcement; violations are warnings only

SOFT_GATE:
  - Model has source_layer assigned
  - If L3: all entries have derived_from links (Gate B passes)
  - Gate violations logged as warnings but do NOT block operations

HARD_GATE:
  - Meets all SOFT_GATE criteria
  - Model has been TLC/Alloy-checked successfully (check-result pass in NDJSON)
  - Gate violations are errors that block operations
```

**Promotion CLI:**
```bash
node bin/promote-gate-maturity.cjs --model "alloy/quorum-votes.als" --level SOFT_GATE
node bin/promote-gate-maturity.cjs --model "alloy/quorum-votes.als" --level HARD_GATE
node bin/promote-gate-maturity.cjs --check  # verify all models meet their current level criteria
```

### Anti-Patterns to Avoid
- **Inventing failure modes without evidence:** Every failure mode MUST derive from either (a) an observed mismatch in mismatch-register.jsonl, (b) a missing_in_model transition from observed-fsm.json, (c) a known invariant violation, or (d) a standard FMEA failure category applied to the state/transition type. Do not fabricate theoretical failure modes that have no L2 backing.
- **Hardcoding FMEA scores:** Severity, Occurrence, and Detection scores must be COMPUTED from data (trace counts, test coverage, formal verification results), not manually assigned. The only exception is Severity, which requires domain judgment but should follow the documented scale.
- **Making Gate B a subprocess of Gate A:** Gate A measures L1-L2 alignment. Gate B measures L2-L3 alignment. They are independent gates in separate scripts. Gate B reads L3 artifacts and verifies L2 links; it never touches L1 data directly.
- **Conflating gate maturity promotion with gate score computation:** `promote-gate-maturity.cjs` changes the model-registry.json `gate_maturity` field. Gate B computes a traceability score. These are separate operations.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| L2 artifact reading | Custom JSON parsers | `require()` + `JSON.parse()` | L2 artifacts are well-structured JSON from v0.29-02 |
| Check result emission | Custom NDJSON writer | `write-check-result.cjs` (existing) | Already validates formalism/result enums |
| Layer manifest updates | Manual JSON editing | `build-layer-manifest.cjs` (v0.29-01) | Already handles classification + schema |
| File path resolution | Hardcoded paths | `planning-paths.cjs` (existing) | Handles monorepo/subdir scenarios |
| Model registry updates | Manual JSON editing | Read/modify/write with schema validation | model-registry.json has a known schema |

**Key insight:** FMEA itself is simple arithmetic (S x O x D = RPN). The real work is mapping S/O/D scores to computable quantities from existing L1/L2 data. The scoring functions ARE the deliverable, not the multiplication.

## Common Pitfalls

### Pitfall 1: FMEA Score Inflation
**What goes wrong:** All transitions get high RPN scores (>200) because Severity is set high for everything, making the heatmap meaningless.
**Why it happens:** Without a disciplined severity scale, every failure "seems important."
**How to avoid:** Use the documented 1-10 scale strictly. Only state transitions that affect quorum correctness (wrong PASS/BLOCK) get Severity >= 8. Most transitions that self-loop in DECIDED state are Severity 2-4 (already in terminal state, self-loop is harmless).
**Warning signs:** All RPNs cluster above 200, or the top-10 and bottom-10 RPNs are within 2x of each other.

### Pitfall 2: Orphaned L3 Artifacts
**What goes wrong:** Hazard model entries reference L2 sources that do not exist or use wrong JSON paths.
**Why it happens:** `derived_from` links are constructed as strings that look plausible but point to non-existent paths in L2 artifacts.
**How to avoid:** Gate B must RESOLVE every `derived_from` link by actually loading the referenced L2 artifact and navigating to the ref path. The link `semantics/observed-fsm.json#observed_transitions.IDLE.QUORUM_START` must be validated by checking `observedFsm.observed_transitions.IDLE.QUORUM_START` exists.
**Warning signs:** Gate B passes with 100% score on first run without any debugging needed.

### Pitfall 3: Failure Mode Explosion
**What goes wrong:** Each of the 16 state-event pairs generates 5+ failure modes, creating 80+ entries that are too numerous to be actionable.
**Why it happens:** FMEA methodology encourages exhaustive enumeration, but software states have fewer failure modes than physical components.
**How to avoid:** Limit to 2-3 failure modes per state-event pair: (1) the transition does not fire when it should (omission), (2) the transition fires when it should not (commission), (3) the transition fires but produces wrong state (corruption). Not all three apply to every pair.
**Warning signs:** Failure mode catalog has >60 entries. Target: 30-50.

### Pitfall 4: Gate Maturity Promotion Without Criteria Validation
**What goes wrong:** A model is promoted to HARD_GATE but does not actually meet the criteria, causing legitimate operations to be blocked.
**Why it happens:** Promotion is done manually without automated criteria checking.
**How to avoid:** `promote-gate-maturity.cjs --check` must validate that every model meets its current level's criteria. Promotion to a new level must pass the target level's criteria check first. If criteria are not met, promotion fails with a specific error.
**Warning signs:** `--check` reports models that do not meet their declared level.

### Pitfall 5: Coverage Gap Data Mismatch
**What goes wrong:** The risk heatmap uses coverage gap data from a different source than the FMEA Detection score, producing contradictory rankings.
**Why it happens:** Coverage gap data exists in multiple places: instrumentation-map.json (emission points), state-candidates.json (missing transitions), failure-taxonomy.json (check failures), observed-fsm.json (model_coverage).
**How to avoid:** Use a single canonical source for the heatmap's coverage component: observed-fsm.json `model_comparison.missing_in_model` (11 transitions not in the XState model) combined with `coverage.model_coverage` (62.5%). The FMEA Detection score should use failure-taxonomy.json for formal verification coverage. These are complementary, not competing data sources.
**Warning signs:** A transition ranks high in the heatmap but has Detection=2 in the FMEA, or vice versa.

### FMEA Hazard Model Generation
```javascript
// Source: derived from observed-fsm.json structure + FMEA methodology
function generateHazardModel(observedFsm, invariantCatalog, traceStats, failureTaxonomy) {
  const hazards = [];
  const transitions = observedFsm.observed_transitions;

  for (const fromState of Object.keys(transitions)) {
    for (const event of Object.keys(transitions[fromState])) {
      const trans = transitions[fromState][event];
      const toState = trans.to_state;
      const count = trans.count;

      const severity = computeSeverity(fromState, event, toState);
      const occurrence = computeOccurrenceScore(count, traceStats.sessions.length);
      const detection = computeDetectionScore(fromState, event, failureTaxonomy);
      const rpn = severity * occurrence * detection;

      hazards.push({
        id: `HAZARD-${fromState}-${event}`,
        state: fromState,
        event: event,
        to_state: toState,
        observed_count: count,
        severity, occurrence, detection, rpn,
        derived_from: [
          { layer: 'L2', artifact: 'semantics/observed-fsm.json', ref: `observed_transitions.${fromState}.${event}` }
        ]
      });
    }
  }

  return {
    schema_version: '1',
    generated: new Date().toISOString(),
    methodology: 'FMEA (IEC 60812)',
    scoring: { severity: '1-10', occurrence: '1-10', detection: '1-10', rpn: 'S x O x D' },
    hazards: hazards.sort((a, b) => b.rpn - a.rpn),
    summary: {
      total_hazards: hazards.length,
      max_rpn: Math.max(...hazards.map(h => h.rpn)),
      critical_count: hazards.filter(h => h.rpn >= 200).length,
      high_count: hazards.filter(h => h.rpn >= 100 && h.rpn < 200).length
    }
  };
}
```

### Severity Mapping for Quorum States
```javascript
// Source: domain analysis of nForma quorum workflow
function computeSeverity(fromState, event, toState) {
  // Critical: transitions affecting quorum correctness
  if (event === 'DECIDE' && fromState === 'DELIBERATING') return 8;  // wrong verdict = incorrect PASS/BLOCK
  if (event === 'VOTES_COLLECTED' && toState === 'DELIBERATING') return 6;  // incomplete votes = stalled quorum
  if (event === 'CIRCUIT_BREAK') return 6;  // false positive break = workflow interruption
  if (event === 'QUORUM_START' && fromState === 'IDLE') return 4;  // start failure = degraded (can retry)

  // Self-loops in DECIDED are low severity (already terminal)
  if (fromState === 'DECIDED' && toState === 'DECIDED') return 2;

  // Default: moderate severity
  return 4;
}
```

### Failure Mode Catalog Entry
```javascript
// Source: FMEA failure mode categories applied to state machine transitions
function enumerateFailureModes(stateEventPair, observedFsm, mismatches) {
  const modes = [];
  const { from, event, to } = stateEventPair;

  // Omission: transition does not fire when expected
  modes.push({
    id: `FM-${from}-${event}-OMISSION`,
    failure_mode: 'omission',
    description: `${event} event not processed in ${from} state when conditions are met`,
    effect: `System remains in ${from} state instead of transitioning to ${to}`,
    severity_class: from === 'IDLE' ? 'degraded' : 'stalled',
    derived_from: [
      { layer: 'L2', artifact: 'semantics/observed-fsm.json', ref: `observed_transitions.${from}.${event}` }
    ]
  });

  // Commission: transition fires spuriously
  if (observedFsm.model_comparison.missing_in_model.some(m => m.from === from && m.event === event)) {
    modes.push({
      id: `FM-${from}-${event}-COMMISSION`,
      failure_mode: 'commission',
      description: `${event} fires in ${from} but is NOT modeled in XState machine`,
      effect: `Unmodeled state transition; actual behavior diverges from specification`,
      severity_class: 'model_gap',
      derived_from: [
        { layer: 'L2', artifact: 'semantics/observed-fsm.json', ref: `model_comparison.missing_in_model` }
      ]
    });
  }

  return modes;
}
```

### Gate B Traceability Check
```javascript
// Source: derived from Gate A pattern in gate-a-grounding.cjs
function checkGateB(reasoningDir, semanticsDir) {
  const artifacts = ['hazard-model.json', 'failure-mode-catalog.json', 'risk-heatmap.json'];
  let totalEntries = 0;
  let groundedEntries = 0;
  const orphans = [];

  for (const artFile of artifacts) {
    const artPath = path.join(reasoningDir, artFile);
    if (!fs.existsSync(artPath)) continue;
    const data = JSON.parse(fs.readFileSync(artPath, 'utf8'));

    // Get the array of entries (hazards, failure_modes, or transitions)
    const entries = data.hazards || data.failure_modes || data.transitions || [];
    for (const entry of entries) {
      totalEntries++;
      if (!entry.derived_from || entry.derived_from.length === 0) {
        orphans.push({ artifact: artFile, entry_id: entry.id, reason: 'no_derived_from' });
        continue;
      }
      // Verify each link resolves
      let allValid = true;
      for (const link of entry.derived_from) {
        if (!resolveL2Link(link, semanticsDir)) {
          orphans.push({ artifact: artFile, entry_id: entry.id, reason: 'broken_link', link });
          allValid = false;
        }
      }
      if (allValid) groundedEntries++;
    }
  }

  return {
    schema_version: '1',
    generated: new Date().toISOString(),
    gate_b_score: totalEntries > 0 ? groundedEntries / totalEntries : 0,
    total_entries: totalEntries,
    grounded_entries: groundedEntries,
    orphaned_entries: orphans.length,
    orphans,
    target: 1.0,
    target_met: orphans.length === 0
  };
}
```

### Gate Maturity Promotion
```javascript
// Source: derived from model-registry.json structure
function promoteModel(registryPath, modelKey, targetLevel) {
  const LEVELS = ['ADVISORY', 'SOFT_GATE', 'HARD_GATE'];
  const registry = JSON.parse(fs.readFileSync(registryPath, 'utf8'));
  const model = registry.models[modelKey];
  if (!model) throw new Error(`Model not found: ${modelKey}`);

  const currentIdx = LEVELS.indexOf(model.gate_maturity || 'ADVISORY');
  const targetIdx = LEVELS.indexOf(targetLevel);
  if (targetIdx <= currentIdx) {
    console.log(`Model already at ${model.gate_maturity}, target ${targetLevel} is not a promotion`);
    return false;
  }

  // Validate criteria for target level
  const criteria = validateCriteria(model, targetLevel);
  if (!criteria.met) {
    console.error(`Cannot promote to ${targetLevel}: ${criteria.reason}`);
    return false;
  }

  model.gate_maturity = targetLevel;
  model.last_updated = new Date().toISOString();
  fs.writeFileSync(registryPath, JSON.stringify(registry, null, 2) + '\n');
  return true;
}
```

## Open Questions

1. **Should Severity scores be hardcoded or data-driven?**
   - What we know: Occurrence and Detection can be computed from trace data and test coverage data. Severity requires domain judgment about the impact of each failure.
   - What's unclear: Whether to hardcode a severity lookup table or derive severity from requirement priority.
   - Recommendation: Use a severity lookup table keyed by (fromState, event) pairs. This is a design judgment, not a data computation. Document the rationale for each severity assignment in the hazard-model.json output. The table has only 16 entries (4 states x 4 events) so manual assignment is feasible.

2. **What is the Gate B target score?**
   - What we know: Gate A targets >= 80%. Gate B should target 100% (no orphaned hazards) since every L3 artifact is generated by our scripts and should always have valid links.
   - What's unclear: Whether 100% is realistic for the initial implementation.
   - Recommendation: Target 100%. Gate B is structural -- it checks whether derived_from links exist and resolve. Unlike Gate A (which depends on external trace data), Gate B checks artifacts we fully control. Any orphan is a bug in our scripts, not a data issue.

3. **How should the risk heatmap combine FMEA RPN with coverage gaps?**
   - What we know: FMEA gives RPN (1-1000). Coverage gaps give a model_coverage percentage (currently 62.5%) and a list of 11 missing_in_model transitions.
   - What's unclear: The formula for combining these into a single ranking.
   - Recommendation: Use a composite score: `risk_score = RPN * (1 + coverage_gap_penalty)` where `coverage_gap_penalty = 0.5` if the transition is in missing_in_model (not modeled), `0.0` otherwise. This boosts risk for transitions that lack formal modeling. The heatmap sorts by risk_score descending.

4. **Should promote-gate-maturity.cjs support demotion?**
   - What we know: GATE-04 says "level can be promoted." No mention of demotion.
   - What's unclear: Whether models that fail criteria after promotion should be automatically demoted.
   - Recommendation: Support demotion via `--check --fix` flag. `--check` reports violations. `--check --fix` demotes models that no longer meet their level's criteria. This prevents stale HARD_GATE models from blocking legitimate operations.
