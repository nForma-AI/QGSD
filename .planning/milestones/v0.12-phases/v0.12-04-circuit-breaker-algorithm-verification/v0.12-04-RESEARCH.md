## Summary

Phase v0.12-04 requires two new TLA+ specifications: `QGSDOscillation.tla` (models the run-collapse oscillation detection algorithm) and `QGSDConvergence.tla` (models state persistence — the `resolvedAt` write-once invariant, log-write-before-state-delete ordering, and Haiku failure safety). These feed into `MCoscillation.cfg` and `MCconvergence.cfg`, which TLC verifies. A runner `bin/run-oscillation-tlc.cjs` invokes TLC, is gated on `JAVA_HOME`, and is tested via `bin/run-oscillation-tlc.test.cjs` with 4 error-path tests.

The algorithm being modeled lives entirely in `hooks/qgsd-circuit-breaker.js`. It is compact, deterministic, and well-suited to TLA+ abstraction. The key modeling challenge for GAP-1 is correctly representing the two-stage algorithm: (1) run-collapse (consecutive identical file sets → groups), (2) group count ≥ depth with net-negative diff. For GAP-5, the challenge is capturing the correct ordering of `writeOscillationLog` before `rmSync(statePath)` and ensuring that a `null` Haiku verdict (API unavailable) causes no state mutation.

The existing TLA+ work in this repo (`QGSDQuorum.tla`, `QGSDCircuitBreaker.tla`) and the runner/test pattern from `run-breaker-tlc.cjs` / `run-breaker-tlc.test.cjs` provide exact templates for both specs and tooling.

**Primary recommendation:** Model the oscillation algorithm as a finite sequence abstraction: `commits` is a sequence of file-set labels (e.g., A/B/A/B/A), `runs` is its run-length collapse, `flagCount` tracks how many distinct run-groups share the same label. The key invariant `OscillationFlaggedCorrectly` asserts that the `flagged` boolean is TRUE iff `flagCount >= Depth /\ netChange <= 0`. The convergence spec models two boolean flags (`logWritten`, `stateDeleted`) with the ordering invariant `stateDeleted => logWritten`.

### Pattern 1: Run-Collapse Algorithm Abstraction (QGSDOscillation.tla)

**What:** Abstract the `commits` array as a sequence of labels over a finite alphabet (e.g., `{A, B, C}`). The model simulates run-collapse and counts occurrences per label.

**State variables:**
- `commits` — Seq of labels (abstract file-set identifiers); models the `hashes`/`fileSets` arrays
- `runs` — Seq of labels after run-collapse (consecutive duplicates merged)
- `flagCount` — Map from label → count of how many run-groups exist with that label
- `netChange` — Integer: sum of (additions - deletions) across all consecutive pairs in the oscillating group; models `hasReversionInHashes` result
- `flagged` — BOOLEAN: whether oscillation was reported

**Key invariant `OscillationFlaggedCorrectly`:**
```tla
OscillationFlaggedCorrectly ==
    flagged <=>
        (\E label \in DOMAIN flagCount :
            flagCount[label] >= Depth /\ netChange <= 0)
```

**Key liveness `AlgorithmTerminates`:**
```tla
AlgorithmTerminates == <>(flagged \/ algorithmDone)
```

Since the algorithm is a finite deterministic loop over a bounded commits sequence (max `CommitWindow`), WF is sufficient.

**Actions:** `AddCommit(label)` (extend commits), `CollapseRuns` (compute run-collapse), `ComputeNetChange` (compute sum of diffs), `EvaluateFlag` (set flagged per the invariant predicate).

### Pattern 2: State Persistence Write-Once Invariant (QGSDConvergence.tla)

**What:** Model the Haiku convergence check's state transitions: the oscillation log entry has `resolvedAt` which is initially `null` and becomes an ISO timestamp once Haiku returns YES. The state file (`circuit-breaker-state.json`) is deleted after the log entry is written.

**State variables:**
- `logWritten` — BOOLEAN: log entry for this oscillation has `resolvedAt` set
- `stateDeleted` — BOOLEAN: `.claude/circuit-breaker-state.json` has been removed
- `haikuVerdict` — One of `"YES"`, `"NO"`, `"UNAVAILABLE"`

**Key invariants:**
```tla
(* resolvedAt is write-once: once logWritten=TRUE, it stays TRUE *)
ResolvedAtWriteOnce ==
    [][logWritten = TRUE => logWritten' = TRUE]_vars

(* State delete can only happen after log is written *)
LogBeforeDelete ==
    stateDeleted => logWritten

(* Haiku unavailable: no state mutation *)
HaikuUnavailableNoCorruption ==
    haikuVerdict = "UNAVAILABLE" =>
        UNCHANGED <<logWritten, stateDeleted>>
```

**Actions:**
- `HaikuReturnsYES` — sets `logWritten = TRUE` then `stateDeleted = TRUE` (in that order)
- `HaikuReturnsNO` — leaves both unchanged
- `HaikuUnavailable` — leaves both unchanged (verdict = "UNAVAILABLE")

### Anti-Patterns to Avoid

- **Modeling file content directly:** Do not try to represent actual JS file content in TLA+. Abstract the file sets as labels (e.g., `{A, B}`) — this is sufficient to verify the counting algorithm.
- **Conflating QGSDCircuitBreaker.tla with QGSDOscillation.tla:** The existing `QGSDCircuitBreaker.tla` models the CLI FSM (`MONITORING`/`TRIGGERED`/`DISABLED`). `QGSDOscillation.tla` models the detection algorithm itself. Keep them separate.
- **Using `SF_vars` for `AlgorithmTerminates`:** The algorithm is a bounded finite-state loop. `WF_vars` on the `EvaluateFlag` action is sufficient. `SF_vars` adds unnecessary state space.
- **Symmetry sets in liveness cfgs:** `MCoscillation.cfg` uses label symmetry (`SYMMETRY LabelSymmetry`) for state space reduction. `MCconvergence.cfg` has no set parameters, so no symmetry needed.
- **Forgetting `--no-deadlock` / `CHECK_DEADLOCK FALSE`:** All configs in this repo use `CHECK_DEADLOCK FALSE`. The TLA+ specs are not deadlock-free by design (terminal states are reached by the algorithm completing).

## Common Pitfalls

### Pitfall 1: State Space Explosion from Long Commit Sequences

**What goes wrong:** TLA+ enumerates all possible sequences of labels up to `CommitWindow` length. If `CommitWindow = 10` and `Labels = {A, B, C}`, the state space is `3^10 = 59049` possible sequences — manageable. But with naive encoding (each commit as a separate state variable), TLC may time out.
**Why it happens:** Modeling the commits array as individual named variables (`commit1`, `commit2`, ...) rather than a sequence gives TLC no structure to exploit.
**How to avoid:** Use TLA+ `Seq(Labels)` with a bounded depth: `commits \in UNION {[1..n -> Labels] : n \in 0..CommitWindow}`. This keeps the state space bounded and enumerable.
**Warning signs:** TLC reports "enumerating initial states" for >30 seconds — reduce CommitWindow or Labels cardinality in the MC config.

### Pitfall 2: Incorrect Run-Collapse Definition

**What goes wrong:** The TLA+ run-collapse operator must match the JS implementation exactly. The JS algorithm (line 172-181 of `hooks/qgsd-circuit-breaker.js`) collapses consecutive identical file sets — the key is that the sort order of file names determines equality. A TLA+ model that uses set equality instead of sorted-join equality will give wrong results.
**Why it happens:** TLA+ sets are unordered; JS sorts file names with `.slice().sort().join('\0')`.
**How to avoid:** In the TLA+ abstraction, represent each file set as a single label (A, B, C) rather than a set of file names. The label is an abstract representative of the sorted-join key. This avoids the ordering concern entirely.
**Warning signs:** TLC finds a violation of `OscillationFlaggedCorrectly` in a case where the human expects no oscillation — the label abstraction may be leaking concrete behavior.

### Pitfall 3: `resolvedAt` Write-Once Modeling vs TLC State Transitions

**What goes wrong:** TLA+ actions must be enabled/disabled via preconditions. If `HaikuReturnsYES` has no precondition preventing it from firing twice, TLC may generate states where `resolvedAt` is "set twice" — which is fine in reality (the JS code uses an `if !resolvedAt` guard), but the TLA+ model needs to express this guard explicitly.
**Why it happens:** Without the guard, TLA+ allows `HaikuReturnsYES` to fire in any state.
**How to avoid:** Add precondition `logWritten = FALSE` to `HaikuReturnsYES`. This mirrors the JS guard on line 556-567 of `qgsd-circuit-breaker.js`: `if (verdict.startsWith('YES'))` is only evaluated once per PostToolUse event, and `log[k].resolvedAt = now` only runs when `resolvedAt` is currently null.
**Warning signs:** TLC's state graph shows `logWritten` transitioning from TRUE to TRUE — indicates the action is firing redundantly.

### Pitfall 4: npm test Not Including run-oscillation-tlc.test.cjs

**What goes wrong:** The new test file is not added to `package.json`'s `test` script, so `npm test` passes locally (because it never runs the new tests) but CI/CD would miss them.
**Why it happens:** `run-breaker-tlc.test.cjs` is also missing from the test script (confirmed: not in `package.json` scripts.test). This is an existing pattern gap.
**How to avoid:** Plan 01 (Wave 0 stubs) MUST add `bin/run-oscillation-tlc.test.cjs` to the `test` script in `package.json`. Confirm `run-breaker-tlc.test.cjs` is also added if it's missing.
**Warning signs:** Running `npm test` separately from `node --test bin/run-oscillation-tlc.test.cjs` gives different results.

### Pitfall 5: Liveness Check Requires -workers 1 for Some TLA+ Versions

**What goes wrong:** Older TLC versions have a known bug where multi-worker liveness checking can produce false positives (violations that don't exist). The existing `run-tlc.cjs` uses `-workers 1` for `MCliveness`.
**Why it happens:** TLC liveness bug in TLC v1.8.0 when `workers > 1`.
**How to avoid:** For `MCoscillation.cfg` (which has a liveness property `AlgorithmTerminates`), use `-workers 1` in the runner, matching the `MCliveness` pattern in `run-tlc.cjs`. For `MCconvergence.cfg` (safety only), use `'auto'`.
**Warning signs:** TLC reports a spurious counterexample for `AlgorithmTerminates` when using `auto` workers.

## Open Questions

1. **TLA+ `netChange` as abstract integer vs. bounded integer**
   - What we know: TLA+ integers are unbounded; TLC requires a bound for model checking.
   - What's unclear: Should `netChange` be bounded to `-CommitWindow..CommitWindow`? In practice the net change per diff pair is small, but TLC needs a finite range.
   - Recommendation: Bound `netChange` in the CONSTANTS to `-10..10` in the MC config. This is sufficient for any realistic 10-commit window.

2. **Whether `MCoscillation.cfg` needs SYMMETRY**
   - What we know: `MCsafety.cfg` uses `SYMMETRY AgentSymmetry` for state space reduction. The oscillation spec uses a label set `{A, B}` or `{A, B, C}`.
   - What's unclear: Whether `SYMMETRY Permutations(Labels)` is valid here given the sequence structure (sequences are ordered, not sets).
   - Recommendation: Do NOT use SYMMETRY on labels in MCoscillation.cfg. Sequences are ordered structures; applying permutation symmetry to the label set can give incorrect results when the labels appear in a specific order in the commits sequence. Keep the MC config simple.

3. **Whether run-breaker-tlc.test.cjs needs to be added to package.json test**
   - What we know: It exists at `bin/run-breaker-tlc.test.cjs` but is NOT in the `npm test` script (confirmed by inspecting `package.json` scripts.test).
   - What's unclear: Whether this omission was intentional (the test was skipped for some reason) or accidental.
   - Recommendation: Include adding `bin/run-breaker-tlc.test.cjs` as a side-fix in Plan 01 alongside `bin/run-oscillation-tlc.test.cjs`. The tests pass without Java and are safe to add.

## Detailed Algorithm Analysis (GAP-1)

The run-collapse algorithm in `hooks/qgsd-circuit-breaker.js` has two stages:

**Stage 1: Run-collapse** (lines 170-181)
```javascript
// Collapse consecutive identical file sets into runs
for (let i = 0; i < fileSets.length; i++) {
  const key = files.slice().sort().join('\0');
  if (runs.length === 0 || runs[runs.length - 1].key !== key) {
    runs.push({ key, files, indices: [i] });
  } else {
    runs[runs.length - 1].indices.push(i);
  }
}
```
This produces a `runs` array where consecutive commits touching the same set of files are merged into one run-group. Example: `[A, A, B, B, A, A, B]` → `[A, B, A, B]` (4 runs).

**Stage 2: Group count check** (lines 183-213)
```javascript
// Count run-group occurrences per key
for (const [key, keyRunList] of keyRuns) {
  if (keyRunList.length >= depth) {
    // Second-pass: check net change (hasReversionInHashes)
    if (!isRealOscillation) return { detected: false, fileSet: [] };
```

A file set is flagged iff it appears in ≥ `depth` (default 3) distinct run-groups AND `hasReversionInHashes` returns `true` (net change ≤ 0).

**TLA+ modeling decision:** The model abstracts file sets as labels `A`, `B` from a finite alphabet. The key predicate is:
- `FlagCount(label, commits)` = number of distinct run-groups in the run-collapse of `commits` that have label `label`
- `OscillationFlaggedCorrectly` = `flagged <=> (\E l : FlagCount(l, commits) >= Depth /\ netChange <= 0)`

This abstraction is sound for verifying the counting correctness property.

**TLA+ modeling decision for `hasReversionInHashes`:** This function computes the sum of `(additions - deletions)` across consecutive commit pairs. In TLA+, model this as an integer variable `netChange` that is nondeterministically chosen from `-CommitWindow..CommitWindow` to represent any possible diff outcome. The invariant then checks that `flagged` is only TRUE when `netChange <= 0`. This verifies the structural correctness of the flag condition without modeling actual file diffs.
