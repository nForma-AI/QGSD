## Summary

Phase v0.20-07 adds a UPPAAL timed automaton model that answers "when do quorum slot races fire relative to each other?" The model uses empirical `runtime_ms` bounds from `check-results.ndjson` as clock guards (not hardcoded constants), and surfaces two annotated properties: minimum inter-slot gap preventing race conditions, and maximum timeout threshold for consensus before the planning gate deadline. The runner `bin/run-uppaal.cjs` wraps `verifyta` CLI and writes a v2.1 NDJSON result. A new STEPS entry `uppaal:quorum-races` is wired into `run-formal-verify.cjs`.

## Domain Analysis

### UPPAAL Timed Automata Overview

UPPAAL is a model checker for real-time systems using timed automata (TA). Key concepts:

- **Clocks**: Real-valued variables reset to 0 and advancing with time. Used in clock guards (`x >= 5`, `x < 10`) and invariants (`x <= 15`).
- **Locations**: States in an automaton (initial, urgent, committed, normal).
- **Edges**: Transitions with guards, synchronization labels, and assignments.
- **Channels**: Synchronization between parallel automata (handshake `a!` / `a?`, broadcast `a!!` / `a?`).
- **Queries**: Properties verified by verifyta (reachability, safety, liveness via TCTL).

**XML model format** (`quorum-races.xml`): UPPAAL uses a proprietary XML schema with `<nta>`, `<template>` (automaton), `<location>`, `<transition>`, and `<system>` sections. The file is human-readable XML but tool-specific — verifyta requires it.

**verifyta CLI**: The command-line model checker shipped with UPPAAL. Typical invocation:
```
verifyta .formal/uppaal/quorum-races.xml .formal/uppaal/quorum-races.q
```
Where `.q` contains the property queries. verifyta exits 0 if all properties satisfied, non-zero otherwise.

**Property query file** (`.q` format): One TCTL query per line, e.g.:
```
A[] not deadlock
E<> Race.fire
```

### Quorum Race Structure

The QGSD quorum protocol concurrently dispatches slot workers and waits for responses. The race scenario:

1. N slots start at time T=0
2. Each slot responds at its own time (empirical `runtime_ms`)
3. A timeout fires at T=TIMEOUT_MS — any non-responded slot is treated as UNAVAILABLE
4. Race condition: if two slot responses arrive at the same logical tick, quorum state may be inconsistent

Key parameters from `check-results.ndjson` empirical data (current values):
- `prism:quorum` runtime_ms: 1-365ms observed (mostly 1-2ms for fast PRISM runs; 365ms is an outlier)
- `ci:conformance-traces` runtime_ms: 90ms
- `ci:trace-schema-drift` runtime_ms: 9ms

For quorum slot workers specifically, the `runtime_ms` represents MCP tool call latency, not PRISM latency. The relevant range is ~100-2000ms for LLM API calls (from MCP logs context in STATE.md).

**Critical measurement points (UPPAAL-03):**
- (a) Minimum inter-slot response gap preventing race: the smallest Δt between two slot responses such that no race fires
- (b) Maximum timeout value for consensus before planning gate deadline

### UPPAAL vs Existing Formalisms

| Aspect | TLA+ | PRISM | UPPAAL |
|--------|------|-------|--------|
| Focus | State/logic | Probabilistic | Timed/real-time |
| Time model | Discrete steps | Discrete steps | Dense real-time clocks |
| Queries | Invariants/liveness | P=?, R=? | Reachability, safety |
| Best for | Protocol correctness | Probabilistic outcomes | Timing constraints, races |
| Tool | TLC/TLAPS | PRISM tool | verifyta |

UPPAAL is the right tool here because we need to reason about *when* concurrent events fire relative to each other — which requires dense-time clocks, not discrete TLA+ steps.

### Empirical Timing Bounds Approach

`run-uppaal.cjs` needs to read `check-results.ndjson` and extract `runtime_ms` bounds for UPPAAL clock guards. The approach:

```javascript
// Read timing bounds from empirical data
const { parseNDJSON } = require('./verify-formal-results.cjs');
const records = parseNDJSON(path.join(process.cwd(), 'formal', 'check-results.ndjson'));

// Extract slot-response timing bounds
const slotRecords = records.filter(r => r.check_id && r.check_id.startsWith('tla:'));
const runtimeValues = slotRecords.map(r => r.runtime_ms).filter(v => v > 0);
const minSlotMs = runtimeValues.length > 0 ? Math.min(...runtimeValues) : 50;   // default fallback
const maxSlotMs = runtimeValues.length > 0 ? Math.max(...runtimeValues) : 500;  // default fallback
const timeoutMs = maxSlotMs * 3;  // planning gate deadline = 3× max slot time
```

These bounds are then passed to verifyta as constants (UPPAAL supports `-C "name=value"` flags or embedded in model).

### UPPAAL XML Model Design

The `quorum-races.xml` model captures:

**Template 1: SlotWorker** — represents a single quorum slot
- Clock `x`: time since dispatch
- Location `Waiting`: invariant `x <= MAX_SLOT_MS`
- Location `Responded`: committed, no time-passing
- Transition `Waiting → Responded`: guard `x >= MIN_SLOT_MS && x <= MAX_SLOT_MS`, sync `response!`

**Template 2: QuorumOrchestrator** — waits for responses within timeout
- Clock `t`: time since first dispatch
- Counter `votes_received`: int
- Location `Collecting`: invariant `t <= TIMEOUT_MS`
- Location `Decided`: absorbing (consensus reached)
- Location `TimedOut`: absorbing (timeout fired)
- Transition `Collecting → Decided`: guard `votes_received >= MIN_QUORUM`
- Transition `Collecting → TimedOut`: guard `t >= TIMEOUT_MS`

**Template 3: RaceDetector** — committed location detecting simultaneous responses
- Urgent location `Race` that becomes reachable if two `response!` synchronizations happen at same clock tick

**Properties (quorum-races.q):**
```
// No race conditions when inter-slot gap >= MIN_GAP_MS
A[] not RaceDetector.fire

// Quorum can always reach consensus before timeout
E<> QuorumOrchestrator.Decided

// Minimum inter-slot gap annotation (UPPAAL-03a)
A[] (gap >= MIN_GAP_MS) imply not RaceDetector.fire

// Maximum timeout annotation (UPPAAL-03b)
A[] (TIMEOUT_MS <= MAX_SAFE_TIMEOUT) imply E<> QuorumOrchestrator.Decided
```

### nonCritical Flag Pattern

From v0.20-06 implementation: `nonCritical: true` means failure shows in summary but doesn't affect exit code.

**Should `uppaal:quorum-races` be nonCritical?** YES — same rationale as ci:triage-bundle:
- verifyta may not be installed on all developer machines
- UPPAAL model results are informational (timing analysis), not blocking formal gates
- TLA+/Alloy/PRISM already handle the safety/liveness correctness gates

### runner exit code convention

Pattern from `run-prism.cjs`:
- Binary not found → write `result: 'fail'`, exit 1
- Model not found → write `result: 'fail'`, exit 1
- Checker passes → write `result: 'pass'`, exit 0
- Checker fails (properties violated) → write `result: 'fail'`, exit 0 (not 1, since runner itself succeeded)

**For UPPAAL**: verifyta not found → write `result: 'inconclusive'` (not fail — it's a missing tool, not a model failure), `triage_tags: ['no-verifyta']`, exit 0 (nonCritical runner should not exit 1 when tool absent).

## Anti-Patterns (from codebase patterns)

1. **Do NOT hardcode timing constants** — read from `check-results.ndjson` with fallbacks
2. **Do NOT exit non-zero when verifyta missing** — use `inconclusive` result, exit 0 (nonCritical step)
3. **Do NOT use `formalism: 'uppaal'` before adding it to VALID_FORMALISMS** — will throw
4. **Do NOT position uppaal:quorum-races after ci:triage-bundle** — triage must capture UPPAAL results
5. **Do NOT fail the NDJSON write if UPPAAL model validation fails** — decouple model creation from runner
6. **Do NOT use inline constants in quorum-races.xml** — use UPPAAL parameters (`const int MIN_SLOT_MS = ...`) set by runner via -C flags or computed from NDJSON at build time
7. **Do NOT require verifyta in PATH for npm test** — all unit tests must run without UPPAAL installed
