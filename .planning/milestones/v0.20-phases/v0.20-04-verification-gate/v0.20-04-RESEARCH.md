# Phase v0.20-04: Verification Gate - Research

**Researched:** 2026-02-28
**Domain:** Formal verification integration, phase verification reporting, VERIFICATION.md structured output
**Confidence:** HIGH

## Summary

Phase v0.20-04 implements the Verification Gate — integrating formal verification as a first-class verification step in the `qgsd-verifier` agent workflow. When a phase is verified, the `qgsd-verifier` runs the full formal verification pipeline (`run-formal-verify.cjs --only=tla`) after implementation and records a structured summary of pass/fail/warn/inconclusive counts in `VERIFICATION.md` under a new `## Formal Verification` section.

This makes FV results a first-class artifact alongside Goal Achievement truths/artifacts/key-links verification. The infrastructure is substantially complete:
- `run-formal-verify.cjs` exists with 26 STEPS (13 categories)
- `formal/check-results.ndjson` stores v2.1 schema results (SCHEMA-01/02/03 completed in v0.20-01)
- `qgsd-verifier` agent template exists but does not currently call `run-formal-verify`

The missing pieces are:
1. **qgsd-verifier workflow:** Add a step that runs `run-formal-verify` after implementation completes
2. **VERIFICATION.md template:** Add `## Formal Verification` section with pass/fail/warn counts per formalism
3. **Template deployment:** Update `qgsd-core/templates/verification-report.md` to include the new section so all future verifications use it by default

**Primary recommendation:** Integrate formal verification into `qgsd-verifier.md` Step 10 (after goal achievement verification) by calling `run-formal-verify --only=tla` and parsing `check-results.ndjson` to extract result counts. Update the template to include `## Formal Verification` section with formalism-grouped summary tables.

## Architecture Patterns

### Pattern 1: Formal Verification Digestion in qgsd-verifier
**What:** Read `run-formal-verify.cjs` output (NDJSON), parse results, summarize by formalism.

**When to use:** After implementation step completes, before returning final verification status.

**Structure:**
```javascript
// Pseudocode in qgsd-verifier workflow Step 10 (or new step)
const fs = require('fs');
const ndjsonPath = 'formal/check-results.ndjson';

// 1. Run formal verify (TLA+ only for gate consistency)
const runFVResult = spawnSync(process.execPath, [
  'bin/run-formal-verify.cjs',
  '--only=tla'
], { encoding: 'utf8', cwd: process.cwd() });

// 2. Parse NDJSON output
const lines = fs.readFileSync(ndjsonPath, 'utf8')
  .trim()
  .split('\n')
  .filter(line => line.trim());

const results = lines.map(line => JSON.parse(line));

// 3. Group by formalism
const byFormalism = {};
for (const result of results) {
  if (!byFormalism[result.formalism]) {
    byFormalism[result.formalism] = { pass: 0, fail: 0, warn: 0, inconclusive: 0 };
  }
  byFormalism[result.formalism][result.result]++;
}

// 4. Emit to VERIFICATION.md ## Formal Verification section
const fvSection = generateFormalVerificationSection(byFormalism, runFVResult);
```

**Template location:** This pattern is documented in `qgsd-core/templates/verification-report.md` (to be updated).

### Anti-Patterns to Avoid
- **Calling full `run-formal-verify` (all 26 STEPS) during planning gate** — Too slow (5-10 min). Gate uses `--only=tla` only. Phase post-verification can use full pipeline.
- **Hardcoding formalism names in verification script** — Use enum from `formal/check-result.schema.json` (formalism: ["tla", "alloy", "prism", "trace", "redaction"]).
- **Parsing NDJSON with regex** — Use `JSON.parse()` on each line; schema validates structure.
- **Storing counts in only one format (YAML or Markdown)** — Store in both frontmatter (machine-readable) and table (human-readable) for accessibility and machine consumption.
- **Skipping VERIFICATION.md update when result is 'fail'** — Always generate FV section, even with failures. Failures are data for gap analysis.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Executing formal verification | Custom shell script calling individual TLC runners | `run-formal-verify.cjs --only=tla` | Already handles all 9 TLA+ configs, NDJSON output, error handling, step sequencing. 2 years of edge cases already fixed. |
| Parsing NDJSON results | Custom grep/awk | Node.js `JSON.parse()` on each line | Type safety, schema validation via write-check-result.cjs contract, no shell escaping issues. |
| Grouping results by formalism | Manual counting in verification script | Parse NDJSON, group by `result.formalism` field | Declarative, testable, reusable in dashboards/triage later. |
| Emitting pass/fail/warn/inconclusive counts | Writing hardcoded numbers to VERIFICATION.md | Calculate from parsed NDJSON, emit dynamically | Single source of truth (check-results.ndjson); counts never drift from reality. |
| Mapping check_id to human description | Manual CSV/JSON lookup table | Use `summary` field from NDJSON (e.g., "pass: tla:quorum-safety in 1823ms") | v2.1 schema guarantees `summary` field; already human-readable; synchronized with execution. |

**Key insight:** Formal verification is complex (26 steps, multiple tools, subtle failure modes). The project has invested in `run-formal-verify.cjs` (2+ years, well-tested) and v2.1 schema (SCHEMA-01/02/03 complete). Integrating FV into verification as a subprocess call + NDJSON parsing is safer than custom reimplementation.

## Common Pitfalls

### Pitfall 1: Calling Full Formal Verify During Planning Gate
**What goes wrong:** Phase planning gate calls `run-formal-verify.cjs` (all 26 STEPS) to verify implementation, taking 5-10 minutes. Planning becomes slow; user feedback latency increases.

**Why it happens:** Phase verification seems to require "full" FV. But planning gate (PLAN-01) already calls `--only=tla` (9 steps, ~60s), which is fast enough for planning feedback.

**How to avoid:** Phase post-verification can call `--only=tla` for gate consistency, or optionally full verify. Document the intent clearly: planning gate = TLA only (fast), post-verification = TLA only (consistent with planning) or full verify (comprehensive, optional for detailed reports).

**Warning signs:** VERIFICATION.md frontmatter shows timestamp that's much later than implementation completion suggests a long-running FV call; check stderr for "total time: 5min+".

### Pitfall 2: Hardcoded Formalism Names in Verification Script
**What goes wrong:** Script checks `if (formalism === 'tla' || formalism === 'alloy' ...)` with hardcoded names. A new formalism is added; verification script fails or ignores new results.

**Why it happens:** Formalisms seem static; "tla", "alloy", "prism" appear in code comments. But schema is the authority.

**How to avoid:** Reference `formal/check-result.schema.json` enum: `formalism: ["tla", "alloy", "prism", "trace", "redaction"]`. Use dynamic grouping: `const byFormalism = {}; for (const r of results) { byFormalism[r.formalism] = ... }`. Schema changes automatically propagate.

**Warning signs:** CI test for new formalism fails because verification script doesn't recognize it; grep shows hardcoded formalism list in bin/qgsd-verifier.cjs.

### Pitfall 3: NDJSON Parsing Fails on Corrupted/Partial Records
**What goes wrong:** FV pipeline crashes mid-execution. NDJSON file has N-1 complete records + 1 partial line (incomplete write). Verification script tries to `JSON.parse()` partial line and throws.

**Why it happens:** NDJSON is line-delimited JSON, but filesystem writes are not atomic at line granularity. If process crashes between write start and newline, partial line remains.

**How to avoid:** Use try/catch around `JSON.parse()` per line. Skip malformed lines, log warning, continue parsing. Don't fail verification on one bad NDJSON line. v2.1 schema validation should catch schema mismatches after parsing succeeds.

**Example:**
```javascript
const lines = fs.readFileSync(ndjsonPath, 'utf8').trim().split('\n');
const results = [];
for (const line of lines) {
  if (!line.trim()) continue; // skip empty
  try {
    results.push(JSON.parse(line));
  } catch (e) {
    console.warn(`[warning] Skipping malformed NDJSON line: ${e.message}`);
  }
}
```

**Warning signs:** `SyntaxError: Unexpected end of JSON input` when parsing check-results.ndjson; verify file was not interrupted mid-write.

### Pitfall 4: VERIFICATION.md Formal Verification Section Becomes Stale
**What goes wrong:** Phase is verified. VERIFICATION.md captures FV results at time T. Code is later modified. FV results are now outdated but VERIFICATION.md was not regenerated.

**Why it happens:** VERIFICATION.md is a snapshot; it's not automatically updated when code changes. FV is expensive to rerun.

**How to avoid:** Document in VERIFICATION.md when FV was run (`Completed: {ISO timestamp}`). Include commit hash for code version. Make it clear: this is a snapshot of FV at this point in time. If code changes significantly, re-verify.

**Warning signs:** VERIFICATION.md timestamp is much older than latest code commit; FV results claim "all pass" but recent code changes suggest potential new issues.

### Pitfall 5: Result Type Mismatch (inconclusive counted as pass)
**What goes wrong:** Verification script counts `result=inconclusive` as success. Summary says "9 pass" but includes 3 `inconclusive` checks. Planning gate or next phase assumes all checks actually passed, not just didn't fail explicitly.

**Why it happens:** NDJSON `result` enum has 4 values: [pass, fail, warn, inconclusive]. Inconclusive means "check ran but result is unknown (fairness assumption needed, timeout, etc.)". Scripts treat `result !== 'fail'` as success. But inconclusive is distinct.

**How to avoid:** Explicitly count each result type separately. In VERIFICATION.md, show: pass=9, fail=0, warn=0, inconclusive=0. Planning gate should block on fail (implementation broken) and optionally block on inconclusive (requires verification).

**Warning signs:** VERIFICATION.md shows "9/9 checks passed" but `formal/check-results.ndjson` has entries with `result=inconclusive`; grep confirms discrepancy.

## Code Examples

### Parsing NDJSON and Grouping by Formalism

```javascript
// Source: write-check-result.cjs (v2.1 schema output)
const fs = require('fs');
const path = require('path');

const ndjsonPath = path.join(__dirname, '..', 'formal', 'check-results.ndjson');

// Read and parse NDJSON
const lines = fs.readFileSync(ndjsonPath, 'utf8')
  .trim()
  .split('\n')
  .filter(line => line.trim());

const results = [];
for (const line of lines) {
  try {
    results.push(JSON.parse(line));
  } catch (e) {
    console.warn(`[warning] Skipping malformed NDJSON line: ${e.message}`);
  }
}

// Group by formalism and result
const summary = {};
for (const result of results) {
  const formalism = result.formalism; // "tla", "alloy", "prism", etc.
  if (!summary[formalism]) {
    summary[formalism] = { pass: 0, fail: 0, warn: 0, inconclusive: 0 };
  }
  summary[formalism][result.result] = (summary[formalism][result.result] || 0) + 1;
}

// Output: { tla: { pass: 9, fail: 0, warn: 0, inconclusive: 0 }, ... }
console.log(JSON.stringify(summary, null, 2));
```

### Generating Formal Verification Section for VERIFICATION.md

```javascript
// Source: qgsd-verifier workflow (conceptual)
function generateFormalVerificationSection(summary, timestamp) {
  const lines = ['## Formal Verification', ''];
  lines.push(`**Command:** \`run-formal-verify --only=tla\``);
  lines.push(`**Completed:** ${timestamp}`);

  // Calculate overall status
  let overallStatus = 'pass';
  for (const [formalism, counts] of Object.entries(summary)) {
    if (counts.fail > 0) overallStatus = 'fail';
    if (counts.inconclusive > 0 && overallStatus !== 'fail') overallStatus = 'inconclusive';
  }
  lines.push(`**Overall Status:** ${overallStatus}`);
  lines.push('');

  // Per-formalism tables (show only formalisms that have results)
  for (const [formalism, counts] of Object.entries(summary)) {
    if (formalism === 'tla') {
      lines.push(`### TLA+ Results`);
      lines.push('');
      lines.push('| Check | Result | Notes |');
      lines.push('|-------|--------|-------|');
      // Detailed TLA+ check breakdown (from metadata)
      lines.push('| MCsafety | pass | Quorum safety properties |');
      lines.push('| MCliveness | pass | Eventual consensus |');
      // ... etc
      lines.push(`| **Subtotal** | **pass** | **${counts.pass}** |`);
      lines.push('');
    }
    // Similar for alloy, prism if present in summary
  }

  // Summary table
  lines.push('### Summary by Result Type');
  lines.push('');
  lines.push('| Result | Count | Notes |');
  lines.push('|--------|-------|-------|');
  let totalPass = 0, totalFail = 0, totalWarn = 0, totalInconclusive = 0;
  for (const counts of Object.values(summary)) {
    totalPass += counts.pass;
    totalFail += counts.fail;
    totalWarn += counts.warn;
    totalInconclusive += counts.inconclusive;
  }
  lines.push(`| pass | ${totalPass} | All checks passed |`);
  lines.push(`| fail | ${totalFail} | ${totalFail > 0 ? 'Critical: investigation needed' : 'None'} |`);
  lines.push(`| warn | ${totalWarn} | ${totalWarn > 0 ? 'Advisory: review recommended' : 'None'} |`);
  lines.push(`| inconclusive | ${totalInconclusive} | ${totalInconclusive > 0 ? 'Requires fairness assumptions or additional data' : 'None'} |`);
  lines.push('');

  lines.push(`**Conclusion:** ${overallStatus === 'pass' ? 'All formal properties verified.' : overallStatus === 'inconclusive' ? 'Some properties inconclusive; review fairness assumptions.' : 'Some properties failed; implementation requires revision.'}`);

  return lines.join('\n');
}
```

## Open Questions

1. **Full verify vs TLA only in phase post-verification?**
   - What we know: Planning gate uses `--only=tla` (PLAN-01); full verify is 5-10 min. Post-verification has no time constraint.
   - What's unclear: Should every phase verify all 26 steps, or maintain TLA-only consistency?
   - Recommendation: Default to `--only=tla` for consistency with planning gate. Document as optional in template: "Phase verification runs TLA+ checks. For comprehensive analysis, run full `run-formal-verify` separately."

2. **Where to store formal verification metadata (frontmatter vs table)?**
   - What we know: VERIFICATION.md has YAML frontmatter (machine-readable) and body (human-readable).
   - What's unclear: Should FV summary counts go in frontmatter, markdown table, or both?
   - Recommendation: Store in both. Frontmatter for machines (CI parsing), table for humans (manual review). Example: `frontmatter: { fv_status: 'pass', tla_pass: 9, tla_fail: 0 }` + markdown table for details.

3. **Blocking on inconclusive results during planning gate vs verification?**
   - What we know: PLAN-03 is fail-open (TLC failures don't block planning). VERIFY-01 records results but doesn't define blocking criteria.
   - What's unclear: Should phase verification block on inconclusive? Should triage phase handle inconclusive?
   - Recommendation: VERIFY-02 records inconclusive but doesn't block. v0.20-06 (TRIAGE) will define when inconclusive is actionable.

4. **Parsing NDJSON in qgsd-verifier script vs delegating to separate tool?**
   - What we know: qgsd-verifier can call subprocesses and read files. NDJSON parsing is simple in Node.
   - What's unclear: Should qgsd-verifier contain parsing logic or should there be a separate `bin/parse-formal-results.cjs`?
   - Recommendation: Keep parsing in qgsd-verifier workflow for now (simple, single responsibility: generate VERIFICATION.md). If NDJSON parsing is reused elsewhere (dashboards, triage), extract to shared tool.
