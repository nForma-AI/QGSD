# Phase v0.20-02: Liveness Fairness Lint - Research

**Researched:** 2026-02-28
**Domain:** TLA+ formal verification, liveness properties, fairness declarations
**Confidence:** HIGH

## Summary

Phase v0.20-02 implements a CI lint step that automatically detects liveness properties in TLA+ model configurations (`.cfg` files) that lack matching fairness declarations in `.formal/spec/{spec}/invariants.md`. When detected, the runner emits `result=inconclusive` instead of `pass`, preventing false-positive liveness proofs that assume fairness without explicitly documenting why.

The infrastructure is **already 80% complete**. The detection logic (`detectLivenessProperties`) exists in `bin/run-tlc.cjs` and is **already called** by `run-tlc.cjs`, `run-oscillation-tlc.cjs`, `run-protocol-tlc.cjs`, and `run-account-manager-tlc.cjs`. These runners **already emit inconclusive results** for liveness properties without fairness declarations (lines 320–342 in run-tlc.cjs). The missing piece is a dedicated `ci:liveness-fairness-lint` STEPS entry in `run-formal-verify.cjs` that runs as an explicit, visible CI step.

**Primary recommendation:** Add a new `ci:liveness-fairness-lint` step to STEPS in `bin/run-formal-verify.cjs` that scans all TLA+ configs for liveness properties without fairness declarations and reports them centrally. This consolidates the per-runner liveness checks into a single CI-visible step that satisfies LIVE-02 and enables LIVE-01 verification during planning-gate checks.

## Architecture Patterns

### Liveness Property Detection Architecture (Existing)

The detection mechanism in `run-tlc.cjs` already implements the core algorithm:

1. **Config parsing:** Extract PROPERTY/PROPERTIES lines from .cfg files (handles multi-name declarations)
   - Regex: `/^PROPERT(?:Y|IES)\s+(.*)/` to extract property names
   - Handles single and multiple properties per line (e.g., `PROPERTIES A B C`)

2. **Surface mapping:** Map config name to spec directory via SURFACE_MAP
   - Example: `'MCliveness'` → `'quorum'` → `.formal/spec/quorum/invariants.md`
   - Unknown configs return [] (fail-open — no false positives on unknown specs)

3. **Fairness verification:** Check invariants.md for matching `## PropertyName` header
   - Example: liveness property `EventualConsensus` requires `## EventualConsensus` section in invariants.md
   - Missing header triggers inconclusive result with triage_tag='needs-fairness'

4. **Result classification:**
   - No PROPERTY lines → [] (safety config, not liveness)
   - Unknown surface → [] (can't validate; don't block)
   - Missing invariants.md → return all property names (file missing entirely)
   - Matching header found → [] (fairness declared ✓)
   - No matching header → return property name (fairness missing → inconclusive)

**Current usage:**
- `run-tlc.cjs`: MCsafety, MCliveness, MCMCPEnv configs
- `run-oscillation-tlc.cjs`: MCoscillation, MCconvergence configs (imports detectLivenessProperties)
- `run-protocol-tlc.cjs`: MCdeliberation, MCprefilter configs (imports detectLivenessProperties)
- `run-account-manager-tlc.cjs`: MCaccount-manager config (imports detectLivenessProperties)
- `run-breaker-tlc.cjs`: MCbreaker config (imports detectLivenessProperties)

All runners emit `result=inconclusive` with triage_tags=['needs-fairness'] and metadata reason when liveness properties lack fairness declarations.

### Pattern: Fairness Declaration Structure

Each liveness spec MUST document why weak fairness is appropriate:

```markdown
# Liveness Fairness Declarations: {spec-name}

**Spec source:** `.formal/tla/{SpecName}.tla`
**Config:** `.formal/tla/{ConfigName}.cfg`

## {PropertyName}

**Property:** `{PropertyName} == <>(...)` or similar eventually-formula
**Config line:** `PROPERTY {PropertyName}` ({ConfigName}.cfg)
**Fairness assumption:** WF_vars on {action list}
**Realism rationale:** [Multi-paragraph explanation of why weak fairness is justified in deployed system]
```

**Current exemplars:**
- `.formal/spec/quorum/invariants.md` — EventualConsensus with WF on 4 quorum actions
- `.formal/spec/breaker/invariants.md` — MonitoringReachable with WF on state transitions
- `.formal/spec/oscillation/invariants.md` — AlgorithmTerminates with WF on algorithm phases

**Anti-exemplar:**
- `.formal/spec/tui-nav/invariants.md` — EscapeProgress property (temporal safety, not liveness) documented but deliberately NOT checked by TLC. Two abandoned liveness properties (EventuallyExits, MainMenuReachable) documented as "user-cooperation properties" requiring fairness on user actions — too strong assumption for abstract model.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Scanning .cfg files for PROPERTY declarations | Custom regex/parser | `detectLivenessProperties()` from run-tlc.cjs | Already handles edge cases: multi-name PROPERTIES lines, unknown configs, missing invariants.md, comment handling |
| Mapping config names to spec directories | Hard-coded lists | SURFACE_MAP in run-tlc.cjs | Centralized, reused by all runners, prevents duplicate config-to-spec mappings |
| Validating fairness declarations against invariants.md | String search for "fairness" or "WF_" in spec | Header search `## PropertyName` in invariants.md | Headers are explicit contract; searching for "fairness" keywords has false positives (comments, rationale text mention fairness) and false negatives (undocumented fairness in TLA+ source) |
| Emitting inconclusive results manually | Custom error handling in each runner | `writeCheckResult()` with result='inconclusive' | v2.1 schema validation ensures required fields, consistent triage_tags, proper NDJSON format; decentralizes validation |

**Key insight:** Liveness verification is complex (fairness assumptions are subtle; different specs require different fairness levels). The project has already invested in pattern-matching and documentation standards (invariants.md headers). Reusing `detectLivenessProperties` and the header-match pattern avoids reimplementing liveness-specific logic and preserves the declarative (documentation-driven) approach.

## Common Pitfalls

### Pitfall 1: Fairness Assumption Mismatch
**What goes wrong:** A liveness property is declared in .cfg (PROPERTY line), TLC passes, but no fairness declaration exists in invariants.md. The proof is unsound — weak fairness assumption is implicit in TLC's liveness checker but not justified in documentation.

**Why it happens:** Developer adds liveness property to .cfg without realizing TLC's liveness mode **requires explicit fairness assumptions**. TLC silently applies weak fairness by default in liveness mode, so the check passes even though no fairness rationale is documented.

**How to avoid:** Before adding PROPERTY line to any .cfg, add corresponding ## PropertyName section to invariants.md with fairness assumption and realism rationale. CI step will catch undeclared properties.

**Warning signs:**
- `run-formal-verify.cjs` output shows `result=inconclusive` for liveness check with `triage_tags=['needs-fairness']`
- NDJSON entry has `reason: 'Fairness declaration missing for: PropertyName'` in metadata
- Planning gate warnings include "fairness declaration missing" before proceeding

### Pitfall 2: Over-Strong Fairness Assumption
**What goes wrong:** Fairness rationale documents fairness assumption that is too strong for the system context (e.g., "user will always press ESC" — not true in abstract model). TLC proves liveness under that assumption, but the proof is vacuous in practice.

**Why it happens:** Developer over-specifies fairness to make a property hold. Abstract models allow behaviors (like infinite user navigation) that defeat weak fairness on some actions.

**How to avoid:** Fairness rationale must justify WF/SF assumption **in deployed system context**. Rationale should cite system guarantees (event loop, timeouts, sequential execution, external API guarantees) that ensure fairness. If rationale requires "user cooperates," fairness assumption is too strong — either weaken property or accept property as unverifiable.

**Warning signs:**
- Fairness rationale begins with "Assume user will..." or "If developer doesn't..." — too strong
- TLC liveness check `pass`, but lasso counterexample shows infinite behavior the system should prevent (e.g., infinite user bouncing between screens)
- Spec comments like "NOT CHECKED" or "User-cooperation property" alongside PROPERTY line

**Real example:** TUINavigation.tla has EventuallyExits and MainMenuReachable properties explicitly NOT checked by MCTUINavigation.cfg because they require fairness on user Select action. Documented as "user-cooperation properties" — too strong for abstract model. Only EscapeProgress (temporal safety, not liveness) is checked.

### Pitfall 3: Config-to-Spec Mapping Drift
**What goes wrong:** New .cfg file added but SURFACE_MAP in run-tlc.cjs not updated. Lint step cannot validate the new config's liveness properties — silently returns [] and config is never linted.

**Why it happens:** Multiple runner scripts have local SURFACE_MAP copies; developer adds new spec but forgets to update all maps. Detection is silent (fail-open), so no warning.

**How to avoid:** Keep SURFACE_MAP in run-tlc.cjs as single source of truth. All runner scripts import detectLivenessProperties from run-tlc.cjs, which uses that map. Add new mapping before adding new config. Central ci:liveness-fairness-lint step will catch unmapped configs during formal verify run (visible failure in run summary).

**Warning signs:**
- New TLA+ spec added but `run-formal-verify.cjs` STEPS updated without corresponding SURFACE_MAP entry
- Lint step shows "Unknown surface" for new config name
- SURFACE_MAP has duplicates or per-runner variations

### Pitfall 4: Confusing Liveness vs Safety vs Temporal Properties
**What goes wrong:** Developer confuses property types:
- **Liveness property** (`<>P` — "eventually P"): Requires fairness; if property fails, TLC produces lasso counterexample (infinite loop without reaching P)
- **Safety property** (`[]P` — "always P"): Requires NO fairness; failure is finite trace violating P
- **Temporal safety** (`[]` on action effects, e.g., `[][A => B']_vars`): Action-based invariant; no fairness needed

Added as PROPERTY in .cfg, but property type mismatches fairness. TLC result is misleading.

**Why it happens:** Liveness syntax is subtle. `<>(state = DONE)` looks like it should always terminate, but weak fairness only guarantees termination if a progress action remains continuously enabled. Developer misses fairness rationale requirement.

**How to avoid:** Before writing PROPERTY line, ask: "What kind of property is this?"
- Liveness (eventually P)? → Needs fairness + WF_vars justification
- Safety (always P)? → No fairness; add to INVARIANT
- Temporal safety (action preserves P)? → Add to INVARIANT or PROPERTY with `[]` operator (no fairness needed)

Only PROPERTY lines with `<>` or `~>` (leads-to) operators need fairness.

**Warning signs:**
- Property is safety (INVARIANT-like) but added as PROPERTY
- No `<>` or `~>` in property definition yet fairness declared
- Lint step reports "Fairness declaration present but property is not liveness-type" (future enhancement)

## Code Examples

Verified patterns from project sources:

### Example 1: Detecting Liveness Properties (run-tlc.cjs lines 57–95)
```javascript
// Source: bin/run-tlc.cjs
function detectLivenessProperties(configName, cfgPath, specDir) {
  const defaultSpecDir = path.join(__dirname, '..', 'formal', 'spec');
  const resolvedSpecDir = specDir || defaultSpecDir;

  let cfgContent;
  try {
    cfgContent = fs.readFileSync(cfgPath, 'utf8');
  } catch (e) {
    return []; // Can't read config — not a detection failure
  }

  // Extract all property names from PROPERTY/PROPERTIES lines (handles multi-name lines)
  const propertyNames = [];
  for (const line of cfgContent.split('\n')) {
    const m = line.trim().match(/^PROPERT(?:Y|IES)\s+(.*)/);
    if (m) {
      propertyNames.push(...m[1].trim().split(/\s+/).filter(n => /^\w+$/.test(n)));
    }
  }
  if (propertyNames.length === 0) {
    return []; // No liveness properties declared
  }

  const surface = SURFACE_MAP[configName] || null;
  if (!surface) {
    return []; // Unknown surface — can't validate; do not block unknown configs
  }

  const invariantsPath = path.join(resolvedSpecDir, surface, 'invariants.md');
  let invariantsContent;
  try {
    invariantsContent = fs.readFileSync(invariantsPath, 'utf8');
  } catch (e) {
    // invariants.md missing entirely — all properties lack declarations
    return propertyNames;
  }

  return propertyNames.filter(propName => !invariantsContent.includes('## ' + propName));
}
```

### Example 2: Handling Inconclusive Results (run-tlc.cjs lines 319–342)
```javascript
// Source: bin/run-tlc.cjs
const passed = (tlcResult.status || 0) === 0;

if (passed) {
  const missingDeclarations = detectLivenessProperties(configName, cfgPath);
  if (missingDeclarations.length > 0) {
    try {
      writeCheckResult({
        tool: 'run-tlc',
        formalism: 'tla',
        result: 'inconclusive',
        check_id: check_id,
        surface: 'tla',
        property: property,
        runtime_ms: _runtimeMs,
        summary: 'inconclusive: fairness missing in ' + _runtimeMs + 'ms',
        triage_tags: ['needs-fairness'],
        metadata: {
          config: configName,
          reason: 'Fairness declaration missing for: ' + missingDeclarations.join(', '),
        }
      });
    } catch (e) {
      process.stderr.write('[run-tlc] Warning: failed to write inconclusive result: ' + e.message + '\n');
    }
    process.stdout.write('[run-tlc] Result: inconclusive — fairness declaration missing for: ' + missingDeclarations.join(', ') + '\n');
    process.exit(0); // inconclusive is not a failure
  } else {
    // Write normal pass result
  }
}
```

## Open Questions

1. **Should check-liveness-fairness.cjs aggregate results or create per-config entries?**
   - What we know: Current per-runner approach creates one entry per runner (e.g., run-tlc.cjs → 1 entry for MCsafety or MCliveness)

   - What we know: All 10 specs with PROPERTY lines have ## headers + multi-paragraph rationales
