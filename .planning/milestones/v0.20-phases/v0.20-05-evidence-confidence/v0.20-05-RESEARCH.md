**Researched:** 2026-02-28
**Domain:** Evidence-driven trace validation confidence tiers; observation_window metadata enrichment
**Confidence:** HIGH

## Summary

Phase v0.20-05 adds evidence-driven confidence annotations to `validate-traces.cjs` output. The tool already computes confidence tiers (`low`, `medium`, `high`) based on quorum rounds (n_rounds) and observation window duration (window_days) from the scoreboard file. The phase requires two changes:

1. **EVID-01**: Annotate `never_observed` path entries (divergences) in stdout with a `confidence` field derived from trace volume and window duration. The logic already exists in `validate-traces.cjs` (lines 24-28, 128-129) but confidence is not currently written to the actual check-result NDJSON output.

2. **EVID-02**: Enhance the check-result NDJSON record written by `writeCheckResult()` to include an `observation_window` object with fields: `window_start`, `window_end`, `n_traces`, `n_events`, `window_days`. This metadata already appears in PRISM calls (run-prism.cjs lines 358-366) as a precedent. The schema already supports `observation_window` as an optional top-level field (.formal/check-result.schema.json).

The implementation builds on v0.20-01 (Schema Enrichment, which added `observation_window` as a schema field) and patterns already visible in run-prism.cjs (CALIB-03, lines 352-366) where `observation_window` is constructed from cold-start timing data and quorum round counts from the scoreboard.

**Primary recommendation:** Task 1 enhances `validate-traces.cjs` to compute and pass `observation_window` metadata to `writeCheckResult()` (currently, the writer call at line 204 and 208 lack this field). Task 2 ensures Wave 0 tests verify both confidence tiers in divergences AND observation_window in NDJSON output. Task 3 integrates validation into the formal verification pipeline step.

## Architecture Patterns

### Pattern 1: Reading Scoreboard Metadata

**What:** `validate-traces.cjs` reads `.planning/quorum-scoreboard.json` to extract quorum round count (`n_rounds`) and observation window duration in days (`window_days`).

**When to use:** At startup of `validate-traces.cjs` (main module only), before processing any traces.

**Existing code structure (validate-traces.cjs lines 30-44):**
```javascript
function readScoreboardMeta() {
  const scoreboardPath = path.join(process.cwd(), '.planning', 'quorum-scoreboard.json');
  try {
    const raw = fs.readFileSync(scoreboardPath, 'utf8');
    const sb = JSON.parse(raw);
    const rounds = Array.isArray(sb.rounds) ? sb.rounds : [];
    const n_rounds = rounds.length;
    if (n_rounds === 0) return { n_rounds: 0, window_days: 0 };
    const dates = rounds.map(r => new Date(r.date).getTime()).filter(t => !isNaN(t));
    const window_days = dates.length < 2 ? 0 : Math.floor((Math.max(...dates) - Math.min(...dates)) / 86400000);
    return { n_rounds, window_days };
  } catch (_) {
    return { n_rounds: 0, window_days: 0 };
  }
}
```

**Call pattern (line 128):**
```javascript
const scoreboardMeta = readScoreboardMeta();
const confidence = computeConfidenceTier(scoreboardMeta.n_rounds, scoreboardMeta.window_days);
```

**Fail-open:** If scoreboard doesn't exist or is malformed, returns `{ n_rounds: 0, window_days: 0 }`, which results in `confidence: low`. No throwing.

### Pattern 2: Confidence Tier Computation

**What:** Three-tier confidence system based on quorum rounds (n_rounds) and observation window duration (window_days).

**Thresholds (lines 18-22):**
```javascript
const CONFIDENCE_THRESHOLDS = {
  low:    { min_rounds: 0,     min_days: 0  },
  medium: { min_rounds: 500,   min_days: 14 },
  high:   { min_rounds: 10000, min_days: 90 },
};
```

**Computation (lines 24-28):**
```javascript
function computeConfidenceTier(n_rounds, window_days) {
  if (n_rounds >= CONFIDENCE_THRESHOLDS.high.min_rounds &&
      window_days >= CONFIDENCE_THRESHOLDS.high.min_days) return 'high';
  if (n_rounds >= CONFIDENCE_THRESHOLDS.medium.min_rounds &&
      window_days >= CONFIDENCE_THRESHOLDS.medium.min_days) return 'medium';
  return 'low';
}
```

**Decision logic:** Both conditions (min_rounds AND min_days) must be satisfied. Low is the default fallback.

**Test coverage:** Full boundary test suite already in validate-traces.test.cjs (lines 119-175).

### Pattern 3: Observation Window Metadata

**What:** An object carrying stochastic check timing and trace data, written to check-result NDJSON.

**Structure (from run-prism.cjs lines 358-366, the precedent):**
```javascript
const observationWindow = {
  window_start: coldStartState.firstRunTimestamp
    ? new Date(coldStartState.firstRunTimestamp).toISOString()
    : new Date().toISOString(),
  window_end: new Date().toISOString(),
  n_traces: coldStartState.quorumRoundCount,    // numeric: quorum rounds
  n_events: coldStartState.ciRunCount,          // numeric: CI run count
  window_days: coldStartState.firstRunTimestamp
    ? (Date.now() - coldStartState.firstRunTimestamp) / (1000 * 60 * 60 * 24)
    : 0,
};
```

**For validate-traces (EVID-02 requirement):**
- `window_start`: ISO timestamp when the earliest quorum round in scoreboard occurred (or now if no scoreboard)
- `window_end`: ISO timestamp when validate-traces runs (now)
- `n_traces`: quorum round count from scoreboard (n_rounds from readScoreboardMeta)
- `n_events`: event count from conformance log (total lines processed)
- `window_days`: days between earliest and latest round timestamps from scoreboard

**Schema support:** .formal/check-result.schema.json already allows `observation_window` as an optional object (line 46-48).

**Writer support:** write-check-result.cjs already handles `observation_window` (lines 81-83) — passes it through without validation.

### Anti-Patterns to Avoid

- **Storing confidence in divergence JSON only, not in NDJSON:** The confidence field in divergence stderr output is for human debugging. EVID-02 requires it in the check-result NDJSON via `observation_window`, not as a separate field.
- **Computing window_days incorrectly:** Should be the time span between EARLIEST and LATEST scoreboard round, not the time from first run to now. Use `Math.floor((max_date - min_date) / 86400000)`.
- **Hardcoding check_id or surface values:** Always derive them from STEPS registry or define them locally in validate-traces.cjs. Do not hardcode `'validate-traces'` as the check_id.
- **Missing fail-open on scoreboard read:** If scoreboard doesn't exist, return `{ n_rounds: 0, window_days: 0 }`, not throw. The validation must be robust to missing data.
- **Computing window_start/window_end incorrectly:** `window_start` should be the earliest date in scoreboard rounds (if available), not the current time.

## Common Pitfalls

### Pitfall 1: Confidence in Divergence Output vs. NDJSON

**What goes wrong:** EVID-01 is implemented by adding confidence to divergence stderr JSON (which already happens at lines 142, 155, 168, 189), but EVID-02 is NOT met because the check-result NDJSON record lacks observation_window.

**Why it happens:** Divergence logging and NDJSON writing are separate code paths. It's easy to miss that EVID-02 is about the NDJSON record, not stderr output.

**How to avoid:** EVID-02 explicitly requires observation_window in check-results.ndjson. Task 1 must pass observation_window to writeCheckResult() on lines 204 and 208. Verify with `node bin/validate-traces.cjs && grep observation_window .formal/check-results.ndjson | tail -1` — the record must contain `observation_window: { window_start, window_end, n_traces, n_events, window_days }`.

**Warning signs:** Divergence objects contain `confidence` field in stderr, but NDJSON records don't have `observation_window`.

### Pitfall 2: Window Days Calculation

**What goes wrong:** `window_days` is computed as days from first run to now, rather than days between first and last scoreboard round.

**Why it happens:** Confusion between "how long has the system been running?" and "what is the observation window for this validation?".

**How to avoid:** In readScoreboardMeta, use:
```javascript
const dates = rounds.map(r => new Date(r.date).getTime()).filter(t => !isNaN(t));
const window_days = dates.length < 2 ? 0 : Math.floor((Math.max(...dates) - Math.min(...dates)) / 86400000);
```
This computes the span between the earliest and latest round in the scoreboard, not the span from first round to now.

**Warning signs:** `window_days` increases even on days when no new rounds are added; it tracks calendar time, not event window time.

### Pitfall 3: Check ID and Surface Values

**What goes wrong:** Hardcoded `check_id: 'validate-traces'` instead of a proper STEPS entry ID like `'ci:conformance-traces'`.

**Why it happens:** Confusion about whether `tool: 'validate-traces'` (the runner name) doubles as the `check_id`.

**How to avoid:** Check `run-formal-verify.cjs` STEPS registry (lines 1-25 or wherever validate-traces step is defined). If it has an `id` field like `'ci:conformance-traces'`, use that as `check_id`. If no STEPS entry exists for validate-traces, add one in a separate task or document in PLAN.md that this is TBD.

**Warning signs:** `check_id` value matches `tool` value; triage bundles (Phase v0.20-06) cannot find the check by ID.

### Pitfall 4: Scoreboard Round Date Format

**What goes wrong:** Scoreboard rounds don't have a `date` field; the actual field is `timestamp` or `ts`, causing `new Date(r.date).getTime()` to return NaN and window_days to be 0.

**Why it happens:** Assumption that the scoreboard format matches run-prism.cjs precedent without verifying actual scoreboard structure.

**How to avoid:** Read the actual scoreboard file (`.planning/quorum-scoreboard.json`) and verify the round object structure. Check the existing readScoreboardMeta (line 38) to see how it handles rounds: `rounds.map(r => new Date(r.date).getTime())`. If the actual field name differs, update the mapping in validate-traces.cjs.

**Warning signs:** `window_days` is always 0; dates array is empty after filter.

### Pitfall 5: Missing Observation Window on Error Paths

**What goes wrong:** writeCheckResult() is called on error/early-exit paths (e.g., if conformance log is empty) without observation_window, while success paths include it.

**Why it happens:** Error paths were written before observation_window was added; easy to miss one code path.

**How to avoid:** Every writeCheckResult() call in validate-traces.cjs must include observation_window. There are 3 call sites: line 114 (no log), line 123 (empty log), line 204 (fail), line 208 (pass). All 4 must include observation_window.

**Warning signs:** Some NDJSON records from validate-traces have observation_window, others don't. Inconsistent schema.

### Pitfall 6: Event Count vs. Round Count

**What goes wrong:** `n_events` and `n_traces` are swapped or conflated: n_traces should be quorum round count, n_events should be conformance event count from the log.

**Why it happens:** Unclear semantics; both are "numbers of things".

**How to avoid:** From run-prism.cjs precedent (lines 363-364):
- `n_traces`: quorum round count from scoreboard (metric of "how much operational data")
- `n_events`: CI run count or event count from another log (metric of "how much validation work")
For validate-traces, use:
- `n_traces`: `scoreboardMeta.n_rounds` (quorum rounds)
- `n_events`: `lines.length` or `total` variable (total conformance events processed)

**Warning signs:** n_events is very small (< 10) when it should be large, or vice versa.
