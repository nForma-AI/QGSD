## Research Summary

### Goal

Phase v0.7-02 extends the `<family>-<N>` naming scheme (established in Phase 39) and `quorum_active` config (from Phase v0.7-01) to allow any number of independently-configured slot instances per family. A user can run `copilot-1` and `copilot-2` as separate MCP server entries, each with different model/provider configuration, and the system will call both during quorum.

### What Phase v0.7-01 Already Provides

The foundation is already in place:

1. **`quorum_active` array** in `qgsd.json` lists which slots participate — adding `copilot-2` to the array is all that's needed at the config layer. The orchestrator reads it dynamically with no code change required.

2. **`SLOT_TOOL_SUFFIX` in `qgsd-prompt.js`**: strips trailing `-N` digit to derive family, then looks up tool suffix. `copilot-2` → family `copilot` → suffix `ask` → `mcp__copilot-2__ask`. This already works for any N.

3. **`buildActiveSlots()` in `install.js`**: reads all mcpServer keys from `~/.claude.json` — already picks up any number of slots per family, no hardcoded limit.

4. **`populateActiveSlots()` in `migrate-to-slots.cjs`**: discovers all keys, populates `quorum_active` — same, no hardcoded limit.

5. **`check-provider-health.cjs`**: iterates `activeMcpServers` (filtered by `quorum_active`) — already handles N slots.

### What Does NOT Yet Work for Multi-Slot

#### MULTI-01 and MULTI-02 — What's the actual gap?

The architectural layer is ready. The gaps are in the "add slot" user flows:

1. **`mcp-setup.md` "Add new agent" template list** only shows `claude-1` through `claude-5`. There is no option to add `copilot-2`, `opencode-2`, `codex-cli-2`, or `gemini-cli-2`. Native CLI agents (copilot, opencode, codex, gemini) each run their own separate MCP server binary — adding a second slot means adding a second `~/.claude.json` mcpServers entry pointing to the same binary but with a different name.

2. **`QGSD_KEYWORD_MAP` in `install.js`** maps family → first detected server name using `keywords.find()`. With multiple copilot slots, it only finds and maps the first one. The `required_models` structure uses `copilot` as the key (singular). This needs to become per-slot when multiple slots of the same family exist.

3. **`buildQuorumInstructions()` in `install.js`** generates the `quorum_instructions` string using `required_models` keys (singular per family). With multi-slot this function isn't the primary path (the dynamic `quorum_active`-driven path in `qgsd-prompt.js` already handles it), but `quorum_instructions` written to `qgsd.json` at install time may have stale single-slot content.

4. **`deriveMissingToolName()` and `findQuorumEvidence()` in `qgsd-stop.js`** use `required_models` keys (codex, gemini, opencode, copilot). With multi-slot `copilot-1` + `copilot-2`, both would need to be tracked. However — the Stop hook's primary enforcement is via `quorum_active`-driven orchestrator call detection. Since the orchestrator is the "preferred method", Stop hook actually checks for the orchestrator Task call. The `required_models`-based individual tool check is the fallback path.

5. **`mcp-setup.md` "Add new agent"** — the template map only supports `claude-1..5` (all claude-mcp-server based). For native CLI slots (copilot, opencode, codex, gemini), the command/args structure is different (each has its own binary, different npm package). Adding a `copilot-2` requires:
   - `command: "node"`, `args: ["/path/to/copilot-mcp-server/dist/index.js"]`
   - No `ANTHROPIC_API_KEY` env (GitHub Copilot uses its own auth)
   - The binary path resolved the same way as `claude-mcp-server` (read from existing entries first)

#### MULTI-03 — Direct config edit support

Direct config edit is already supported: user adds an entry to `~/.claude.json` mcpServers manually, then runs `npx qgsd@latest` → `buildActiveSlots()` picks up the new slot → `quorum_active` gets updated (if absent/empty). This works today for any slot name including `copilot-2`.

The only gap is **verification**: the migration/install script doesn't validate that the new slot name follows the `<family>-<N>` scheme. But given MULTI-03's success criterion ("running `npx qgsd@latest` causes the new slot to appear in `quorum.active`"), this already works structurally. The missing piece is the wizard path.

### Implementation Plan (MULTI-01, MULTI-02, MULTI-03)

#### What needs to be built:

**Plan A — Native CLI multi-slot support in mcp-setup.md (MULTI-02 wizard path)**

Extend the "Add new agent" template list in `mcp-setup.md` to include native CLI slots:
- `codex-cli-2` (second codex-mcp-server instance)
- `gemini-cli-2` (second gemini-mcp-server instance)
- `opencode-2` (second opencode-mcp-server instance)
- `copilot-2` (second copilot-mcp-server instance)

The native CLI agents use their own binary (not claude-mcp-server). The add-agent Step C script needs a variant that doesn't write ANTHROPIC_API_KEY/ANTHROPIC_BASE_URL/CLAUDE_DEFAULT_MODEL env vars — native CLI agents have their own auth mechanisms. The binary path is resolved by reading existing entries for that family from `~/.claude.json`.

After writing the new mcpServers entry, the flow runs:
- `populateActiveSlots()` → adds new slot to `quorum_active` if not already present
- Alternatively: direct `quorum_active` update inline in the apply script

**Plan B — MULTI-01: Multiple claude-mcp slots (already partially supported)**

The existing "Add new agent" claude template map (`claude-1..5`) already provides up to 5 claude-mcp-server slots. MULTI-01 success criterion says "both listed in `quorum.active` sees both slots called independently." Since `quorum_active` already drives the orchestrator, this works if both are in `~/.claude.json` and in `quorum_active`. The gap is only UX: the "Add new agent" list doesn't exclude already-configured slots from the template list properly for the 5 claude slots (it says "omit names already in mcpServers" but the templates are fixed identities not families).

**Plan C — `quorum_active` update on slot add (MULTI-03 automated path)**

After any new slot is added via wizard, the apply script must also update `quorum_active` in `qgsd.json` to include the new slot. Currently `populateActiveSlots()` only runs as part of `migrate-to-slots.cjs`. The install.js reinstall path backfills `quorum_active` from all discovered slots but only if currently empty. After wizard adds a slot, we need an incremental add-to-quorum_active step.

### Key Technical Constraints

1. **Native CLI agent binary paths**: `codex-cli-2` would use the same binary as `codex-cli-1`. Path resolution reads the existing `codex-cli-1` entry's args[0]. If no existing entry: use `npx codex-mcp-server` (run-time resolution via npm exec).

2. **No env vars for native CLI agents**: copilot/opencode/codex/gemini use their own auth (GitHub token, OpenCode config, Codex API, Gemini key). The mcpServers entry for a second slot of the same family copies the `command`/`args` from the first slot — same binary, same env pattern.

3. **`quorum_active` incremental update**: A new helper `addSlotToQuorumActive(slotName, qgsdJsonPath)` should append the new slot to `quorum_active` if not already present. Idempotent. Used by the wizard apply step.

4. **Model preferences for multi-slot**: `model_preferences` in `qgsd.json` is keyed by slot name (`claude-1`, `copilot-1`). Additional slots work naturally — `copilot-2` gets its own `model_preferences` entry if desired. No structural change needed.

5. **`QGSD_KEYWORD_MAP` is install-time only**: It's used to populate `required_models` at install time. Post-v0.7-01, `quorum_active` is the runtime source of truth. `required_models` is only consulted by the Stop hook's fallback path. For multi-slot, `quorum_active` is already correct and the Stop hook's orchestrator-detection path handles it without `required_models` changes.

### Wave Structure

### Risk Assessment

- **Low risk**: quorum_active already supports any slot name; orchestrator already handles N slots
- **Medium risk**: mcp-setup.md native CLI template — need to handle "no ANTHROPIC credentials" case cleanly; binary path resolution from existing entries must be robust
- **Low risk**: `addSlotToQuorumActive()` is a simple idempotent append with JSON r/w
- **No regression risk**: existing single-slot behavior is entirely additive

### MULTI-01 vs MULTI-02 Scope Clarification

MULTI-01 ("multiple claude-* slots") is already architecturally supported end-to-end. The mcp-setup "Add new agent" flow for claude slots already works (claude-1..5). The success criterion only requires that "both see both slots called independently" — this is satisfied by `quorum_active` + orchestrator. No new code needed for MULTI-01 beyond ensuring the exclusion list in "Add new agent" correctly shows only un-configured slots.

MULTI-02 ("multiple copilot-N, opencode-N, codex-cli-N, gemini-cli-N slots") requires explicit wizard support since these have different mcpServers structures than claude-mcp-server entries.

MULTI-03 ("adding new slot supported by both direct edit and wizard") — direct edit already works via `buildActiveSlots()` at install time. Wizard path needs the `addSlotToQuorumActive()` call after adding any new slot.
