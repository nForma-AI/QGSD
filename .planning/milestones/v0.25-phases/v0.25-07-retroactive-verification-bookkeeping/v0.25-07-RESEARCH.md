## Architecture Patterns

### Pattern 1: Extract Annotations Parser

**What:** `bin/extract-annotations.cjs` scans formal model files, extracts `@requirement` comments, and outputs a structured JSON map with property-to-requirement associations.

**When to use:** Primary data source for traceability matrix (reads annotations first, falls back to model-registry).

**Format per formalism:**
- TLA+: `\* @requirement REQ-ID` (above property definition)
- Alloy: `-- @requirement REQ-ID` (above assertion)
- PRISM: `// @requirement REQ-ID` (above property formula)

**Example output:**
```json
{
  ".formal/tla/QGSDQuorum.tla": [
    {
      "property": "TypeOK",
      "requirement_ids": ["QUORUM-01"]
    },
    {
      "property": "QuorumCeilingMet",
      "requirement_ids": ["QUORUM-02", "QUORUM-03"]
    }
  ]
}
```

**Key implementation notes:**
- Use model-registry.json for file discovery (not globbing) — enforces central registry ownership
- Filter paths starting with `..` or `/` (security)
- Handle multi-requirement annotations (one `@requirement` line per ID, stacked)
- Return `.slice()` copy of requirement_ids arrays to prevent caller mutation

### Pattern 2: Bidirectional Validation

**What:** Detect asymmetric links between model-registry `requirements` arrays and requirements.json `formal_models` arrays.

**When to use:** As part of traceability matrix generation (TRACE-04).

**Asymmetric link types:**
1. Model claims requirement, but requirement doesn't claim model
2. Requirement claims model, but model doesn't claim requirement
3. Stale references to non-existent models or requirements

**Output:** Warnings to stderr + `bidirectional_validation` section in matrix JSON with:
```json
{
  "asymmetric_links": [
    {
      "type": "model_claims_requirement_only",
      "model_file": ".formal/tla/QGSDQuorum.tla",
      "requirement_id": "QUORUM-02",
      "reason": "model-registry lists QUORUM-02 but requirements.json QUORUM-02 does not list this model"
    }
  ],
  "stale_links": [
    {
      "model_file": ".formal/nonexistent.tla",
      "reason": "file does not exist on disk"
    }
  ]
}
```

### Pattern 3: Coverage Regression Guard

**What:** Compare current traceability matrix coverage percentage against a baseline snapshot, warn if drop exceeds threshold.

**When to use:** As part of formal verification pipeline and in CI checks.

**Commands:**
```bash
node bin/check-coverage-guard.cjs                  # Compare current vs baseline, exit 0
node bin/check-coverage-guard.cjs --save-baseline # Save current matrix as baseline
node bin/check-coverage-guard.cjs --threshold 10  # Override default 15% threshold
```

**Interpretation of "drops below threshold":** If baseline is 27% coverage and current is 11%, the drop is 16 percentage points > 15pp default threshold → warning fires.

**Output:** Message like `Coverage dropped by 16% (threshold: 15%) — from 27.0% to 11.0%`

### Pattern 4: State-Space Analysis

**What:** Estimate state-space size of each TLA+ model and classify risk (MINIMAL/LOW/MODERATE/HIGH).

**When to use:** After model construction or decomposition, as part of traceability matrix generation (DECOMP-01..04).

**Analysis inputs:**
- TLA+ .cfg file (variable domains)
- Model variables and their declared ranges
- Check for unbounded types (Nat, Int, String without constraints)

**Risk classification rules:**
- **MINIMAL:** All variables bounded, state-space < 10^4
- **LOW:** All variables bounded, 10^4 ≤ state-space < 10^6
- **MODERATE:** Mixed bounded/unbounded or 10^6 ≤ state-space < 10^8
- **HIGH:** Any unbounded variable OR state-space > 10^8

**Output in matrix:**
```json
{
  "state_space": {
    ".formal/tla/QGSDQuorum.tla": {
      "estimated_state_space": "2.5e7",
      "risk_classification": "MODERATE",
      "unbounded_domains": ["quorumConfig"],
      "reason": "quorumConfig declared as Nat without upper bound"
    }
  }
}
```

### Anti-Patterns to Avoid
- **Don't manually insert baseline data** — Use `check-coverage-guard.cjs --save-baseline` to capture authoritative baseline
- **Don't rely on single data source** — Always cross-verify model-registry ↔ requirements.json bidirectionally
- **Don't skip requirement updates in bookkeeping** — Every completed requirement must have [ ] checkbox updated to [x] and requirements-completed field in SUMMARY.md frontmatter
- **Don't regenerate VERIFICATION.md artifacts without re-running test suites** — Verification must reflect actual test execution, not assumptions

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Bidirectional data consistency checking | Custom diff algorithm | Structured cross-reference in generate-traceability-matrix.cjs with asymmetric link detection | Edge cases: stale references, null values, array order variations; project pattern established in v0.25-03 |
| Coverage regression detection | Manual baseline comparison in shell script | Dedicated `bin/check-coverage-guard.cjs` with percentage point calculation and configurable threshold | Precision errors with floating-point comparison; hard to maintain threshold config across deployments |
| State-space estimation | Manual TLA+ parsing | `bin/analyze-state-space.cjs` with TLA+ .cfg parser + variable domain scanner | Complex domain syntax (Int 1..100, Int 1..k, Nat); unbounded type detection requires semantic awareness |
| Requirement ID extraction from code | Regex-based grep with post-processing | `bin/extract-annotations.cjs` with formalism-specific comment parsers | Multi-line annotations, multi-requirement properties, generated file preservation notes, encoding edge cases |

**Key insight:** All four of these are deceptively complex due to semantic concerns (domain boundedness, multi-line structure, formalism syntax variations) that regex or simple scripts handle poorly. The existing project patterns and v0.25-03/04 research establish that domain-aware tools are necessary.

## Common Pitfalls

### Pitfall 1: Baseline File Staleness
**What goes wrong:** Coverage baseline (`.formal/traceability-matrix.baseline.json`) is not updated after legitimate coverage changes, causing false-positive warnings when someone fixes a gap or refactors models.

**Why it happens:** Baseline update workflow is unclear; developers forget to commit baseline updates; baseline is easy to ignore since it's a JSON file without prominent status.

**How to avoid:**
- Document baseline update workflow in DECOMP-05-PLAN.md (first task)
- Add a `node bin/check-coverage-guard.cjs --save-baseline` step to the plan that updates baseline whenever coverage legitimately changes
- Include baseline file in git; make updates visible in commit history

**Warning signs:** CI warnings appear for legitimate changes; baseline timestamp is much older than recent code changes.

### Pitfall 2: Asymmetric Links From Model-Registry Seeding
**What goes wrong:** TRACE-04 bidirectional validation reports many "asymmetric links" during first run because model-registry.json and requirements.json were seeded independently in v0.25-01 without careful cross-referencing.

**Why it happens:** v0.25-01 seeded model-registry from TRACEABILITY_RESEARCH.md Section 5 (~22 models, some partial requirements arrays); requirements.json was aggregated from milestone files independently. The two sources were never cross-validated before v0.25-07.

**How to avoid:**
- Run bidirectional validation early (first task) to surface all asymmetries
- Create a "reconciliation" plan that fixes the most critical asymmetries (models/requirements that should be linked but aren't)
- Accept that some asymmetries may be intentional (e.g., models that don't verify any formal requirement)

**Warning signs:** First run of `node bin/generate-traceability-matrix.cjs` produces 10+ asymmetric link warnings.

### Pitfall 3: Incomplete REQUIREMENTS.md Checkbox Updates
**What goes wrong:** REQUIREMENTS.md is updated for some requirements but not others; the file becomes a patchwork of [x] Complete and [ ] Pending even after phase completion.

**Why it happens:** Manual checkbox updates are error-prone; easy to miss a few entries when updating 15+ checkboxes; no automated verification that all checkboxes match phase status.

**How to avoid:**
- Create a single task that updates ALL v0.25 requirement checkboxes at once (not scattered across multiple plans)
- Write a validation script that checks: all 18 v0.25 requirements have [ ] or [x]; all that should be [x] per REQUIREMENTS.md section are [x]; SUMMARY.md frontmatter requirements-completed field matches REQUIREMENTS.md status
- Run validation before commit

**Warning signs:** `grep "v0.25 requirements" REQUIREMENTS.md | grep "\[ \]"` shows remaining Pending items after phase completion.

### Pitfall 4: SUMMARY.md Frontmatter Incompleteness
**What goes wrong:** v0.25-02-01 and v0.25-02-02 SUMMARY.md files have frontmatter with `provides` field but lack `requirements-completed` field, breaking the VERIFICATION.md → SUMMARY.md → REQUIREMENTS.md chain.

**Why it happens:** Frontmatter structure evolved during v0.25 phases; early phases didn't include `requirements-completed`; those files were never retrofitted.

**How to avoid:**
- Add `requirements-completed` field to v0.25-02-01 and v0.25-02-02 SUMMARY.md files as part of bookkeeping work
- Establish pattern: every SUMMARY.md must have both `provides` and `requirements-completed` fields
- Future phases enforce this pattern in planner templates

**Warning signs:** `grep -L "requirements-completed" .planning/phases/v0.25-*/*.md | grep SUMMARY` identifies missing fields.

### Pitfall 5: Generated File Annotation Preservation
**What goes wrong:** Generated model files (QGSDQuorum.tla, QGSDQuorum_xstate.tla, quorum-votes.als, quorum.props) are regenerated by spec-regen hook and lose annotations.

**Why it happens:** Spec-regen hook (qgsd-spec-regen.js) regenerates from source templates without preserving manually-added @requirement comments.

**How to avoid:**
- This was fixed in v0.25-06 (annotation regression fix) — spec-regen now preserves annotations
- Verify no generated files lose annotations by running `extract-annotations.cjs --validate` after any regeneration
- If this regresses, the fix pattern is: either annotation-aware regeneration in spec-regen.js, or post-regeneration annotation injection

**Warning signs:** `extract-annotations.cjs --validate` reports "X unannotated properties" in generated files after a code change.

### Extract Annotations Parser Pattern

**Source:** Established in v0.25-02, patterns in v0.25-02-RESEARCH.md

```javascript
// bin/extract-annotations.cjs — parser entry point
const fs = require('fs');
const readline = require('readline');
const registry = JSON.parse(fs.readFileSync('.formal/model-registry.json', 'utf8'));

const PATTERNS = {
  tla: /^\s*\\\*\s+@requirement\s+([A-Z]+\-\d+)/, // TLA+ comments
  alloy: /^\s*--\s+@requirement\s+([A-Z]+\-\d+)/, // Alloy comments
  prism: /^\s*\/\/\s+@requirement\s+([A-Z]+\-\d+)/ // PRISM comments
};

async function extractFromFile(modelFile) {
  const rl = readline.createInterface({
    input: fs.createReadStream(modelFile),
    crlfDelay: Infinity
  });

  const properties = [];
  let currentProperty = null;
  let currentRequirements = [];

  for await (const line of rl) {
    // Detect @requirement comments (all formalisms)
    for (const [formalism, pattern] of Object.entries(PATTERNS)) {
      const match = line.match(pattern);
      if (match) {
        currentRequirements.push(match[1]);
        continue;
      }
    }

    // Detect property definition (formalism-specific)
    const propMatch = detectProperty(line, modelFile);
    if (propMatch && currentRequirements.length > 0) {
      properties.push({
        property: propMatch,
        requirement_ids: currentRequirements.slice() // Copy to prevent mutation
      });
      currentRequirements = [];
    }
  }

  return properties;
}

// Usage: extract-annotations.cjs [--validate]
const output = {};
for (const model of registry.models) {
  try {
    output[model.path] = await extractFromFile(model.path);
  } catch (err) {
    // Fail-open: skip unreachable models
    console.warn(`Warning: Could not parse ${model.path}: ${err.message}`);
  }
}

console.log(JSON.stringify(output));
```

### Bidirectional Validation Pattern

**Source:** v0.25-04-RESEARCH.md, integrate into generate-traceability-matrix.cjs

```javascript
// Validate model-registry ↔ requirements.json links
function validateBidirectionalLinks(modelRegistry, requirements) {
  const asymmetricLinks = [];
  const staleLinks = [];

  // Type 1: Model claims requirement, but requirement doesn't claim model
  for (const model of modelRegistry.models) {
    for (const reqId of model.requirements || []) {
      const req = requirements[reqId];
      if (!req) {
        staleLinks.push({
          model_file: model.path,
          requirement_id: reqId,
          reason: `requirement ${reqId} does not exist in requirements.json`
        });
        continue;
      }
      if (!req.formal_models?.includes(model.path)) {
        asymmetricLinks.push({
          type: 'model_claims_requirement_only',
          model_file: model.path,
          requirement_id: reqId,
          reason: `model-registry lists ${reqId} but requirements.json ${reqId} does not list ${model.path}`
        });
      }
    }
  }

  // Type 2: Requirement claims model, but model doesn't claim requirement
  for (const req of Object.values(requirements)) {
    for (const modelPath of req.formal_models || []) {
      const model = modelRegistry.models.find(m => m.path === modelPath);
      if (!model) {
        staleLinks.push({
          model_file: modelPath,
          reason: `file does not exist on disk or not in model-registry`
        });
        continue;
      }
      if (!model.requirements?.includes(req.id)) {
        asymmetricLinks.push({
          type: 'requirement_claims_model_only',
          requirement_id: req.id,
          model_file: modelPath,
          reason: `requirements.json lists ${modelPath} but model-registry ${modelPath} does not list ${req.id}`
        });
      }
    }
  }

  return { asymmetric_links: asymmetricLinks, stale_links: staleLinks };
}

// Integrate into generate-traceability-matrix.cjs:
const validation = validateBidirectionalLinks(modelRegistry, requirements);
if (validation.asymmetric_links.length > 0) {
  console.warn(`Warning: ${validation.asymmetric_links.length} asymmetric links detected`);
  validation.asymmetric_links.forEach(link => console.warn(`  ${link.type}: ${link.model_file} <-> ${link.requirement_id}`));
}

matrixOutput.bidirectional_validation = validation;
```

### Coverage Guard Pattern

**Source:** v0.25-04-RESEARCH.md, standalone script

```bash
#!/usr/bin/env node
// bin/check-coverage-guard.cjs

const fs = require('fs');
const path = require('path');

const args = process.argv.slice(2);
const saveBaseline = args.includes('--save-baseline');
const thresholdIdx = args.indexOf('--threshold');
const threshold = thresholdIdx >= 0 ? parseFloat(args[thresholdIdx + 1]) : 15.0;

const matrixFile = '.formal/traceability-matrix.json';
const baselineFile = '.formal/traceability-matrix.baseline.json';

const currentMatrix = JSON.parse(fs.readFileSync(matrixFile, 'utf8'));
const currentCoverage = currentMatrix.coverage_summary.coverage_percentage;

if (saveBaseline) {
  fs.writeFileSync(baselineFile, JSON.stringify(currentMatrix, null, 2));
  console.log(`Saved baseline: ${currentCoverage}%`);
  process.exit(0);
}

if (!fs.existsSync(baselineFile)) {
  console.log(`No baseline found. Create one with: node bin/check-coverage-guard.cjs --save-baseline`);
  process.exit(0);
}

const baseline = JSON.parse(fs.readFileSync(baselineFile, 'utf8'));
const baselineCoverage = baseline.coverage_summary.coverage_percentage;
const drop = baselineCoverage - currentCoverage;

if (drop > threshold) {
  console.warn(`Coverage dropped by ${drop.toFixed(1)}% (threshold: ${threshold}%) — from ${baselineCoverage}% to ${currentCoverage}%`);
  process.exit(0); // Non-gating warning
}

console.log(`Coverage stable: ${currentCoverage}% (baseline: ${baselineCoverage}%)`);
process.exit(0);
```
