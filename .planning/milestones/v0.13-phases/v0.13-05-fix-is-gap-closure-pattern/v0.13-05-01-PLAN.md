# v0.13-05-01 — Fix TECH-01: IS_GAP_CLOSURE Grep Pattern

## Goal

Fix the false-positive IS_GAP_CLOSURE detection in `qgsd-core/workflows/transition.md` by:
1. Anchoring the grep to the `^### Phase X:` heading block (prevent cross-phase bleed)
2. Using `-A 4` instead of `-A 3` (the `**Gap Closure:**` field is at offset 4, not 3)
3. Correcting all stale `-A 3` references in ROADMAP.md that carried the wrong value

## Background

TECH-01 was identified in the v0.13 re-audit (v0.13-MILESTONE-AUDIT.md). The original fix
recommended `grep -A 3 "^### Phase ${COMPLETED_PHASE}:"` but this was wrong: the `**Gap Closure:**`
field consistently appears at **offset 4** from the `^### Phase X:` header in all 7 gap-closure
phases in ROADMAP.md (v0.10-07, v0.10-08, v0.12-09, v0.12-10, v0.13-03, v0.13-04, v0.13-05 —
all verified at offset 4).

The `-A 3` value would cause IS_GAP_CLOSURE to always return 0, breaking LOOP-02 (gap closure
re-audit routing) entirely. Using `-A 4` is the correct fix.

The ROADMAP.md itself also contained the wrong `-A 3` value in 4 places — these must also be
corrected.

This was caught by Gemini's BLOCK in quorum Round 1 deliberation (2026-02-25).

## Tasks

### Task 1 — Edit transition.md

**File:** `qgsd-core/workflows/transition.md`

**Change the IS_GAP_CLOSURE assignment line:**

From:
```bash
IS_GAP_CLOSURE=$(grep -A 15 "Phase ${COMPLETED_PHASE}" .planning/ROADMAP.md | grep -c '\*\*Gap Closure:\*\*')
```

To:
```bash
IS_GAP_CLOSURE=$(grep -A 4 "^### Phase ${COMPLETED_PHASE}:" .planning/ROADMAP.md | grep -c '\*\*Gap Closure:\*\*')
```

**Why `-A 4`:** The `**Gap Closure:**` field is at line offset 4 from the `^### Phase X:` header
(confirmed across all 7 gap-closure phases in ROADMAP.md). Using `-A 3` would always miss it.

**Why `^### Phase`:** Anchoring to the heading prevents the false-positive where a downstream
gap-closure phase's "Depends on: Phase X" line appears within 15 lines of a `**Gap Closure:**`
entry, causing primary phase X to be incorrectly detected as a gap closure phase.
