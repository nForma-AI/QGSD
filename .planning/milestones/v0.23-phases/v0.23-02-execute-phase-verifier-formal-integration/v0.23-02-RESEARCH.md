# Phase v0.23-02: execute-phase + Verifier Formal Integration - Research

**Researched:** 2026-03-02
**Domain:** Workflow orchestration — formal verification gate injection into execute-phase and qgsd-verifier
**Confidence:** HIGH

## Summary

Phase v0.23-02 wires formal verification into the execute-phase workflow and the qgsd-verifier agent so that real TLC/Alloy/PRISM tool output (not LLM estimation) gates workflow advancement. The implementation pattern is already established in quick.md Steps 6.3 and 6.5 — execute-phase needs an identical gate inserted between the last executor wave completion and the verifier spawn. The verifier agent needs a `<formal_context>` block injected into its prompt carrying the actual `FORMAL_CHECK_RESULT` JSON (passed-in, not re-run), and must treat any `failed > 0` result as a hard failure that sets `status: counterexample_found` in VERIFICATION.md.

The three key behaviors are: (1) execute-phase runs `bin/run-formal-check.cjs` after executor waves complete and before the verifier fires, (2) when `run-formal-check.cjs` exits 1 the workflow hard-blocks with a new `counterexample_found` status, (3) the user can override the block with an explicit acknowledgment that is audited in VERIFICATION.md. Fail-open is the fourth requirement: missing Java or PRISM binary causes a skip warning in VERIFICATION.md, never a block.

The principal design challenge is how execute-phase recovers `FORMAL_SPEC_CONTEXT` (the module list matched during plan-phase) at execution time. Because `FORMAL_SPEC_CONTEXT` is an in-memory variable during plan-phase and is not persisted anywhere, execute-phase must re-run the same keyword-match scan (using the phase description) or use the phase description directly to build the module list. The cleanest approach — and the one consistent with the project's existing pattern — is to re-run the Step 4.5 formal scope scan at execute-phase time using the phase description, identical to plan-phase's algorithm.

**Primary recommendation:** Add a formal check step to execute-phase after the last executor wave using the same `FORMAL_SPEC_CONTEXT` keyword-scan from plan-phase Step 4.5. Pass `FORMAL_CHECK_RESULT` to the verifier Task prompt. Update qgsd-verifier to parse and act on `FORMAL_CHECK_RESULT` from the prompt. Add `counterexample_found` status with acknowledgment override path. Preserve fail-open throughout.

### FORMAL_CHECK_RESULT JSON Schema

Produced by `bin/run-formal-check.cjs` on stdout line starting with `FORMAL_CHECK_RESULT=`:

```json
{
  "passed": 2,
  "failed": 0,
  "skipped": 1,
  "counterexamples": ["quorum:tlc", "breaker:alloy"]
}
```

Key fields the verifier must read:
- `failed`: if > 0, treat as hard failure
- `skipped`: if all results are skipped, treat as tooling absent (skip warning, not block)
- `counterexamples`: array of `module:tool` pairs identifying which checks failed

## Architecture Patterns

### Pattern 1: Formal Check Step in execute-phase (after executor waves, before verifier)

The `verify_phase_goal` step in execute-phase.md currently:
1. Reads phase requirement IDs
2. Sets activity tracking
3. Spawns qgsd-verifier Task

This phase inserts a new step **between** wave completion and verifier spawn:

```
After last executor wave completes:
  Step: formal_check_post_wave
    1. Re-run keyword-match scan to populate $FORMAL_SPEC_CONTEXT (same algorithm as plan-phase Step 4.5)
    2. If FORMAL_SPEC_CONTEXT is empty → skip (set FORMAL_CHECK_RESULT=null, display skip message)
    3. Run: node bin/run-formal-check.cjs --modules=${MODULES} 2>&1
    4. Parse FORMAL_CHECK_RESULT from output
    5. Route on exit code (0 = pass/skip, 1 = counterexample)
    6. Store FORMAL_CHECK_RESULT and FORMAL_CHECK_EXIT for verifier spawn
  Step: verify_phase_goal (existing, now receives FORMAL_CHECK_RESULT)
    1. Spawn qgsd-verifier with FORMAL_CHECK_RESULT passed in prompt
    2. Route on status: passed | human_needed | gaps_found | counterexample_found
```

**Placement in execute-phase.md:** New step `formal_check_post_wave` inserted INSIDE `verify_phase_goal` step, BEFORE the Task spawn. This avoids adding a new top-level step and keeps the formal gate logically coupled to the verify step.

### Pattern 2: FORMAL_SPEC_CONTEXT Reconstruction at Execute Time

During plan-phase, `FORMAL_SPEC_CONTEXT` is an in-memory bash variable. It is NOT persisted to disk (task-envelope.json does not store it; PLAN frontmatter's `formal_artifacts:` only records `none`/`update`/`create`, not the module list).

**Solution:** Execute-phase re-runs the identical keyword-match scan using the phase description, which is available via `gsd-tools roadmap get-phase`. This is the same algorithm as plan-phase Step 4.5 — deterministic, no state required.

```bash
# In execute-phase, after last wave completes, before verifier spawn:
PHASE_DESC=$(node ~/.claude/qgsd/bin/gsd-tools.cjs roadmap get-phase "${PHASE_NUMBER}" | jq -r '.section // empty' | head -3)
FORMAL_SPEC_CONTEXT=()
if [ -d ".formal/spec" ]; then
  for MODULE_DIR in .formal/spec/*/; do
    MODULE=$(basename "$MODULE_DIR")
    INVARIANTS_FILE=".formal/spec/${MODULE}/invariants.md"
    if [ -f "$INVARIANTS_FILE" ]; then
      DESC_LOWER=$(echo "$PHASE_DESC" | tr '[:upper:]' '[:lower:]')
      MODULE_LOWER=$(echo "$MODULE" | tr '[:upper:]' '[:lower:]')
      if echo "$DESC_LOWER" | grep -qF "$MODULE_LOWER" || echo "$MODULE_LOWER" | grep -qF "$DESC_LOWER"; then
        FORMAL_SPEC_CONTEXT+=("{\"module\":\"${MODULE}\",\"path\":\"${INVARIANTS_FILE}\"}")
      fi
    fi
  done
fi
```

If `FORMAL_SPEC_CONTEXT` is empty → set `FORMAL_CHECK_RESULT=null`, skip run-formal-check.cjs. Verifier receives `null` and skips formal section entirely.

### Pattern 3: Verifier Formal Context Block

**From quick.md Step 6.5 (source of truth):**

```markdown
<formal_context>
${FORMAL_CHECK_RESULT !== null ?
`Formal check result (real tool output, not estimated):
${JSON.stringify(FORMAL_CHECK_RESULT)}

Rules:
- If failed > 0: this is a HARD FAILURE — set status: counterexample_found in VERIFICATION.md. No must_haves can pass when a counterexample exists.
- If skipped only (failed == 0, passed == 0): tooling was absent — include skip warning in VERIFICATION.md but do NOT set counterexample_found. Proceed normally.
- If passed > 0 and failed == 0: formal checks passed — include pass counts in VERIFICATION.md as evidence.` :
'No formal check performed (FORMAL_SPEC_CONTEXT was empty or tooling unavailable). Skip formal section.'}
</formal_context>
```

**The verifier does NOT re-run `run-formal-check.cjs`.** The `FORMAL_CHECK_RESULT` is pre-computed by execute-phase and passed into the verifier prompt. This enforces a single canonical result — the verifier is a consumer, not a runner.

### Pattern 4: counterexample_found Status and Override Path

**New status in VERIFICATION.md:** `counterexample_found`

This is a fourth status alongside `passed`, `gaps_found`, `human_needed`. It represents "formal model checker found a counterexample — workflow cannot advance without explicit user override."

**VERIFICATION.md frontmatter when counterexample found:**

```yaml
---
phase: v0.23-02
verified: 2026-03-02T10:00:00Z
status: counterexample_found
formal_check:
  passed: 0
  failed: 2
  skipped: 0
  counterexamples:
    - "quorum:tlc"
    - "quorum:alloy"
counterexample_override:  # Only present if user provided override
  acknowledged_at: 2026-03-02T10:05:00Z
  reason: "This is a known race in the model that does not affect runtime — tracked in #123"
  override_by: "user"
---
```

**Execute-phase routing table update:**

| Status | Current Action | New Action (v0.23-02) |
|--------|----------------|----------------------|
| `passed` | update_roadmap | (unchanged) |
| `human_needed` | quorum resolution | (unchanged) |
| `gaps_found` | gap closure | (unchanged) |
| `counterexample_found` | (NEW) | Present counterexample details. Prompt user for explicit acknowledgment reason. If user provides reason → write to VERIFICATION.md `counterexample_override:`, set effective status = passed, continue to update_roadmap. If user declines → halt, report. |

**Override audit trail requirement (ENF-02):** The override MUST be written to VERIFICATION.md with the reason and timestamp. There is NO silent bypass — the user must type a non-empty acknowledgment string.

### Pattern 5: Skip Warning for Missing Tooling (ENF-03)

When `run-formal-check.cjs` reports all `skipped` (Java or jars absent), `FORMAL_CHECK_RESULT.failed == 0`. Execute-phase continues without blocking. Verifier writes a skip note in VERIFICATION.md but does NOT set `counterexample_found`.

**Key distinction:** `failed: 0, skipped: N` (tooling absent) ≠ `failed: 1` (counterexample found). Only `failed > 0` triggers the counterexample gate.

When `FORMAL_SPEC_CONTEXT` is empty (no matching modules), `FORMAL_CHECK_RESULT` is never set — the entire formal gate is skipped. VERIFICATION.md will contain no `formal_check:` section.

### Anti-Patterns to Avoid

- **Verifier re-running run-formal-check.cjs:** The verifier is a consumer of `FORMAL_CHECK_RESULT`, not a runner. Re-running would create a second independent tool invocation that could disagree with the gate that already blocked (or passed) the workflow.
- **Persisting FORMAL_SPEC_CONTEXT to task-envelope.json:** Unnecessary complexity — re-running the keyword-match scan from phase description is deterministic and requires no state. Adding storage creates a sync problem.
- **Using `formal_artifacts:` value to reconstruct module list:** The `formal_artifacts:` field stores `none`/`update`/`create` — it records what the PLANNER plans to do with formal files, not which modules were matched. These are different concerns.
- **Treating tooling absence as counterexample:** A `skipped` result means the gate could not run, not that a violation was found. Gate absence must be fail-open (skip warning), not fail-closed (block).
- **Silent override path:** Any override of a `counterexample_found` block must produce a VERIFICATION.md `counterexample_override:` entry with timestamp and reason. An empty acknowledgment string is not acceptable.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Running TLC/Alloy/PRISM | Custom spawn logic in execute-phase | `bin/run-formal-check.cjs` | Already handles Java detection, JAR existence, fail-open, exit codes, FORMAL_CHECK_RESULT output format |
| Keyword-match to get module list | New filesystem scanner | Copy the Step 4.5 bash block from plan-phase.md | Identical algorithm; duplication is acceptable for workflow files that can't `import` each other |
| Parsing FORMAL_CHECK_RESULT JSON | Manual string parsing | `FORMAL_CHECK_RESULT=$(echo "$OUTPUT" | grep '^FORMAL_CHECK_RESULT=' | cut -d= -f2-)` then pass as string | run-formal-check.cjs emits valid JSON; grep+cut is reliable; no jq dependency needed |
| Writing override to VERIFICATION.md | Execute-phase writing directly | Verifier agent writes it when receiving override reason in prompt | Verifier controls VERIFICATION.md content; orchestrator passes override reason to verifier via continuation prompt |

**Key insight:** All formal running logic lives in `bin/run-formal-check.cjs`. The workflow files (execute-phase.md, qgsd-verifier.md) are orchestrators — they invoke the tool and route on its output. This separation keeps the formal gate testable independently of the workflow.

## Common Pitfalls

### Pitfall 1: FORMAL_SPEC_CONTEXT Not Available in execute-phase

**What goes wrong:** execute-phase cannot access the `FORMAL_SPEC_CONTEXT` variable set during plan-phase because plan-phase and execute-phase are separate Claude sessions.

**Why it happens:** `FORMAL_SPEC_CONTEXT` is an in-memory bash variable in plan-phase.md. It is not persisted to any file. It is not in task-envelope.json, PLAN frontmatter, or STATE.md.

**How to avoid:** Re-run the keyword-match scan inside execute-phase using the phase description from ROADMAP. The scan is deterministic — given the same phase description and the same `.formal/spec/` directory, it always produces the same modules list. This is the canonical approach.

**Warning signs:** If execute-phase attempts to read `FORMAL_SPEC_CONTEXT` from task-envelope.json or PLAN frontmatter and fails → this pitfall hit.

### Pitfall 2: counterexample_found Blocks Even When Tooling Absent

**What goes wrong:** execute-phase sets `status: counterexample_found` when `run-formal-check.cjs` exits 0 because all results were `skipped` (Java absent).

**Why it happens:** Developer treats any non-`passed` result as a block.

**How to avoid:** Gate is ONLY on `FORMAL_CHECK_EXIT == 1`. Exit code 0 means no counterexample — regardless of whether checks ran or were skipped. The `failed` count in JSON provides additional signal but the exit code is the canonical gate.

**Warning signs:** Execute-phase showing `counterexample_found` when Java is not installed.

### Pitfall 3: Verifier Gets FORMAL_CHECK_RESULT From Wrong Wave

**What goes wrong:** Execute-phase runs multiple waves. The formal check should run ONCE after ALL executor waves complete (before the single verifier spawn). Running it after each wave and passing only the last result loses counterexamples from earlier waves.

**Why it happens:** Placement of the formal check step at per-wave level instead of post-all-waves level.

**How to avoid:** Place the formal check step inside `verify_phase_goal`, not inside `execute_waves`. The verifier fires once per phase, and the formal check must aggregate across all modifications made by all waves.

**Warning signs:** Multiple `FORMAL_CHECK_RESULT` lines in execute-phase output, one per wave.

### Pitfall 4: Override Without Audit Trail

**What goes wrong:** User types "yes" to override the counterexample block, but VERIFICATION.md is never updated with the override reason. Next time someone reads VERIFICATION.md, it shows `status: passed` with no record of the override.

**Why it happens:** Execute-phase routes past the block without passing the reason to the verifier.

**How to avoid:** Execute-phase must pass the user's override reason string to the verifier via a continuation prompt. The verifier writes `counterexample_override:` to VERIFICATION.md with the received reason and a timestamp. Execute-phase then reads the updated VERIFICATION.md and confirms the field is present before advancing.

**Warning signs:** VERIFICATION.md status changed from `counterexample_found` to `passed` with no `counterexample_override:` section.

### Pitfall 5: Install Step Forgotten

**What goes wrong:** Changes made to `qgsd-core/workflows/execute-phase.md` and agent files but not synced to `~/.claude/`. The live workflow at `~/.claude/qgsd/workflows/execute-phase.md` still has the old (no formal gate) version.

**Why it happens:** QGSD has a two-copy system: source in `qgsd-core/workflows/` and installed copy at `~/.claude/qgsd/workflows/`. Edit source → installer syncs → `~/.claude/` is what actually runs.

**How to avoid:** Every plan that modifies workflow files MUST include an install task: `node bin/install.js --claude --global`. Verify with diff between source and installed copy.

**Warning signs:** grep for new formal gate content in `~/.claude/qgsd/workflows/execute-phase.md` returns nothing after executor successfully ran.

## Code Examples

### Formal Scope Scan in execute-phase (copied from plan-phase Step 4.5)

**Source:** `qgsd-core/workflows/plan-phase.md` lines 68-107

```bash
# Insert in execute-phase.md before verifier spawn (inside verify_phase_goal step)
# Formal scope scan (same algorithm as plan-phase Step 4.5)
FORMAL_SPEC_CONTEXT=()
if [ -d ".formal/spec" ]; then
  PHASE_DESC=$(node ~/.claude/qgsd/bin/gsd-tools.cjs roadmap get-phase "${PHASE_NUMBER}" | jq -r '.goal // empty')
  for MODULE_DIR in .formal/spec/*/; do
    MODULE=$(basename "$MODULE_DIR")
    INVARIANTS_FILE=".formal/spec/${MODULE}/invariants.md"
    if [ -f "$INVARIANTS_FILE" ]; then
      DESC_LOWER=$(echo "$PHASE_DESC" | tr '[:upper:]' '[:lower:]')
      MODULE_LOWER=$(echo "$MODULE" | tr '[:upper:]' '[:lower:]')
      MATCHED=0
      for KEYWORD in $(echo "$DESC_LOWER" | tr ' -' '\n' | grep -v '^$'); do
        if echo "$MODULE_LOWER" | grep -qF "$KEYWORD"; then
          MATCHED=1
          break
        fi
      done
      if [ $MATCHED -eq 1 ]; then
        FORMAL_SPEC_CONTEXT+=("{\"module\":\"${MODULE}\",\"path\":\"${INVARIANTS_FILE}\"}")
      fi
    fi
  done
fi
```

### Formal Check Invocation (execute-phase.md)

**Source:** `qgsd-core/workflows/quick.md` Step 6.3

```bash
# After last executor wave, before verifier spawn:
if [ ${#FORMAL_SPEC_CONTEXT[@]} -gt 0 ]; then
  MODULES=$(echo "${FORMAL_SPEC_CONTEXT[@]}" | node -e "
    const chunks = require('fs').readFileSync('/dev/stdin','utf8').trim().split(/\s+(?=\{)/);
    const mods = chunks.map(c => { try { return JSON.parse(c).module; } catch(e) { return null; } }).filter(Boolean);
    process.stdout.write(mods.join(','));
  ")
  FORMAL_CHECK_OUTPUT=$(node bin/run-formal-check.cjs --modules=${MODULES} 2>&1)
  FORMAL_CHECK_EXIT=$?
  FORMAL_CHECK_RESULT=$(echo "$FORMAL_CHECK_OUTPUT" | grep '^FORMAL_CHECK_RESULT=' | cut -d= -f2-)

  if [ "$FORMAL_CHECK_EXIT" -eq 0 ]; then
    echo "◆ Formal check: PASSED"
  else
    echo "◆ Formal check: COUNTEREXAMPLE FOUND — see output above"
  fi
else
  FORMAL_CHECK_RESULT=null
  FORMAL_CHECK_EXIT=0
fi
```

**Simpler module extraction** (bash-only, avoids node):

```bash
MODULES=""
for ctx_entry in "${FORMAL_SPEC_CONTEXT[@]}"; do
  MOD=$(echo "$ctx_entry" | sed 's/.*"module":"\([^"]*\)".*/\1/')
  MODULES="${MODULES:+${MODULES},}${MOD}"
done
```

### counterexample_found Routing in execute-phase

**New routing row** added to execute-phase's `verify_phase_goal` step status table:

```markdown
| `counterexample_found` | Present counterexample details from VERIFICATION.md. Prompt user:
  "Formal model checker found counterexamples: {counterexamples}.
   To override, provide an acknowledgment reason. This will be written to VERIFICATION.md.
   Type your reason, or 'abort' to stop."

  If user provides reason (non-empty, not 'abort'):
  1. Spawn continuation verifier Task with `{override_reason}` in prompt
  2. Verifier updates VERIFICATION.md: adds counterexample_override: { acknowledged_at, reason, override_by: "user" }
  3. Read updated VERIFICATION.md status — must now be 'passed'
  4. Continue to update_roadmap

  If user types 'abort':
  - Report phase halted due to counterexample.
  - No update_roadmap, no phase completion.
  - STATE.md remains at current position.
```

## Open Questions

1. **Where to place formal check in execute-phase structure**
   - What we know: quick.md places it AFTER executor, BEFORE verifier, as a distinct step. Execute-phase has a `verify_phase_goal` step that spawns the verifier.
   - What's unclear: Should the formal check be its own named `<step>` in execute-phase, or embedded inside `verify_phase_goal`?
   - Recommendation: Embed inside `verify_phase_goal` BEFORE the Task spawn. This keeps the gate logically coupled to verification and avoids adding a new top-level step that might get skipped by resumption logic.

2. **Agent file location for qgsd-verifier.md**
   - What we know: Installed copy at `~/.claude/agents/qgsd-verifier.md`. Source at `qgsd-core/agents/qgsd-verifier.md` OR just in `agents/qgsd-verifier.md`.

   - Recommendation: Planner should verify installer reads agents from `agents/` directory (not `qgsd-core/agents/`) before writing the plan task. Check `bin/install.js` agent sync logic.

3. **Module list for execute-phase: keyword-scan vs. aggregate formal_artifacts from plans**
   - What we know: FORMAL_SPEC_CONTEXT must be reconstructed. Two options: (A) keyword-scan from phase description (same as plan-phase Step 4.5), (B) aggregate `formal_artifacts: update:` paths from all PLANs and extract module names.
   - What's unclear: Option B would be more targeted (only modules the planner declared changes to) but requires parsing multiple PLAN.md files. Option A is simpler and consistent.
   - Recommendation: Option A (re-run keyword-scan from phase description). Consistent with plan-phase, no PLAN parsing needed.

4. **Continuation verifier for override path**
   - What we know: execute-phase has a checkpoint continuation mechanism (spawn fresh agent with prior state). The override path needs the verifier to update an existing VERIFICATION.md.
   - What's unclear: Should execute-phase spawn a separate "update VERIFICATION.md" task, or spawn the verifier again with `override_reason` in prompt?
   - Recommendation: Spawn verifier again with a simplified prompt: "Update VERIFICATION.md at {path}: user acknowledged counterexample with reason: '{reason}'. Write counterexample_override: { acknowledged_at: {timestamp}, reason: '{reason}', override_by: 'user' }. Change status to passed." This is simpler than a dedicated tool invocation.
