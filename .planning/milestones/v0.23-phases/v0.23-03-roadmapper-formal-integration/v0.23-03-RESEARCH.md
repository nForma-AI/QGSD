## Summary

Phase v0.23-03 has three sequential responsibilities. First, it must fix two pre-condition issues (ISSUE-1, ISSUE-2) before the roadmapper wiring is even possible. Second, it wires the roadmapper agent to discover and inject `.formal/spec/*/invariants.md` files into its context when designing phases. Third, it installs all changed workflow files and agents to the `~/.claude/` runtime.

ISSUE-1 is a three-part inconsistency in the keyword-match algorithm between `plan-phase.md` (Step 4.5) and `execute-phase.md` (verify_phase_goal step): plan-phase uses `.phase_name` as the source string, execute-phase uses `.goal // empty`; plan-phase splits on `' -/'` (3 separators), execute-phase on `' -'` (2, missing `/`); plan-phase runs bidirectional substring matching (`KEYWORD in MODULE OR MODULE in KEYWORD`), execute-phase is unidirectional only. The fix standardizes both to `.goal // .phase_name`, `' -/'` separator, and bidirectional matching.

ISSUE-2 is that `gsd-tools.cjs` parses the Goal field with regex `\*\*Goal:\*\*` (colon inside asterisks) but ROADMAP.md writes goals as `**Goal**:` (colon outside asterisks), causing `.goal` to return `null` for all phases. Confirmed by running `roadmap get-phase v0.23-03` which returns `"goal": null` even though the ROADMAP.md entry clearly states `**Goal**: qgsd-roadmapper reads...`. The fix is to update the gsd-tools regex to match `\*\*Goal\*\*:` (and also the fallback in execute-phase to `.goal // .phase_name` so a null goal safely falls back to `.phase_name`).

**Primary recommendation:** Fix gsd-tools parser first (ISSUE-2), then fix both workflow files to use standardized matching (ISSUE-1), then wire the roadmapper. All three tasks must install updated files to `~/.claude/` before validation tests can pass.

## Architecture Patterns

### Pattern 1: Formal Scope Scan (Pre-Roadmapper Injection)

The scan runs in the `new-milestone.md` orchestrator, immediately before spawning the roadmapper Task. The pattern is identical to plan-phase Step 4.5 — copy it verbatim with the standardized algorithm.

**Source string:** Use the milestone scope/description from MILESTONE-CONTEXT.md or the user's stated goals (not a phase description, since phases haven't been designed yet). Keyword-match milestone keywords against module names.

**Where in new-milestone.md:** Add as Step 9.5 (between requirements definition and roadmap creation), or inline at the start of Step 10 before the Task spawn.

```bash
# Step 9.5 / Step 10 pre-roadmapper: Formal scope scan
FORMAL_SPEC_CONTEXT=()

if [ -d ".formal/spec" ]; then
  # Use milestone scope description — this is the goal text from MILESTONE-CONTEXT.md or user input
  MILESTONE_DESC_LOWER=$(echo "${MILESTONE_GOAL}" | tr '[:upper:]' '[:lower:]')
  for MODULE_DIR in .formal/spec/*/; do
    MODULE=$(basename "$MODULE_DIR")
    INVARIANTS_FILE=".formal/spec/${MODULE}/invariants.md"
    if [ -f "$INVARIANTS_FILE" ]; then
      MODULE_LOWER=$(echo "$MODULE" | tr '[:upper:]' '[:lower:]')
      MATCHED=0
      for KEYWORD in $(echo "$MILESTONE_DESC_LOWER" | tr ' -/' '\n' | grep -v '^$'); do
        if echo "$MODULE_LOWER" | grep -qF "$KEYWORD" || echo "$KEYWORD" | grep -qF "$MODULE_LOWER"; then
          MATCHED=1
          break
        fi
      done
      if [ "$MATCHED" -eq 1 ]; then
        FORMAL_SPEC_CONTEXT+=("{\"module\":\"${MODULE}\",\"path\":\"${INVARIANTS_FILE}\"}")
      fi
    fi
  done
fi
```

**Fail-open:** If `.formal/spec/` does not exist → `FORMAL_SPEC_CONTEXT=()`, no warning. If no keyword matches → empty array. Roadmapper always proceeds.

### Anti-Patterns to Avoid

- **Scanning .formal/spec/ inside the roadmapper agent:** The scan runs in the orchestrator (new-milestone.md), not the agent. Agent receives pre-scanned context via prompt injection — same pattern as plan-phase/planner. Agent reads files via `files_to_read`, never scans directories.
- **Requiring formal_artifacts: in ROADMAP.md phase entries:** That field is for PLAN.md frontmatter only. Roadmapper creates ROADMAP.md, not PLAN.md — no `formal_artifacts:` declaration needed at roadmap time.
- **Blocking roadmapper on missing .formal/spec/:** Fail-open. If the directory is absent or has no keyword matches, roadmapper proceeds normally with `FORMAL_SPEC_CONTEXT=()`.
- **Running gsd-tools fix only in installed copy:** Fix must go in `qgsd-core/` source, then install. If only the installed copy is patched, the next `node bin/install.js` will overwrite the fix.

## Common Pitfalls

### Pitfall 1: Fixing Only the Installed gsd-tools

**What goes wrong:** Developer edits `~/.claude/qgsd/bin/gsd-tools.cjs` directly. The next `node bin/install.js` overwrites the fix.

**Why it happens:** install.js copies from `qgsd-core/` to `~/.claude/`. Editing installed copy only is futile.

**How to avoid:** Always edit `qgsd-core/bin/gsd-tools.cjs`, then run `node bin/install.js --claude --global`.

**Warning signs:** `node /Users/jonathanborduas/.claude/qgsd/bin/gsd-tools.cjs roadmap get-phase v0.23-03` returns `"goal": null` after installation.

### Pitfall 2: Forgetting the Three gsd-tools Regex Locations

**What goes wrong:** Fix one occurrence at line 1027, but lines 2612 and 4370 still use old pattern. Some callers get `goal`, others get `null`.

**Why it happens:** Same regex pattern is copy-pasted to multiple parsing functions.

**How to avoid:** Before committing, grep gsd-tools.cjs for `\*\*Goal:\*\*` — zero matches required.

**Verification:**
```bash
grep -n '\*\*Goal:\*\*' /Users/jonathanborduas/code/QGSD/qgsd-core/bin/gsd-tools.cjs
# Must output nothing after the fix
```

### Pitfall 3: execute-phase Separator Fix Breaks Existing Matches

**What goes wrong:** Adding `/` as a separator causes an existing working phase to no longer match a formal module because a keyword that previously matched now gets split differently.

**Why it happens:** Separator change alters tokenization of phase description.

**How to avoid:** The fix makes matching MORE permissive (adds `/` splitting creates more keywords), not less. False negatives cannot be introduced — only false positives could theoretically increase. Test with example: "TUI/Navigation" → with `tr ' -/'`, produces tokens ["TUI", "Navigation"] instead of ["TUI/Navigation"] — the former is more likely to match "tui-nav" module.

**Warning signs:** After fix, a phase that previously matched a module no longer matches. (Unlikely — but if it happens, the issue is in the bidirectional condition, not the separator.)

### Pitfall 4: Roadmapper Agent Not Knowing What to Do with Formal Context

**What goes wrong:** Formal invariants are injected into roadmapper's context, but agent doesn't know how to use them. Success criteria remain generic boilerplate.

**Why it happens:** Agent prompt only reads files — doesn't have explicit instructions to derive criteria from invariants.

**How to avoid:** Update `qgsd-roadmapper.md` execution_flow Step 5 with explicit instruction: "when formal_context is non-empty, include at least one criterion per formal-matched phase that reflects the invariant's observable behavior."

**Warning signs:** ROADMAP.md success criteria read "Quorum works correctly" instead of "Quorum reaches DECIDED state given responsive slots (EventualConsensus)."

### Pitfall 5: ROADMAP.md Phases Already Have `.goal` Null — Audit Needed

**What goes wrong:** After gsd-tools fix, calling `roadmap get-phase` on older phases may still return `null` if those phases used a different Goal format.

**Why it happens:** Pre-v0.23 ROADMAP.md phases all use `**Goal**:` format (colon outside), which after the fix will correctly parse. Spot-check confirms: `**Goal**: plan-phase discovers...` → after fix, gsd-tools returns non-null.

**How to avoid:** After gsd-tools fix, run spot-check: `node gsd-tools.cjs roadmap get-phase v0.23-01` — should return non-null goal. If still null, regex is still wrong.

**Verification command:**
