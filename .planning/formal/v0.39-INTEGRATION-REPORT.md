## Executive Summary

Milestone v0.39 implements a **dual-cycle formal reasoning architecture** where:
1. **Cycle 1 (Diagnostic)** analyzes why a model fails to capture a bug and generates evidence-based correction proposals
2. **Cycle 2 (Solution Simulation)** tests fix ideas in model space before code application, with three automated convergence gates

All three phases are fully integrated with complete cross-phase wiring. Both cycles respect shared configuration (iteration limits). Diagnostic feedback from Cycle 1 flows into quorum worker prompts. All 170+ tests across phases pass with no failures.

## E2E Flow Verification

### Flow 1: Full Diagnostic Cycle (Cycle 1)

**Trigger:** Model fails to reproduce bug in refinement-loop.cjs

**Path:**
1. verifyBugReproduction() with bugTraceJsonPath option
2. run-tlc/run-alloy with --verification-mode=diagnostic
3. interpretGateResult() returns INCOMPLETE (no violation = model incomplete)
4. **Diagnostic generation triggered:**
   - extractFinalStates(bugTraceJsonPath) from model-state-extractor
   - extractFinalStates(modelFile) from model-state-extractor
   - generateStateDiff() from diagnostic-diff-generator
   - formatDiffAsMarkdown() from diagnostic-diff-generator
   - generateCorrectionProposals() from diagnostic-proposal-generator
5. onDiagnosticGenerated callback fires (line 295-296)
6. Caller serializes diagnostic as JSON
7. Passes to quorum-slot-dispatch as --review-context
8. buildModeAPrompt/buildModeBPrompt detect JSON
9. formatDiagnosticForPrompt() creates "## Model Diagnostic Feedback" section
10. Workers receive diagnostic context in prompt

### Flow 2: Full Simulation Cycle (Cycle 2)

**Trigger:** User provides fix idea via quorum or diagnostic proposals

**Path:**
1. simulateSolutionLoop(fixIdea, ...) called with fix idea
2. **Iteration loop (maxIterations from config.json):**
   - normalizeFixIntent() parses three channels (constraints, variables, code snippets)
   - generateConsequenceModel() applies mutations to reproducing model
   - runConvergenceGates() executes three gates:
     - Gate 1: Invariants Hold (verification_mode=validation)
     - Gate 2: Bug Resolved (verification_mode=diagnostic, inverted)
     - Gate 3: No Regressions (neighbor models)
   - Verdict persisted to gate-verdicts.json with write-once semantics
   - Progress displayed (iteration banner + table)
3. Loop control: converged → exit, max_iterations exhausted → escalate, else iterate

## Known Limitations and Design Notes

### 1. Cross-Cycle Bridge (Flow 3) is Manual
- Diagnostic proposals from Cycle 1 are not automatically fed into Cycle 2
- This is **intentional** — allows user to understand and validate fix ideas before model simulation
- Future enhancement: Auto-feed proposals as constraint/variable/code snippets to normalizeFixIntent

### 2. Config Propagation Method Divergence
- Cycle 1 reads config via import: `const { getMaxIterations } = require('./config-update.cjs')`
- Cycle 2 reads config inline: `fs.readFileSync(configPath)`
- Both produce same result (max_iterations: 3), but inconsistent method
- **Note:** This is acceptable but could be refactored for consistency in v0.40

### 3. Verification Mode Semantics
- FND-01 defines four-cell truth table (diagnostic/validation × violation/no-violation)
- SIM-03 (convergence gates) uses verification_mode correctly:
  - Gate 1: validation (normal semantics)
  - Gate 2: diagnostic (inverted semantics, no violation = success)
  - Gate 3: validation for regression detection
- Implementation is correct per specification

### 4. Session Directory Isolation
- Cycle 1 uses: `.planning/formal/.tmp/session-{hex16}/` (temp artifacts)
- Cycle 2 uses: `.planning/formal/cycle2-simulations/{sessionId}/` (persistent)
- Both use `crypto.randomBytes(8)` for collision resistance (96-bit entropy)
- No collision risk or directory pollution detected
